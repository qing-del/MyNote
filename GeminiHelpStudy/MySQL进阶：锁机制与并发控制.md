## 概述

- **概念**：锁是计算机协调多个进程或线程并发访问某一资源的机制 。在数据库中，数据也是一种供许多用户共享的资源。
- **目的**：为了保证数据的一致性和有效性，在并发访问时需要通过加锁来控制数据的读写。
- **分类**：按照锁的粒度分，MySQL 中的锁主要分为三类：**全局锁**、**表级锁**和**行级锁** 。

---

## 一、 全局锁

> [!info] 核心逻辑 全局锁就是对**整个数据库实例**加锁 。加锁后，整个实例处于只读状态，后续的 DML 的写语句、DDL 语句以及更新操作的事务提交语句都将被阻塞。

### 1. 作用场景

- 典型的使用场景是做**全库逻辑备份**（一致性数据备份） 。获取全局锁后，再通过 `mysqldump` 工具备份，确保备份出的数据在时间逻辑上是完全一致的。

### 2. 语法结构

```SQL
-- 加全局读锁
FLUSH TABLES WITH READ LOCK;

-- 执行数据备份操作 (在命令行终端执行，不是在SQL窗口)
-- mysqldump -u root -p 密码 > backup.sql

-- 释放全局锁
UNLOCK TABLES;
```

> [!warning] 性能注意点
> 数据库加全局锁是一个比较重的操作，会让整个业务停摆（只能读不能写）。
> **更优雅的备份方案**：在 InnoDB 引擎中，备份时加上 `--single-transaction` 参数，可以在不加全局锁的情况下完成一致性备份（底层利用了 MVCC 机制） 。

## 二、 表级锁
- **概念**：每次操作锁住整张表。锁定粒度大，发生锁冲突的概率最高，并发度最低 。主要应用在 MyISAM、InnoDB 、DBD 等存储引擎中。

### 1. 表锁 (Table Lock)

分为**表共享读锁 (Read Lock)** 和 **表独占写锁 (Write Lock)** 。

|**锁类型**|**对当前客户端**|**对其他客户端**|**语法**|
|---|---|---|---|
|**读锁 (S锁)**|可读、不可写|可读、不可写 (写操作会被阻塞)|`LOCK TABLES 表名 READ;`|
|**写锁 (X锁)**|可读、可写|不可读、不可写 (读写皆阻塞)|`LOCK TABLES 表名 WRITE;`|
|**释放锁**|-|-|`UNLOCK TABLES;`|

### 2. 元数据锁 (Meta Data Lock, MDL)

- **概念**：MDL 加锁过程是**系统自动控制**的，无需显式使用 。
- **目的**：维护表元数据的数据一致性。简单来说，就是**防止在执行 CRUD（增删改查）时，有其他线程修改表结构** 。
- **执行逻辑**：
    - 对一张表进行增删改查（DML/DQL）时，自动加 **MDL 读锁 (共享)**。
    - 对表结构进行修改（DDL，如 `ALTER TABLE`）时，自动加 **MDL 写锁 (排他)**。

|对应SQL|锁类型|说明|
|---|---|---|
|`lock tables xxx read/write`|`SHARED_READ_ONLY`/`SHARED_NO_READ_WRITE`||
|`select、select ... lock in share mode`|`SHARED_READ`|与`SHARED_READ`、`SHARED_WRITE`兼容，与`EXCLUSIVE`互斥|
|`insert、update、delete、select ... for update`|`SHARED_WRITE`|与`SHARED_READ`、`SHARED_WRITE`兼容，与`EXCLUSIVE`互斥|
|`alter table ...`|`EXCLUSIVE`|与其他的MDL都互斥|

> [!info] 查看元数据锁：
> `select object_type,object_schema,object_name,lock_type,lock_duration from performance_schema.metadata_locks;`
> - **`object_type`**：**锁的对象类型**。它告诉你锁是加在什么东西上的。常见的取值包括：
>     - `TABLE`：最常见，表示表级锁。
>     - `SCHEMA`：数据库级锁（比如当你删除一个数据库时）。
>     - `USER LEVEL LOCK`：用户自定义锁。
>     - `BACKUP`：备份相关的锁。
> - **`object_schema`**：数据库名称。告诉你这个锁属于哪个库。
> - **`object_name`**：对象名称。如果是表锁，这里就是具体的表名。
>  - **`lock_type`**：**锁的具体模式**。这是核心字段，决定了锁的强弱，例如：
>     - `SHARED_READ`：共享读锁（通常不阻塞其他读，但阻塞写）。
>     - `EXCLUSIVE`：排他锁（修改表结构时使用，会阻塞所有其他操作）。
> - **`lock_duration`**：**锁的持续时间**。
>     - `TRANSACTION`：锁会在事务提交或回滚时释放。
>     - `EXPLICIT`：显式锁，需要手动释放
> 
> > [!warning] **注意**
> > 如果你发现查询结果为空，可能是因为 `performance_schema` 的监控开关没打开。元数据锁监控默认在某些版本是开启的，但你可以通过以下语句确认： `SELECT * FROM performance_schema.setup_instruments WHERE NAME = 'wait/lock/metadata/sql/mdl';`

### 3. 意向锁 (Intention Lock)

- **产生背景**：假设线程 A 给表中的某一行加了行锁；此时线程 B 想给这张表加“表锁”，它就必须去全表扫描，看看有没有哪一行被加了行锁。这种全表扫描效率极低 。
- **概念**：为了解决行锁与表锁的冲突，InnoDB 引入了意向锁。线程在给某行加行锁之前，会先在**表级别**加上意向锁 。这样后续线程想加表锁时，只需看表上有没有意向锁即可，无需逐行检查 。
- **分类**：
    - **意向共享锁 (IS)**：与表锁共享锁 (read) 兼容，与表锁排他锁 (write) 互斥。
    - **意向排他锁 (IX)**：与表锁的 read 和 write 都互斥。_(注：意向锁之间不会互斥)_

## 三、 行级锁 (InnoDB 核心)

- **概念**：每次操作锁住对应的行数据。锁定粒度最小，发生锁冲突的概率最低，并发度最高 。

> [!danger] 底层原理警告 InnoDB 的数据是基于索引组织的，**行锁是通过对索引上的索引项加锁来实现的，而不是对记录加的锁** 。如果 SQL 语句没有走索引，那么 InnoDB 将把整张表的所有记录都锁住，**行锁将退化为表锁**！

### 1. 行级锁的分类

|**分类**|**概念简述**|**核心作用**|**课程对应**|
|---|---|---|---|
|**行锁 (Record Lock)**|锁定单个行记录的锁，防止其他事务对此行进行 `UPDATE` 和 `DELETE`。|在 RC (读已提交) 和 RR (可重复读) 隔离级别下都支持。||
|**间隙锁 (Gap Lock)**|锁定索引记录之间的间隙（不含该记录），确保索引记录的间隙不变。|防止其他事务在这个间隙进行 `INSERT`，产生幻读现象。||
|**临键锁 (Next-Key Lock)**|行锁和间隙锁的组合，同时锁住数据及其前面的间隙。|RR (可重复读) 隔离级别的默认行级锁。||

### 2. 行锁 (Record Lock)

行锁依然分为共享锁（S）和排他锁（X） 。

- **常见 SQL 的加锁情况**：
    - 普通的 `SELECT`：不加任何锁（利用 MVCC 快照读）。
    - `SELECT ... LOCK IN SHARE MODE`：加共享锁（S）。
    - `INSERT` / `UPDATE` / `DELETE`：自动加排他锁（X）。
    - `SELECT ... FOR UPDATE`：加排他锁（X）。


### 3. 间隙锁 & 临键锁 加锁规则

默认情况下，InnoDB 运行在 `REPEATABLE READ` 隔离级别下，使用 **Next-Key Lock** 进行搜索和索引扫描 。

- **等值查询 (唯一索引 / 主键)**：
    - 如果查询的记录存在，退化为 **行锁 (Record Lock)**。
    - 如果查询的记录不存在，退化为 **间隙锁 (Gap Lock)**。

- **等值查询 (普通非唯一索引)**：
    - 不仅会给匹配到的记录加锁，还会给该记录前后的间隙加上 **间隙锁 (Gap Lock)**，以防止其他事务在此处插入相同的值。

## 💡 避坑与进阶指导

### 实际开发常见坑

1. **未建索引引发的血案**：执行 `UPDATE table SET status = 1 WHERE name = 'Tom';`。如果在 `name` 字段上没有索引，或者索引失效，InnoDB 无法进行精确的行级锁定，会**将行锁直接升级为表锁** ！这会导致整个系统针对该表的写操作全部阻塞，并发能力瞬间跌至谷底。
2. **死锁 (Deadlock)**：多个事务以不同的顺序去请求互相持有的锁，就会发生死锁。在业务代码（特别是 Java 并发编程中），需要保证多个表或多行数据的更新顺序是一致的。
