## 条件化简
- MySQL在准备执行SQL语句之前，会对我们的搜索条件进行一些“**重写**”或者“**化简**”，以达到更高效执行的目的。这种纯粹依靠改变SQL语句结构的优化称为**基于规则的优化**。

### 移除不必要的括号
- 优化器会把那些多余的括号给去掉，让结构更加清晰：
```sql
-- 优化前
((a = 5 AND b = c) OR ((a > c) AND (c < 5)))
-- 优化后
(a = 5 AND b = c) OR (a > c AND c < 5)
```

### 常量传递
- 有时候一个表达式中某列的值被定义为常量，如果其他条件中有与该表达式的等值传递，优化器会直接将该列替换为常量：
```sql
-- 优化前
a = 5 AND b > a
-- 优化后
a = 5 AND b > 5
```

### 等值传递
- 如果涉及到了多个列的等值匹配，那么它们之间可以互相传递：
```sql
-- 优化前
a = b AND b = c AND c = 5
-- 优化后
a = 5 AND b = 5 AND c = 5
```

### 移除没用的条件
- 对于那些明显永远为TRUE或者永远为FALSE的条件，MySQL会直接把它们干掉：
```sql
-- 优化前
(a < 1 AND b = b) OR (a = 6 OR 5 != 5)
-- 优化后 (其中 b=b 相当于 TRUE，5 != 5 永远为 FALSE)
(a < 1 AND TRUE) OR (a = 6 OR FALSE)
-- 最终优化为
a < 1 OR a = 6
```

### 表达式计算
- 在查询开始前，如果表达式中只包含常量，那它的值就会被先计算出来：
```sql
-- 优化前
a = 5 + 1
-- 优化后
a = 6
```
> [!failure] 警告
> 如果索引列不是以单独形式出现在表达式中（比如给列做了函数调用，或者是算术运算：`ABS(a) > 5` 或 `-a < -8`），优化器是**不会**对其进行计算优化的，这也是为什么说**对索引列进行函数或表达式操作会导致索引失效**！

### HAVING子句和WHERE子句的合并
- 如果查询中没有出现 `GROUP BY` 等分组语句，也没有聚集函数（如 `SUM`、`MAX`），那么 `HAVING` 子句和 `WHERE` 子句其实没什么区别，优化器会把它们合并到一起。

### 常量表检测
- MySQL觉得这两种查询速度非常快（几乎为 `O(1)`）：
  1. 查询的表中只有一条记录或没有记录
  2. 使用主键或唯一二级索引与常量进行等值匹配（如 `const` 访问方法）
- 优化器会把这种查询对应的表称为 **常量表（constant tables）**。在分析其他表的查询成本前，就会先把常量表的数据查出来，然后在后续查询中将其作为常量对待。

## 外连接消除
- 我们知道，外连接和内连接的本质区别在于：如果被驱动表中找不到与驱动表匹配的记录，**内连接会直接丢弃该记录，而外连接会保留驱动表记录，并将被驱动表的字段都填上 `NULL`**。

### 空值拒绝（rejected-NULL）
- 如果我们在外连接的 `WHERE` 子句中直接指明了**被驱动表的某些列不能为 `NULL`**，这种条件就叫做**空值拒绝（rejected-NULL）条件**。
```sql
SELECT * FROM t1 LEFT JOIN t2 ON t1.m1 = t2.m2 WHERE t2.n2 IS NOT NULL;
```
- 一旦出现了空值拒绝条件，那些因为没有匹配上而被填充为 `NULL` 的记录肯定是不符合 `WHERE` 条件的，最终还是要被丢弃。这也就意味着，**此时外连接和内连接的效果是一模一样的**。

### 外连接转内连接的意义
> [!tip] 核心思考
> 为什么 MySQL 要费尽心机把能够转为内连接的外连接给转过去？
- 其实根本原因在于：**外连接的驱动表和被驱动表是固定死的**（比如LEFT JOIN左表就是驱动表），这意味着优化器无法去选择最优的连接顺序。
- 但是一旦转换成了**内连接**，SQL优化器就可以自行**评估不同表作为驱动表的成本**，从而选出成本最低、性能最好的连接顺序（比如小表驱动大表），大大提升查询效率。

## 子查询优化

### 子查询基础
- 子查询可以在很多地方出现，一般我们将出现在 `SELECT` 子句中的称为 `标量子查询/列子查询`，而出现在 `FROM` 子句中的称为 **派生表**。
- 按与外层查询的关系可以分为：
  - **不相关子查询**：子查询单独执行，不需要外层查询的值。
  - **相关子查询**：子查询的执行需要依赖外层查询中某些列的值。

### 标量子查询、行子查询的执行
- **不相关标量/行子查询**：先单独执行子查询，得出一个结果，再把这个结果当成常量替换到外层查询中，再去执行外层查询。
- **相关标量/行子查询**：
  1. 先从外层查询中获取一条记录
  2. 从获取的记录中找出子查询中涉及的值，执行子查询
  3. 基于子查询得到的结果，再来判断外层记录是否符合 `WHERE` 条件
  4. 重复以上步骤，直到外层查询全表扫描或索引扫描完毕
> [!warning] 注意
> 这个过程非常类似于嵌套循环连接，外层表每读取一次，就要进去执行一次子查询，成本相对较高。

### IN子查询优化
- `IN` 子查询可以说是最常见、也是优化器下了最大功夫优化的类型。例如：`SELECT * FROM s1 WHERE key1 IN (SELECT common_field FROM s2)`。如果子查询里的结果集非常大，直接把结果集作为外层的 `IN` 参数，不仅内存装不下，外层基于该长列表的查询也是灾难。

#### 物化表（Materialization）
- 既然子查询结果集大，MySQL就会把它**“物化”**（也就是存到临时表中）。
- 这种临时表有几个特点：
  - **写入前去重**：临时表所有的列组成联合唯一主键，自动帮你去除掉重复的数据。
  - **Memory引擎与哈希索引**：如果结果集不太大，临时表是用 `Memory` 引擎在内存中创建的，并且会为其建立**哈希索引**，`IN` 查询的等值匹配速度非常接近 `O(1)`。
  - **转磁盘表**：如果结果集太大了（超过 `tmp_table_size` 或 `max_heap_table_size`），就会转成基于磁盘的存储引擎（InnoDB），索引就会变成 `B+树` 索引。

#### 物化表转连接
- 虽然物化表查得很快，但我们终究是在做这件事情：`遍历外层表获取每一条记录，然后拿着记录跑到物化表中去哈希查找是否在IN列表中`。
- 如果外层表是个几千万行记录的无敌大表，而物化表反而只有几条记录，那让外层表去驱动物化表进行嵌套循环查询显然是不划算的。
- **神来之笔**：因为物化表没有重复记录，所以我们可以**直接将物化表与外层表进行内连接**！
- 变成了内连接后，优化器就可以大显身手，自由计算究竟是让“外层表去驱动物化表”，还是让“物化表去驱动外层表”成本更低。

#### 半连接（Semi-Join）
> [!question] 为什么不用全连接，而要用半连接？
> 因为对于 `IN` 子查询来说，我们到底不关心子查询表里有多少条记录匹配外层表的某一条记录，我们**只关心“存不存在匹配”**。如果被驱动表里有100条符合要求的记录，全表连接就会让结果集平白无故暴增出100条重复数据，最后我们还得去做去重操作。
- 为了只判断匹配的存在性，MySQL提出了**半连接**的黑科技。所谓半连接就是：**对于驱动表的一条记录，不管被驱动表中有多少条记录与之匹配，最终结果集里只保留一条该驱动表记录**。

#### Semi-Join适用的前置条件
并不是所有的 `IN` 子查询/ `= ANY` 子查询都能转换为半连接，它需要满足一些极为苛刻的**前置条件**：
> [!warning] 适用条件
> 1. 该子查询**必须且只能以 `IN` 或 `= ANY` 的形式出现在外层查询的 `WHERE` 或 `ON` 子句中**。
> 2. 外层查询和子查询都必须是纯粹的 `SELECT` 语句（不能是 `UPDATE`，`DELETE`）。
> 3. 子查询中**不能包含 `GROUP BY`、`HAVING` 子句以及诸如 `MAX()`、`MIN()` 等聚集函数**。
> 4. 子查询**不能包含 `UNION` 语句**。
> 5. 此 `IN` 子查询不能使用 `OR` 运算符与其他搜索条件连接（因为 `OR` 的语义是只要满足其一即可，而 `Semi-Join` 的本质上依赖于过滤驱动表数据）。

> [!tip] 成功与失败的案例对比
> **满足 Semi-Join 优化**（查询s1中存在于s2且符合某些条件的记录，内查询毫无花哨，就是普通的查询寻找存在性）：
> ```sql
> SELECT * FROM s1 WHERE key1 IN (SELECT common_field FROM s2 WHERE key3 = 'a');
> ```
> 
> **失败案例 1：子查询被 `OR` 连接，无法使用半连接**
> 本质上，使用半连接意味着 `key1` 不在子查询里面的行会被直接过滤掉，这违背了 `OR` 的逻辑（因为如果有另外的条件如 `key2 > 100`，不管在不在 `IN` 里面，这行本该保留的）。这时候就只能走之前的**物化表**策略或者回退到**循环匹配**。
> ```sql
> SELECT * FROM s1 WHERE key1 IN (SELECT common_field FROM s2 WHERE key3 = 'a') OR key2 > 100;
> ```
> 
> **失败案例 2：子查询包含 `GROUP BY` 或聚集函数，无法半连接**
> 因为带有分组或聚合的查询，其语义不仅仅是查找“存在性”，还会涉及到分组计算本身，这与半连接的初衷不符。
> ```sql
> SELECT * FROM s1 WHERE key1 IN (SELECT common_field FROM s2 GROUP BY common_field);
> ```

#### Semi-Join执行策略
- MySQL并没有向用户暴露 `SEMI JOIN` 的语法，它只是一种内部优化手段。为了实现这种过滤/去重，优化器搞出了好几种策略：
  1. **Table pullout（子查询表上拉）**：如果子查询的查询列表恰好是其主键或唯一索引，天然就不会重复。那么直接把子查询提到外层，当成普通的内连接来执行即可！太爽了！
  2. **Duplicate weedout（重复值消除）**：先当成普通内连接执行，将产生的结果集的驱动表ID塞到一个建有唯一索引的临时表里。利用唯一键冲突来过滤掉因为一对多而产生的重复驱动表记录。
  3. **LooseScan（松散扫描）**：如果我们将子查询表作为驱动表，由于子查询表可能会产生重复值，MySQL就在驱动表扫描二级索引的时候，对拥有相同值的记录只取第一条去跟被驱动表（外层表）进行匹配。
  4. **Semi-join Materialization（半连接物化）**：就是上面提到的，先把子查询物化成没有重复值的临时表，然后再跟外层表进行内连接。
  5. **FirstMatch（首次匹配）**：这就是最原始的做法，取外层表的一条记录，到子查询表中去匹配，一旦匹配到第一条不往下找了，直接把这条外层记录加进结果集。其实这就是**相关子查询**的默认执行方式。

### ANY与ALL子查询的优化
- 带有 `ANY`、`ALL` 或者 `SOME` 的子查询，其实本质上就是比较大小的数学问题。MySQL往往可以直接把它们转换成聚集函数：
  - `< ANY (SELECT xxx)` ——> 只要小于最大值就可以 ——> `< (SELECT MAX(xxx))`
  - `> ALL (SELECT xxx)` ——> 必须大于最大值才可以 ——> `> (SELECT MAX(xxx))`
> [!tip] 关联思考
> 通过换算成聚集函数，MySQL就可以不用执行整个大子查询进行比对，而是依靠索引直接快速算出MAX或MIN，时间复杂度大幅度下降。

### EXISTS子查询的执行
- 如果 `EXISTS` 子查询是不相关子查询，那么优化器会先执行因子查询。
  - 如果查到记录，原语句的 `EXISTS (...)` 就被替换为 `TRUE`
  - 如果查不到，就被替换为 `FALSE`，然后再去简化外层的WHERE条件。
- 如果是相关子查询，那执行方式就回到了我们前面介绍的循环方式：
  - 取外层记录 -> 带入子查询运算 -> 发现结果则保留外层记录并马上停止内层寻找 -> 取下一条外层记录。
