### 原子性（Atomicity）
- 将每个SQL语句都看作是一个原子
- 将所有本次操作的原子都放入到一个事务中
- 操作完毕之后可以选择**提交**或者**回滚**

> [!info] 回滚机制
> 跟之前学习的`Undo Log`有关
> 一条数据的一次改动对应一条`Undo Log`日志
> 复杂一点的更新操作可能会对应两条
> 当出现**异常**的时候
> 就会出现对`Undo Log`中的日志进行**反向操作**

- 详细过程可见 -> [[MySQL执行原理详细#“写”指令]]

### 隔离性
- 不同事务之间不能相互查看未提交的内容
- 可以分为：**写与写**、**写与读** 这两种隔离
- 通过 **锁** 与 **MVCC** 来实现

#### 主要问题
- 多个事务同时执行的时候，会出现**三个主要问题**
	- **脏写（Dirty Read）**
		- 事务B先将数据设置为`关羽`且**未提交**
		- 事务A修改了事务B中未提交的数据，将值修改为`张飞`且事务A**提交**
		- 最后事务B进行了**回滚**/**提交**
	> 如果是最后**回滚**，会导致事务A相当于什么都没改；如果是最后**提交**，会导致逻辑混乱（可能是同一个库存被过度扣减），按照上面的例子来说，最后值是`张飞`；
	> > [!failure] 首先这个是最严重的，所有隔离等级都不允许脏读的存在；
	- **脏读（Dirty Read）**
		- 事务B有**未提交**的数据
		- 事务A读到了事务B中未提交的数据
		- 且最后事务B进行了**回滚**
	> 注意**脏读**是要读取到不存在的数据，如果发生了提交，数据就存在了，不算是脏读；
	- **不可重复读** - （事务读取到两次不同的数据）如下是大致流程
		- 事务A读取`数据1`
		- 事务B修改`数据1`，并完成了提交
		- 事务A再次读取`数据1`，与之前读的不同
	- **幻读** （读到了两次行数据不一样的数据）流程如下
		- 事务A读取`集合a`中的数据（假设获取了2行）
		- 事务B修改了（新增）`集合a`中的部分数据
		- 事务A再次读取`集合a`中的数据（只获取了3行）
	> 注意**幻读**强调的是读取到了之前没有读到的数据，所以删除不算；
	> 先前读到，后续读不到，这其实是每条记录发生**不可重复读**

#### 事务隔离
- **四种隔离等级**（并发性能从搞到低）（隔离力度从低到高）
	- `Read Uncommited`
		- 隔离性最弱（**缺点**）
		- 三大问题都会出现
		- 性能最强
	- `Read Commited`
		- 能够解决脏读
	- `Repeatable Read`（Default）
		- 解决脏读
		- 解决不可重复读
		- 能大概率避免幻读（不保证不发生）
	- `Serializable`
		- 解决三大问题
		- 性能最慢（**缺点**）

> [!info] 调整方式
> ```sql
> -- 隔离级别
> SHOW VARIABLES LIKE 'transaction_isolation'
> -- InnoDB 默认隔离级别是 REPEATABLE-READ 可重复读
> SET transaction_isolation = 'REPEATABLE-READ';
>
> -- 另一种修改方式
> SET [GLOBAL|SESSION] TRANSACTION ISOLATION LEVEL level;
> level: { 
> 	REPEATABLE READ | 
> 	READ COMMITTED | 
> 	READ UNCOMMITTED | 
> 	SERIALIZABLE 
> }
> ```
> - 使用 **GLOBAL** 关键字（在全局范围影响）
> 	- 只对执行完该语句之后产生的会话起作用
> 	- 当前已经存在的会话无效
> - 使用 **SESSION** 关键字（在会话范围影响）
> 	- 对当前会话的所有后续的事务有效
> 	- 该语句可以在已经开启的事务中间执行，但不会影响当前正在执行的事务
> 	- 如果在事务之间执行，则对后续的事务有效
> - 上述两个关键字**都不用**（只对执行语句后的下一个事务产生影响）
> 	- 只对当前会话中下一个即将开启的事务有效
> 	- 下一个事务执行完后，后续事务将恢复到之前的隔离级别
> 	- 该语句不能在已经开启的事务中间执行，会报错的

### 持久性
- 事务一旦提交，就不会由于突发情况（电源故障、系统崩溃、竞争条件）发生变化
- 这种突发情况会非常多且复杂，但是一般出现**两种问题**

#### 问题
- **写入前数据损坏/丢失**
	- 在向磁盘写入数据（刷盘）时，**内存缓冲区**（Buffer Pool）中的数据在**写入磁盘前**发生意外
	- `InnoDB`通过`Redo Log`机制，保证内存丢失的数据会被恢复到磁盘中
- **写入（刷盘）过程中数据损坏/丢失**
	- 页（最小传输单元）的默认大小是 16KB
	- 由于操作系统是以最小 4KB 来向磁盘中写入数据
	- 在刷盘过程中要是突然断电，就会出现**残缺页**
	- 通过`Double Write`区的同步恢复机制来解决

- 详细过程可见 -> [[MySQL执行原理详细#“写”指令]]

### 一致性
- 数据库在
	- 执行前
	- 执行中
	- 执行后
- **任意时刻**的状态都是**稳定的**
- 都可以**精准**的判断数据变化，而非是数据随机、杂乱混合

> [!warning] MYISAM
> 在MySQL中另一款常用的存储引擎
> 并没有“事务”这个概念

## 机制解析

### Undo Log

#### 执行流程
- 执行器在写入内存之前，第一时间会先在回滚段`Undo Log Segment`中申请`Undo Log`
- 根据SQL信息构建对应的`Undo Log`内容，并将其写入磁盘，保证执行真正写入前，`Undo Log`是完整的
- 为了后续操作出现异常之后可以有回滚的操作
- 写完`Undo Log`之后才会写入`Redo Log Buffer`以及后续的一系列操作

#### 申请机制
- 默认“页”大小是 16KB，对于每个**回滚段**来说，**最多**可以申请 1024 个；如果把“页”的大小改为 32KB，就只能申请 512个 了；
> 可以理解为`总回滚段大小(16384KB) = 页的大小 * 页数`

- 设置回滚段的数量
```sql
-- 回滚段数量，也可以用 innodb_undo_log（被弃用）
SHOW VARIABLES LIKE 'innodb_rollback_segments';
SET innodb_rollback_segments = '128'; -- 默认 128 个
```
> 最大能支持的`Undo Log`页数 = 1024 * 128

> [!info] 细节
> 由于一个事务可能申请多个 `Undo Log`页
> 而且每个页可能被多个事务重复利用
> 然后 `25%` 是为了临时表准备的
> **结论**：这个数量和**最大事务数**没关系

#### 页类型
> [!tip] 前提
> 由于新增（Insert）操作在事务提交之后就可以直接删除了
> 而删除（Delete）和更新（Update）操作还需要服务后续的`MVCC`
> 所以`Undo Log`会分成两类

> **记录新增操作类** - Insert Undo链表
> **记录更新和删除操作类** - Update Undo链表

![[MySQL-Undo_Log学习3.png]]
- **普通事务**
	- 一个`Insert Undo链表`/`Update Undo链表`
	- 一个临时表
- **有新增又有修改**
	- 一个`Update Undo链表`
	- 一个`Insert Undo链表`
	- 两个临时表

> 由于为每个事务都分配`Undo`链的话会很消耗空间
> 所以还会提供**重用`Undo`页**的机制

##### 版本链
- `trx_id` ：每次一个事务对某条聚簇索引记录进行改动时，都会把该事务的 事务id 赋值给 `trx_id` 隐藏列
- `roll_pointer` ：每次对某条聚簇索引记录进行改动时，都会把旧的版本写入到 undo日志 中，然后这个隐藏 列就相当于一个指针，可以通过它来找到该记录修改前的信息

### InnoDB锁机制
- 默认的`InnoDB`中的**锁信息**是存放在内存当中的
- 锁信息会分两种信息
	- **基本信息**
		- 上锁事务的信息 - 事务`id`
		- 被锁的索引信息 - 聚簇索引 / 二级索引
		- 锁的类型和模式信息 - 32BIT 数来表示
			- 锁模式（`LOCK_MODE`）
				- **共享锁**（`LOCK_S`）
				> ```sql
				> -- 对查询结中的每行，都加 [共享锁]
				> SELECT user WHERE id < 10 LOCK IN SHARE MODE;
				> ```
				> 事务A给其加入**共享锁**（S锁）之后，别的事务（例如事务B）也可以给它加共享锁；但是不可以给它加**排他锁**（X锁）；可以防止读取数据的时候，数据被其他事务修改数据；
				- **独占锁**（`LOCK_X`）
				> 别名-`排他锁` or `写锁`
				> ```sql
				> -- 对查询结中的每行，都加 [独占锁]
				> SELECT user WHERE id < 10 FOR UPDATE;
				> ```
				> 事务A给数据加上**独占锁**之后，别的事务（事务B）就不可以再次给它添加锁；直到事务A完成并把独占锁放开之后，别的事务B才可以给其添加其他锁；
				- **共享意向锁**（`INTENTION_LOCK_IS`）
				- **独占意向锁**（`INTENTION_LOCK_IX`）
				> [!info] 意向锁
				> 就是添加锁之后，利用数据**标记**被上锁的**表**
				> 可以让接下来操作这个表的事务在**添加锁的时候**
				> **不用**一条条去**遍历**查看是否添加锁，提升了查询效率

				- 
				> [!tip] 关键理解（个人总结）
                > **IS、IX锁是表级锁**，这两个锁出现是为了在==加表级S锁/X锁的时候==做一个**快速判断**
                > 其实可以是理解为==行级S/X锁上锁时==，会在“门口”做的**标记锁** -- 给表级S/X锁看的
                > 实际上并*不会锁住*数据，只是用于**做标记的**
                
				- **自增锁**（`LOCK_AUTO_INC`）
				> [!info] 自增列
				> - 在插入数据的时候，会增加**自增锁**，并生成自增值，阻塞其他的插入操作，保证值的唯一
				> > [!warning] 自增值生成之后，尽管是事务归滚，自增值也不会回退，所以自增值有可能是不连续的
				> ```sql
				> -- 自增锁的机制
				> SHOW VARIABLES LIKE 'innodb_autoinc_lock_mode';
				> SET innodb_autoinc_lock_mode='1';
				> ```
				> - `0` - 自增场景都加锁
				> - `1`(`default`) - 确认插入数量时，采用`2`机制；插入数量不确定时，采用`0`机制；
				> - `2` - 自增场景不加锁，直接生成自增值；但这样即使不会滚，并发插入的时候依旧会导致自增值不连续
				> 
				> > 细节原理：[[#自增锁原理]]
			- 锁类型（`LOCK_TYPE`）
				- **表锁**
					- 对整张表加锁
					- 在**释放之前**，**拒绝**任何事务对这张表进行操作
				- **行锁**
			- 行子信息（`REC_LOCK_TYPE`）
				- **精准行锁**（`LOCK_REC_NOT_GAP`）
					- 精确锁住某一行（例如锁住`id=10`的行）
				- **GAP锁**（`LOCK_GAP`）
					- 锁住行与行之间的间隙，防止这个间隙中插入数据
					> [!warning] 提示
					> **仅仅是为了防止插入幻影记录而提出的**
					> 实际上不会锁到真正数据行
				- **Next Key锁**（`LOCK_ORDINARY`）
					- 精准行锁 + GAP锁的组合
				- **插入意向锁**（`LOCK_INSERT_INTENTION`）
					- 这是一个共享锁，在**GAP锁**生效的时候
					- 别的事务要在这个行间隙插入数据，就会加入**插入意向锁**
					- 多个事务都可以加这个锁，当**GAP锁**释放之后
					- 多个**插入意向锁**可以同时进行，不需要串行执行
	- **特有信息**
		- 被锁表的信息 - 表锁
		- 表空间+页号 - 行锁
		- 被锁行行号（`bits`） - 行锁

![[MySQL-锁机制学习1.png]]

#### 自增锁原理
- 采用 AUTO-INC 锁，也就是在执行插入语句时就在表级别加一个 AUTO-INC 锁，然后为每条待插入记录 的 AUTO_INCREMENT 修饰的列分配递增的值，在该语句执行结束后，再把 AUTO-INC 锁释放掉。这样一个 事务在持有 AUTO-INC 锁的过程中，其他事务的插入语句都要被阻塞，可以保证一个语句中分配的递增 值是连续的。 
	- 如果我们的插入语句在执行前不可以确定具体要插入多少条记录（无法预计即将插入记录的数量），比 方说使用 INSERT ... SELECT 、 REPLACE ... SELECT 或者 LOAD DATA 这种插入语句，一般是使用 AUTO-INC 锁为 AUTO_INCREMENT 修饰的列生成对应的值。 
	> 小贴士： 需要注意一下的是，这个AUTO-INC锁的作用范围只是单个插入语句，插入语句执行完成后， 这个锁就被释放了，跟我们之前介绍的锁在事务结束时释放是不一样的。 
- 采用一个轻量级的锁，在为插入语句生成 AUTO_INCREMENT 修饰的列的值时获取一下这个轻量级锁，然 后生成本次插入语句需要用到的 AUTO_INCREMENT 列的值之后，就把该轻量级锁释放掉，并不需要等到 整个插入语句执行完才释放锁。
	- 如果我们的插入语句在执行前就可以确定具体要插入多少条记录，比方说我们上边举的关于表 t 的例子 中，在语句执行前就可以确定要插入2条记录，那么一般采用轻量级锁的方式对 AUTO_INCREMENT 修饰的 列进行赋值。这种方式可以避免锁定表，可以提升插入性能。 
	> 小贴士： 设计InnoDB的大叔提供了一个称之为innodb_autoinc_lock_mode的系统变量来控制到底使用上 述两种方式中的哪种来为AUTO_INCREMENT修饰的列进行赋值，当innodb_autoinc_lock_mode值 为0时，一律采用AUTO-INC锁；当innodb_autoinc_lock_mode值为2时，一律采用轻量级锁；当 innodb_autoinc_lock_mode值为1时，两种方式混着来（也就是在插入记录数量确定时采用轻 量级锁，不确定时使用AUTO-INC锁）。不过当innodb_autoinc_lock_mode值为2时，可能会造 成不同事务中的插入语句为AUTO_INCREMENT修饰的列生成的值是交叉的，在有主从复制的场景 中是不安全的。


#### 隐式锁
- 我们前边说一个事务在执行 INSERT 操作时，如果即将插入的 间隙 已经被其他事务加了 gap锁 ，那么本次 INSERT 操作会阻塞，并且当前事务会在该间隙上加一个 插入意向锁 ，否则一般情况下 INSERT 操作是不加 锁的。那如果一个事务首先插入了一条记录（此时并没有与该记录关联的锁结构），然后另一个事务：
	- 立即使用 SELECT ... LOCK IN SHARE MODE 语句读取这条事务，也就是在要获取这条记录的 S锁 ，或者 使用 SELECT ... FOR UPDATE 语句读取这条事务或者直接修改这条记录，也就是要获取这条记录的 X 锁 ，该咋办？ 
	> 如果允许这种情况的发生，那么可能产生 脏读 问题。
	- 立即修改这条记录，也就是要获取这条记录的 X锁 ，该咋办？
	> 如果允许这种情况的发生，那么可能产生 脏写 问题。
- 这时候我们前边唠叨了很多遍的 事务id 又要起作用了。我们把聚簇索引和二级索引中的记录分开看一 下：
	- 情景一：对于聚簇索引记录来说，有一个 trx_id 隐藏列，该隐藏列记录着最后改动该记录的 事务id 。 那么如果在当前事务中新插入一条聚簇索引记录后，该记录的 trx_id 隐藏列代表的的就是当前事务的 事务id ，如果其他事务此时想对该记录添加 S锁 或者 X锁 时，首先会看一下该记录的 trx_id 隐藏列 代表的事务是否是当前的活跃事务，如果是的话，那么就帮助当前事务创建一个 X锁 （也就是为当前事 务创建一个锁结构， is_waiting 属性是 false ），然后自己进入等待状态（也就是为自己也创建一个 锁结构， is_waiting 属性是 true ）。
	- 情景二：对于二级索引记录来说，本身并没有 trx_id 隐藏列，但是在二级索引页面的 Page Header 部 分有一个 PAGE_MAX_TRX_ID 属性，该属性代表对该页面做改动的最大的 事务id ，如果 PAGE_MAX_TRX_ID 属性值小于当前最小的活跃 事务id ，那么说明对该页面做修改的事务都已经提交 了，否则就需要在页面中定位到对应的二级索引记录，然后回表找到它对应的聚簇索引记录，然后再重 复 情景一 的做法。

> [!tip] 在某些时候InnoDB中的insert是会获取锁的

#### 进阶：内存锁结构
![[MySQL内存锁结构.png]]

- 三个重要信息
	- Space ID ：记录所在表空间。
	- Page Number ：记录所在页号。
	- n_bits ：对于行锁来说，一条记录就对应着一个比特位，一个页面中包含很多记录，用不同的比 特位来区分到底是哪一条记录加了锁。为此在行锁结构的末尾放置了一堆比特位，这个 n_bits 属 性代表使用了多少比特位。

### 事务机制原理
#### 实现`MVCC`机制的原理
- 类似于Java中的“同步锁”
- 当事务A对数据`XX`进行操作的时候，就会对这个数据上一把锁
- 事务B想要操作`XX`，就要等事务A把这把锁放出来（一般只能等事务A完成之后）
- 等到事务A完成把锁放开之后，事务B就可以对`XX`操作并上锁

#### MVCC （写+读）
- Multi-version Concurrency Control 多版本并发控制机制
- 基于`Undo Log`版本链 & `Read View`

#### Read View
> [!info] 隔离等级的可读区别
> - `READ UNCOMMITTED`：由于可以读到未提交事务修改过的记录，所以直接读取记录 的最新版本就好了
> - `READ COMMITTED` 和 `REPEATABLE READ`：都必须保证读到已经提交了的事务修改过的记录，也就是说假如另一个事务已经修改了记录但是尚未提交， 是不能直接读取最新版本的记录的，**核心问题**就是：_==需要判断一下版本链中的哪个版本是当前事务可见的==_
> - `SERIALIZABLE`：使用**加锁**的方式来访问记录
- 在`select`事务构建时就会构造一个`Read View`
	- 里面会记录版本链的**统计信息**，在后续查询处理时，就不需要遍历所有版本链
		- `m_ids`：表示在生成`ReadView`时当前系统中活跃的读写事务的`事务id`列表，即统计所有**没有提交**的事务
		- `min_trx_id`：版本链**尾**的`事务id`（**未提交**的**最早**`事务id`，如果没有活跃事务的时候，会直接与`max_trx_id`相等）
		- `max_trx_id`：下一个将被分配的事务，版本链**头**的`事务id + 1`
		> [!tip] Tip
		> 注意`max_trx_id`并不是`m_ids`中的最大值，`事务id`是递增分配的。比方说现在有`id`为1，2，3这三 个事务，之后`id`为3的事务提交了。那么一个新的读事务在生成`ReadView`时，`m_ids`就包括1和2，`min_trx_id`的值就是1，`max_trx_id`的值就是4
		- `creator_trx_id`：创建这个`Read View`的`事务id`
        > [!tip] Tip
		> 我们前边说过，只有在对表中的记录做改动时（执行`INSERT`、`DELETE`、`UPDATE`这些语句时）才会 为事务分配`事务id`，否则在一个只读事务中的`事务id`值都默认为0
	- **查找规则**
		- 先从链头版本开始遍历整个版本链
		- 根据**四步查找规则**，判断每一个版本：（规定`list`为需要被访问**版本链**，`item`为版本链中的**版本**，`now`为**当前**的`Read View`）
			1. 判断当前版本是否为事务创建的，即为`item.trx_id = now.creator_trx_id`；意味着读取自己修改的数据，可以读；否则**进入第二步**；
			2. 若是`item.trx_id < now.min_trx_id`，意味着该版本在`Read View`生成前就已经提交了，可以直接访问；否则**进入第三步**；
			3. 若是`item.trx_id >= now.max_trx_id`，说明生成该版本的事务A是在生成当前`Read View`的事务B之后出现的（==开启事务B （产生该`now.trx_id`）--后续--> 开启事务A（产生该`item`）==），所以当前版本不可被访问，并直接**跳过第四步**；否则**进入第四步**；
			4. 能跑到这步，已经有`now.max_trx_id > item.id && item.id > now.min_trx_id`这个条件了，现在对这个条件进行判断`now.m_ids.contain(item.trx_id)`，如果返回`false`，说明是可以访问；否则不可以；
		- 
		> 如果某个版本的数据对当前事务不可见的话，那就顺着版本链找到下一个版本的数据，继续按照上边的步骤判断 可见性，依此类推，直到版本链中的最后一个版本。如果最后一个版本也不可见的话，那么就意味着该条记录对 该事务完全不可见，查询结果就不包含该记录。

> [!tip] 隔离等级`Read View`策略
> - `Read Commited`
> 	- 事务中每个`select`都会新建一个`Read View`
> - `Repeatable Read`
> 	-为事务中的第一条`select`创建`Read View`，后续的`select`不会创建新的`Read View`
> 	- 尽管这个事务中包含`update`/`insert`/`delete`这种修改语句，也不会创建新的`Read View`
> 	- 只是会为这些带有修改属性的操作赋予新的全局`事务id(trx_id)`，然后将`Read View`中的`creator_trx_id`改为这个修改语句导致的新`trx_id`
> 	- 然后根据**四步法则**，这个事务可以查询到自己的修改
