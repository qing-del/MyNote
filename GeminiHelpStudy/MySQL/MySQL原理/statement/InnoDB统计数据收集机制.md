## 概述

- **概念**：MySQL 查询优化器在生成执行计划时，是基于**成本**（Cost-Based Optimizer）进行决策的。而计算成本的核心依据，就是 InnoDB 收集的各种统计数据（如表行数、索引基数等）。
- **目的**：理解统计数据是如何存储、更新以及产生误差的，从而在遇到“MySQL选错索引”或“执行计划异常”时，能够从底层找到原因并手动干预。

## 一、 统计数据的存储方式

> [!info] 分析思路
> 优化器需要知道表的行数、索引的不重复值数量等。这些数据可以存放在内存中（极易丢失），也可以持久化到磁盘中。目前 MySQL 默认推荐持久化。

### 存储模式对比

| **模式** | **对应系统参数** | **执行逻辑说明** | **核心价值/特点** |
| --- | --- | --- | --- |
| **永久性统计数据** | `innodb_stats_persistent=ON` (5.6版本起默认开启) | 将统计数据持久化存储在磁盘的专门系统表中（`mysql.innodb_table_stats` 等）。 | 避免服务器重启或表结构变更时重新计算，保证执行计划的**稳定性**。 |
| **非永久性统计数据** | `innodb_stats_persistent=OFF` | 仅在内存中保存，重启后丢失。表首次打开或数据变动较大时会重新计算。 | 属于历史遗留模式，目前生产环境极少使用。 |

### 核心系统表解析

InnoDB 将永久性统计数据存放在 `mysql` 数据库下的两张系统表中。

#### 1. `mysql.innodb_table_stats`（表级别统计）
记录了表级别的整体统计信息。
```sql
-- 查看某张表的全局统计数据
SELECT * FROM mysql.innodb_table_stats WHERE table_name = 'your_table_name';
````

> [!note] 关键字段解析
> - `n_rows`：表中的**估算**总行数。注意：由于 InnoDB 的 MVCC 机制，这个值是通过采样估算出来的，绝不是绝对精确的。
> - `clustered_index_size`：聚簇索引占用的页面数量。
> - `sum_of_other_index_sizes`：其他所有二级索引占用的页面数量之和。

#### 2. `mysql.innodb_index_stats`（索引级别统计）

记录了每个索引的详细信息。
```SQL
-- 查看某张表所有索引的详细统计数据
SELECT * FROM mysql.innodb_index_stats WHERE table_name = 'your_table_name';
```

> [!note] 关键字段解析 (`stat_name`)
> - `n_diff_pfxNN`：代表索引包含的**不重复值的数量（基数 Cardinality）**。比如 `n_diff_pfx01` 是仅看索引第一列的基数，`n_diff_pfx02` 是前两列联合的基数。
> - `n_leaf_pages`：索引的叶子节点占用的页面数。
> - `size`：该索引总共占用的页面数。

## 二、 统计数据的收集与更新规则

统计数据并非实时绝对精确的，频繁更新会拖垮数据库性能，因此它采用**采样**和**条件触发**的更新机制。

### 1. 自动更新机制
- **触发条件**：当表中发生变动的记录数超过表总记录数的 **10%**（且开启了 `innodb_stats_auto_recalc=ON` 默认开启）。
- **执行逻辑**：InnoDB 会在后台异步启动一个线程，重新对表进行采样并计算统计数据。

### 2. 手动更新机制

- **作用场景**：当发现执行计划选错索引，或者批量导入大量数据后，优化器基于过时的统计数据做出错误决策时。
- **标准语法**：

```SQL
-- 强制重新收集并更新表的统计数据
ANALYZE TABLE 表名;
```

> [!warning] 警告
> `ANALYZE TABLE` 会对表加读锁。虽然收集速度通常较快，但在高并发的大表上执行仍可能造成一定的请求阻塞，**强烈建议在业务低峰期执行**。

### 3. 采样机制 (Sampling)

- **概念**：InnoDB 收集统计数据时**不会全表扫描**（代价太大），而是随机抽取一部分 B+树的叶子节点进行采样分析，然后推算出全表的数据。
- **控制参数**：`innodb_stats_persistent_sample_pages`（默认 20 个页）。
- **优化目标**：
    - 值越大：采样的页面越多，统计数据越准确，但收集过程耗时越长。
    - 值越小：收集速度快，但遇到数据分布极其不均的情况时，容易造成估算误差导致索引选错。

> [!info] `InnoDB` 默认是以表为单位来收集和存储统计数据的
> - 设置方式就是在创建或修改表的时候通过指定 STATS_SAMPLE_PAGES 属性来指明该表的统计 数据存储方式
> ```sql
> CREATE TABLE 表名 (...) Engine=InnoDB, STATS_SAMPLE_PAGES = 具体的采样页面数量;
> 
> ALTER TABLE 表名 Engine=InnoDB, STATS_SAMPLE_PAGES = 具体的采样页面数量;
> ```
> > 创建表的语句中并没有指定 `STATS_SAMPLE_PAGES` 属性
> > 默认使用系统变量 `innodb_stats_persistent_sample_pages` 的值

### 4. NULL 值的特殊处理

- **问题背景**：如果有大量记录的某列值为 `NULL`，在计算基数（不重复值数量）时，这些 `NULL` 算是同一个值，还是互不相同的值？
- **控制参数**：`innodb_stats_method`
    - `nulls_equal` (默认值)：认为所有 NULL 值**都是相等的**。如果表里 NULL 极多，会让优化器认为该列区分度极低，从而拒绝使用索引。
    - `nulls_unequal`：认为所有 NULL 值**都不相等**。
    - `nulls_ignored`：直接在统计时**忽略** NULL 值。

> 详细可见 -> [[MySQL索引进阶：性能分析、使用规则与设计原则#一、 索引性能分析]] 中关于优化器如何评估 EXPLAIN 的部分。

## 💡 避坑与进阶指导

### 实际开发常见坑

1. **`COUNT(*)` 的行数幻觉**：
    很多开发者疑惑，为什么 `SELECT COUNT(*)` 查出的真实行数和 `innodb_table_stats` 表里的 `n_rows` 或者 `EXPLAIN` 中的 `rows` 不一样？
    **真相**：因为 `n_rows` 只是通过采样几十个页**估算**出来的，加上 MVCC 机制导致不同事务看到的行数本就不同。**绝对不要**把系统表里的估算行数当作业务逻辑中的精确行数使用！
    
2. **批量导数据后索引突然失效**：
    使用脚本一次性导入了几百万条数据，紧接着的查询发现 MySQL 走了全表扫描。
    **真相**：数据暴增后，自动更新统计数据的异步线程还没跑完或者还没触发。优化器看到的是老的统计数据（以为表里没多少数据）。
    **对策**：在大量数据变更脚本执行完毕后，紧跟一句 `ANALYZE TABLE 表名;`。

### 性能注意点

1. **基数 (Cardinality) 是优化器的命脉**：
    优化器决定是否走某个索引，最看重的是 `mysql.innodb_index_stats` 中的 `n_diff_pfxNN`（基数）。如果一个列有100万条数据，但只有2个唯一值（比如性别），基数极低，优化器算账后会认为“走索引+回表的成本比全表扫描还高”，从而**果断放弃索引**。
    
2. **NULL值分布不均引发的惨案**：
    如果业务表中某个字段大部分是 `NULL`（例如“订单取消时间”字段），但在查询时又是以非 NULL 值为条件。默认的 `nulls_equal` 会让系统认为这个列基数极低，可能导致选错索引。这时候可以考虑修改为 `nulls_ignored` 或者在业务上用默认值代替 NULL。
    

### 面试高频问题

- **面试题：MySQL 是怎么知道走索引的成本比全表扫描低的？**
    - **核心答法**：MySQL 依赖基于采样收集的 `innodb_table_stats` 和 `innodb_index_stats` 中的统计数据（特别是行数和不重复值的基数），结合 CPU 读取和磁盘 I/O 的成本常数，通过数学公式估算出来的。
        
- **面试题：明明建了联合索引，而且满足最左前缀法则，为什么 MySQL 还是选择了全表扫描？**
    - **核心答法**：除了数据本身分布导致回表成本过高外；很大可能是**统计数据过期或采样误差**，导致优化器把基数估算得过低。可以通过 `EXPLAIN` 观察，并通过执行 `ANALYZE TABLE` 手动重新收集统计数据来解决。
        
- **面试题：InnoDB 为什么不能像 MyISAM 那样，实时统计一张表的准确行数？**
    - **核心答法**：因为 InnoDB 支持事务和 **MVCC（多版本并发控制）**。同一时刻，不同的事务由于隔离级别不同，能看到的可见行数是不一样的。因此 InnoDB 无法在内部维护一个全局统一的准确行数，只能在执行 `COUNT(*)` 时实时去扫描遍历。
