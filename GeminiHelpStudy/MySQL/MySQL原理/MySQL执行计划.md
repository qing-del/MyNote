## 引入案例
```sql
CREATE TABLE single_table ( 
	id INT NOT NULL AUTO_INCREMENT, 
	key1 VARCHAR(100), 
	key2 INT, 
	key3 VARCHAR(100), 
	key_part1 VARCHAR(100), 
	key_part2 VARCHAR(100), 
	key_part3 VARCHAR(100), 
	common_field VARCHAR(100), 
	PRIMARY KEY (id), 
	KEY idx_key1 (key1), 
	UNIQUE KEY idx_key2 (key2), 
	KEY idx_key3 (key3), 
	KEY idx_key_part(key_part1, key_part2, key_part3) 
) Engine=InnoDB CHARSET=utf8;
```

## 访问方法（access method）
- InnoDB中访问方法大致分为两种：
	- **全表扫描**
	- **使用索引**
		- 主键、唯一二级索引等值查询
		- 普通二级索引的等值查询
		- 索引列的范围查询
		- 直接扫描整个索引

> [!warning] 个人界定的工程思想复杂度分析
> 单次索引查询 = `O(1)`
> 多次回表查询（随机IO）= `O(log n)`
> 多次顺序IO = `O(n½)`
> 全表扫描 = `O(n)`
> > 其实我感觉`多次回表随机IO`和`多次顺序IO`挺难界定一个准确值的
> > 因为太多回表的随机IO，不如直接做几次顺序IO
> > 不过大概就i这样区分一下吧

### 单表访问

#### const
- 可以使用主键列（或者唯一二级索引）精确定位一条记录（唯一性）
```sql
SELECT * FROM single_table WHRER id = 1438;
```
- 这个时候MySQL会利用这个**唯一性**在索引树中快速定位到数据行

> [!tip] 关联思考
> 在这个查找的==底层==是 `O(log n)` 的时间复杂度，因为是基于 B+树
> 但是在==工程思想==上，你可以是将其视作 `O(1)` 的时间复杂度，因为只需要**常数次索引查询**即可找到数据


#### ref
- 使用普通二级索引的时候，有可能这个二级索引的等值搜索结果并**不是唯一**的（丢失**唯一性**）
- 就会需要进行**多次回表查询**
> [!warning] PS：一次回表相当于一次随机的磁盘IO -- 这是一个很慢的操作

> [!failure] 警告
> 如果第一次查询到的**数据集很多**，那么回表的时候也会进行大量随机IO
> 这个时候SQL优化器就有可能走**全表扫描**了

> [!tip] 关联思考
> 在这个查找的==底层==是 `O(log n + m)` 的时间复杂度，基于 B+树 找到对应范围内的第一个链表节点之后开始遍历`m`个结果
> 但是在==工程思想==上，你可以是将其视作 `O(log m)` 的时间复杂度，因为需要**m次回表索引查询**才可以找到所有数据；

#### ref_or_null
- 在普通二级索引（**使用索引**）的`ref`的基础上加入`is null`这种判定
> 对于复杂度的分析，其实和`ref`类似，在工程思想的复杂度上就是可能常数大了点

#### range
- 出现范围查询这种，不用管是聚簇索引还是二级索引
> 重要的前提就是，出现**索引区间**并且是**使用索引**，而非是全表扫描

```sql
SELECT * FROM single_table WHERE key2 IN (1438, 6328) OR (key2 >= 38 AND key2 <= 79);
```

> [!tip] 关联思考
> 底层：`O(k(log n (+ m))( + mlog n))`，总查询范围有`k`个索引，`(+m)`与`(+mlog n)`是如果是二级索引就会出出现;
> 工程思想：`O(log k + log m)`

#### index
- 结合 [[#引入案例]] 来看一条语句即可
```sql
SELECT key_part1, key_part2, key_part3 FROM single_table WHERE key_part2 = 'abc';
```
- 首先存在`(key_part1, key_part2, key_part3)`这个联合索引
- 但是又因为条件不满足最左前缀法则
- 最后会选择是遍历二级索引树，但是不用做回表

> [!tip] 联想思考
> 有点类似于`all`+`using index`（索引覆盖）
> 这里的`all`，常数比正常的`all`小，因为二级索引表的数据量不大

#### all
- 最原始最简单的方法，也是最慢的方法了

#### 多条件查询的细节
- 在==一般情况下==，先找个SQL语句来举例先
```sql
SELECT * FROM single_table WHERE key1 = 'abc' AND key2 > 1000;
```
- SQL优化器会先分析出有两个查询条件
	- key1 = ’abc‘
	- key2 > 1000
- 这里SQL优化器会根据数据统计表中找到能筛选出数据量最小的作为第一个筛选条件
- 例如，`key1='abc'`有1000条，而`key2>1000`有10000条，那么SQL优化器肯定会先选择前者
- 既定选了优先过滤条件，就会有这两个步骤
	- 先通过`key1='abc'`过滤并进行1000次回表的随机IO得到所有被过滤之后的数据
	- 然后再利用`key2>1000`来过滤剩下的这个1000条数据，最终得到结果

> ==一般情况下==只会用到单个二级索引，不代表不会出现使用多个的情况

#### 索引合并

##### Intersection合并
- Intersection 翻译过来的意思是`交集`。这里是说某个查询可以使用**多个二级索引**，将从多个二级索引中查询到的结果取交集，比方说下边这个查询：
```sql
SELECT * FROM single_table WHERE key1 = 'a' AND key3 = 'b';
```
- **查询流程**
	- 从 idx_key1 二级索引对应的 B+ 树中取出`key1 = 'a'`的相关记录
	- 从 idx_key3 二级索引对应的 B+ 树中取出`key3 = 'b'`的相关记录
	- 二级索引的记录都是由`索引列 + 主键`构成的，所以我们可以计算出这两个结果集中 id 值的**交集**
	- 按照上一步生成的 id 值列表进行回表操作，也就是从聚簇索引中把指定 id 值的完整用户记录取出来，返回给用户

> [!question] 为什么不先选一个二级索引进行回表查询之后，再利用其他条件进行过滤呢？
> - 分析两种方案中进行回表 id 的数据量（回表的次数）
> - 可以从`随机IO`和`顺序IO`消耗性能的角度去思考

- **只有**以下情况才会出现`Intersection`合并
	- 二级索引列是等值匹配的情况，对于联合索引来说，在联合索引中的每个列都必须等值匹配，不能出现只出现匹配部分列的情况
	- 主键列可以是范围匹配

> [!info] 思考
> 至于为什么二级索引必须是完全等值匹配，而主键又可以范围匹配呢
> 因为再细致一点，其实 B+树的二级索引树可以看作是这样的`(完整二级索引, 主键索引)`
> 在**完全二级索引等值匹配**的下，所得到的**主键索引结果集是有序的**，才会进行索引合并

> [!warning] 注意
> 以上说的两种情况并不是充分条件，仅仅只是必要条件
> 还得看优化器的选择，只有出现单个条件查询回表量很大，然后合并之后可以大大减少回表量的时候才会出现合并

##### Union合并
- 这个就是`并集`了，一般出现在`or`关键连接词上面
- **只有**以下情况才会出现`Union`合并
	- 二级索引列是等值匹配的情况，对于联合索引来说，在联合索引中的每个列都必须等值匹配，不能出现只出现匹配部分列的情况
	- 主键列可以是范围匹配
	- 使用`Intersection`索引合并的搜索条件
	> 可以将使用索引合并的部分看成一个整体与别的条件进行`or`连接查询

> [!warning] 注意
> 同样的，以上条件出现了也不定会是`Union`合并
> 优化器只有在单独根据搜索条件从某个二级索引中获取的记录数比较少
> 通过 Union 索引合并后进行访问的代价比全表扫描更小时才 会使用 Union 索引合并

##### Sort-Union合并
- 由于`Union`合并条件较为苛刻，但是有时候有些条件我们又想使用`Union`合并
```sql
SELECT * FROM single_table WHERE key1 < 'a' OR key3 > 'z'
```
- 发生**以下流程**：
	- 先根据 key1 < 'a' 条件从 idx_key1 二级索引总获取记录，并按照记录的主键值进行排序
	- 再根据 key3 > 'z' 条件从 idx_key3 二级索引总获取记录，并按照记录的主键值进行排序
	- 因为上述的两个二级索引主键值都是排好序的，剩下的操作和 Union 索引合并方式就一样了

> [!info] 思考
> 有`Sort-Union`而没有`Sort-Intersection`
> 因为`Union`是获取的聚簇索引集合结果少 -> 尽管加入排序代价也不大
> 而`Intersection`其中就是聚簇索引结果多才触发交集，所以排序的代价都可能大过回表的代价

##### 使用联合索引代替索引合并
- 其实使用索引合并就是因为需要分开读取两颗索引树
- 还不如直接就建立联合索引，如下SQL语句：
```sql
-- 建立联合索引
ALTER TABLE single_table drop index idx_key1, idx_key3, add index idx_key1_key3(key1, key 3);

-- SQL查询
SELECT * FROM single_table WHERE key1 = 'a' AND key3 = 'b';
```
- 这样我们把没用的 idx_key1 、 idx_key3 都干掉，再添加一个联合索引 idx_key1_key3 ，使用这个联合索引进行查询简直是又快又好，既不用多读一棵 B+ 树，也不用合并结果，何乐而不为？

> [!tip] 提示
> 如果有单独查询`key3`的业务，那么还得加入`idx_key3`这个索引

### Join连接
- 两张表的连接是通过笛卡尔积，如下图所示
![[MySQL联表笛卡尔积.jpg]]
- 例如以下的SQL语句
```sql
SELECT * from t1, t2 WHTER t1.m1 > 1 AND t1.m1=t2.m2 AND t2.n2<'d';
```
- **流程步骤**：
	- 首先优化器会先确定哪张表作为驱动表（这里我们将t1作为驱动表）
	- 此时优化器会通过 `t1.m1>1` 这个条件来过滤掉一部分数据 -> 在这个例子里面我们明确会得到两条数据
	- 然后我们将结果中的每条数据去 `t2` 表中去进行连接条件的匹配 `t1.m1 = t2.m2` 进行过滤
	> （其实这里设计了嵌套`for`循环，原因就是需要对第一张表中得到的数据，然后每一条都去第二张表中进行遍历筛选数据行）
- 那如果我们不设定 `t1.m1 > 1` 这个条件的话，就会出现 `t1` 表的结果需要被执行 3 条，然后第二张表就要被循环查询 3 次

#### 外连接
- 使用外连接的时候会有两种连接方式：左(外)连接、右(外)连接
- 当使用内连接会丢失部分需要的数据
> 比如就是部分A表中的数据与B表中的数据没有任何交集
> 但是我们又需要查询这一部分数据，就是使用外连接的时候

> [!tip] 关于 `on` 关键字
> 在外连接中，只需要满足 `on` 关键字的数据行都会被加入到结果集中，但是连接中另一张表没有的数据会被赋予 `NULL` 值
> 所以在外连接中，`on`关键字是**必须写的**
> 但是要知道，在内连接中，`on` 和 `where` 关键词的作用是等价的

#### 连接计算
- 按照上述的基础，我们不难发现如果是多张表进行联表查询的时候，可以等价为以下的代码：
```Java
t1.foreach(e1 -> {
	// 检查是否满足 t1 表的过滤条件
	t2.foreach(e2 -> {
		// 检查是否满足 t2 表的过滤条件
		t3.foreach(e3 -> {
			// 检查是否满足 t3 表的过滤条件
			result.add(e3);
		})
	})
})
```
- 以上这个就是**嵌套循环连接**（`Nested-Loop Join`），这个是最慢的方法
- 但是也是不难发现，嵌套内的每次循环都是要对应去被驱动表中进行遍历查询
- 所以我们可以做以下优化
	- 获取并过滤驱动表的数据集 -> Set1
	- 遍历每条 `Set1` 的数据行时，是通过 `on` 上面的条件进行连接的，我们可以去优化 `on` 上面过滤条件的查询优化，比如给 t2 表加入索引
	> 但是我们要注意一个点，如果我们连接条件上使用的是二级索引的时候，还需要满足以下条件
	> 当 `二级索引 ＋ 回表` 的代价 < 全表扫描的代价时，才会使用索引查询加速

##### 块的嵌套循环连接（Block Nested-Loop Join）
> [!info] 前提
> 这个适用于当数据量很大，比如被驱动表的数据量都是几百万行的时候
> 那么我们对于驱动表的每一条数据都进行一次IO读取数据来做过滤的话，那么IO开销太大了
> 这个时候就可以通过加入 `join buffer` 内存块来**多行条件匹配**

- 可以通过 `join_buffer_size` 这个变量来控制缓存区的大小，默认是（`262144字节`，即为`256kb`），最小是`128KB`

- `Join Buffer`的匹配方式并不是一种降低时间复杂度的加速，它更是一种降低常数（降低IO开销的方式进行优化的）
- 将驱动表结果集中的数据行丢进去 `Join Buffer` 里面去
> 这里尽量不要防止 `*` 的查询结果，因为这样放置的数据量会减少，所以只需要放置我们所需要的过滤条件即可
- 当读取一次被驱动表的数据进行匹配时，会一下子将所有的 `Join Buffer` 里面的数据行都和被驱动表的数据行进行匹配

> [!warning]
> 再注意一次 `Join Buffer` 中存放的是驱动表过滤后的数据集
> 其实通过这个方式并不会降低 **时间复杂度**，仅仅只是常数优化而已
> 并且是用于数据量很大，反复IO读取会消耗很多性能的时候才会有明显的优化效果

> [!tip] 推荐
> 最优解还是让被驱动表中的数据被匹配的时候，使用索引匹配来加速，这个才是真正的**降低**时间复杂度的方式

### 成本计算与优化
- IO成本：读取一页数据的时候，默认成本是`1.0`
- CPU成本：通过索检数据行、结果集排序的时候消耗的CPU资源

> 读取或者是索检单行数据的时候默认成本都算为`0.2`

- **执行计划的计算流程**
	- 列出所有可能使用的索引
	- 计算全表扫描的代价
	- 计算使用各种索引的代价
	- 选出总代价最小的计划去执行

- 接下来我们分析这一条SQL语句
```sql
SELECT * FROM single_table WHERE   
	key1 IN ('a', 'b', 'c') AND   
	key2 > 10 AND key2 < 1000 AND   
	key3 > key2 AND  
	key_part1 LIKE '%hello%' AND   
	common_field = '123';
```

#### 全表扫描的代价
- 可以通过这条语句来查看数据库表的统计信息
```sql
SHOW TABLE STATUS LIKE '[表名]';
```
- 假设我们得到以下的信息：
```txt
mysql> SHOW TABLE STATUS LIKE 'single_table'\G  
*************************** 1. row ***************************  
  Name: single_table  
  Engine: InnoDB   
  Version: 10  
  Row_format: Dynamic   
  Rows: 9693  
  Avg_row_length: 163   
  Data_length: 1589248  
  Max_data_length: 0  
  Index_length: 2752512   
  Data_free: 4194304  
  Auto_increment: 10001   
  Create_time: 2018-12-10 13:37:23   
  Update_time: 2018-12-10 13:38:03   
  Check_time: NULL  
  Collation: utf8_general_ci   
  Checksum: NULL  
  Create_options:   
  Comment:  
1 row in set (0.01 sec)
```
> [!info] 详细信息
> - `Rows`：表示数据行的数量 -- （`InnoDB`中展示的是大概值，`MySIAM`存储引擎就是准确值）
> - `Data_length`：表示聚簇索引的页面数据量的总大小（单位：字节）
> > 聚簇索引的页面数量 = `Data_length` / 16 / 1024  -- 16代表每页的KB大小，1024代表1KB=1024字节

> 我们需要知道一个前提，就是计算的时候会加入一个微调值，跟硬编码有关
- 根据上述所说的数据分析，可以得到是一共有 97 个聚簇索引数据页，所以我们可以计算出`IO代价 = 97 * 1.0 + 1.1 = 98.1`，其中`1.1`就是一个微调值
- CPU代价 = 9693 * 0.2 + 1.0 = 1939.6
- 总代价 = IO代价 + CPU代价 = 2037.7

#### 索引扫描的代价
> [!info] 关键前提
> 我们需要知道就是 `InnoDB` 中计算某个区间（范围区间、单点区间）内的索引时，都算为一次随机IO的代价（`1.0`）

- 根据语句进行分析，我们不难知道可以使用的索引有：
	- `key2` -- `unique index`
	- `key3` -- `index`
- 那么根据 InnoDB 的计算方式，优先会先去计算唯一二级索引的代价：
> `key2 > 10 and key < 1000`
- 分析这个条件其实只会读取一个索引区间（10，1000），根据我们上述所说的计算方式来计算：
	- 索引搜索的代价：IO代价 = 1.0 * 1 = 1.0
	- 回表的代价：
    > [!info] Details
    > 统计回表的代价：
    > 1. 通过 `key2 > 10` 找到符合记录的第一条数据 `最左区间记录`，通过 `key2 < 1000` 来找到最右边第一条符合条件的记录 `最右区间记录`
    > 2. 接下来就是统计 `最左区间记录` 和 `最右区间记录` 之间有多少个数据行（假设我们算到有 95 行） -> [[#两条记录之间的数据行统计]]
    > 3. 计算读取所有二级索引数据行的CPU代价：`95 * 0.2 + 0.01 = 19.01`
    > 4. 计算回表所需要付出的IO代价：`95 * 1.0 = 95.0`
    > 5. 计算读取回表所需要的所有数据行的CPU代价：`95 * 0.2 = 19.0`
	- 总代价 = 范围读取二级索引的IO代价 + 读取所有符合的二级索引数据行CPU代价 + 回表读取完整数据行的IO代价 + 回表读取所有数据行的CPU代价 = 1.0 + 19.01 + 95.0 + 19.0 = `134.01`
- 所以通过 key2 来进行过滤的代价为 `134.01`
- 利用我们用同样的方式来计算 `key1` 这个条件，假设我们得到的是 `168.02`

###### 两条数据之间的数据行统计
- 首先我我们可以规定 `最左区间记录`-`a`  `最右区间记录`-`b`
- 第一种：a与b之间差的记录页不超过10页，会进行**精确计算**
	- 在 `InnoDB` 中会直接通过 B+树 匹配到a的位置
	- 然后会向右遍历直到统计到 b；（10个数据页以内）
- 第二种：a与b之间差的记录页超过10页
	- 会先和第一种一样遍历10个数据页，但是没有发现 b 这个数据行
	- 就会统计刚刚遍历这 10 个数据页中的**平均数据行的量**
	- 接下来利用 `平均数据行的量 * 数据页的数量` 来计算数据行

> [!info] 关于数据页的量统计
> - 第一种：a和b的父节点都在同一个非叶子节点中
> 	- 会直接统计这两个父节点之间有多少个节点来算数据页
> - 第二种：a和b的父节点并不在同一个非叶子节点中（a与b相隔非常远的情况）
> 	- 会直接通过叶子节点的链表指针来遍历并计算（相当于链表遍历来一个一个算）


#### 汇总
- 通过计算我们就可以得到三种路线的代价：
	- 全表扫描：2037.7
	- 使用 key2 索引搜索： 134.01
	- 使用 key1 索引搜索： 168.02
- 那么就选用 key2 来作为索引进行搜索咯

> [!tip] 提示
> 这里略过了索引合并的算法
> 这也只是大致的一种计算方式，再不需要看MySQL源码的情况下，基本是够用的


#### 索引数据统计估算
- 我们正常情况下是直接通过 `index dive` （**实地考察**）的方式去统计大概有的数据行
> 就是针对每个索引值，我们都去算 `最左区间匹配` 和 `最右区间匹配` 然后来计算的方式
- 比如 `select * from single_table where key2 in (1,2,3)` 这种索引匹配数量比较少的情况
- 若是出现 `in` 里面包了差不多有 2w 个这样，那么如果再使用 `index dive` 这种方式去算，“测量”的代价都大过“查询”的代价了

> [!info] 界定数量
> 可以通过MySQL中的 `eq_index_dive_limit` 这个变量来控制
> 在MySQL 5.7.3之前的版本默认是 10，之后的默认是 200；

- 接下来我们会使用
```sql
SHOW INDEX FROM [表名];
```
- 我们会得到有以下属性的表：
![[MySQL索引表的属性.jpg]]

- 那么此时MySQL会通过
	- `SHOW TABLE STATUS LIKE '[表名]'`得到`Rows`
	- `SHOW INDEX FROM [表名]`得到 `Cardinality`
	- 利用 d = Rows / Cardinality 得到平均值
	- 获取 `in` 中的索引数量 * d 来得到一个大概代价

> [!warning] 注意
> 这种计算方式得到的值非常的==不准确==

### 连接查询的成本
> [!warning] 提示
> 我们采用 `single_table as s1 join single_table as s2` 这样的方式来作为连接

#### Condition filtering
- 一般是MySQL需要开始猜测符合条件的**驱动表扇出**的数据量的时候才会出现
- 比如驱动表中过滤条件中，没有带有索引的方式，这个时候数据库就会去猜数据量
- 我们拿以下SQL语句来进行举例说明：
```sql
-- 第一条SQL语句
SELECT * FROM single_table AS s1 INNER JOIN single_table2 AS s2;

-- 第二条SQL语句
SELECT * FROM single_table AS s1 INNER JOIN single_table2 AS s2 
WHERE s1.key2 >10 AND s1.key2 < 1000;

-- 第三条SQL语句
SELECT * FROM single_table AS s1 INNER JOIN single_table2 AS s2   
WHERE s1.common_field > 'xyz';

-- 第四条SQL语句
SELECT * FROM single_table AS s1 INNER JOIN single_table2 AS s2   
WHERE s1.key2 > 10 AND s1.key2 < 1000 AND  
	s1.common_field > 'xyz';
	
-- 第五条SQL语句
SELECT * FROM single_table AS s1 INNER JOIN single_table2 AS s2   
WHERE s1.key2 > 10 AND s1.key2 < 1000 AND  
    s1.key1 IN ('a', 'b', 'c') AND   
    s1.common_field > 'xyz';
```

- 对于上述的举例中，第三、四、五都是需要猜的
	- 第三条中，因为 `s1.common_field` 是没有索引的，MySQL就会去猜有多少条数据满足这个条件
	- 第四条中，MySQL会先通过 `s1.key2` 来过滤出部分数据，然后猜这些剩余的数据中有多少条数据是满足 `s1.common_field` 条件的数据量
	- 第五条中，MySQL依旧是会先通过 `s1.key2` 过滤出部分数据，然后猜剩余的数据中有多少条是满足剩余两个条件的

#### 连接成本的计算
- 引入一条公式：`总成本 = 驱动表的查询成本 + 驱动表的扇出 * 被驱动表的查询成本`
- 通过以下这条SQL语句来进行分析
```sql
SELECT * FROM single_table AS s1 
INNER JOIN single_table2 AS s2   
ON s1.key1 = s2.common_field  
WHERE s1.key2 > 10 AND s1.key2 < 1000 AND   
	s2.key2 > 1000 AND s2.key2 < 2000;
```

- 我们可以分析不同表作为驱动表的情况来进行成本的计算
	- 使用 `s1` 作为驱动表
		- 驱动表的查询有两种可能的索引：`key2`
		- 但是由于前文已经有过分析了，不难发现是 `key2` 作为索引的查询成本会更低
		- 然后对于被驱动表 `s2`，我们发现有两个条件：`key2`和`common_field`，但是由于后者不是索引，我们只有一个选择
		> 以 `s1` 作为驱动表的**总代价** = `s1`以索引`key2`查询的代价 + `s1`用`key2`查询的扇出 * `s2`以`key2`查询的代价
	- 使用 `s2` 作为驱动表
		- 驱动表的查询有两个条件：`s2.key2 > 1000 AND s2.key2 < 2000`，只能选用`key2`作为查询条件
		- 对于`s1`表就可以通过选用代价更低的`key1`，而不是比较`key2`
		> [!info] Details
		> 通过 `key1=常量`，会通过上面所说过的方法来计算一个`key1`平均所拥有的数据量
		> 然后`key2`就相当于会使用`最左区间匹配`和`最右区间匹配`来计算
		> 总的来说，其实使用`key1`查询就是用`ref`，使用`key2`就是使用`range`
		- 选定好两个表所使用的访问方式之后，就可以得出总代价了
		> 以 `s2` 作为驱动表的**总代价** = `s2`以索引`key2`查询的代价 + `s2`用`key2`查询的扇出 * `s1`以`key1`查询的代价

- 被驱动表是要被多次查询的，所以我们要==对被驱动表建立合适的索引来降低单次查询的代价==

#### 多表连接的成本分析
- 首先我们可以考虑多表查询的连接顺序
	- 对于两表连接，比如表A和表B连接：只有 AB、BA这两种连接顺序。其实相当于 2 × 1 = 2 种连接顺序。
	- 对于三表连接，比如表A、表B、表C进行连接：有ABC、ACB、BAC、BCA、CAB、CBA这么6种连接顺序。其实相当于 3 × 2 × 1 = 6 种连接顺序。
	- 对于四表连接的话，则会有 4 × 3 × 2 × 1 = 24 种连接顺序。
	- 对于 n 表连接的话，则有 n × (n-1) × (n-2) × ··· × 1 种连接顺序，就是n的阶乘种连接顺序， 也就是 n! 。

> 有 n 个表进行连接， MySQL 查询优化器要每一种连接顺序的成本都计算一遍么？那可是 n! 种连接顺序呀。其实 真的是要都算一遍，不过设计 MySQL 的大叔们想了很多办法减少计算非常多种连接顺序的成本的方法：

- 提前结束某种顺序的成本评估
> 类似于递归搜索中的剪枝，其实就是当找到一种较优解的时候，存为全局变量，然后搜索其他顺序代价大小的时候，如果超过了这个值，就直接掐断这次搜索，直接进入下一种可能的计算；
- 系统变量`optimizer_search_depth`
> 当连接表的可能数量小于这个值的时候，就会进行枚举每一个可能的代价去精确计算；
- 根据某些规则压根儿就不考虑某些连接顺序
> 即使是有上边两条规则的限制，但是分析多个表不同连接顺序成本花费的时间还是会很长，所以设计 MySQL 的大叔干脆提出了一些所谓的 启发式规则 （就是根据以往经验指定的一些规则），凡是不满足这些规则的 连接顺序压根儿就不分析，这样可以极大的减少需要分析的连接顺序的数量，但是也可能造成错失最优的执 行计划。他们提供了一个系统变量`optimizer_prune_level`来控制到底是不是用这些启发式规则。
> - 类似于**启发式搜索**吧
