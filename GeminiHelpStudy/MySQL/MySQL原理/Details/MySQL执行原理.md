## 一、 通信方式

- 客户端（你的 Java 程序）通过数据库连接池（如 HikariCP）与 MySQL 建立 TCP/IP 连接。
- 双端采用**半双工模式**通信：
    - 就像**对讲机**，同一时刻只能有一方在讲话（发送数据）。一端发送消息，另一端必须**完整**接收后才能响应，不能中途打断。
    - 客户端用单独数据包传输数据，服务端通过 `max_allowed_packet` 限制单次传输上限，超限会拒收并抛出异常。

## 二、 查询流程（Server 层执行链路）

### 查询缓存 (Query Cache)

- 采用 Map 结构，`key = SQL语句`，`value = 查询结果`。

- **状态：** 在 5.7 版本默认关闭，**8.0 版本后被彻底移除**。
- **被淘汰的原因（命中率太低，副作用大）：**
    1. 缓存查找本身消耗性能。
    2. 只要表内有一条数据发生修改，该表相关的所有缓存**全部失效**（批量清空极慢）。
    3. 发生数据写入时，会使用**表级锁**来保证数据一致性，严重拖慢并发。

### SQL解析器 (Parser) —— “听懂你在说什么”

将人类写的 SQL 字符串，转化为 MySQL 能看懂的数据结构（抽象语法树 AST）。- **词法分析：**
- 将 SQL 语句切分为一个个 Token（词汇）。
- 比如：`SELECT id FROM test;` 会被切分为 `SELECT`(关键字)、`id`(非关键字)、`FROM`(关键字)、`test`(非关键字)。
- **语法分析：**
    - 验证语法是否符合规则（比如有没有漏掉 `FROM`，括号有没有闭合）。
    - 最终生成一棵**语法树**。如果语法报错（如 `You have an error in your SQL syntax`），就是在这个阶段拦截的。

### 预处理器 & SQL优化器 (Optimizer) —— “决定怎么做最快”

- **预处理：** 检查表名、列名是否存在，展开 `*` 为具体的列名。
- **SQL优化（核心）：**
    - 基于成本的优化器（CBO, Cost-Based Optimizer），从多种可能的执行方案中，算出**IO成本**（读磁盘）和**CPU成本**（计算/排序），选择总成本最小的一条路。
    - **示例：** `SELECT * FROM user WHERE age = 20 AND name = '张三';`。如果 `age` 和 `name` 都有索引，优化器会预估哪个索引扫描的行数更少，最终只挑最优的那个走。

- **常见优化动作：** 关联查询重排、索引优选、优化 `min/max`、等价变换（如 `a=5 and b=a` 优化为 `a=5 and b=5`）。
> [!info] 查阅优化路径 (排查慢查询的高级手段) 
> ```SQL
> -- 打开优化成本分析轨迹
> SET optimizer_trace="enabled=on";
> -- 执行你的复杂查询
> SELECT * FROM test WHERE name LIKE 'test';
> -- 查询优化器是怎么想的（查看最终选择了哪个索引，放弃了哪个）
> SELECT * FROM information_schema.OPTIMIZER_TRACE;
> ```
> ```sql
> -- 注意：查阅之后一定要关闭，这东西非常吃性能！
> SET optimizer_trace="enabled=off";
> ```
> - 查看最近一次 SQL 消耗的总成本：
> ```sql
> SHOW STATUS LIKE 'last_query_cost';
> ```

## 三、 执行器与 Handler API（存储引擎层交互）

### 执行器 (Executor) —— “发号施令的包工头”
- 优化器输出**执行计划**后，交接给执行器。
- 执行器本身不处理数据，它会根据表的引擎类型（如 InnoDB），去调用对应的 **Handler API**。
- 权限校验、构建临时表等也是在执行器阶段完成的。


### Handler API 的底层魔法（重点掌握）

存储引擎收到请求后，会将操作拆分为一个个微事务 `MTR`（Mini Transaction）。对于底层的“16KB数据页”来说，每个 MTR 都是原子访问。

#### 1. 读数据的流程 (Read)

- 优先去内存的 `Buffer Pool` 找。结合 **自适应哈希索引 (AHI)** ，如果是热点数据，瞬间就能定位并返回，如同用 HashMap 取值。- 如果内存没有，就触发**磁盘 I/O**，把对应的“数据页”从 `.ibd` 文件读取到 `Buffer Pool` 中（由 `Free链表` 管理空闲内存块）。


#### 2. 写数据的流程 (Write / Update / Delete)

_这是保证 ACID 特性的核心流，一定要按照顺序记：_

1. **写 Undo Log：** 先把旧数据记到 `Undo Buffer`，再刷到磁盘。万一你想 `ROLLBACK`，就靠它反向推演。
2. **写内存数据：** 更新 `Buffer Pool` 里的那条数据。此时该页变成了**脏页**。如果是普通非唯一索引的插入，可能会先写进 `Change Buffer` 延后合并。
3. **写 Redo Log：** 把这次“修改了什么”记录到 `Redo Log Buffer`，并根据配置（`innodb_flush_log_at_trx_commit`）刷入磁盘里的 `Redo Log`。防止服务器突然断电导致内存脏页丢失。
4. **双写缓冲保护 (Double Write Buffer)：** 在把内存脏页真正刷回磁盘 `.ibd` 文件前，先顺序写一份到磁盘的系统表空间里。防止“页断裂”（16KB的页刚写了8KB就断电了，磁盘数据损坏，可以用 Double Write 恢复）。
5. **真实数据落盘：** 后台 IO 线程慢慢把脏页刷入 `.ibd` 磁盘文件（由 `Flush链表` 记录哪些页被修改过）。

## 四、 记忆锚点：核心逻辑线串联（以业务场景为例）

为了不让你死记硬背，我们把上面的所有概念，代入到一个非常具体的开发场景中。
**场景：** 在后台管理系统中，你点击了“起售”某个停售的菜品。Java 代码最终向 MySQL 发送了一条语句：
`UPDATE dish SET status = 1 WHERE id = 1001;`
**MySQL 的内部通关纪实：**

1. **【连接器】（查绿码）：** Java 线程通过 TCP 发来请求，MySQL 校验你的账号密码和权限。半双工通信建立。
2. **【解析器】（切词拆解）：** MySQL 提取出 `UPDATE`、`dish`、`status`，验证语法没错，生成抽象语法树（AST）。
3. **【优化器】（导航寻路）：** 看到 `WHERE id = 1001`，发现 `id` 是主键。优化器一拍大腿：“直接走聚簇索引成本最低！生成执行计划，交给执行器。”
4. **【执行器】（下达指令）：** 呼叫 InnoDB 引擎的 Handler API：“老弟，去给我把 `dish` 表里 `id=1001` 的数据拿出来，我要改状态！”
5. **【Handler 读阶段】（找数据）：** InnoDB 去 `Buffer Pool` 里摸了一下，发现 `id=1001` 的这一页（16KB）刚好在内存里（或者走磁盘加载到内存）。
6. **【Handler 写阶段 - 留退路】：** 动手改之前，先把 `status=0` 这个旧状态写进 **Undo Log**（万一等下 Spring 抛异常了，还要靠它回滚库存和状态）。
7. **【Handler 写阶段 - 改内存】：** 在 `Buffer Pool` 里，把这条数据的 `status` 字段改成 `1`。这时候，内存数据变了，但磁盘上的文件还没变（产生**脏页**）。
8. **【Handler 写阶段 - 记账本】：** 万一现在机房停电，内存清空怎么办？赶紧把“把页X里的status改成了1”这个动作，写进 **Redo Log** 并火速刷到磁盘。（此时，事务其实就可以宣告 Commit 成功并返回给 Java 端了）。
9. **【Handler 写阶段 - 偷偷干活】：** 夜深人静时，后台 IO 线程准备把内存里的脏页刷回磁盘。为了防止刷到一半断电导致页损坏，先写一份到 **Double Write Buffer**，然后再把那个 16KB 的页安稳地覆写到 `dish.ibd` 文件里。任务圆满结束。
