## 通信方式
- 客户端通过连接驱动与链接池连接（这个连接通常使用`TCP/IP`协议底层连接）
- 双端采用**半双工模式**通信
	- 在任意时刻，一端发送消息，另一端要**完整**接收才可以响应
	- 客户端用单独数据包传输数据
	- 服务端可以通过设置**最大单次传输上限**来限制大小，超过限制会拒收并抛出异常 ->在[[MySQL大体框架.canvas]]中的**连接器**模块有详细细节

## 查询流程

### 查询缓存
- 采用的是 Map 结构，`key = SQL语句`，`value = 查询结果`
- 在 5.7 版本是**默认关闭**的
```sql
-- “查询缓存”开关
show variable like 'query_cache_type';
set query_cache_type=0;

-- query_cache_type=2时，代表按需使用，用SQL_CACHE对某表开启缓存
select SQL_CACHE * FROM test;
```
> [!failure] 在 8.0 之后的版本被移除了，命中率太低了。
> ①缓存查找会消耗性能 -> 慢
> ②如果没找到要新建缓存 -> 慢
> ③如果发生数据写入操作 会使用**表级锁**来保证数据一致 -> 慢
> ④当表内数据修改后 要**批量设置失效** -> 慢

### SQL解析器
- **词法分析**
	- 通过 `sql` / `sql_lex.cc` 将SQL语句切分为各种Token词
	- 利用定义了各类关键字、操作符的数组`symbols[]`进行**关键字**、**非关键字**的标记
	- 例如：
	```sql
	select id from test;
	-- select <-> 关键词
	-- id <-> 非关键词
	-- from <-> 关键词
	-- test <-> 非关键词
	```
- **语法分析**
	- 通过`sql`/`sql_yacc.cc`规则代码将语句解析为了一个”语法树
	- 进行正确性验证
		- 关键字
		- 关键字顺序

### 预处理器 & SQL优化器
> [!info] 涉及编译原理的相关内容

#### 预处理

- 先提供模板语句，再提供参数
   > [!tip] 性能提升 
   > 对于多次执行相同模板，参数不同的SQL语句有性能上的提升
   > 但是依旧是会进行
   
#### SQL优化
- 基于CBO(`CostBased Optimizer`)从诸多执行路径中找到**最小成本路径**
- 成本：数据表、数据量、索引
- 根据成本算出**IO成本消耗**和**CPU成本消耗**
- 会涉及到索引、表、行等数据

> [!info] 查阅优化路径
>```sql
>-- 打开优化成本分析轨迹
>SHOW VARIABLES LIKE 'optimizer_trace';
>SET query_cache_type='enabled=on';
>-- 执行查询 同时查看trace
>SELECT * FROM test WHERE name LIKE 'test';
>SELECT * FROM information_schema.OPTIMIZER_TRACE;
>```
>> [!failure] 查阅之后一定要关闭，否则会消耗性能
>>```sql
>>-- 关闭优化成本轨迹分析
>>SET query_cache_type='enabled=off';
>>```
> - 也可以通过如下命令查看最近一次执行SQL消耗的成本
> ```sql
> -- 查询成本
> SELECT * FROM test WHERE name LIKE 'test';
> SHOW STAUTS LKE 'last_query_cost';
> ```
> > 会查询出来是 `IO消耗` + `CPU消耗` 的和

- **优化规则**
	- 关联查询重排
	- 索引优选
	- 连接查询重组
	- 优化排序
	- 优化`min`/`max`等函数
	- 提前终止查询
	- 等价变换

### 执行器
- 通过优化器之后，会得到`执行计划`
- 将这个计划丢给**执行器**
- 执行器会在“存储引擎”中查找`Handler API`，来进行**组合**并**调用**，最终达成“执行计划”

> [!info] 细节
> 在优化器处理的时候，就已经为每一张表创建了一个 `Handler` 实例
> 优化器就可以根据这个实例来获取所需的表、列、索引等信息
> `Handler` 会提供几十个API接口

- 平时所说的**索引查询**、**遍历查询**、**构建临时表**都是在这个时候完成的

---

### 返回数据 & 执行流程图
- 在`Handler`处理完毕之后
- 将`查询结果`通过`TCP`协议返回给客户端
- 如果这里开启了缓存，会同时更新缓存

### Handler API
![[MySQL执行原理3.png]]
- 收到请求会将请求进行**微事务拆分**，即`MTR`（`Mini Transaction`）
- 对于底层的“页”来说，每个`MTR`都是一个**原子访问**
- 同时`Buffer Pool`中的`free`链表、`flush`链表、`LRU`链表来管理数据页的`写入位置`、`刷盘位置`、`数据页淘汰`

#### ”读“指令
> [!info] 以下拿“读”指令来举例说明
> - 首先`MTR`会先去内存的`Buffer Pool`中，通过“自适应哈希索引”(`AHI` - `Adaptation Hash Index`)进行查询并返回
> > 因为 `InnoDB` 会为**热点页**(被频繁查询的页)，创建自适应哈希表，用于提升查询速度
> - 如果内存中没有，那么就会“预读”**磁盘表空间**中的页数据到内存，然后返回数据
> > 这里的“预读”是使用的是`free`链表来优化操作
> 
> > [!warning] 为了获得“执行计划”的最终结果，可能会进行多次"读"操作，知道找到最终结果；如果建立了索引，通常会减少这个“读”的操作次数

#### “写”指令
> [!info] 以下是“写”指令举例说明
> 先写入到`Undo Buffer`，然后写入到磁盘的`Undo Log Buffer`中，为数据回滚做准备
> 以及是写入到`Redo Log Buffer`中，然后按照一定的刷盘决策，刷到磁盘的`Redo Log`中，防止断电发生数据丢失
> 这里会**根据规则做判定**，是否要写入到`Change Buffer`中，然后再后续**某些时机合并**到`Buffer Pool`中
> `Log`写完之后，就会将真实数据写入到`Buffer Pool`中
> > 这里会使用`flush`链表来优化刷盘
> 首先为了保证数据一致性，会先刷到`Double Write Buffer`中，然后写入磁盘中**系统表空间**的`Double Write`中；保证“页”传输到一半的时候断电了，不用担心产生残缺的“页”，可以通过`Double Write`进行**恢复**
> 最后将**真实数据**刷到磁盘中，就完成了“写”指令

> [!extend] 其他模块
> **内存**还有以下几个模块
> 	- 锁信息区 - 锁信息
> 	- 数据字典区 - 字典信息
> 	- Additional Memory Pool - 内部共享信息
> **磁盘**的其他模块
> 	- “通用表空间” - 用户自定义表
> 	- “临时表空间” - 存储临时数据
