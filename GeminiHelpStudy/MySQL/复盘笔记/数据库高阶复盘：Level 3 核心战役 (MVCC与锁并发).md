## 一、 MVCC 破窗效应：UPDATE 打断快照与事务 ID 延迟分配

在构建高并发的电商交易平台时，底层数据的快照读与当前读的交叉往往是导致幻读或诡异数据的根源。

### 1. 事务 ID (trx_id) 的延迟分配机制
- 纯粹的快照读（`SELECT`）在 RR 隔离级别下，初期并不会被分配真正的 `trx_id`。
- 其 Read View 中的 `creator_trx_id` 初始实际上是 0。
- 只有当事务真正执行了 DML 修改语句（如 `UPDATE` / `INSERT`）时，InnoDB 才会为其分配一个全局递增的事务 ID。

### 2. 为什么 UPDATE 必须是“当前读”？
> [!danger] 防御“脏写”（丢失更新 Lost Update）
> 假设初始 `amount=150.00`，其他事务将其修改为 `amount=999.00` 并提交。如果当前事务基于快照历史版本（150.00）去执行金额扣减或修改，就会直接覆盖掉那 `999.00` 的真金白银。
- 因此，所有的 DML 操作强制走**当前读（Current Read）**。它会跳过快照，直接在最新的物理版本上加 X 锁并修改。

### 3. “我看到了别人未提交的数据？” —— 错，是你修改了版本链！
- 执行 `UPDATE` 后，原本的快照状态会被打断。
- `id=5` 这行数据的最新物理版本的 `trx_id` 变成了当前事务自己的 ID。
- 随后的快照读会触发特权规则：**`trx_id == creator_trx_id`（自己做的修改自己可见）**，从而导致事务读到了包含其他事务修改结果的混合版本（例如读到了 999.00）。

## 二、 锁的正交二维属性（最易混淆的难点）

锁的体系在脑海中必须拆解为两个完全正交（互相独立）的维度：

1. **锁的模式 (Lock Mode)**：S锁（共享）与 X锁（排他）。决定了并发请求是否互斥。
2. **锁的类型/范围 (Lock Type)**：
   - Record Lock（纯记录锁）
   - Gap Lock（纯间隙锁）
   - Next-Key Lock（临键锁 = 记录 + 间隙）

> [!tip] 核心顿悟：S-Gap 与 X-Gap 的奇妙兼容
> - 在纯记录锁的世界里，对同一行加 S 锁和 X 锁是绝对互斥的。
> - **但是在间隙锁（Gap Lock）的世界里，S 型 GAP 锁和 X 型 GAP 锁是完全兼容的！**
> - 因为 Gap Lock 的唯一使命是防止其他事务在这个间隙中进行 `INSERT` 操作（即阻塞插入意向锁）。只要大家的目的是一致对外的“防插入”，不论是 S 还是 X，都不会互相阻塞。

## 三、 二级索引的加锁边界：Next-Key 与 Gap 的接力

当执行 `WHERE user_id = 20` 并在非唯一二级索引上加锁时，加锁行为不是粗暴的“锁住 20 这个数字”。

### 1. 索引树上的精准加锁轨迹
- 命中目标记录时，加 **Next-Key Lock**：例如锁住 `((10,1), (20,3)]`，以及 `((20,3), (20,5)]`。
- 向后扫描直到第一条**不匹配**的记录（如 `user_id=30`），退化为加 **Gap Lock**：锁住右侧的间隙 `((20,5), (30,8))`。
- 这三把锁共同构成了 RR 级别下防止幻读的完整防线。

### 2. 聚簇索引的联动优化
- 联动聚簇索引时，只对精确命中的主键（`id=3` 和 `id=5`）加纯 **Record Lock (`REC_NOT_GAP`)**。
- **原因**：间隙保护已经在二级索引上完成了，主键聚簇索引上没必要多此一举再去加 Gap。

## 四、 死锁检测与 Wait-for Graph

> [!bug] 阻塞不等于死锁
> 事务的互相等待并不一定会死锁。只有形成**环形等待 (Circular Wait)** 才会触发死锁机制。例如：事务 X 等待 Y 的 Gap Lock，而事务 Y 又反过来等待事务 X 的 Gap Lock。

**InnoDB 是如何处理死锁的？**
- InnoDB 内部有一个死锁检测器，它会实时维护并遍历**事务等待图（Wait-for Graph）**。
- 一旦在图中检测到有向环路（如 X → Y → X），就会立刻判定为死锁。
- 检测到死锁后，InnoDB 会选择**代价较小的事务**（通常是 undo log 更少、修改量更小的那一个）作为 Victim（牺牲者）强制回滚，释放其所有锁，以拯救另一个事务。
