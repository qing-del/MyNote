## 💡 核心顿悟：CBO 优化器与一次算错的加法

永远不要凭借直觉去猜测优化器的行为，MySQL 优化器是严格基于成本（Cost-Based Optimizer）的。

* **全表扫描并非十恶不赦：** 全表扫描是**顺序 I/O**。走二级索引获取大量数据并回表，是**随机 I/O**。
* **成本计算复盘：** * 全表扫描估算：1,500,000 * 0.2 (CPU) + 15,000 * 1.0 (I/O) = **315,000**。
* 二级索引估算：**987,000**（大量的随机回表 I/O）。
* *错误记录*：之前草稿纸上算加法多写了一个 0（算成了 3,150,000），导致误以为全表扫描成本更高。


* **结论：** 当过滤条件（如 `gender=1`）的区分度极低，需要提取表中接近一半的数据时，优化器会果断抛弃索引，选择成本更低的全表平推。

## 🔍 技术盲区修正

### 1. EXPLAIN 的 `rows` 到底是什么？

* **误区：** 以为 `rows` 显示的是这张表最终过滤出的总行数。
* **真相：** `rows` 代表的是**每次循环预计扫描的行数**。
* **实战应用：** 当 `order_items` 表通过 `idx_order_id` 作为被驱动表关联时，`type` 为 `ref`，`rows` 应该大约是 **2**（因为每个订单平均只有 2.1 个商品），而不是几万或几十万。总扫描行数 = 驱动表 rows × 被驱动表 rows。

### 2. 真正的“延迟关联”（Deferred Join）规范

* **错误写法：** 在 `CTE`（公用表表达式）内部把 4 张表全部 JOIN 完，最后再在外层取数据。这完全没有起到“瘦身”的作用，前 50 万行的无用联查依然发生。
* **标准范式：** **内层只碰一张表，只 SELECT 主键。**
```sql
-- 1. 极其轻量级的内层：单表走索引完成排序和分页，只拿 20 个主键 ID
WITH page_ids AS (
    SELECT o.id
    FROM orders o
    WHERE o.created_at >= '2024-01-01 00:00:00' 
      AND o.order_status IN (3, 4)
    ORDER BY o.created_at DESC
    LIMIT 500000, 20
)
-- 2. 外层：用这 20 个 ID 去做复杂的 JOIN 和数据装配
SELECT ...
FROM page_ids t
JOIN orders o ON o.id = t.id
JOIN users u ON u.id = o.user_id ...

```



## 🛠️ 避坑指南与最佳实践

### 1. 索引的“断舍离”

优化不仅是建索引，更要删废索引。废索引不仅占用空间、拖慢写入，还会干扰优化器的判断。

* **区分度极低的索引必须删：** 比如 `gender` (仅3个值)、`is_deleted` (仅2个值且极度倾斜)。
* **被完全覆盖的索引必须删：** `idx_order_id` 已经被 `idx_order_product(order_id, product_id)` 的最左前缀完全覆盖，属冗余索引。

### 2. 联合索引的字段顺序哲学

* **目标：** `(order_status, created_at)` 优于 `(created_at, order_status)`。
* **原因：** `order_status IN (3, 4)` 是等值匹配（多个离散值），放在前面可以让 MySQL 做两次 `ref` 查找，并且在每个状态值内部，`created_at` 依然是有序的，可以直接用于范围查询和排序。

### 3. 函数计算绝对不能包住索引字段

* `DATE(o.created_at) >= '2024-01-01'` 导致索引彻底报废。
* 永远改写为右侧常量计算：`o.created_at >= '2024-01-01 00:00:00'`。

### 4. 降维打击：冗余设计（架构思维）

为了解决多表 JOIN 带来的性能损耗，在设计电商交易平台时，可以在 `order_items` 表中适当冗余 `unit_price` 和 `product_name`。牺牲少量空间换取时间，把 4 表联查降维成 3 表甚至 2 表，这是高并发架构中极具价值的工程思维。
