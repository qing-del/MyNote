### 1. 核心背景：InnoDB 的数据存储结构
在 MySQL 的 InnoDB 引擎中，表都是根据主键顺序以 B+ 树的形式存放的。
- **聚簇索引（Clustered Index）：** 主键 `id` 所在的 B+ 树。**它的叶子节点存储的是整行数据的完整记录。**
- **二级索引（Secondary Index）：** 其他非主键列的 B+ 树。它的叶子节点存储的是**索引列的值 + 主键 `id`**。

---

### 2. 为什么 `SELECT *` 会慢到令人发指？
当你执行 `select * from user limit 10000000, 10` 时，MySQL 的执行过程如下：
1. **Server 层**向 **InnoDB 引擎**请求第一条记录。
2. InnoDB 扫描**聚簇索引**，取出完整的第一行数据，返回给 Server 层。
3. Server 层拿到完整数据后，发现 offset 还没到 10000000，于是**把这条完整的数据丢弃**。
4. 重复上述过程 **10,000,010** 次。

**痛点在哪里？**
你以为仅仅是磁盘 I/O 吗？不止！InnoDB 需要把 1000 万条完整的行记录从数据页中解析出来，拷贝到内存中，传给 Server 层，然后 Server 层再无情地丢弃它们。这中间产生了**巨大的 CPU 开销、内存拷贝开销，并且会把 Buffer Pool（缓冲池）里的热点数据挤出去**。

---

### 3. 为什么 `SELECT id` 会快很多？
正如你所问，`id` 也是主键，既然都是扫表，为什么 `select id from user limit 10000000, 10` 会快？这分为两种情况：

#### 情况 A：表里有其他二级索引（最常见）
虽然你查的是主键 `id`，但 MySQL 优化器非常聪明。它发现如果要查的仅仅是 `id`，它**根本不需要去扫描庞大的聚簇索引**。它会随便挑一个最小的**二级索引**来扫描。
- 为什么？因为二级索引的叶子节点只存了“索引值 + `id`”，它的体积比聚簇索引小得多！一个数据页能装载更多的二级索引记录。
- 扫描二级索引获取 1000 万个 `id` 的磁盘 I/O 和内存加载量，远远小于扫描聚簇索引获取 1000 万条完整行记录。这在 MySQL 中叫做**索引覆盖（Covering Index）**。

#### 情况 B：表里除了主键，没有任何其他索引（极端情况）
哪怕强制扫描聚簇索引，`SELECT id` 依然比 `SELECT *` 快。

- **正如你猜想的那样：** InnoDB 依然要遍历叶子节点，但它只需要从每行数据中提取 `id` 这个字段，然后传给 Server 层。Server 层丢弃的也仅仅是 `id`。
- 这省去了大量解析其余字段、构建完整行数据并在内存中传输的 CPU 和内存消耗。

---

### 4. 终极解法：为什么“延迟关联”是最优解？

你提到的这条语句：
`select t.* from user t join (select id from user limit 10000000, 10) d on t.id=d.id;`
我们来算一笔账，看看它到底省了什么：

1. **内部子查询 (`select id ... limit 10000000, 10`)**：
    利用了前面提到的**索引覆盖**原理，在体积很小的二级索引上快速跳过 1000 万条记录，极其轻量地拿到了**最终需要的 10 个 `id`**。
2. **外部回表查询 (JOIN)**：
    拿着这 10 个精确的 `id`，去聚簇索引（主键索引）里做 10 次点查（Point Lookup），获取这 10 行的完整数据 (`t.*`)。

**对比总结：**
- **直接 `LIMIT *`：** 扫描并提取 10,000,010 条**完整行记录**，丢弃前 10,000,000 条。
- **延迟关联：** 扫描并提取 10,000,010 个 **`id`（极小代价）**，丢弃前 10,000,000 个，然后拿着剩下的 10 个 `id`，去提取 10 条**完整行记录**。
它并没有避免“跳过 1000 万条记录”这个动作，但它把这 1000 万次操作的**单次成本降到了最低**。
