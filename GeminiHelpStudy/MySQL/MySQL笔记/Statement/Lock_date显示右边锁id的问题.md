> [!warning] Question❓
> 假设当一张表中只有id:1,3,8的数据，当我在事务中执行这条语句的时候`update stu set age = 10 where id = 5;`
> 整张表只有id这个聚簇索引
> 然后此时使用`select object_schema,object_name,index_name,lock_type,lock_mode,lock_data from performance_schema.data_locks;`查询语句查询时
> 会看见一条`lock_mode`为`X,GAP`的锁，然后`lock_data`是`8`
> 我知道其实锁住的位置只有(3,8)不包含端点，但是我想知道为什么不显示3，而是显示8？

---

这个问题非常刁钻，且切中了 InnoDB 锁机制的物理实现核心！你对锁范围 `(3, 8)` 的理解是完全正确的，这是一个纯粹的间隙锁（Gap Lock）。
至于为什么 `lock_data` 显示的是 `8` 而不是 `3`，这要归结于 **InnoDB 底层对锁的物理存储方式**。
简单来说：**InnoDB 的锁不能悬空存在，它必须“依附”在一个具体的索引记录上。**
下面为您详细拆解背后的底层逻辑：

### 1. 锁的“锚点”机制：向右寻找第一个存在的记录
在 InnoDB 的底层实现中，并没有一个专门的数据结构去凭空描述一个“区间”。所有的行锁（包括间隙锁和临键锁）都是挂载在具体的 **B+ 树索引节点**上的。
当你尝试操作 `id = 5` 时，由于 5 不存在，InnoDB 会在 B+ 树中寻找 5 应该所在的位置。它会找到 5 的上界（即 8）和下界（即 3）。为了锁住这中间的间隙，InnoDB 会采取**向右找锚点**的策略：

- 它会将这个间隙锁挂在大于 5 的**第一个现有记录**上，也就是 `id = 8` 的这条记录上。
- 所以，在 `data_locks` 表中，这个锁的宿主记录（`lock_data`）就是 `8`。

### 2. `lock_mode` 的真实语意定义
在 InnoDB 的语境中，锁的模式和锁的数据是配合起来表达完整的锁范围的：
- **`lock_data = 8`**：代表这个锁的物理依附点是主键值为 8 的索引记录。
- **`lock_mode = X, GAP`**：这里的 `GAP` 标志位是一个极其重要的修饰符。它的严格语意是：**锁定依附点（8）之前的那个间隙，且不包含依附点本身**。
所以，这两者结合在一起，翻译成人类语言就是：“在记录 8 的前面拉起一道警戒线，直到遇到上一个存在的记录（3）为止”。

### 3. 为什么不需要显示下界（3）？

你可能会问，那 InnoDB 内部是怎么知道间隙的下界是 3 的呢？为什么不在锁结构里把 3 也存下来？
这是为了**极致节省内存**，并且利用了 B+ 树的天然物理结构：
- InnoDB 的 B+ 树叶子节点是一个**双向链表**。
- 当 InnoDB 知道锁的锚点在 `8` 时，它只要顺着底层链表的指针向左看一眼，瞬间就能找到前驱节点 `3`。
- 既然通过底层数据结构天然就能确定下界，在内存的锁结构（Lock Struct）中再去额外存储一个“下界值”就纯属浪费内存了。当系统中存在海量锁时，这种极简设计能节省巨大的开销。

### 一个通俗的比喻

想象一下，这条街上有 1号、3号、8号 三根电线杆。你想把 3号 到 8号 之间的空地围起来不让人停车（插入数据 5）。
你不能把横幅挂在空气中。你的做法是：拿着写着“禁止停车”的牌子（`X, GAP`），把它**绑在 8号 电线杆上**（`lock_data = 8`），牌子上的规定是：“从这根电线杆往回走，直到碰到上一根电线杆为止，这段距离不准停车”。
所以别人检查的时候，看到的是 8号 杆子上挂了牌子，而不是 3号 杆子。
