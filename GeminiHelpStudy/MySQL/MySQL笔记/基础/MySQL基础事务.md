## 概述
- **概念**：事务是一组操作的集合，它是一个不可分割的工作单位。事务会把所有的操作作为一个整体一起向系统提交或撤销操作请求，即这些操作**要么同时成功，要么同时失败** 。
- **目的**：保证业务逻辑的数据完整性和一致性。

![[MySQL事务简介图.png]]
> [!warning] 注意
> MySQL 中事务默认是自动提交的（`autocommit = 1`），即执行一条 DML 语句，MySQL 会立即隐式提交事务。

## 事务基本操作
- **作用场景**：涉及多个步骤的数据变更，且要求这些步骤必须全部成功才能生效（例如：银行转账，A账户扣钱的同时B账户必须加钱） 。
- **执行逻辑**：开启事务后，后续的 DML 操作只会改变内存中的临时状态。只有执行提交操作后，数据才会真正持久化到磁盘；若中途出错执行回滚，则所有临时变更作废 。

#### 标准语法与示例
```SQL
-- 方式一：修改事务提交方式
SELECT @@autocommit;
SET @@autocommit = 0; -- 设置为手动提交

-- 方式二：显式开启事务（推荐）
START TRANSACTION; 
-- 或
BEGIN;

-- 提交事务（成功时执行）
COMMIT;

-- 回滚事务（失败时执行）
ROLLBACK;
```

- **示例代码 (转账场景)**
```SQL
START TRANSACTION;

-- 1. 查询张三账户余额
SELECT * FROM account WHERE name = '张三';
-- 2. 将张三账户余额-1000
UPDATE account SET money = money - 1000 WHERE name = '张三';
-- 3. 将李四账户余额+1000
UPDATE account SET money = money + 1000 WHERE name = '李四';

-- 如果上述操作全部成功
COMMIT;
-- 如果中途出现报错
ROLLBACK;
```

## 事务四大特性 (ACID)
这是面试和理论的核心重点 。

|**特性**|**英文缩写**|**概念说明**|**核心意义**|
|---|---|---|---|
|**原子性**|**A**tomicity|事务是不可分割的最小操作单元，要么全部成功，要么全部失败。|保证操作不出现“半截子”状态。|
|**一致性**|**C**onsistency|事务完成时，必须使所有的数据都保持一致状态。|业务逻辑上的正确（如转账前后总金额不变）。|
|**隔离性**|**I**solation|数据库系统提供的隔离机制，保证事务在不受外部并发操作影响的独立环境下运行。|多个事务并发执行时互不干扰。|
|**持久性**|**D**urability|事务一旦提交或回滚，它对数据库中的数据的改变就是永久的。|宕机或断电也不丢失数据。|

## 并发事务问题
在多个事务同时运行（并发）时，如果不加以隔离，会引发以下数据读取问题 。

|**问题名称**|**概念简述**|**执行逻辑说明**|
|---|---|---|
|**脏读**|一个事务读到另外一个事务**还没有提交**的数据。|事务A修改了数据但未提交，事务B此时读取了该数据。若事务A回滚，事务B读到的就是“脏”数据。|
|**不可重复读**|一个事务先后读取同一条记录，但两次读取的数据不同。|事务A第一次读取某行后，事务B**修改**了该行并提交，事务A第二次读取同一行发现数据变了。|
|**幻读**|一个事务按照条件查询数据时，没有对应的数据行，但是在插入数据时，又发现这行数据已经存在，好像出现了“幻影”。|事务A查询发现无某条记录，准备插入；此时事务B**插入**了该记录并提交；事务A再插入时报错（主键冲突），再查询却发现有这条记录了。|

## 事务隔离级别
为了解决并发事务问题，数据库提供了不同的隔离级别 。

### 隔离级别对比表

|**隔离级别**|**脏读**|**不可重复读**|**幻读**|**说明**|
|---|---|---|---|---|
|**Read Uncommitted** (读未提交)|√|√|√|性能最高，但数据安全性最差，基本不用。|
|**Read Committed** (读已提交)|×|√|√|Oracle默认级别，解决了脏读。|
|**Repeatable Read** (可重复读)|×|×|√|**MySQL默认级别**，解决了不可重复读。|
|**Serializable** (串行化)|×|×|×|事务排队执行，绝对安全，但并发性能极差。|

> [!info] 提示
> `√` 表示可能出现该问题，`×` 表示不会出现该问题。隔离级别越高，数据越安全，但并发性能越低 。

#### 设置与查看隔离级别语法

```SQL
-- 查看当前事务隔离级别
SELECT @@TRANSACTION_ISOLATION;

-- 设置事务隔离级别
SET [ SESSION | GLOBAL ] TRANSACTION ISOLATION LEVEL { READ UNCOMMITTED | READ COMMITTED | REPEATABLE READ | SERIALIZABLE };
```

## 💡 避坑与进阶指导
### 实际开发常见坑
1. **长事务阻塞**：开启事务后进行了大量的耗时操作（如调用第三方外部接口、复杂计算），导致数据库锁迟迟不释放，拖垮整个数据库性能。**规范：事务只包裹数据库 DML 操作，越短越好。**
2. **Spring 声明式事务失效**：在 Java 后端开发中，常使用 `@Transactional` 注解控制事务。如果方法内部 `try-catch` 捕获了异常且没有向外抛出（或者抛出的不是 `RuntimeException`），会导致事务不回滚。
3. **忘记提交/回滚**：手动管理事务时，若因为异常跳过了 `COMMIT` 或 `ROLLBACK` 步骤，会导致连接一直挂起。


### 性能注意点
1. 绝大多数互联网项目使用 MySQL 默认的 `Repeatable Read` 即可，或者为了更高的并发降低为 `Read Committed`。极少使用 `Serializable`。
2. 避免在业务高峰期执行涉及大批量数据的事务更新。
