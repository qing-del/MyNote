## 概述
- **概念**：指从多张表中查询数据的操作 。
- **多表关系**：在业务系统中，表与表之间通常存在联系（一对一、一对多、多对多） 。
- **核心逻辑**：不加条件的笛卡尔积（A表行数 * B表行数）会产生大量无效数据，多表查询的核心在于通过连接条件消除无效的笛卡尔积 。

![[MySQL多表查询概念图.png]] 
> [!warning] 注意
> 在进行多表查询时，务必明确表之间的关联字段，否则极易引发笛卡尔积导致数据库性能骤降甚至宕机。

## 连接查询
多表查询主要分为内连接、外连接、自连接和联合查询 。
### 连接方式对比
|**连接类型**|**关键字**|**查询范围**|**适用场景**|
|---|---|---|---|
|**内连接**|`[INNER] JOIN`|交集数据（两表完全匹配的部分）|查询两个业务实体的关联数据（如：员工及其所属部门）|
|**左外连接**|`LEFT [OUTER] JOIN`|左表所有数据 + 右表匹配数据|以左表为主表查询（如：查询所有员工，即使没有部门也要显示）|
|**右外连接**|`RIGHT [OUTER] JOIN`|右表所有数据 + 左表匹配数据|以右表为主表查询|
|**自连接**|`JOIN`|把一张表当成两张表进行连接|具有层级结构的数据表（如：查询员工及其直属领导的名字）|

---

#### 内连接
- **作用场景**：查询两个表中存在对应关系的数据记录。
- **执行逻辑**：遍历主表，根据连接条件去从表寻找匹配行，只返回完全匹配的记录。

```SQL
-- 隐式内连接
SELECT 字段列表 FROM 表1, 表2 WHERE 条件 ... ;

-- 显式内连接（推荐，语义更清晰）
SELECT 字段列表 FROM 表1 [INNER] JOIN 表2 ON 连接条件 ... ;
```

- **示例代码**
```SQL
-- 查询员工姓名及其对应的部门名称
SELECT e.name, d.name FROM employee e INNER JOIN department d ON e.dept_id = d.id;
```

> [!info] 限制
> 如果说是`employee`表中有员工的`dept_id=null`
> 那么使用内连接查询是不会显示这条数据的

#### 外连接
- **作用场景**：需要保证某个业务实体（左表或右表）的数据完整性，即使另一方没有关联数据也要展示。
- **执行逻辑**：保留基准表的所有行，如果在连接表中找不到匹配行，则对应列返回 NULL。

```SQL
-- 左外连接
SELECT 字段列表 FROM 表1 LEFT [OUTER] JOIN 表2 ON 条件 ... ;
```

- **示例代码**

```SQL
-- 查询所有部门的信息，以及该部门下的员工信息（即使没有员工的部门也会列出）
SELECT d.*, e.name FROM department d LEFT JOIN employee e ON d.id = e.dept_id;
```

#### 联合查询 (UNION)
- **作用场景**：将多次查询的结果合并为同一个结果集。
- **执行逻辑**：分别执行多个SELECT语句，然后将结果集垂直拼接。`UNION ALL` 直接拼接，`UNION` 会进行去重操作。

```SQL
SELECT 字段列表 FROM 表A ...
UNION [ALL]
SELECT 字段列表 FROM 表B ...;
```

> [!warning] 要求
> 联合查询的多张表的列数必须保持一致，字段类型也需要保持一致。

> [!tip]
> `union all`会将全部的数据直接合并在一起，`union`会对合并之后的数据**去重**

子查询 (嵌套查询)
- **概念**：SQL语句中嵌套SELECT语句，称为嵌套查询，又称子查询 。
- **分类**：根据子查询结果的不同，分为标量、列、行、表子查询 。

### 子查询分类与示例
|**分类**|**结果特征**|**常用操作符**|**作用场景示例**|
|---|---|---|---|
|**标量子查询**|单个值（一行一列）|`=`, `<>`, `>`, `<`|查询“研发部”的所有员工（先查研发部ID，再根据ID查员工）|
|**列子查询**|一列（多行一列）|`IN`, `NOT IN`, `ANY`, `ALL`|查询和“张三”或“李四”在同一部门的员工|
|**行子查询**|一行（一行多列）|`=`, `<>`, `IN`, `NOT IN`|查询与“王五”的薪资及直属领导相同的员工|
|**表子查询**|多行多列|`IN`|查询入职日期在某个时间之后的员工信息及其部门信息（将其作为临时表再进行联查）|

- **示例代码 (列子查询)**
```SQL
-- 查询 "销售部" 和 "市场部" 的所有员工信息
SELECT * FROM employee WHERE dept_id IN (SELECT id FROM department WHERE name = '销售部' OR name = '市场部');
```

#### 操作符
|**操作符**|**描述**|
|:---:|:---|
|`IN`|**在**指定的集合范围之内，多选一|
|`NOT IN`|**不在**指定的集合范围之内|
|`ANY`|子查询返回列表中，有**任意一个**满足即可|
|`SOME`|和`ANY`等同|
|`ALL`|子查询返回列表的**所有**值都**必须**满足|

#### 列子查询
- `ALL`的示例
```sql
-- 查询比"研发部"所有人工资都高的员工
select * 
from employee 
where salary > all(
	select salary 
	from employee
	where dept_id = (
		select id
		from dept
		where name='研发部'
	)
);
```

#### 行子查询
- 另一种等值匹配方式（多列）
```sql
-- 查询和“张无忌”的薪资和管理人相同的员工
select *
from emp
where (salary, managerId) = (
	select salary,managerId
	from emp
	where name = '张无忌'	
)
```

#### 表子查询
```sql
-- 查询和“鹿杖客”，“宋远桥”的职位和薪资相同的员工信息
select *
from emp
where (job, salary) in (
	select job, salary
	from emp
	where name='鹿杖客' or name='宋远桥'
)
```

## 💡 避坑与进阶指导
### 实际开发常见坑
1. **隐式内连接的滥用**：早期遗留代码常在 `WHERE` 中写多个表的关联条件，导致关联逻辑与过滤逻辑混杂，可读性和维护性极差。**规范要求：推荐使用 `JOIN ... ON` 的显式写法。**
2. **不加表别名**：多表查询时，若多个表有同名字段（如 `id`, `name`），不使用表别名会报 `ambiguous` 错误。
3. **外连接条件错放**：将外连接 `LEFT JOIN` 从表的过滤条件写在了外层的 `WHERE` 中，会导致外连接退化为内连接。正确的做法是写在 `ON` 条件里。

### 性能注意点
1. **小表驱动大表**：在进行 JOIN 查询时，尽量让数据量小的表作为驱动表去关联数据量大的表（虽然优化器通常会自动处理，但在复杂 SQL 中可能需要人工干预）。
2. **关联字段加索引**：虽然互联网开发中常为了性能不建立物理外键约束，但在逻辑上用于关联的字段（如 `dept_id`）**强烈建议建立索引**，否则表连接时容易导致全表扫描，严重拖慢查询速度。
3. **子查询 vs JOIN**：尽量避免在 `SELECT` 列表中使用相关子查询，因为外层表每返回一行，子查询就会执行一次。在多数情况下，推荐将子查询改写为 `JOIN` 查询。
