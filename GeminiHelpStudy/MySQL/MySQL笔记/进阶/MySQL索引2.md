## 概述

- **概念**：建立索引只是第一步，核心在于如何排查慢查询，以及掌握索引的生效机制与设计规范，从而避免索引失效带来的性能瓶颈。
- **目的**：通过工具定位慢SQL，利用规则优化SQL语句，最终构建高效的数据库查询体系。

## 一、 索引性能分析

> [!info] 分析思路
> 排查数据库性能问题通常遵循：看全局频次 -> 抓慢查询日志 -> 查单条耗时分布 -> EXPLAIN深度分析。

### 性能分析工具对比

|**分析工具**|**作用场景**|**执行逻辑说明**|**核心价值**|
|---|---|---|---|
|**执行频次**|评估当前数据库是以读为主还是写为主 。|统计 `Com_select` / `Com_insert` 等指令的执行次数。|决定是否需要做主从分离或偏向查询优化。|
|**慢查询日志**|定位执行时间超过预设阈值的“慢 SQL” 。|开启后，MySQL 会将耗时超过 `long_query_time` 的语句记录到日志文件中。|找出系统中真正拖慢响应的罪魁祸首。|
|**Show Profiles**|查看某条 SQL 在各个阶段的耗时细节 。|记录最近执行的 SQL 语句在底层硬件层面的 CPU/IO 耗时分布。|排查耗时是在网络、解析还是执行阶段。|
|**EXPLAIN**|获取 MySQL 优化器如何执行该条 SQL 语句的详细计划 。|模拟优化器执行 SQL，输出表连接顺序、索引使用情况等。|**日常优化的最核心工具**，直接判断索引是否生效。|

### 执行关键字的总次数
- 用于判断当前表是**查询**为主还是**更新**为主
```sql
-- 首先需要执行一条 SQL 语句
select * from user;

-- 使用下面这条指令可以查看执行次数 （7个下划线,后面跟7个字符）
show global status like 'Com_______';
```

### 慢查询日志
> [!warning] Linux的步骤

- 慢查询日志记录了所有执行时间**超过指定参数**（`long_query_time`，单位：秒，默认10秒）的**所有**SQL语句的日志
- MySQL的慢查询**默认不开启**，需要在MySQL的配置文件（`/etc/my.cnf`）中配置如下信息：
```bash
# 开启MySQL慢查询日志开关
slow_query_log=1
# 设置慢查询日志的时间为 2 秒，SQL语句执行超过 2 秒，就会视为慢查询，记录慢查询日志
long_query_time=2
```
> 配置完毕之后重启MySQL进行测试
> 查看慢日志文件中记录的信息`/var/lib/mysql/localhost-slow.log`

### Profile
- 可以在做SQL优化的时候帮助我们知道时间耗费
```sql
-- 查看 Profile 开关是否打开
select @@have_profiling;

-- 设置开关参数
set profiling = 1;
```

- 功能用法：
```sql
-- 查看当前会话中所有 SQL 语句的耗时
show profiles;

-- 查看指定 query_id 的 SQL 语句各阶段的耗时情况
show profile for query [query_id];

-- 查看指定 query_id 的 SQL 语句 CPU 使用情况
show profile cpu for query [query_id];
```

### EXPLAIN 执行计划核心字段

- **作用场景**：在编写复杂查询或遇到慢查询时，必须在 SQL 前加上 `EXPLAIN` 或 `DESC` 进行分析 。
- **标准语法**：
```SQL
EXPLAIN SELECT 字段列表 FROM 表名 WHERE 条件;
```

- **核心字段解析**：

- `id`
	- 如果用`explain`查询出来`id`相同就是从上往下执行
	- 如果是不同，就是越大越先被执行
- `select_type`
	- `SIMPLE`简单表查询
	- `PRIMARY`主键查询
	- `UNION`在含有次关键词的查询中属于第二个或者后者的查询
	- `SUBQUERY`（`SELECT/WHERE`之后包含了子查询）

|**字段名**|**含义简述**|**优化目标**|
|---|---|---|
|`type`|访问类型。表示找到所需行的方式。|争取达到 `ref` 或 `range`，避免 `ALL` (全表扫描)。|
|`possible_keys`|可能应用在这张表上的索引。|仅供参考，不一定实际使用。|
|`key`|**实际使用的索引**。|如果为 NULL，说明没有使用索引或索引失效。|
|`key_len`|索引中使用的字节数。|联合索引中，越长代表使用的字段越多，精度越高。|
|`Extra`|额外信息。|期望看到 `Using index` (覆盖索引)，警惕 `Using filesort` (文件排序)。|

> `Type`字段详细 -> [[MySQL索引原理#Type 列]]

## 二、 索引使用规则

索引并非建了就一定生效，需要遵循特定的使用规则 。

### 1. 最左前缀法则 (针对联合索引)

- **概念**：如果索引了多列（联合索引），查询从索引的最左列开始，并且不跳过索引中的列 。
- **执行逻辑**：如果跳跃了某一列，索引将部分失效（后面的字段索引失效）。
- **示例SQL**：

```SQL
-- 假设存在联合索引 idx_user_pro_age_sta (profession, age, status)

-- ✅ 完全生效：包含最左前缀 profession，且连续
EXPLAIN SELECT * FROM user WHERE profession = '软件工程' AND age = 20 AND status = '0';

-- ❌ 部分生效：跳过了 age，status 的索引失效
EXPLAIN SELECT * FROM user WHERE profession = '软件工程' AND status = '0';

-- ❌ 完全失效：没有最左前缀 profession
EXPLAIN SELECT * FROM user WHERE age = 20 AND status = '0';
```
> [!tip] 可以思考 B+树 中的排序结构

> [!warning] 特别说明：
> - 联合索引中 `>` 导致“索引失效”的真相 很多教程称在联合索引 `(a, b, c)` 中使用 `b > 30` 会导致右侧索引失效，这是一种简化且易误导的说法。**底层并未发生全表扫描，依然在走索引树。**
>
> - **为什么 `>=` 显示用到了完整索引？** > `>= 30` 提供了绝对精确的边界。MySQL 的 B+ 树能利用完整的联合索引精确降落（Seek）到数据的唯一起点。此时 `EXPLAIN` 中的 `key_len` 最长，代表所有字段都参与了**起点定位**。
>   
> - **为什么 `>` 显示部分索引失效？** `> 30` 无法确定下一个确切值（如 30.1 还是 31）。因此引擎只能用前面的字段寻找大致边界，后面的字段无法参与“起点定位”，而是退化为顺着叶子节点扫描时的**过滤条件**（Scan / 索引下推）。如果 `> 30` 匹配的数据量极大，优化器（CBO）甚至会直接跳过用它找边界，导致 `key_len` 变得极短。
>   
> **💡 最佳实践：** “失效”仅代表未参与 B+ 树的精确检索，而非没用上索引。但在业务逻辑允许且字段为整数时（例如将 `> 30` 等价替换为 `>= 31`），**强烈建议优先使用 `>=`**，明确边界，压榨出最极致的 B+ 树定位性能。
> 
> > 详细内容 -> [[MySQL联合索引范围查询失效问题]]

### 2. 索引失效的常见情况

在使用时，以下操作会导致索引失效，从而退化为全表扫描 。

|**失效场景**|**示例写法**|**执行逻辑说明**|
|---|---|---|
|**函数运算**|`WHERE substring(phone,10,2) = '15'`|在索引列上做运算，破坏了 `B+Tree` 的原有排序，无法使用索引定位。|
|**类型隐式转换**|`WHERE phone = 13800000000` (phone为VARCHAR)|字符串不加单引号，MySQL 内部会自动调用铸型函数进行转换，等同于函数运算。|
|**模糊匹配**|`WHERE name LIKE '%三'`|**头部模糊**匹配导致无法利用 `B+Tree` 的前缀比较特性。**尾部模糊不会**|
|**OR连接条件**|`WHERE id = 10 OR age = 20` (age无索引)|只要 OR 两侧有一个条件没有索引，为保证数据不遗漏，数据库只能全表扫描。|

> [!info] 额外情况
> - **数据分布影响**
> 	- 如果MySQL评估使用索引比全表更慢，则不使用索引
> 	- 大概是`1/3`这样；这个走不走索引并**不是固定**的，而是根据表中数据分布来确定的；

### 3. 覆盖索引与回表查询

- **概念**：查询使用了索引，并且需要返回的列，在该索引中已经全部能够找到，这就叫**覆盖索引** 。如果需要返回的列不在索引中，就需要通过主键去聚集索引中查整行数据，这就叫**回表查询** 。
- **作用场景**：高频查询的接口，通过联合索引直接覆盖返回值，避免二次磁盘 I/O。

> 详细可见 -> [[MySQL索引原理#Extra 列]]

### 4. 其他高级规则

- **前缀索引**：针对 `VARCHAR` / `TEXT` 等长文本字段，只提取前几个字符建立索引，以减小索引体积，提高查询效率 。
> [!info] 详细
> 可以先获取字符选择度
> ```sql
> -- 如果是 1.0 就是性能最好的 因为选择性最高
> select count(distinct email)/count(*) from user;
> 
> -- 获取截取之后的选择性
> select count(distinct substring(email, 1, 5))/count(*) from user;
> 
> -- 根据（大小，区分度）最优的选择截取长度来创建索引
> create index idx_email_5 on user(email(5));
> ```
> 
> - 建立这个索引之后，使用`where email = lvbu666@163.com`来查询
> > 会先截取**前五个**来进行索引查询，如果有重复再进行范围扫描
- **SQL提示**：在 SQL 中显式告诉优化器使用哪个索引（`USE INDEX`(**建议**)、`IGNORE INDEX`(**忽略**)、`FORCE INDEX`(**强制**)） 。
> `select * from user force index([索引名称]) where profession = 'Xxx';`
- **单列与联合索引的选择**：在多个字段都需要做查询条件时，**强烈建议建立联合索引**，而非多个单列索引 。

## 三、 索引设计原则

> [!warning] 警告
> 索引是把双刃剑。虽然提升了查询效率，但会占用磁盘空间，并拖慢 DML（增删改）操作速度。

1. **针对表**：数据量较大，且查询比较频繁的表。
2. **针对字段**：常作为查询条件（`WHERE`）、排序（`ORDER BY`）、分组（`GROUP BY`）操作的字段 。
3. **区分度**：尽量选择区分度高的列作为索引，尽量建立唯一索引（如身份证号），区分度越高，使用索引的效率越高 。
4. **字符串类型**：如果是字符串类型字段，字段长度较长，可以建立前缀索引 。
5. **联合索引优先**：尽量使用联合索引，减少单列索引。联合索引很多时候可以覆盖索引，节省存储空间，避免回表查询 。
6. **控制数量**：要控制索引的数量，索引越多，维护代价越大 。

> 索引不能存储`NULL`值，需要建立索引就需要用`not null`来约束
> 当优化器知道每列是否包含`NULL`值时，它可以更好确定哪个索引在查询中更有效

## 💡 避坑与进阶指导

### 实际开发常见坑

1. **隐式转换翻车**：后端 Java 代码中传入的参数类型是 `Integer`，但 MySQL 对应的表字段是 `VARCHAR`，由于没有严格校验，导致生产环境慢查询频发。
    
2. **`ORDER BY` 导致的文件排序**：业务逻辑需要对结果集进行排序，但排序字段没有包含在索引中，导致出现 `Using filesort`。需要在联合索引的设计中将排序字段一并考虑进去。
    

### 性能注意点

1. **数据分布影响优化器抉择**：有时候即使 SQL 完全符合最左前缀法则，索引依然失效了。这是因为 MySQL 优化器在评估后认为：全表扫描比走索引再回表查询的代价更低（例如查询的数据占据了全表 80% 以上）。
    
2. **联合索引字段顺序**：在设计联合索引时，应将查询频率最高、区分度最大的字段放在最左侧。
