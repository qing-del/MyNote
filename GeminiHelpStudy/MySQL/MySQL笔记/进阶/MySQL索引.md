## 概述

- **概念**：索引（Index）是帮助 MySQL **高效获取数据**的**数据结构**（有序）。在数据之外，数据库系统还维护着满足特定查找算法的数据结构，这些数据结构以某种方式引用（指向）数据，这样就可以在这些数据结构上实现高级查找算法。
- **目的**：极大地减少磁盘 I/O 次数，提高查询效率。
- **优缺点**：
    - **优点**：提高数据检索效率，降低数据库 IO 成本；通过索引列对数据进行排序，降低数据排序的成本，降低 CPU 消耗。
    - **缺点**：索引列也是要占用磁盘空间的；索引大大提高了查询效率，但同时也降低了更新表（`INSERT`、`UPDATE`、`DELETE`）的速度，因为更新数据时还要同步维护索引。

> [!warning] 注意
> 并不是所有的字段都适合加索引。索引只应建立在经常作为查询条件（`WHERE`）、排序（`ORDER BY`）或分组（`GROUP BY`）的字段上。

## 索引结构
MySQL 的索引是在存储引擎层实现的，不同的存储引擎有不同的索引结构。

### 核心索引结构对比

|**索引结构**|**特点说明**|**适用存储引擎**|
|---|---|---|
|**B+Tree 索引**|最常见的索引类型，大部分引擎都支持 B+ 树索引。叶子节点包含了所有数据，且形成一个双向链表，便于范围查询。|InnoDB, MyISAM, Memory|
|**Hash 索引**|底层数据结构是用哈希表实现的，只有精确匹配索引列的查询才有效，**不支持范围查询**。|Memory (InnoDB 具有自适应 Hash 功能)|
|**R-Tree (空间索引)**|空间索引是 MyISAM 引擎的一个特殊索引类型，主要用于地理空间数据类型，通常使用较少。|MyISAM|
|**Full-text (全文索引)**|是一种通过建立倒排索引，快速匹配文档的方式。类似于 Lucene, Solr, ES。|InnoDB (5.6+), MyISAM|

> [!info] 执行逻辑说明：为什么选 B+Tree？
> 相比于二叉树（容易退化为链表）和红黑树（树层级依然较深），B+Tree 是一种多路平衡查找树。它的**非叶子节点只存储指针，不存储数据**，这使得一个固定大小的磁盘页能够容纳更多的键值，从而极大降低了树的层高（通常 3 层就能支撑千万级数据）。并且**叶子节点形成了双向链表**，非常适合范围查询（如 `age > 20`）。
> 参考 -> [[MySQL索引原理#B+树]]

## 索引分类

### 1. 按照逻辑功能分类

|**分类**|**关键字**|**作用场景**|**特点**|
|---|---|---|---|
|**主键索引**|`PRIMARY KEY`|表的主键字段|针对该表中主键创建的索引，默认自动创建，**只能有一个**。|
|**唯一索引**|`UNIQUE`|手机号、身份证号等唯一字段|避免同一个表中某数据列中的值重复，**可以有多个**。|
|**常规索引**|无|经常被查询的普通字段（如姓名）|快速定位特定数据，**可以有多个**。|
|**全文索引**|`FULLTEXT`|文章内容、备注等长文本|全文索引查找的是文本中的关键词，而不是比较索引中的值，**可以有多个**。|

### 2. 按照物理存储形式分类 (InnoDB 核心)
在 InnoDB 存储引擎中，根据索引的存储形式，又可以分为以下两种：

|**分类**|**概念简述**|**执行逻辑说明**|
|---|---|---|
|**聚集索引 (Clustered Index)**|将数据存储与索引放到了一块，**叶子节点保存了整行数据**。|必须有且只有一个。如果存在主键，主键索引就是聚集索引；若无主键，将使用第一个唯一索引；若都无，InnoDB 会自动生成一个隐式聚集索引（rowid）。|
|**二级索引 (Secondary Index)**|将数据与索引分开存储，**叶子节点关联的是对应的主键**。|也叫非聚集索引。查询时先走二级索引找到主键值，再用主键值去聚集索引中查找整行数据，这个过程称为**回表查询**。|

## 语法与示例

#### 标准语法

```SQL
-- 创建索引 (默认是 B+Tree 索引)
CREATE [ UNIQUE | FULLTEXT ] INDEX index_name ON table_name ( index_col_name,... ) ;

-- 查看某张表的索引
SHOW INDEX FROM table_name ;

-- 删除索引
DROP INDEX index_name ON table_name ;
```

- **示例代码**

```SQL
-- 为 user 表的 name 字段创建一个常规索引
CREATE INDEX idx_user_name ON user(name);

-- 为 user 表的 phone 字段创建一个唯一索引
CREATE UNIQUE INDEX idx_user_phone ON user(phone);

-- 为 user 表的 profession, age, status 字段创建一个联合索引
CREATE INDEX idx_user_pro_age_sta ON user(profession, age, status);
```

## 💡 避坑与进阶指导
### 实际开发常见坑
1. **滥用索引**：为了查询快，给表里所有字段都加上索引。这会导致每次 `INSERT` / `UPDATE` 极慢，且占用大量磁盘空间。
2. **模糊查询导致全表扫描**：使用 `LIKE '%关键字'`（左侧带百分号）会导致索引失效，引发全表扫描。
3. **数据类型隐式转换**：如果索引字段是 `VARCHAR`，但在查询时传入了 `INT` 类型（如 `WHERE phone = 13800000000` 而不是 `'13800000000'`），会导致触发内部函数转换，从而使索引失效。


### 性能注意点

1. **覆盖索引 (Covering Index)**：尽量使用覆盖索引（查询使用了索引，并且需要返回的列，在该索引中已经全部能够找到），避免使用 `SELECT *`。因为 `SELECT *` 极大概率会导致**回表查询**，增加一次磁盘 I/O。
2. **前缀索引**：当字段类型为字符串（`VARCHAR`，`TEXT`等）时，有时候需要索引很长的字符串，这会让索引变得很大。可以只将字符串的一部分前缀建立索引，以节约空间。
    

### 面试高频问题

- 面试题：为什么 InnoDB 存储引擎选择 B+Tree 作为索引结构？（对比 B-Tree、Hash 和二叉树回答）。
- 面试题：什么是聚集索引和二级索引？什么是回表查询？
- 面试题：如果查询语句是 `SELECT id, name FROM user WHERE name = '张三';`（name 上有常规索引），会发生回表查询吗？（提示：如果 id 是主键，这叫覆盖索引，不需要回表）。
