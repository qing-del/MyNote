## 概述

- **概念**：视图、存储过程和触发器是数据库中用于**逻辑封装、代码复用和自动化控制**的高级数据库对象。
- **目的**：
    - **视图**：简化复杂的 SQL 操作，保护数据底层的表结构。
    - **存储过程**：将复杂的业务逻辑直接下沉到数据库层执行，减少网络交互。
    - **触发器**：实现数据的自动化审计、日志记录和多表数据同步。

> [!info] 学习定位
> 在现代基于微服务和 Java/Go 后端的开发模式中，**存储过程和触发器的使用频率正在大幅下降**（业务逻辑更倾向于写在代码里以便于维护和扩展）。这部分内容作为进阶知识，重在**理解其运行机制和语法框架**，以应对遗留系统维护或特定性能场景。

## 一、 视图 (View)

- **概念**：视图是一种**虚拟表** 。视图本身不存储数据，它只**保存**了生成该视图的 SQL **查询逻辑**。**每次查询**视图时，数据库会**动态执行**底层的 **SQL 语句**并返回结果。

### 1. 语法结构

```SQL
-- 创建或替换视图（同名就会替换视图）
CREATE [OR REPLACE] VIEW 视图名称[(列名列表)] AS SELECT语句 [ WITH [ CASCADED | LOCAL ] CHECK OPTION ];

-- 查询视图 (完全等同于查询普通表)
SHOW CREATE VIEW 视图名称;  -- 查询视图创建的参数
SELECT * FROM 视图名称;  -- 查询视图中的数据

-- 修改视图
ALTER VIEW 视图名称 AS SELECT语句;

-- 删除视图
DROP VIEW [IF EXISTS] 视图名称;
```

### 2. 检查选项 (CHECK OPTION)

当通过视图去 `INSERT`、`UPDATE`、`DELETE` 数据时，MySQL 会检查这些操作是否符合视图定义的 `WHERE` 条件。

|**检查选项**|**执行逻辑说明**|**核心差异**|
|---|---|---|
|**CASCADED (默认)**|**级联检查**。不仅会检查当前视图的条件，还会**向上追溯并强制**检查所有底层依赖视图的条件。|极度严格，任何底层视图的条件都不能违背。|
|**LOCAL**|**本地检查**。**只会检查当前**视图的条件，如果底层依赖的视图没有定义检查选项，则不管底层条件。|相对宽松，只对明确加了检查选项的层级负责。|

> [!info] 依赖关系
> - 如果`view_2`依赖`view_1`
> 	- 修改`view_2`时，需要检查是否满足`view_1`的条件
> 	- 修改`view_1`时，不需要检查是否满足`view_2`的条件

> [!warning] `CASCADED`的“传染性”
> - **假设**：
> 	- **视图 1 (Base View)**：假设没有检查选项。
> 	- **视图 2 (LOCAL)**：依赖视图 1。由于是 `LOCAL`，它只检查**落入自己定义范围**的条件。
> 	- **视图 3 (CASCADED)**：依赖视图 2。
> 
> - **第一步：** 检查修改的数据是否符合 **视图 3** 的 `WHERE` 条件。
> - **第二步：** 向上级联，检查是否符合 **视图 2** 的 `WHERE` 条件（即使视图 2 是 `LOCAL`）。
> - **第三步：** 继续向上级联，检查是否符合 **视图 1** 的 `WHERE` 条件。
> 
> **结论：** 修改视图 3 时，**会检查到视图 1 为止**。`CASCADED` 具有“传染性”，一旦顶层使用了它，它会强制要求整条依赖链上的所有约束都要满足。

### 3. 视图的更新规则

> [!warning] 警告
> 并不是所有的视图都能进行增删改。视图的数据必须与底层基础表的数据存在==**一对一**==的关系。

如果视图包含以下任何一项，则该视图**不可更新**：
- 聚合函数（`SUM()`, `MIN()`, `MAX()`, `COUNT()` 等）
- `DISTINCT`、`GROUP BY`、`HAVING`、`UNION` 或 `UNION ALL`

### 视图的实际作用

#### 1. 简单（简化复杂查询）

如果你的数据库表结构很复杂，经常需要写很长的 `JOIN` 语句来关联五六张表，视图可以把这些复杂的逻辑封装起来。以后你只需要查询这个视图，就像查询单张表一样简单。
- **例子：** 假设要查“订单详情”，平时需要关联 `Orders`、`Products`、`Users` 三张表。
- **视图化：** 你创建一个名为 `view_order_summary` 的视图。
- **结果：** 以后想看数据，只需执行 `SELECT * FROM view_order_summary;`，再也不用写几十行 SQL 了。

#### 2. 安全（权限控制）

视图可以实现“列级别”的权限控制。你可以只给用户访问视图的权限，而不给他们访问原表的权限。这样就能隐藏敏感字段（如：身份证号、工资、密码）。
- **例子：** 员工表 `Employees` 中包含 `姓名`、`电话` 和 `基本工资`。
- **安全做法：** 为普通 HR 创建一个视图，只包含 `姓名` 和 `电话`。
- **结果：** HR 只能通过视图看到工作联系方式，无法窥探员工的薪资隐私。

#### 3. 数据独立（逻辑解耦）

当数据库的底层表结构发生变化时（比如拆分表、修改字段名），只要视图的定义逻辑不变，前端应用程序的代码就不需要做任何修改。
- **例子：** 原本有一张 `User_Info` 表，后来因为性能优化，你把它拆分成了 `User_Base` 和 `User_Extra` 两张表。
- **逻辑平滑：** 你可以创建一个名为 `User_Info` 的视图，通过 `JOIN` 把这两张新表连起来。
- **结果：** 对于程序员来说，他们查询的还是 `User_Info`，完全感觉不到底层表已经“大换血”了。

## 二、 存储过程与函数 (Stored Procedure & Function)

- **概念**：事先经过编译并存储在数据库中的一段 SQL 语句的集合 。
- **特点**：封装、复用、可以接收参数并返回数据、减少网络交互次数。

### 1. 变量体系

|**变量类型**|**作用域与生命周期**|**定义与赋值方式**|
|---|---|---|
|**系统变量**|分为全局 (`GLOBAL`) 和会话 (`SESSION`) 级别。由 MySQL 服务器提供。|`SHOW VARIABLES LIKE '...';`<br><br>  <br><br>`SET SESSION @@变量名 = 值;`|
|**用户定义变量**|当前客户端会话级别。不需要提前声明，直接使用。|`@变量名`<br><br>  <br><br>`SET @age = 20;`|
|**局部变量**|仅在存储过程的 `BEGIN ... END` 块内有效。必须提前声明。|`DECLARE 变量名 类型 [DEFAULT 值];`<br><br>  <br><br>`SET 变量名 = 值;`|

> 如果设置变量的时候，需要区别设置`global`/`seesion`的话
> `set [global/seesion] @@变量名 = Xx;`（`＠＠`表示系统变量，`＠`表示用户变量）
> 如果初始化／创建用户变量的时候没有对其进行赋值，直接进行查询并不会出现报错，会查询到`NULL`

> 在MySQL中`=`能表示等值，也能表示赋值
> 为了做区分，表示等值的时候可以用`:=`来表示等值

### 2. 存储过程核心语法与参数
存储过程支持三种参数类型：
- `IN`：输入参数（默认）。
- `OUT`：输出参数，该参数可以在存储过程内部被赋值，并返回给调用者。
- `INOUT`：既可以作为输入，也可以作为输出。

基本结构与控制流示例 ：

```SQL
-- 修改结束分隔符，避免与内部的 ; 冲突
DELIMITER $$

-- 创建存储过程
CREATE PROCEDURE p_calc_score(IN score INT, OUT result VARCHAR(10))
BEGIN
    -- IF 判断示例
    IF score >= 85 THEN
        SET result = '优秀';
    ELSEIF score >= 60 THEN
        SET result = '及格';
    ELSE
        SET result = '不及格';
    END IF;
END $$

DELIMITER ;  -- 重新改为";"

-- 调用存储过程
CALL p_calc_score(75, @res);
-- 查看输出结果
SELECT @res;
```
> 要在命令行中执行就需要使用到`delimiter`指令
> 如果是在console中执行 可以忽略

查看存储过程

```sql
-- 查看指定数据库的存储过程及状态信息
SELECT * FROM INFORMATION_SCHEMA.ROUTINES WHERE ROUTINE_SCHEMA = '数据库名字';

-- 查看某个存储过程的定义
SHOW CREATE PROCEDURE 存储过程名称;
```

删除存储过程

```sql
DROP PROCEDURE [IF EXISTS] 存储过程名称;
```

> [!info] 在存储过程中的常见逻辑关键词
> `if` 
> `case`(相当于Java中的switch) 
> `while`
> `repeat`（相当于和前面的`while`差不多）
> `loop`（其实也和while差不多，不过可以对某个变量进行特别操作）

### 3. 游标 (Cursor) 与 条件处理程序 (Handler)
- **游标**：用来存储查询**结果集**的数据类型，允许在存储过程中逐行遍历、处理这些数据。
- **条件处理程序**：类似于 Java 中的 `try-catch`，用于在发生错误或游标遍历结束（状态码 `02000`）时，执行特定的应对逻辑。

```SQL
-- 游标与 Handler 标准处理框架
DECLARE 变量1 数据类型;
-- 1. 声明游标
DECLARE u_cursor CURSOR FOR SELECT id FROM user;
-- 2. 声明条件处理程序 (当找不到数据时，将标记变量设为 1)
DECLARE EXIT HANDLER FOR NOT FOUND SET has_no_data = 1;

-- 3. 打开游标
OPEN u_cursor;

-- 4. 循环获取数据
WHILE has_no_data = 0 DO
    FETCH u_cursor INTO 变量1;
    -- 处理逻辑...
END WHILE;

-- 5. 关闭游标
CLOSE u_cursor;
```

> [!info] 也可以利用状态码来关闭游标
> ```sql
> -- 即为检测到出现 02000 状态码的时候会自己关闭游标
> DECLARE EXIT HANDLER FOR SQLSTATE '02000' CLOSE u_cursor;
> ```

### 4. 存储函数

存储函数是有返回值的存储过程，**它的参数只能是 `IN` 类型**，并且必须包含 `RETURN` 语句。

```SQL
CREATE FUNCTION fun_name(参数列表) RETURNS 返回类型
RETURN [结果类型] [characteristic]
BEGIN
    -- 逻辑代码
    RETURN 结果;
END;
```

> [!warning] 关于`characteristic`的说明
> 在高版本的MySQL中，会不允许`characteristic`为空
> - `characteristic`的值：
> 	- `DETERMINSTIC`: 相同的输入总是产生相同的输出
> 	- `NO SQL`: 不包含SQL语句
> 	- `READS SQL data`: 包含读取数据的语句，但不包含写入数据的语句

## 三、 触发器 (Trigger)

- **概念**：与表有关的数据库对象，在满足定义条件时（如对表进行 `INSERT`、`UPDATE`、`DELETE` 时）自动触发并执行内部的 SQL 集合。
- **核心作用**：数据完整性约束、日志记录（审计）、数据校验。

### 1. `NEW` 与 `OLD` 关键字

在触发器内部，可以使用这两个别名来获取变更前后的数据：

|**触发器事件**|**OLD 含义**|**NEW 含义**|
|---|---|---|
|`INSERT`|无效 (NULL)|将要 / 已经新增的数据|
|`UPDATE`|修改前的数据|修改后的新数据|
|`DELETE`|将要 / 已经删除的数据|无效 (NULL)|

### 2. 基本示例

```SQL
-- 需求：记录 user 表的数据变更日志
CREATE TRIGGER tb_user_insert_trigger
AFTER INSERT ON user FOR EACH ROW  -- 行级触发器
BEGIN
    -- 往日志表插入一条记录，使用 NEW 关键字获取刚刚插入的 user_id
    INSERT INTO user_logs(user_id, action, create_time) 
    VALUES (NEW.id, 'INSERT', NOW());
END;
```

## 💡 避坑与进阶指导

### 实际开发常见坑

1. **触发器导致的神坑**：在排查线上数据异常时，经常看代码逻辑完全没问题，最后才发现是某个被遗忘的触发器在后台默默地修改数据。**强烈建议业务逻辑不要过度依赖触发器**。
2. **存储过程的调试噩梦**：存储过程的调试极其困难，且消耗数据库服务器的 CPU 资源。目前主流架构推倡“胖应用，瘦数据库”，即数据库只做存储，逻辑统统放到后端应用（如 Java）中实现。
3. **视图的性能假象**：不要以为查视图就一定会变快。视图本质上依然是在查底层的表，如果底层表没有建好索引，或者视图嵌套层级过深，查询性能会灾难性下降。

- 面试题：视图和物理表有什么区别？视图能提升查询性能吗？（不能，视图仅封装逻辑，不存数据）
- 面试题：解释一下视图的 `CASCADED` 和 `LOCAL` 检查选项的区别？
- 面试题：什么是游标？在什么场景下会使用游标？
- 面试题：触发器中的 `NEW` 和 `OLD` 关键字分别代表什么？在 `DELETE` 触发器中能使用 `NEW` 吗？
- 面试题：为什么现在的互联网大厂开发规范中，通常禁止或不推荐使用存储过程和触发器？
