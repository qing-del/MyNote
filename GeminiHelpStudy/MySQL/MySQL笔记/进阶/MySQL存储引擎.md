## 概述

- **概念**：存储引擎就是存储数据、建立索引、更新/查询数据等技术的实现方式 。
- **核心逻辑**：存储引擎是基于表的，而不是基于库的。所以存储引擎也可被称为表类型。我们可以为一个数据库下的不同表指定不同的存储引擎。
- **体系结构**：MySQL体系结构主要分为四层：网络连接层、服务层、引擎层、存储层 。存储引擎处于**引擎层**，是MySQL的核心，支持可插拔。

> [!warning] 注意
> 在绝大多数业务场景下，直接使用 MySQL 默认的 InnoDB 引擎即可，切勿为了追求极端查询速度盲目切换引擎，这可能导致数据丢失或并发问题。

## 存储引擎基本操作

- **作用场景**：在建表时根据业务需求（如是否需要事务、是否高并发读写）指定数据存储和处理的底层实现。
- **执行逻辑**：若不特别指定，MySQL（5.5版本之后）默认采用 InnoDB 引擎。

#### 标准语法与示例

```SQL
-- 查看当前数据库支持的存储引擎
SHOW ENGINES;

-- 创建表时指定存储引擎
CREATE TABLE 表名(
    字段1 字段1类型 [COMMENT 字段1注释],
    ...
    字段n 字段n类型 [COMMENT 字段n注释]
) ENGINE = INNODB [COMMENT 表注释];
```

- **示例代码**

```SQL
-- 创建一张日志表，指定使用 MyISAM 引擎
CREATE TABLE user_log(
    id INT PRIMARY KEY AUTO_INCREMENT,
    log_info VARCHAR(255)
) ENGINE = MyISAM COMMENT '用户操作日志表';
```

## 核心存储引擎详解

### 1. InnoDB
- **作用场景**：要求保证数据完整性、一致性、支持高并发读写的核心业务表（如订单表、用户表、支付表） 。
- **概念简述**：MySQL 默认存储引擎，提供高可靠性和高性能。
- **核心特性**：
    - 支持**事务（ACID）**，保证数据安全。
    - 支持**行级锁（Row-level locking）**，提高并发访问性能。
    - 支持**外键约束（Foreign Key）**，保证数据的完整性和正确性。
- **底层文件**：`xxx.ibd` 文件。InnoDB 的每张表都会对应这样一个表空间文件，存储该表的表结构、数据和索引。

### 2. MyISAM

- **作用场景**：以读操作和插入操作为主，极少更新和删除，对事务完整性要求不高的场景（如早期博客系统的文章表、单纯的访问日志表） 。
- **概念简述**：MySQL 早期默认的存储引擎。
- **核心特性**：
    - 不支持事务，不支持外键。
    - 支持**表锁**，不支持行锁。
    - 访问速度快。
- **底层文件**：`.sdi`（存储表结构）、`.MYD`（存储数据）、`.MYI`（存储索引）。

### 3. Memory

- **作用场景**：临时表、字典表或数据量小且需极速访问的缓存层数据 。
- **概念简述**：表数据存储在内存中，受硬件问题或断电影响会导致数据丢失。
- **核心特性**：
    - 内存存放，访问极快。
    - 默认使用 Hash 索引。

## 存储引擎对比分析

|**特性 / 引擎**|**InnoDB**|**MyISAM**|**Memory**|
|---|---|---|---|
|**事务安全**|支持|不支持|不支持|
|**锁机制**|行锁（默认）、表锁|表锁|表锁|
|**外键约束**|支持|不支持|不支持|
|**存储介质**|磁盘|磁盘|内存|
|**适用场景**|绝大多数核心业务|纯读写、日志归档|临时数据、高速缓存|

## 💡 避坑与进阶指导

### 实际开发常见坑
1. **老旧项目迁移风险**：接手老旧项目时，如果发现表是 MyISAM 引擎，在做并发高频 `UPDATE` 操作时会遭遇严重的表锁等待（锁表），导致接口大规模超时。
2. **内存溢出风险**：切勿将业务主表设置为 Memory 引擎，一旦数据量激增会耗尽服务器内存，且服务重启后数据全部丢失。
3. **外键级联灾难**：虽然 InnoDB 支持外键，但在大型互联网工程化开发中，**强制要求禁止使用物理外键约束**（会严重影响插入/更新性能且极易产生死锁），外键的逻辑一致性应由 Java 代码（业务层）来保证。

### 性能注意点

1. InnoDB 的性能高度依赖于内存缓冲池（Buffer Pool）。在服务器配置 MySQL 时，通常会将 `innodb_buffer_pool_size` 设置为物理内存的 50% - 70%，以极大减少磁盘 I/O。
2. `COUNT(*)` 性能差异：MyISAM 内部维护了一个计数器，执行 `COUNT(*)` 极快；而 InnoDB 需要逐行扫描统计（因为 MVCC 版本控制），在海量数据下 InnoDB 的 `COUNT(*)` 性能较差，通常需要借助 Redis 等外部组件优化。
