### C代码的栈帧
- 假设我们有以下`c`代码
```c
void method(int a, int b) {
	int x = a + 1;
	int y = a + b;
}
```
- 栈帧运行图
```txt
┌─────────────┐ ← 高地址
│    var b    │  ← 参数从右到左入栈（cdecl约定），b 先入栈，地址更高
├─────────────┤
│    var a    │  ← a 后入栈
├─────────────┤
│   返回地址   │  ← call 指令自动 push，表示执行完后跳回的位置
├─────────────┤
│  saved rbp  │  ← callee 保存调用方的帧指针（简化图可忽略）
└─────────────┘ ← 低地址（ESP/SP 当前指向此处，即栈顶）

执行 int x = a + 1（忽略寄存器，假设直接压栈）
	 ⬇
	 
┌─────────────┐ ← 高地址
│    var b    │
├─────────────┤
│    var a    │
├─────────────┤
│   返回地址   │
├─────────────┤
│  saved rbp  │
├─────────────┤
│    var x    │ ← 新分配的局部变量，ESP 下移后写入
└─────────────┘ ← 低地址（栈顶 ESP 指向此处）
```
> [!info] 说明
> - CPU 原生栈：**运行到对应代码行才会动态压栈**，局部变量随用随分配
> - 栈向低地址方向增长，ESP（栈顶指针）始终指向当前已使用的最低地址

### JVM栈帧
- 假设有以下代码
```java
public void method(int a, int b) {
	int x = a + 1;
	int y = a + b;
}
```
- 栈帧运行图
```txt
调用方栈帧（caller frame，简化）：
┌─────────────┐ ← 高地址
│  ...其他...  │
└─────────────┘

即将进入 method 方法，JVM 为其分配新的栈帧
	  ⬇

method 的 JVM 栈帧（局部变量表 Local Variable Table）：
┌─────────────┐
│  slot 0     │  this（实例方法隐含的第一个参数，静态方法无此项）
├─────────────┤
│  slot 1     │  var a
├─────────────┤
│  slot 2     │  var b
├─────────────┤
│  slot 3     │  var x（尚未赋值时slot已存在，值为默认值）
├─────────────┤
│  slot 4     │  var y
└─────────────┘
+ 操作数栈（Operand Stack）用于中间计算
+ 返回地址 / 动态链接 等其他栈帧组成部分
```
> [!info] 解释
> - JVM 模拟了一套比 CPU 原生栈更简单、统一的抽象架构
> - JVM 在加载 `.class` 文件时，从字节码的 `Code` 属性中读取 `max_locals`（局部变量表大小）和 `max_stack`（操作数栈深度），**在方法调用时就一次性预分配好所有 slot 的空间**
> - 不像 CPU 原生栈那样随代码执行动态增长，JVM 局部变量表大小在编译期已完全确定
> - 等执行到对应赋值语句时，再往对应 slot 填入实际的值
> - 调用链路：`Java 代码 → JVM 字节码解释/JIT编译 → CPU 原生栈帧`
