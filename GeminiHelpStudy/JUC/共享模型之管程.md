## 线程安全问题引入

### 小故事引入
- 想象两个人合用一个银行账户（**共享资源**），各自同时去 ATM 取钱
- 如果两人同时读到"余额 1000"，然后各取 500，账户扣了两次 500，余额可能变成 0 或 500，取决于操作的交叉顺序
- 在 CS 术语中，"两个人"就是两个**线程**，"银行账户"就是**共享变量**，"同时操作"产生了**竞态条件（Race Condition）**

```java
// 经典的线程安全问题
static int counter = 0;

public static void main(String[] args) throws InterruptedException {
    Thread t1 = new Thread(() -> {
        for (int i = 0; i < 5000; i++) counter++;
    }, "t1");

    Thread t2 = new Thread(() -> {
        for (int i = 0; i < 5000; i++) counter--;
    }, "t2");

    t1.start(); t2.start();
    t1.join();  t2.join();

    // 期望结果: 0，实际结果: 不确定（可能为正、为负、或为0）
    System.out.println("counter = " + counter);
}
```

> [!warning] 为什么结果不确定？
> `counter++` 在 **JVM 字节码**层面不是原子操作，实际分为三步：
> 1. `getstatic` — 从主内存中读取 counter 的值到操作数栈
> 2. `iadd` — 在操作数栈中执行 +1
> 3. `putstatic` — 将结果写回主内存
>
> 如果 t1 执行完第 1 步（读到 0），但还没来得及写回，此时 **CPU 发生上下文切换**，t2 也读到 0 并 -1 写回 -1；然后 t1 恢复执行，将 0+1=1 写回 → 两次操作的效果只保留了一次

### 临界区与竞态条件

- **临界区（Critical Section）**：访问**共享资源**的那段代码
  > 类比：ATM 机的操作界面就是临界区，一次只能一个人在上面操作
  > CS 术语：一段对共享资源进行读写操作的代码区域，需要互斥访问（Mutual Exclusion）

- **竞态条件（Race Condition）**：多个线程在临界区内执行，由于代码的执行序列不同而导致结果不可预期
  > 类比：两个人同时伸手拿最后一个苹果，谁先拿到取决于手速（调度顺序）
  > CS 术语：多线程对共享资源的非原子操作交叉执行，导致最终状态依赖于实际调度顺序

> [!info] OS 知识扩展：上下文切换触发点
> 操作系统的调度器在以下时刻可能进行线程切换：
> 1. **时间片耗尽**（时钟中断 — Timer Interrupt）
> 2. **线程主动阻塞**（IO/sleep/wait）
> 3. **更高优先级线程就绪**（抢占式调度 — Preemptive Scheduling）
>
> 竞态条件正是因为上下文切换可以发生在**任意两条字节码指令之间**，而 `counter++` 需要 3 条指令才能完成

## synchronized 解决方案

### synchronized 的基本使用

- `synchronized` 关键字用于给临界区加**互斥锁**（也叫**管程锁** / **Monitor Lock**）
- 同一时刻，只有持有锁的线程才能进入同步代码块

```java
static int counter = 0;
static final Object lock = new Object();  // 锁对象

Thread t1 = new Thread(() -> {
    for (int i = 0; i < 5000; i++) {
        synchronized (lock) {  // 获取锁
            counter++;
        }  // 释放锁
    }
});

Thread t2 = new Thread(() -> {
    for (int i = 0; i < 5000; i++) {
        synchronized (lock) {
            counter--;
        }
    }
});
// 现在 counter 一定等于 0
```

> [!tip] 理解 synchronized 的关键
> - `synchronized(lock)` 中的 `lock` 是一个**Java 对象**，它充当"钥匙"，所有线程竞争同一把钥匙
> - **必须用同一个锁对象**：如果 t1 锁 `lockA`、t2 锁 `lockB`，那等于没锁
> - 锁的粒度应尽可能小，只保护真正的临界区代码

> [!question] 思考
> 如果把 `synchronized(lock)` 放到 `for` 循环外面（即整个 for 循环是一个大同步块），虽然正确，但会让两个线程完全串行执行，失去并发的意义，性能极差。

### synchronized 加在方法上

```java
class Account {
    private int balance;

    // 等价于 synchronized(this) { ... }
    public synchronized void withdraw(int amount) {
        balance -= amount;
    }

    // 等价于 synchronized(Account.class) { ... }
    public static synchronized void staticMethod() {
        // 锁的是 Class 对象，所有实例共享同一把锁
    }
}
```

| 写法 | 等价形式 | 锁对象 |
|------|---------|--------|
| `synchronized void method()` | `synchronized(this)` | 当前实例对象 |
| `static synchronized void method()` | `synchronized(ClassName.class)` | 类的 Class 对象 |

> [!failure] 经典陷阱
> 如果两个不同的 Account 实例分别调用 `withdraw()`，**不互斥**！因为各自锁的是各自的 `this`
> ```java
> Account a = new Account(1000);
> Account b = new Account(1000);
> // a.withdraw() 和 b.withdraw() 锁的是不同对象，不互斥！
> ```

### 锁如何生效的理解
- 我们可以思考两个问题：
	- 线程A：`synchronized (lock1)`，线程B：`synchronized (lock2)`
	- 线程A：`synchronized (lock1)`，线程B：不加锁
- 对于**第一个问题**：
	- 其实你可以当作是A从前门进入算数房，并锁住了前门，然后A时间片结束了，但是A还没“解锁前门”（释放锁）
	- 结果B从后门进入了并锁住了后门，并开始操作
	- 最终导致线程安全问题
- 对于**第二个问题**：
	- 其实你可以当作是A从前门进入算数房，并锁住了前门，然后A时间片结束了，但是A还没“解锁前门”（释放锁）
	- 结果B从后门进入了，但是没上锁，然后开始操作
	- 最终导致线程安全问题

> [!info] 真正锁住的位置
> 其实真正锁住的是别人获取锁这个过程
> 并不是真的把那段内存锁住不让别人操作
> 如果别人不获取锁，直接就可以进来操作，那么锁就无法起到应有的作用

### 线程安全分析

#### 局部变量是线程安全的
- 局部变量存在于**线程私有的栈帧**中，每个线程都有自己的一份副本
- 不存在共享，天然线程安全

> [!info] JVM 知识扩展
> 在 JVM 运行时数据区中：
> - **堆（Heap）**：所有线程共享 → 存放对象实例 → **需要考虑线程安全**
> - **虚拟机栈（VM Stack）**：线程私有 → 存放栈帧（局部变量表等）→ **天然安全**
> - **方法区（Method Area）/ 元空间（Metaspace）**：共享 → 存放类信息、静态变量 → **需要考虑安全**

#### 局部变量引用的对象不一定安全
```java
public void method() {
    List<String> list = new ArrayList<>();  // 局部变量，引用安全
    // 但如果将 list 传给其他线程，另一个线程也持有了这个引用
    // 那 ArrayList（非线程安全集合）就不安全了
}
```

#### 暴露引用
- 如果一个 `private` 方法被子类 `override` 时暴露了原本的局部变量引用（如传给新线程），就可能破坏线程安全性
- 这也是为什么**不可变设计**和 **`final class`** 在并发场景中很重要

#### 常见线程安全类
- **线程安全类**：`StringBuffer`、`Hashtable`、`Vector`、`ConcurrentHashMap`、`AtomicInteger` 等
  > 每个方法是原子的，但**方法的组合调用不是原子的**
- **线程不安全类**：`StringBuilder`、`HashMap`、`ArrayList`、`SimpleDateFormat` 等

```java
// 单个方法调用是原子的
Hashtable table = new Hashtable();  // 假设这个是全局变量
table.put("key", "val");  // 线程安全

// 组合调用不是原子的！这里是不同线程对全局变量进行操作
if (table.get("key") == null) {    // 第一步
    table.put("key", "value");     // 第二步
}
// get 和 put 之间可能被其他线程插入
```

> [!warning] 不可变类的线程安全
> `String`、`Integer` 等**不可变类**是线程安全的，因为它们的状态一旦创建就不能被修改
> 看似修改的操作（如 `"abc" + "def"`）实际上创建了一个新的 String 对象

#### 进一步理解线程安全
- **内存角度理解**
	- 其实再说细节一点，线程安全问题就是考虑到会不会有同一块内存同时被不同线程操作
	- 在`String`类的源码中，它的成员变量是被`final`修饰，每次修改都会开辟`new String()`，相当于不在同一块内存区
	- 在`Date`类的源码中，它的成员变量并没有`final`修饰，所以每次修改都会改动他的成员变量值（对内存里的值进行修改），所以尽管是`final`修饰`Date`类，依旧是线程不安全的

> [!tip] 外星方法
> 如果说一个类中，有一个无`final`修饰的成员变量，在同一个线程中只被一个方法调用（抽象方法、可重写方法）
> > 看似线程安全了：“只在一个线程中调用这个方法啊，没问题啊！”
> 
> 实际上如果这个方法重写之后，如果使其出现多个并发线程来使用这个被继承且无`final`修饰的成员变量
> 就会导致线程安全问题，此现象成为==**外星方法**==

- **组合问题**
	- 关于两个加了同步锁的函数，为什么进行组合之后就导致线程不安全了呢？
	- 其实可以是将其看为下图：
```txt
┌───────────────────┐
│        get        │  -- 单个方法是原子的
└─────────-─────────┘
┌───────────────────┐
│       remove      │  -- 单个方法是原子的
└─────────-─────────┘

---

如果他们组合到一起的时候就会变成下图

 Thread_1                    Thread_2
┌───────────────────┐       ┌───────────────────┐
│        get        │	    │        get        │
└─────────-─────────┘		└─────────-─────────┘
													<--- 若是在此处出现了时间切片超出，就会导致线程安全，因为这样整个操作过程就不是原子的了
┌───────────────────┐       ┌───────────────────┐
│       remove      │	    │       remove      │
└─────────-─────────┘		└─────────-─────────┘
```
> 线程安全要是整个代码临界区都要是保证原子性才能是线程安全

## Monitor 底层原理

### OS底层前置补充
- 不管是什么操作，都其实只是在OS的三个线程队列中移来移去
- **三个队列**：
	- 就绪队列（Ready Queue）
	- 定时睡眠队列（(Timer Wait / Sleep)  Queue）
	- 阻塞/条件等待队列（(Block / Condition Wait) Queue）

##### 1）就绪队列（Ready Queue）
- 作用：所有能运行、只差CPU时间片的线程，都在这里排队。
- 状态：就绪（Ready）
- 什么时候进：
	- 线程刚创建
	-  sleep  时间到
	- 被  notify  唤醒 + 拿到锁后
	- 时间片用完被切下来
- 出队：被CPU选中，开始运行。
 
##### 2）定时睡眠队列（Timer Wait Queue / Sleep Queue）
- 作用：要等一段时间再运行的线程。
- 状态：定时等待（Sleep/ TimedWaiting）
- 谁会进：
	-  Thread.sleep(1000) 
	-  lock.wait(1000) （带超时的等待）
- 关键点：
	- 由系统时钟驱动，时间到自动弹回 就绪队列。
	- sleep 进的就是这个队列。
 
 
##### 3）阻塞/条件等待队列（Block Queue / Condition Wait Queue）
- 作用：线程在等某个事件/某个锁/某个条件，没到时永远不醒。
- 状态：阻塞（Blocked / Waiting）
	- 谁会进：
	- 抢  synchronized  锁抢不到
	- 调用  lock.wait() （无超时）
	- 等待 IO、等待信号量
- 唤醒方式：
> 必须由别人主动唤醒（释放锁、notify、IO完成），自己醒不了。
 
#### 二、Java 里的 monitor 队列（EntryList、WaitSet）是什么？
 
它们不是 OS 原生队列，是 JVM 自己用链表维护的“软件队列”，
最终还是会把线程丢进上面那 3 个 OS 队列里。
 
- 1）EntryList（monitor 入口集）
	- 作用：想抢锁、但没抢到的线程在这排队。
	- 对应 OS 队列：阻塞队列（Block Queue）
 
- 2）WaitSet（monitor 等待集）
	- 作用：调用了  wait() 、主动释放锁的线程在这排队。
	- 对应 OS 队列：条件等待队列 / 定时睡眠队列
 
#### 三、sleep / wait / 抢锁 分别进哪个队列？
 
1. Thread.sleep(1000)
→ 进 OS 定时睡眠队列
→ 不释放锁
→ 时间到 → 回到 就绪队列
​
2. synchronized 抢锁失败
→ 进 JVM EntryList
→ 对应 OS 阻塞队列
→ 锁释放 → 回到 就绪队列
​
3. lock.wait()
→ 释放锁
→ 进 JVM WaitSet
→ 对应 OS 条件等待队列
→ 被 notify → 移到 EntryList → 重新抢锁

4. lock.wait(1000)
→ 释放锁
→ 进 WaitSet
→ 对应 OS 定时睡眠队列
→ 时间到/notify → 移到 EntryList

#### 四、终极极简总结（OS 层面只有 3 个）
1. 就绪队列：能跑就等CP
2. 睡眠队列：定时到自动醒（sleep 用）
3. 阻塞队列：必须别人唤醒（抢锁、wait 用）
Java 所有线程状态、锁、wait/sleep，底层全都是在这 3 个 OS 队列里移来移去。



### Java 对象头（Object Header）

每个 Java 对象在内存中都有一个**对象头**，其中最关键的部分是 **Mark Word**：

```
普通对象（64位 JVM）:
┌──────────────────────────────────────────────┬────────────┐
│                  Mark Word (64 bits)         │ Klass Word │
│ [hashcode:31 | age:4 | biased:1 | lock:2 ]   │  (指向类)   │
└──────────────────────────────────────────────┴────────────┘
```

Mark Word 的最后 2 位（lock flag）标识当前锁状态：

| Mark Word 内容 | 锁标志位 | 锁状态 |
|---------------|---------|--------|
| hashcode / age / 0 / 01 | `01` | 无锁（Normal） |
| 线程ID / epoch / age / 1 / 01 | `01` | 偏向锁（Biased） |
| 指向 Lock Record 的指针 / 00 | `00` | 轻量级锁 |
| 指向 Monitor 的指针 / 10 | `10` | 重量级锁 |
| 空 / 11 | `11` | GC 标记 |

> [!info] JVM 知识扩展 —— Klass Pointer
> Mark Word 后面紧跟着 **Klass Word**（Klass Pointer），它指向对象所属的**类元数据**，JVM 通过它可以确定对象的类型
> 在 64 位 JVM 中开启**指针压缩（Compressed Oops）** 后，Klass Pointer 占 4 字节而非 8 字节

### Monitor 工作原理

- **Monitor**（管程/监视器）是操作系统底层的同步原语，Java 中每个对象都可以关联一个 Monitor
- 当 `synchronized` 升级为**重量级锁**时，Mark Word 中存储的就是指向 Monitor 的指针

```
Monitor 结构:
┌──────────────────────────────────────────┐
│            Monitor 对象                    │
│  ┌─────────┐                              │
│  │  Owner  │ ← 当前持有锁的线程            │
│  └─────────┘                              │
│  ┌──────────────┐                         │
│  │  EntryList   │ ← 等待获取锁的线程队列    │
│  │  [t2] [t3]   │    （BLOCKED 状态）       │
│  └──────────────┘                         │
│  ┌──────────────┐                         │
│  │  WaitSet     │ ← 调用 wait() 的线程队列  │
│  │  [t4]        │    （WAITING 状态）       │
│  └──────────────┘                         │
└──────────────────────────────────────────┘
```

**流程说明**：
1. 线程 t1 进入 `synchronized` 块，成为 Monitor 的 **Owner**
2. 其他线程（t2, t3）试图进入同一个 `synchronized` 块时发现 Owner 已被占用 → 进入 **EntryList** 等待 → 进入 `BLOCKED` 状态
3. 当 t1 执行 `wait()` 后，主动释放锁，进入 **WaitSet** → 进入 `WAITING` 状态
4. t1 退出 `synchronized` 块或调用 `wait()`，Owner 为空 → EntryList 中的线程竞争锁

> [!tip] 类比理解
> 把 Monitor 想象成一个**会议室管理系统**：
> - **Owner** = 正在使用会议室的人
> - **EntryList** = 在门口排队等着用会议室的人（等锁，BLOCKED）
> - **WaitSet** = 已经进了会议室、但主动说"我先不开，等某个条件满足再叫我"的人（等条件，WAITING）
>
> 用 CS 术语说：Monitor 维护了 **互斥（Mutual Exclusion）** 和 **同步（Synchronization）** 两种语义，EntryList 实现互斥等待，WaitSet 实现条件等待

### 从字节码角度看 synchronized

```java
synchronized (lock) {
    counter++;
}
```

对应字节码（简化版）：
```
monitorenter         // 获取 lock 对象的 Monitor，成功则进入临界区
  getstatic counter
  astore_1           // 将锁对象存到 slot_1
  iconst_1
  iadd
  aload_1            // 从 slot_1 中获取锁对象
  putstatic counter
monitorexit          // 释放 Monitor
goto <end>           // 没有出现异常就会直接跳转到结束（返回语句）
  astore_2           // 将 e 存到 slot_2 中
  aload_1            // 从 slot_1 中获取锁对象
monitorexit          // 异常路径的释放（确保异常时也能释放锁）
  aload_2            // 从 slot_2 中获取 e
  athrow             // throw e
return
```

> [!info] 细节
> 编译器会自动生成**两个 `monitorexit`**：一个用于正常退出，一个用于异常退出
> 这就是为什么 `synchronized` 不需要手动在 `finally` 中释放锁 —— 由 JVM 保证

## synchronized 锁优化

> [!info] JVM 知识扩展 —— 锁升级路径
> Java 6 之后，JVM 对 `synchronized` 做了大量优化，引入了**锁升级**机制：
> `无锁 → 偏向锁 → 轻量级锁 → 重量级锁`
> 锁**只能升级、不能降级**（也有争议说重量级锁在某些情况会降级，但一般认为不降级）
> 这样做的目的是：大多数场景下并没有真正的多线程竞争，没必要一上来就用"最重的锁"

### 轻量级锁

**适用场景**：多个线程**交替**（而非同时竞争）进入同步块

**流程**（类比：用便利贴占位的自习室座位）：
1. 线程在自己的**栈帧**中创建一个 **Lock Record**（锁记录）
2. 尝试用 **CAS** 操作将对象 Mark Word 替换为指向 Lock Record 的指针
3. 如果 CAS 成功 → 获取轻量级锁成功，Mark Word 锁标志变为 `00`
4. 如果 CAS 失败 → 说明有竞争，进行**锁膨胀**

> [!tip] CAS（Compare And Swap）
> CAS 是一种无锁的硬件级原子操作：比较内存中的值是否 = 期望值，若是则替换成新值，否则失败
> 它是整个 JUC 无锁并发体系的基石（后续第六章会深入讲解）
> 在 CPU 层面，CAS 对应 `CMPXCHG` 指令（x86 架构）

```
轻量级锁获取过程（Lock Record 交换）:

线程栈帧:                      对象头:
┌─────────────┐              ┌─────────────────┐
│ Lock Record │              │    Mark Word     │
│ ┌─────────┐ │    CAS交换    │  [hashcode|01]   │
│ │存储原始  │ │ ←──────────→ │        ↓         │
│ │MarkWord │ │              │  [→LockRecord|00]│
│ └─────────┘ │              └─────────────────┘
└─────────────┘
```

```java
static Object lock = new Object();  // 它的对象头就是一把锁

public void method1() {
	synchronized (lock) {
		method2();
	}
}

public void method2() {
	synchronized (lock) {
		// 逻辑
	}
}
```

![[轻量级锁上锁过程.jpg]]

- **上锁**
	- 这里虽然也会是出现上锁失败的情况，但是在`method2`在上锁的时候，依旧会创建一个新的`Lock Record`
	- 然后获取`lock`指向的锁，然后去检查被`lock`指向的`Monitor`发现其锁地址指向了自己的线程
	- 这个时候method2创建的Lock Record中地址值（`obj preference`上面的）不会设置新的值，只会设置为`null`，同时记入为==**重入锁**==，使其重入锁的记录值`+1`
- **解锁**
	- 如果是值为`null`的解锁，并不是真的去修改`Monitor`，而是去修改重入锁的状态值使其`-1`
	- 最后值不为`null`时，就会尝试解除轻量级锁
		- 成功，正常CAS然后解锁
		- 失败，可能被其他线程修改为重量级锁，然后进入重量级锁的解锁流程

### 锁膨胀

- 当轻量级锁 CAS 失败（说明有**真正的竞争**），锁就会**膨胀**为重量级锁
- 此时 JVM 会为对象关联一个 **Monitor 对象**
- Mark Word 变为指向 Monitor 的指针，锁标志变为 `10`
- 竞争失败的线程进入 Monitor 的 **EntryList**，被操作系统挂起（**阻塞 = 用户态到内核态的切换**）

![[重量级锁过程.jpg]]

- **加锁**
	- 假设线程`t1`已经占着这把锁了，此时线程`t2`需要进行加锁，然后出现了*线程之间的锁竞争*
	- `t2`就会为锁对象的对象头申请`Monitor`，`Mark Word`变为指向`Monitor`的指针，锁标志变为 `10`
	> 此时原本存放`t1`指针的位置，会被直接*覆盖*掉，`Monitor`此时会存放`obj.hashCode`，同时执行`Monitor.owner = t1`
	- 然后`t2`将自己加入到`Monitor`的`EntryList`队列中（即为`BLOCKED`状态）
- **解锁**
	- `t1`线程通过`obj`的对象头地址找到原本的对象头，但是发现变成重量级锁了，就会通过对象头的`Monitor`地址找到`Monitor`
	- 执行`Monitor.owner = null`，唤醒在`EntryList`中的`t2`线程
	> 此时`t2`的`Lock Record`就废弃了，不会再使用
	- 现在`t2`执行完毕了，要解锁了

> [!warning] 提示
> `obj`的对象头指向`Monitor`之后，尽管是`Owner`、`EntryList`、`WaitSet`都为空值，`Monitor`依旧不会被释放
> 后续有人用`obj`上锁的话，没有轻量级锁，会直接进入重量级锁
> 直到`obj`被`GC`机制回收之后，才会释放掉`Monitor`

> [!failure] 锁膨胀的代价
> 重量级锁涉及操作系统层面的**互斥量（Mutex）**，线程的阻塞和唤醒需要**系统调用（syscall）**
> 这意味着**用户态 → 内核态**的切换，代价远大于用户态的 CAS 自旋
> 这也是 JVM 不一上来就用重量级锁的原因


### 自旋优化

- 在轻量级锁 CAS 失败 → 膨胀为重量级锁之前，JVM 会先让线程做**自旋**（忙等一小段时间）
- 自旋的本质：让 CPU 空转几个循环，赌对方很快就释放锁
- 如果自旋期间锁被释放了 → 获取成功，避免了进入重量级锁的代价
- 如果自旋超过阈值 → 还是得膨胀

> [!tip] 类比
> 就像你去洗手间发现有人在用，你可以**在门口站一会儿等着**（自旋），而不是回到座位上打电话叫人通知你（阻塞唤醒，代价大）
> 如果等了太久还没出来，你才选择回座位等通知
>
> CS 术语：**自旋锁（Spin Lock）** 适合**临界区很短**的场景；如果锁持有时间长，自旋就会浪费 CPU 资源

- Java 6 之后采用**自适应自旋（Adaptive Spinning）**：如果线程之前自旋成功过，JVM 会认为这次成功概率也高，允许更长的自旋时间；反之缩短
- Java 7 之后取消了自选的开关控制

### 偏向锁

**适用场景**：只有一个线程**反复进入**同一个同步块（无竞争）

- 偏向锁的核心思想：如果一把锁始终被同一个线程获取，那么连 CAS 都省掉
- 第一次获取时，用 CAS 把线程 ID 写入 Mark Word，之后再进入时只需检查线程 ID 是否相同
- 如果相同 → 直接进入（零额外开销）

```
偏向锁状态的 Mark Word:
┌──────────────────────────────────────────────┐
│ [thread_id:54 | epoch:2 | age:4 | 1 | 01]    │
│   偏向的线程ID   epoch   分代年龄 偏向标记 标志位 │
└──────────────────────────────────────────────┘
```

#### 偏向锁撤销的场景
1. **其他线程竞争**：发现 Mark Word 中的线程 ID 不是自己 → 偏向锁撤销 → 升级为轻量级锁
2. **调用 `hashCode()`**：对象的 hashCode 需要存在 Mark Word 中，但偏向锁状态下 Mark Word 被线程 ID 占据 → 撤销偏向锁
3. **调用 `wait()/notify()`**：需要 Monitor，偏向锁直接升级为重量级锁

#### 批量重偏向 & 批量撤销
- **批量重偏向**：如果一个类的大量对象都被同一个线程T1偏向后，又有另一个线程T2来获取它们，JVM 不会逐个撤销，而是到达阈值（默认20次）后批量将这些对象重偏向给 T2
- **批量撤销**：如果撤销次数到达更高阈值（默认40次），说明这个类的对象不适合偏向锁，JVM 会给整个类**禁用偏向锁**

> [!warning] JDK 15 之后的变化
> 从 JDK 15 开始，偏向锁被**默认关闭**（`-XX:-UseBiasedLocking`），因为现代应用大多是多线程竞争场景，偏向锁的维护成本（撤销需要 safepoint）反而成了负担

### 锁消除

- **JIT 编译器**在==运行时==分析代码，如果发现某个锁对象**不可能被其他线程访问**（逃逸分析），就会直接消除该锁
> 这里是程序启动之后，在热点（循环、反复执行）代码处，`JIT`会即时查看代码运行状况
> 如果`JIT`发现一个变量不会逃逸出局部范围，它就会将其重新进行优化

```java
public void method() {
    Object lock = new Object();  // 局部变量，不会逃逸（说明只在当前方法内使用）
    synchronized (lock) {         // JIT 会直接消除这个 synchronized
        // ...
    }
}
```

> [!info] JVM 知识扩展 —— 逃逸分析（Escape Analysis）
> JIT 编译器通过逃逸分析判断对象的作用域：
> - **不逃逸**：对象只在当前方法内使用 → 可以锁消除、栈上分配、标量替换
> - **方法逃逸**：对象作为方法返回值或参数传出
> - **线程逃逸**：对象被其他线程引用
>
> 锁消除针对的是"不逃逸"的情况——只有当前线程能看到这个锁对象，加锁毫无意义

## wait & notify 机制

### wait/notify 工作原理

- `wait()` 和 `notify()` 是 `Object` 类的方法（因为锁是对象层面的概念）
- **必须在 `synchronized` 块内部**调用（否则抛出 `IllegalMonitorStateException`）

```
wait/notify 的 Monitor 视角：

  1. Owner线程调用 lock.wait()
     ┌───────────┐        ┌──────────┐
     │  Owner=t1 │ ──→    │ WaitSet  │
     │           │  释放锁 │ [t1]     │
     └───────────┘        └──────────┘
                          t1 → WAITING

  2. 某个线程调用 lock.notify()
     ┌──────────┐         ┌───────────┐
     │ WaitSet  │ ──→     │ EntryList │
     │ [t1]     │ 唤醒一个 │ [t1]      │
     └──────────┘         └───────────┘
                          t1 → BLOCKED（等锁）

  3. t1 重新获取到锁
     ┌───────────┐
     │ Owner=t1  │
     │           │   t1 → RUNNABLE（从 wait() 返回）
     └───────────┘
```

**关键 API**：

| 方法 | 功能 | 备注 |
|------|------|------|
| `wait()` | 释放锁，进入 WaitSet 等待 | 无限期等待 |
| `wait(long timeout)` | 同上，但最多等 timeout 毫秒 | 超时后自动唤醒 |
| `notify()` | 随机唤醒 WaitSet 中的**一个**线程 | 不释放锁 -- 不会让持有锁的线程释放锁 |
| `notifyAll()` | 唤醒 WaitSet 中的**所有**线程 | 所有被唤醒的线程进入 EntryList 竞争锁 |

### wait vs sleep

| 对比项 | `wait()` | `sleep()` |
|--------|----------|-----------|
| 所属类 | `Object` | `Thread` |
| 是否释放锁 | ✅ **释放锁** | ❌ **不释放锁** |
| 使用条件 | 必须在 synchronized 内 | 任何地方 |
| 唤醒方式 | `notify()`、`notifyAll()`、`interrupt()` | 超时自动唤醒、`interrupt()` |
| 状态 | `WAITING` / `TIMED_WAITING` | `TIMED_WAITING` |
| 打断后标记 | 清除（`isInterrupted() = false`） | 清除（`isInterrupted() = false`） |

> [!warning] 最关键的区别
> **wait 释放锁，sleep 不释放锁！**
> 如果在 `synchronized` 块中调用 `Thread.sleep()`，其他线程无法进入该同步块（锁还被持有着）
> 而 `wait()` 会释放锁，让其他线程有机会进入

### wait/notify 的正确使用姿势

**错误写法**：
```java
synchronized (lock) {
    if (条件不满足) {
        lock.wait();  // 被唤醒后直接往下执行
    }
    // 执行业务...
}
```

**问题**：`notify()` 唤醒后，条件可能又不满足了（**虚假唤醒 — Spurious Wakeup**）

**正确写法**：
```java
synchronized (lock) {
    while (条件不满足) {   // 用 while 而不是 if！
        lock.wait();       // 被唤醒后重新检查条件
    }
    // 条件已满足，执行业务...
}
```

> [!tip] 关联思考
> `while` 而不是 `if` 是一个经典的并发编程范式，几乎所有 wait 场景都应该用 `while`：
> 1. 防止**虚假唤醒**（JVM 规范允许 wait 在没有 notify 的情况下返回）
> 2. 防止 `notifyAll()` 唤醒了多个线程，但条件只满足一个线程的需求

## 设计模式

### 保护性暂停（Guarded Suspension）

**核心思想**：一个线程等待另一个线程的结果。

> 类比：你在餐厅点了一杯咖啡，然后在座位上等着（wait），直到服务员把咖啡做好送到（notify + 设置结果）
> CS 术语：**同步阻塞式等待**，一个生产者线程产出结果，一个消费者线程阻塞等待该结果

```java
class GuardedObject {
    private Object response;

    // 获取结果（带超时）
    public synchronized Object get(long timeout) {
        long begin = System.currentTimeMillis();
        long passedTime = 0;
        while (response == null) {
            long waitTime = timeout - passedTime;
            if (waitTime <= 0) break;  // 超时退出
            try {
                this.wait(waitTime);   // 等待（释放锁）
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            passedTime = System.currentTimeMillis() - begin;
        }
        return response;
    }

    // 设置结果
    public synchronized void complete(Object response) {
        this.response = response;
        this.notifyAll();  // 唤醒等待的线程
    }
}
```

> [!info] join 的底层原理
> `Thread.join()` 的实现就是保护性暂停模式：
> ```java
> // Thread.join() 简化版源码
> public final synchronized void join(long millis) throws InterruptedException {
>     while (isAlive()) {
>         wait(millis);  // 主线程在 Thread 对象上 wait
>     }
>     // 当目标线程执行完毕后，JVM 会自动调用 notifyAll()
> }
> ```

#### 扩展：解耦等待与生产（Futures 信箱）
- 多对多的情况下，可以引入一个"中间邮局（Mailbox）"，按 ID 分配 `GuardedObject`
- 生产者和消费者不需要直接引用对方，通过 ID 从邮局中匹配

### 生产者消费者模式

**与保护性暂停的区别**：

| 对比 | 保护性暂停 | 生产者消费者 |
|------|-----------|------------|
| 耦合度 | 一对一绑定 | 通过消息队列解耦 |
| 同步性 | 同步等待结果 | 异步通信 |
| 中间结构 | 无 | **阻塞队列** |

```java
class MessageQueue {
    private LinkedList<Message> queue = new LinkedList<>();
    private int capacity;

    public MessageQueue(int capacity) {
        this.capacity = capacity;
    }

    // 消费者 — 取消息
    public synchronized Message take() {
        while (queue.isEmpty()) {
            try { this.wait(); } catch (InterruptedException e) { e.printStackTrace(); }
        }
        Message msg = queue.removeFirst();
        this.notifyAll();  // 通知生产者：队列有空位了
        return msg;
    }

    // 生产者 — 放消息
    public synchronized void put(Message msg) {
        while (queue.size() == capacity) {
            try { this.wait(); } catch (InterruptedException e) { e.printStackTrace(); }
        }
        queue.addLast(msg);
        this.notifyAll();  // 通知消费者：队列有消息了
    }
}
```

> [!tip] 工程映射
> 生产者消费者模式是消息中间件（RabbitMQ、Kafka、RocketMQ）的核心抽象：
> - 生产者 → 消息发布者
> - 阻塞队列 → Broker（消息代理服务器）中的 Queue / Topic
> - 消费者 → 消息订阅者
> 通过引入消息队列实现了**流量削峰**和**服务解耦**

## park & unpark

`LockSupport.park()` / `LockSupport.unpark(thread)` 是比 `wait/notify` 更底层、更灵活的线程暂停/恢复机制。

| 特性 | wait/notify | park/unpark |
|------|------------|-------------|
| 调用条件 | 需要在 synchronized 内 | **不需要获取锁** |
| 粒度 | notify 随机唤醒一个 | unpark 精确唤醒**指定线程** |
| 顺序 | 必须先 wait，后 notify | **可以先 unpark，后 park**（不会阻塞） |

```java
Thread t1 = new Thread(() -> {
    System.out.println("t1 即将 park...");
    LockSupport.park();            // 暂停当前线程
    System.out.println("t1 恢复运行");
});
t1.start();

Thread.sleep(500);
LockSupport.unpark(t1);          // 精确唤醒 t1
```

> [!info] park/unpark 的底层原理 — 许可证（Permit）机制
> 每个线程都有一个 **permit**（许可证），最多只有 1 个：
> - `unpark(t)` → 给线程 t 一张许可证（permit = 1）
> - `park()` → 消耗一张许可证（permit 1→0，不阻塞）；如果没有许可证（permit = 0）→ 阻塞
>
> 类比：把 permit 想象成**共享单车的骑车券**。`unpark` = 发一张券，`park` = 用掉一张券。如果已经有券，直接骑走不等待；如果没有券，就收到券之前必须等着
>
> 这也解释了为什么 unpark 可以在 park 之前调用——券先发好了放在那里，等 park 来了直接使用

## 线程状态转换详解

结合第三章的六种状态与本章的同步工具，完整的状态转换如下：

```
                       ┌──────────────────────────┐
                       │   NEW                     │
                       │  （调用 start()）           │
                       └─────────┬────────────────┘
                                 ↓
                       ┌─────────────────────┐
              ┌───────→│     RUNNABLE          │←────────────┐
              │        │  （正在运行或等待CPU）   │             │
              │        └─┬───┬───┬───┬───────┘             │
              │          │   │   │   │                      │
              │          │   │   │   └─── run() 结束 ──→ TERMINATED
              │          │   │   │
   获取到锁    │          │   │   └── wait()/join()/park()
              │          │   │        → WAITING
              │          │   │          (notify/unpark/interrupt → 回到 RUNNABLE)
              │          │   │
              │          │   └── sleep(n)/wait(n)/join(n)/parkNanos(n)
              │          │        → TIMED_WAITING
              │          │          (超时/唤醒/打断 → 回到 RUNNABLE)
              │          │
              │          └── 等待进入 synchronized 块
              │               → BLOCKED
              └────────────── (获取到锁 → 回到 RUNNABLE)
```

> [!question] BLOCKED vs WAITING 的本质区别
> - **BLOCKED**：线程想进入 synchronized 块，但**锁被其他线程持有** → 被动等待
> - **WAITING**：线程主动调用 `wait()`/`park()`/`join()` → 主动等待某个条件
>
> 从 OS 角度看，两者都是"不在运行"状态，但 Java 将它们细分为不同语义

## 活跃性问题

### 死锁

**定义**：两个或多个线程互相持有对方需要的锁，并且都在等待对方释放，导致永远阻塞。

> 类比：两个人面对面走在窄桥上，谁也不让谁，都在等对方先退（**循环等待**）
> CS 术语：满足**互斥、占有且等待、不可抢占、循环等待**四个条件时即产生死锁（Coffman 条件）

```java
Object lockA = new Object();
Object lockB = new Object();

Thread t1 = new Thread(() -> {
    synchronized (lockA) {     // t1 持有 lockA
        Thread.sleep(100);
        synchronized (lockB) { // t1 等待 lockB（被 t2 持有）
            // ...
        }
    }
});

Thread t2 = new Thread(() -> {
    synchronized (lockB) {     // t2 持有 lockB
        Thread.sleep(100);
        synchronized (lockA) { // t2 等待 lockA（被 t1 持有）
            // ...
        }
    }
});
// 死锁！t1 等 t2 释放 lockB，t2 等 t1 释放 lockA
```

> [!info] OS 知识 —— 死锁的四个必要条件
> 1. **互斥条件**（Mutual Exclusion）：资源一次只能被一个线程持有
> 2. **占有且等待**（Hold and Wait）：持有一个资源的线程等待获取其他资源
> 3. **不可抢占**（No Preemption）：已获取的资源只能由持有者主动释放
> 4. **循环等待**（Circular Wait）：线程之间形成等待环路
>
> 打破任一条件即可预防死锁。实践中最常用的方案：**按固定顺序加锁**（打破循环等待）

**定位死锁**：
- 使用 `jstack <PID>` 或 `jconsole` 检测死锁
- JVM 会在线程 dump 中直接报告 `Found one Java-level deadlock`

> [!info] 检查流程
> 先打开`Terminal`
> 先输入`jps`获取正在运行的所有进程（会有进程`id`）
> - **第一种**
> 	- 然后再通过`jstack <PID>`，`<PID>`是进程`id`来获取进程中所有线程的信息
> - **第二种**
> 	- 打开`jconsole`然后连接对应的进程，然后进去查看

#### 哲学家就餐问题
- 5 个哲学家围桌而坐，每人两侧各一支筷子（共 5 支），吃饭需要同时拿到左右两支
- 如果每人都先拿左边的筷子，然后等右边的 → 所有人都拿着一支等另一支 → **死锁**
- 解法之一：让编号最大的哲学家先拿右边的筷子，打破循环等待
- 另一种解法：使用 `ReentrantLock.tryLock()` + 超时机制（见下文）

### 活锁

**定义**：线程没有阻塞，但不断重试导致永远无法推进。

> 类比：两个人在走廊对面走来，互相让路但总是让到同一边，无限循环
> CS 术语：线程一直在**活跃地执行重试逻辑**，但无有效推进（Live Lock）

```java
// 活锁示例：两个线程互相修改对方的状态，导致无法退出
static volatile int count = 10;

Thread t1 = new Thread(() -> {
    while (count > 0) { count--; Thread.sleep(200); }
});
Thread t2 = new Thread(() -> {
    while (count < 20) { count++; Thread.sleep(200); }
});
// count 在 0~20 之间震荡，两个线程永远无法结束
```

**解决方案**：引入**随机等待时间**，打破同步节奏

### 饥饿

**定义**：某些线程始终无法获得执行所需的资源（锁、CPU 时间），一直被"饿着"。

> 类比：公司食堂每次都让领导先打饭，普通员工永远等不到（**优先级反转**）
> CS 术语：低优先级线程因高优先级线程持续获取资源而得不到调度（Starvation）

- **常见原因**：
  - 使用不公平锁（`synchronized` 就是不公平的），某些线程运气差总是抢不到
  - 线程优先级设置不合理
- **解决方案**：使用**公平锁**（如 `ReentrantLock(true)`），按先到先得顺序分配锁

## ReentrantLock

`ReentrantLock`（可重入锁）是 `java.util.concurrent.locks` 包提供的锁实现，是 `synchronized` 的增强替代方案。

```java
ReentrantLock lock = new ReentrantLock();
lock.lock();       // 获取锁
try {
    // 临界区
} finally {
    lock.unlock();  // 必须在 finally 中释放！
}
```

> [!warning] 与 synchronized 的核心区别
> `synchronized` 由 JVM 自动释放锁（即使发生异常），而 `ReentrantLock` **必须手动在 finally 中调用 `unlock()`**
> 如果没写 finally → 异常时锁不释放 → 其他线程永远无法获取锁

### 可重入

**可重入（Reentrant）**：同一个线程可以多次获取同一把锁而不会死锁。

```java
lock.lock();
try {
    lock.lock();   // 同一线程再次获取，不会死锁（重入计数器 +1）
    try {
        // ...
    } finally {
        lock.unlock(); // 重入计数器 -1
    }
} finally {
    lock.unlock();    // 重入计数器归 0，真正释放锁
}
```

> [!info] `synchronized` 也是可重入的
> ```java
> synchronized (lock) {
>     synchronized (lock) {  // 同一线程再次进入，不会死锁
>         // ...
>     }
> }
> ```

### 可打断

- `lock.lockInterruptibly()` 可以在等待锁的过程中被 `interrupt()` 打断
- 普通的 `lock.lock()` 不可被打断（即使 interrupt 也会继续等待）

```java
Thread t = new Thread(() -> {
    try {
        lock.lockInterruptibly();  // 可被打断的获取锁
        try {
            // 临界区
        } finally {
            lock.unlock();
        }
    } catch (InterruptedException e) {
        System.out.println("等锁的过程中被打断，放弃获取");
    }
});
```

> [!tip] 实际价值
> 可打断锁可以**防止线程无限期等待**，在需要快速取消任务的场景（如超时取消远程调用）特别有用

### 锁超时

- `tryLock()` — 尝试获取锁，成功返回 `true`，失败**立即**返回 `false`（不阻塞）
- `tryLock(long time, TimeUnit unit)` — 尝试获取锁，等待指定时长，超时返回 `false`

```java
if (lock.tryLock(1, TimeUnit.SECONDS)) {  // 最多等 1 秒
    try {
        // 获取成功，执行业务
    } finally {
        lock.unlock();
    }
} else {
    System.out.println("获取锁超时，放弃操作");
}
```

**解决哲学家就餐问题**：
```java
// 每个哲学家使用 tryLock 尝试获取两支筷子
if (left.tryLock()) {
    try {
        if (right.tryLock()) {
            try {
                // 吃饭
            } finally { right.unlock(); }
        }
    } finally { left.unlock(); }  // 没拿到右边的筷子，放下左边的
}
// tryLock 失败就放弃，不会死锁
```

### 公平锁

```java
// 默认不公平锁
ReentrantLock unfairLock = new ReentrantLock();        // fair = false

// 公平锁：按照等待顺序获取锁（FIFO）
ReentrantLock fairLock = new ReentrantLock(true);      // fair = true
```

| 类型 | 吞吐量 | 饥饿问题 | 实现 |
|------|--------|---------|------|
| 不公平锁 | 高（减少上下文切换） | 可能饥饿 | `synchronized`、`new ReentrantLock()` |
| 公平锁 | 相对低 | 不会饥饿 | `new ReentrantLock(true)` |

> [!info] 为什么不公平锁吞吐量更高？
> 当一个线程释放锁时，如果恰好有一个新线程来获取锁，不公平锁允许它**直接抢**（避免唤醒等待队列中的线程），减少了一次线程上下文切换
> 公平锁必须唤醒队列中等待最久的线程，多了一次阻塞→唤醒的开销

### 条件变量

- `synchronized` 只有一个 WaitSet（所有 `wait()` 的线程都在一起）
- `ReentrantLock` 可以创建**多个 `Condition`**，每个 Condition 有自己的等待队列

```java
ReentrantLock lock = new ReentrantLock();
Condition waitCigarette = lock.newCondition();  // 等烟的条件
Condition waitBreakfast = lock.newCondition();  // 等早餐的条件

// 等烟的线程
lock.lock();
try {
    while (!hasCigarette) {
        waitCigarette.await();      // 进入"等烟"队列（释放锁）
    }
    // 拿到烟，开始工作...
} finally { lock.unlock(); }

// 送烟的线程
lock.lock();
try {
    hasCigarette = true;
    waitCigarette.signal();         // 只唤醒"等烟"队列中的线程
} finally { lock.unlock(); }
```

> [!tip] 精细化唤醒
> `synchronized` 的 `notifyAll()` 会唤醒所有 wait 的线程，即使有些线程等待的是不同的条件
> Condition 可以做到**精确唤醒**：只唤醒等待特定条件的线程，减少不必要的竞争和上下文切换

## 同步模式应用

### 固定运行顺序

**需求**：保证线程A的某段代码先于线程B执行

**方案1**：wait & notify
```java
static final Object lock = new Object();
static boolean t1Finished = false;

// t1 先执行
Thread t1 = new Thread(() -> {
    synchronized (lock) {
        System.out.println("1");
        t1Finished = true;
        lock.notify();
    }
});

// t2 等 t1
Thread t2 = new Thread(() -> {
    synchronized (lock) {
        while (!t1Finished) {
            try { lock.wait(); } catch (InterruptedException e) {}
        }
        System.out.println("2");
    }
});
t2.start(); t1.start();  // 无论谁先 start，都保证输出 1 2
```

**方案2**：park & unpark（更简洁）
```java
Thread t2 = new Thread(() -> {
    LockSupport.park();   // t2 先 park 等着
    System.out.println("2");
});
t2.start();

Thread t1 = new Thread(() -> {
    System.out.println("1");
    LockSupport.unpark(t2);  // t1 执行完后唤醒 t2
});
t1.start();
```

### 交替输出

**需求**：多个线程交替打印 A B C，循环 N 次

**方案1**：wait & notify（轮询标记）
```java
class WaitNotifyPrinter {
    private int flag;  // 0=A, 1=B, 2=C
    private int loopCount;

    public synchronized void print(String str, int waitFlag, int nextFlag) {
        for (int i = 0; i < loopCount; i++) {
            while (flag != waitFlag) {
                try { this.wait(); } catch (InterruptedException e) {}
            }
            System.out.print(str);
            flag = nextFlag;
            this.notifyAll();
        }
    }
}
// 线程A: print("A", 0, 1) — 当 flag=0 时打印A，然后 flag→1
// 线程B: print("B", 1, 2) — 当 flag=1 时打印B，然后 flag→2
// 线程C: print("C", 2, 0) — 当 flag=2 时打印C，然后 flag→0
```

**方案2**：Condition（精确唤醒，效率更高）
```java
ReentrantLock lock = new ReentrantLock();
Condition condA = lock.newCondition();
Condition condB = lock.newCondition();
Condition condC = lock.newCondition();

// 线程A：await 在 condA，打印后 signal condB
// 线程B：await 在 condB，打印后 signal condC
// 线程C：await 在 condC，打印后 signal condA
```

> [!tip] 方案对比
> `notifyAll()` 会唤醒所有等待线程，只有一个满足条件，其余做无用功
> `Condition.signal()` 精确唤醒下一个该执行的线程，避免浪费

**方案3**：park & unpark（最简洁）
```java
// 每个线程打印后 unpark 下一个线程，自己 park
static Thread ta, tb, tc;

ta = new Thread(() -> {
    for (int i = 0; i < 5; i++) {
        LockSupport.park();
        System.out.print("A");
        LockSupport.unpark(tb);
    }
});
// tb, tc 类似...
```

## 第四章小结

| 概念 | 核心要点 |
|------|---------|
| 临界区 & 竞态条件 | 多线程访问共享资源时需要互斥保护 |
| synchronized | 对象级锁，自动获取释放，不可中断 |
| Monitor 结构 | Owner + EntryList + WaitSet |
| 锁升级 | 无锁 → 偏向锁 → 轻量级锁 → 重量级锁 |
| wait/notify | 条件等待，必须在 synchronized 内，wait 释放锁 |
| park/unpark | 线程暂停/恢复，permit 机制，可先 unpark |
| 活跃性 | 死锁、活锁、饥饿 |
| ReentrantLock | 可重入、可打断、可超时、公平锁、多条件变量 |

> [!tip] synchronized vs ReentrantLock 选型指南
> - 简单场景用 `synchronized`（写法简单，JVM 自动优化）
> - 需要**超时获取锁、可打断、公平锁、多条件变量**时用 `ReentrantLock`
> - `tryLock` 解决死锁问题是 `ReentrantLock` 的杀手级特性
