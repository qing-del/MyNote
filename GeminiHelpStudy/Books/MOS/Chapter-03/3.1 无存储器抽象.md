## 最早的内存模型

在最早期的计算机中，**没有任何内存抽象**——程序直接使用物理内存地址。

```
早期内存布局模型（三种历史方案）:

方案1: OS在RAM底部              方案2: OS在ROM顶部             方案3: 驱动在ROM，OS在RAM
┌──────────────┐ 0xFFFF    ┌──────────────┐ 0xFFFF     ┌──────────────┐ 0xFFFF
│              │           │  操作系统     │            │  设备驱动ROM │
│   用户程序    │           │  (ROM)       │            ├──────────────┤
│              │           ├──────────────┤            │              │
├──────────────┤           │              │            │   用户程序    │
│  操作系统     │           │   用户程序    │            │              │
│  (RAM)       │           │              │            ├──────────────┤
└──────────────┘ 0x0000    └──────────────┘ 0x0000     │  操作系统RAM │
                                                       └──────────────┘ 0x0000
```

> [!info] 关键特征
> - 每个程序直接操作**物理地址**（Physical Address）
> - 同一时刻只能运行**一个程序**（加上操作系统）
> - 程序可以读写任何内存地址 → **没有任何保护机制**

> 类比：这就像一间只有一张大桌子的办公室，桌子上没有任何隔板。一次只能有一个人在桌子上工作，而且他可以随意碰到桌子上的任何东西——包括操作系统的"文件"
> CS 术语：这种模型中程序使用的是**裸机物理地址（Raw Physical Address）**，没有**地址空间隔离（Address Space Isolation）**

## 多程序运行的困境

当试图在没有内存抽象的情况下运行多个程序时，问题立刻显现：

```
问题：两个程序都使用绝对物理地址

程序A（加载在地址 0）:       程序B（加载在地址 16384）:
┌─────────────────────┐     ┌─────────────────────┐
│ JMP 28              │     │ JMP 28              │  ← 程序B也跳转到28
│ ...                 │     │ ...                 │     但28是程序A的区域！
│ ADD                 │     │ ADD                 │
└─────────────────────┘     └─────────────────────┘
```

> [!failure] 重定位问题
> 程序在编译时已经确定了物理地址。当程序B被加载到物理地址 16384 处时，它内部的 `JMP 28` 指令仍然跳转到地址 28（属于程序A的空间）而不是 16384+28 = 16412
> 这就是**静态重定位（Static Relocation）** 问题

**IBM 360 的解决方案**：在程序加载时，由加载器扫描整个程序，把所有地址引用加上偏移量。

```
静态重定位过程：

加载前（程序编译后）:    加载到地址 16384 后:
JMP 28          →       JMP 16412      (+16384)
LOAD 100        →       LOAD 16484     (+16384)
STORE 200       →       STORE 16584    (+16384)
```

> [!warning] 静态重定位的缺陷
> 1. **加载速度慢**：需要遍历整个程序修改所有地址引用
> 2. **无法区分地址和常量**：如何区分 `MOVE REGISTER1, 28` 中的 28 是地址还是数值 28？
> 3. **无保护机制**：程序仍然可以访问任意物理地址，一个坏程序可以破坏操作系统

## 💡 架构师视角映射

| 操作系统概念 | Java 后端映射 |
|------------|-------------|
| 无内存抽象 → 程序直接操作物理地址 | 早期 Servlet 直接操作 Socket → 没有框架抽象，编码复杂且不安全 |
| 静态重定位 | Java 类加载器的重定位：`.class` 文件中的符号引用在加载时被解析为直接引用（Resolution） |
| 无地址保护 | 无权限控制的 API → 任何调用者都能访问任何资源 → 需要引入 Spring Security 等安全框架 |

> [!tip] JVM 的对比
> JVM 的内存模型就是对"裸机物理地址"的一层抽象——程序使用的是 JVM 虚拟地址（堆中的对象引用），而不是物理内存地址。Java 程序永远不会直接操作物理内存（除非用 `Unsafe` 类）
