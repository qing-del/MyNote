## 文件命名

文件是操作系统对磁盘数据的**抽象**——用户不需要关心数据存在哪个磁盘扇区，只需要通过**文件名**来访问。

> 类比：文件系统就像一个图书馆的管理系统。你不需要知道某本书在仓库的第几个架子第几层（磁盘扇区），你只需要知道书名和分类号（文件名和路径），图书管理员（文件系统）会帮你找到它
> CS 术语：文件是对**持久化存储（Persistent Storage）** 的抽象，提供了**命名（Naming）** 和**定位（Location Transparency）** 的能力

```
文件命名规则（因系统而异）:

┌──────────────┬───────────┬───────────────────────┐
│    系统       │ 大小写    │ 文件名长度限制          │
├──────────────┼───────────┼───────────────────────┤
│ UNIX/Linux   │ 区分大小写 │ 255 字符               │
│ Windows      │ 不区分     │ 255 字符（保留大小写）   │
│ FAT (旧DOS)  │ 不区分     │ 8.3 格式（8+3字符）    │
└──────────────┴───────────┴───────────────────────┘

文件扩展名:
┌──────────┬───────────────────────┐
│ 扩展名    │ 含义                   │
├──────────┼───────────────────────┤
│ .c       │ C 语言源代码            │
│ .java    │ Java 源代码             │
│ .class   │ Java 字节码             │
│ .o / .obj│ 目标文件（编译中间产物）  │
│ .html    │ HTML 网页               │
│ .pdf     │ 便携文档格式             │
│ .zip     │ 压缩归档文件             │
└──────────┴───────────────────────┘
```

> [!info] UNIX vs Windows 的关键区别
> - **UNIX**：文件扩展名仅是约定，操作系统不强制关联。`gcc` 通过扩展名判断是 C 源码还是目标文件
> - **Windows**：扩展名与应用程序**强绑定**。双击 `.docx` 自动用 Word 打开，这由注册表中的文件关联决定

## 文件结构

操作系统对文件内部结构的理解程度不同：

```
三种文件结构模型:

① 字节序列（Byte Sequence）—— UNIX/Windows 采用
┌───────────────────────────────────┐
│ B B B B B B B B B B B B B B B B B │
└───────────────────────────────────┘
OS 看到的只是一串字节，不理解内部含义
文件的结构由应用程序自行解释

② 记录序列（Record Sequence）—— 大型机时代
┌─────┬─────┬─────┬─────┬─────┐
│记录1 │记录2 │记录3 │记录4 │ ... │
└─────┴─────┴─────┴─────┴─────┘
每条记录固定长度（如 132 字符/行）
读写以"记录"为单位

③ B树结构 —— 商业大型机数据处理
      ┌───────┐
      │   K5  │
      ├───┬───┤
  ┌───┤   │   ├───┐
  │   │   │   │   │
┌─┴─┐ ┌─┴─┐ ┌─┴─┐
│K1 │ │K3 │ │K7 │
│K2 │ │K4 │ │K8 │
└───┘ └───┘ └───┘
按键值（Key）组织，支持快速查找
```

> [!tip] 现代操作系统的选择
> UNIX 和 Windows 都选择了**方案①：字节序列** → 最大的灵活性
> 操作系统不关心文件内容的含义，将解释权完全留给应用程序
> 这也是为什么你可以用文本编辑器打开任何文件（只是可能看到乱码）

## 文件类型

```
常见文件类型:

┌──────────────┬──────────────────────────────────┐
│ 文件类型      │ 说明                              │
├──────────────┼──────────────────────────────────┤
│ 普通文件      │ 包含用户数据（文本文件或二进制文件）│
│             │ - ASCII 文件：可用文本编辑器查看    │
│             │ - 二进制文件：有特定内部结构        │
├──────────────┼──────────────────────────────────┤
│ 目录         │ 管理文件系统结构的系统文件          │
├──────────────┼──────────────────────────────────┤
│ 特殊文件      │ - 字符特殊文件：串口、打印机等     │
│  (UNIX)     │ - 块特殊文件：磁盘设备             │
└──────────────┴──────────────────────────────────┘
```

```
二进制文件的"魔数"（Magic Number）:

UNIX 可执行文件头部:
┌───────────┬──────┬───────┬──────┬────────┬─────┐
│ Magic Num │ 代码段│ 数据段 │ BSS  │ 符号表  │ ... │
│ (标识格式) │ 大小  │ 大小  │ 大小  │ 大小   │     │
└───────────┴──────┴───────┴──────┴────────┴─────┘

常见魔数:
  ELF: 0x7F 'E' 'L' 'F'  (Linux 可执行文件)
  PE:  'M' 'Z'            (Windows 可执行文件)
  PDF: '%' 'P' 'D' 'F'    (PDF 文档)
  ZIP: 'P' 'K'            (ZIP 压缩文件)
  Java: 0xCA 0xFE 0xBA 0xBE (Class 文件 → "CAFEBABE")
```

> [!tip] Java 的 CAFEBABE
> 每个 `.class` 文件的前 4 个字节都是 `0xCAFEBABE`（咖啡宝贝）→ 这就是 **魔数（Magic Number）**
> JVM 加载 class 文件时首先检查这 4 个字节，如果不匹配则拒绝加载
> 这与 UNIX 检查 ELF 魔数的机制完全一致

## 文件访问方式

```
两种访问方式:

① 顺序访问（Sequential Access）:
   从头开始，按顺序读取，不能跳跃
   ┌───┬───┬───┬───┬───┬───┐
   │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ → 只能按 1,2,3,4,5,6 顺序读
   └───┴───┴───┴───┴───┴───┘
   适用：磁带时代，日志文件处理
   
② 随机访问（Random Access）:
   可以按任意顺序读取任意位置
   ┌───┬───┬───┬───┬───┬───┐
   │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ → 可以直接跳到 4，再跳到 1
   └───┴───┴───┴───┴───┴───┘
   实现方式:
   - read() 后由 seek() 指定下次读取位置
   - 或 read(position, buffer) 直接指定位置
   适用：数据库文件、视频播放（快进/倒退）
```

> 类比：顺序访问就像听磁带——只能按顺序播放。随机访问像 MP3 播放器——可以跳到任何一首歌
> CS 术语：现代磁盘和 SSD 都支持**随机访问（Random Access）**，但磁盘的顺序读取性能远优于随机读取（因为寻道时间），SSD 则差距较小

## 文件属性

```
常见文件属性（元数据）:

┌────────────────┬──────────────────────────────────┐
│ 属性            │ 含义                              │
├────────────────┼──────────────────────────────────┤
│ 保护/权限       │ 谁能以何种方式访问（rwxrwxrwx）    │
│ 创建者/所有者   │ 文件的拥有者                      │
│ 创建时间        │ 文件首次创建的时间                 │
│ 最后访问时间    │ 上次读取的时间                     │
│ 最后修改时间    │ 上次写入的时间                     │
│ 当前大小        │ 文件的字节数                      │
│ 最大大小        │ 文件允许增长到的最大值             │
│ 只读标志        │ 0=可读写，1=只读                  │
│ 隐藏标志        │ 0=正常，1=不在目录列表中显示       │
│ 系统标志        │ 0=普通文件，1=系统文件             │
│ 记录长度        │ 固定长度记录中每条记录的字节数      │
│ 键的位置/长度   │ 用于按键查找的记录内偏移和长度      │
└────────────────┴──────────────────────────────────┘
```

> [!info] UNIX 的文件权限模型
> ```
> -rwxr-xr-- 1 user group 4096 Mar 27 15:00 file.txt
>  │││ │││ │││
>  │││ │││ └── 其他用户：读
>  │││ └── 同组用户：读+执行
>  └── 所有者：读+写+执行
> ```
> 9 个权限位 = 3 组（owner/group/others）× 3 种权限（read/write/execute）

## 文件操作

| 操作 | 系统调用 | 说明 |
|------|---------|------|
| 创建 | `create()` | 创建空文件，设置属性 |
| 删除 | `delete()` | 释放磁盘空间 |
| 打开 | `open()` | 将文件属性和磁盘地址加载到内存 → 返回**文件描述符** |
| 关闭 | `close()` | 释放内存中的文件信息 |
| 读取 | `read()` | 从当前位置读取数据到缓冲区 |
| 写入 | `write()` | 从缓冲区写入数据到当前位置 |
| 追加 | `append()` | 只能在文件末尾添加数据 |
| 定位 | `seek()` / `lseek()` | 设置读写位置（用于随机访问） |
| 获取属性 | `stat()` | 读取文件元数据 |
| 设置属性 | `chmod()` / `chown()` | 修改权限、所有者等 |
| 重命名 | `rename()` | 修改文件名 |

> [!warning] 文件描述符（File Descriptor）
> `open()` 返回的**文件描述符（fd）** 是一个整数，代表打开的文件
> 后续的 `read()`、`write()`、`close()` 都基于这个 fd 操作
> 这是操作系统管理打开文件的核心抽象
> 
> UNIX 中 fd=0（stdin）、fd=1（stdout）、fd=2（stderr）是预定义的

## 💡 架构师视角映射

| 操作系统概念 | Java 后端映射 |
|------------|-------------|
| 文件 = 持久化存储的抽象 | Java 的 `File` / `Path` 类；数据库表是对文件的更高层抽象 |
| 字节序列结构 | Java 的 `InputStream` / `OutputStream` 将一切视为字节流 |
| 魔数检查 | JVM 加载 `.class` 检查 `CAFEBABE`；HTTP Content-Type 头部判断数据格式 |
| 文件描述符 fd | Java NIO 的 `FileChannel`；JDBC 的 `Connection` 对象类似 fd 的角色 |
| 顺序访问 vs 随机访问 | Kafka 消费者顺序消费 vs MySQL B+Tree 随机查询 |
| 文件属性/元数据 | 数据库表的 schema 信息（表名、列类型、索引等）→ 类似文件的元数据 |
| `open/read/write/close` 模式 | JDBC 的 `getConnection → executeQuery → close` 生命周期 |

## 🔍 深挖指南

> [!note] 核心要点
> 1. 文件是对持久化存储的核心抽象——命名 + 字节序列

> 3. 文件描述符是操作系统管理打开文件的关键机制
