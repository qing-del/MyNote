## 竞态条件与临界区

**进程间通信（Interprocess Communication, IPC）** 核心要解决三个问题：
1. **怎么传递信息？**
2. **怎么确保不交叉重叠（互斥）？**
3. **怎么保证顺序依赖（同步）？**

> [!failure] 竞态条件（Race Condition）
> 两个或多个进程读写某些共享数据，而**最后的结果取决于进程运行的精确时序**。

```
经典的银行转账例子（A 和 B 同时给同一个账户打100元）：

代码：
balance = balance + 100; // 这其实是 3 条机器指令

进程 A 视角:
1. r1 = balance (读到 1000)
====== 💥 这里发生了时钟中断，OS 把 CPU 给了进程 B ======
进程 B 视角:
1. r1 = balance (也读到 1000)
2. r1 = r1 + 100 (变 1100)
3. balance = r1 (写回 1100)
====== OS 把 CPU 还给进程 A ======
进程 A 视角继续:
2. r1 = r1 + 100 (1000 + 100 = 1100)
3. balance = r1 (写回 1100)

结果：明明打了 200 元，账户只涨了 100 元！100 元凭空消失！
```

**临界区（Critical Region）**：对共享内存进行访问的程序片段。
为了避免竞态条件，必须保证**互斥（Mutual Exclusion）**：不能有两个进程同时处于其临界区。

一个好的互斥方案需要满足 **4 个条件**：
1. 任何两个进程不能同时处于临界区
2. 不应对 CPU 的速度和数量做任何假设
3. 临界区外的进程不能阻塞其他进程
4. 不得使进程无限期等待进入临界区（饥饿）

## 忙等待的互斥方案

这些方案通过**死循环不断检查（忙等待，Busy Waiting）** 来实现互斥。

1. **屏蔽中断（单核有效）**：在进入临界区后关中断，出来再开中断。现代系统不用因为：用户态进程不该有此特权，且多核系统无效。
2. **锁变量（软件）**：引入一个共享变量 `lock`（初始 0）。检查 `lock==0` 时设为 1 并进入。→ ❌ **自身也有竞态条件**。
3. **严格轮换（软件）**：引入一个变量 `turn` 记录轮到谁。→ ❌ 违反了条件 3：慢进程会拖死快进程。
4. **Peterson 算法（软件）**：结合了 `turn` 和 `interested` 数组的软件完美解法，但需要编译器不乱序执行（通常需内存屏障）。
5. **TSL 指令（硬件）**：**测试并加锁（Test and Set Lock）**。这是一条硬件原语，将读取内存和修改内存捆绑为**一条不可分割（原子）的机器指令**。多核 CPU 会锁住总线。这是所有现代自旋锁的基石。

> [!warning] 忙等待的致命缺陷：浪费 CPU + 优先级反转问题
> - **浪费 CPU**：当一个进程在里面，另一个一直死循环，白白空转（这就是自旋锁）。
> - **优先级反转（Priority Inversion）**：高优先级线程 H 需要读，低优先级 L 在临界区。由于 H 优先级高，CPU 一直给 H 空转检查，导致 L 永远分不到时间片去把它手头的事做完并退出临界区！

## 睡眠与唤醒（Sleep & Wakeup）

为了解决忙等待浪费 CPU 的问题，引入了阻塞机制的系统调用原型：
- `sleep()`：将调用进程阻塞（让出 CPU）
- `wakeup(pid)`：唤醒某个指定进程

```
经典的生产者-消费者问题（有界缓冲区）
问题：
生产者：如果满了就 sleep，没满就装，然后 wakeup 消费者。
消费者：如果空了就 sleep，没空就吃，然后 wakeup 生产者。

💥 致命的唤醒丢失（Wakeup Waiting）问题：
1. 缓冲区为空。消费者检查 count == 0。
2. 调度器切到生产者！
3. 生产者放入了一个货，count=1。它觉得之前是 0，说明消费者肯定在睡觉，所以发了一个 wakeup(消费者) 信号。
4. 此时消费者还没真正睡下（醒着的）。这个 wakeup 信号发到空气中，丢失了！
5. 调度器切回消费者！消费者继续执行之前准备 sleep 的指令 → 睡死过去！
6. 生产者不断生产，最后填满，自己也睡死过去。死锁 (Deadlock)！
```

## 信号量（Semaphore）

E.W. Dijkstra（迪杰斯特拉）在 1965 年提出的革命性方案。
它是一个**整型变量**，不仅可以累计唤醒次数，而且引入了两个**原子操作**（由系统/硬件保证期间不被打断）：

- **Down（也叫 P，减操作）**：
    - 如果信号量 > 0，减 1，继续执行。
    - 如果信号量 == 0，**进程睡眠**（此时连同减 1 的操作作为一个不可分割的整体）。
- **Up（也叫 V，加操作）**：
    - 信号量加 1。
    - 如果有一个或多个进程在这个信号量上睡眠，**唤醒其中一个**（被唤醒的进程完成减1操作并继续）。

> 类比：**信号量就像一家餐厅的等位号码机和座位数**。
> `Down`：客人想吃饭。看座位数（信号量）。有座位（>0）就领筹码进去坐下；没座位（==0）就在门口长椅上睡觉等（阻塞）。
> `Up`：客人吃完了出来，还筹码（座位+1）。然后喊醒长椅上第一个等的人。
> 整个拿筹码看等位过程是瞬间完成（原子）的！

```java
// 使用信号量解决生产者消费者（Java 版伪码对照）
Semaphore mutex = 1; // 互斥，确保同一时间只有一人能操作缓冲区
Semaphore empty = N; // 同步，记录空槽位数量
Semaphore full  = 0; // 同步，记录满槽位数量

生产者():
  down(&empty); // 等待空槽位 (没空槽我就睡)
  down(&mutex); // 锁住缓冲区
  插入物品();
  up(&mutex);   // 解锁
  up(&full);    // 满槽位+1 (顺便叫醒消费者)

消费者():
  down(&full);  // 等待充满 (空盆子我就睡)
  down(&mutex); // 锁住缓冲区
  拿走物品();
  up(&mutex);   // 解锁
  up(&empty);   // 空槽位+1 (顺便叫醒生产者)
```

## 互斥量（Mutex）

如果信号量只需要用来锁临界区（只取值 0 和 1），那就是一个**互斥量（Mutex）**。
它是用来处理“谁能进房间”的最简化的信号量版本，无需关注累计次数机制，性能极高，适合于管理一块内存或一段代码的互斥访问。

## 管程（Monitor）

信号量很强大，但在大型程序中只要少写一个 `up`，或者 `down` 的顺序写反了（比如先 `down(&mutex)`，发现没容量再去 `down(&empty)` 就会死锁），整个系统就会崩溃。死锁极难 debug！

**管程（Monitor）** 是一种**高级语言层面**的同步原语（编译器帮你搞定，OS 不管）。
在管程里：
- **互斥由编译器保证**：包含变量声明和过程（函数）。**任何时刻只能有一个进程（线程）活跃在管程内！**
- **同步靠条件变量（Condition Variables）**：引入了 `wait(c)` 和 `signal(c)`。
    - 当进程在管程内发现条件不满足，调用 `wait(c)` 会让自己在 `c` 队列上睡眠，**同时自动释放管程互斥锁**（让别人能进来）！
    - 别人满足条件后，调 `signal(c)` 叫醒那个睡着的人。

> [!important] 为什么说你在天天用管程？
> **Java 就是在底层直接支持了管程的概念！**
> 在 Java 中，每个 `Object` 都有一个隐式的管程！
> `synchronized` 关键字：就是一头扎进管程，编译器保证了互斥。
> `Object.wait()` 和 `Object.notify()`：就是管程里的条件变量操作！

## 消息传递（Message Passing）

上述所有方案（信号量、管程）都依赖于一个前提：**几个进程能看到那块共享内存（共享变量所在的内存区）！**
但在**分布式系统（不同机器）**或者严密隔离的系统中没有共享内存怎么办？
只能用系统调用原语：`send(destination, &message)` 和 `receive(source, &message)`。

面临的架构问题：确认包（Ack）、重传（Retransmission）、身份认证（Authenticity）、寻址等。
（这直接衍生出了现代的 RPC 框架以及 Message Queue 中间件的基石理论）

## 💡 架构师视角映射

| 操作系统概念 | Java 后端映射 |
|------------|-------------|
| 忙等待（自旋检测） | `java.util.concurrent` (JUC) 中的 `AtomicInteger.compareAndSet()` 以及自旋锁（SpinLock），适用场景是极短的临界区代码，为了避免线程阻塞/唤醒的重型开销 |
| 信号量（Semaphore） | JUC 完全对应的实现：`java.util.concurrent.Semaphore`，常用于限流器、如限流 100个并发 HTTP 连接 |
| 互斥锁（Mutex） | JUC 下的 `ReentrantLock` 这个重量级独占锁 |
| 管程（Monitor） | `synchronized` 隐式锁，或配合 `Lock` 使用的 `Condition` 对象接口 （`await()`, `signal()`） |
| 原子 CPU 指令（TSL） | x86 的 `LOCK CMPXCHG` 指令。Java 中的 `Unsafe.compareAndSwapInt` 就是由 JVM 直呼这一条 CPU 指令，这就是赫赫有名的 CAS 无锁编程基础！ |
| 消息传递 | RabbitMQ, Kafka 发送事件通知。不同微服务进程不在同一主机无法用管程锁，改用 TCP 消息报文同步 |

## 🔍 深挖指南

> [!note] 核心要点
> 1. 无保护的并发读写同一块内存一定会导致数据凭空丢失或紊乱。
> 2. 软件锁/硬件锁解决了同时进入房间的问题（互斥）。休眠与唤醒解决了浪费资源空转的问题。
> 3. 信号量通过原子操作解决了唤醒丢失。管程把复杂易错的上锁逻辑交给了编译器。

- CAS 自旋是怎么运作的？ → 参考 Java 的 `AtomicInteger` 源码以及 JMM 内存控制逻辑
- Java 中 Synchronized 的底层实现（Monitor Object 模式） → 参考《深入理解Java虚拟机》第13章
- 从管程扩展到 Java 的并发控制包 → 强烈推荐阅读 Doug Lea 的神作：AQS (`AbstractQueuedSynchronizer`) 的源码设计
