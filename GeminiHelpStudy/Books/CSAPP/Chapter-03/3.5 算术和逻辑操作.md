## 加载有效地址 (leaq) - 最精妙的黑客指令

`leaq` 指令（Load Effective Address Quad）被认为是 x86-64 汇编集的一座神来之笔。

从表面上看：
```gas
leaq 7(%rdx, %rdx, 4), %rax
```
这句话如果在前面的 `movq` 意味着：“从地址对应的抽屉里拿东西放进 `%rax` 里面。”
**但在 `leaq` 面前，它只算地址公式，绝不去取内存！**
`%rax = 7 + %rdx + %rdx * 4` 
即计算出来的等式总和直接被写入了目标寄存器，而根本不去读取这个可能实际上什么都不包含的外包内存位。

> [!tip] leaq 的双重意义
> 1. C 语言生成指针的幕后推手，它能轻而易举为目标变量计算一个指针（就是 C 的取地址符 `&` 的真身）。
> 2. 也是 C 编译器极度精通的**加法和乘法运算平替手段**！！上面这个式子，由于比例因子 `s` 最高为 8，所以如果你想用 C 语言求 `x * 5 + 7` 这么长串式子，一条硬件层面的简短的 `leaq` 就通过极其高效的单周期运算器结束了它，甚至不会改变任何进位或者溢出的状态标记控制码！

## 一元操作与二元操作

其他算数操作都是极其纯粹直接的（大部分类似 C 的 `++`, `--`, `+=`, `^=` 等）：

**一元操作（只有一个操作数）**：
- `incq D` : 也就是 `D++`，或 `D = D + 1`，可用于寄存器也可用于内存对应处增加。
- `decq D` : 也就是 `D--`
- `negq D` : 数学取负操作
- `notq D` : 二进制按位取反（就是 C 中的 `~` 操作）

**二元操作（源端 Src 结合进目的端 Dest）**：
- `addq S, D` : 计算 `D = D + S`。注意！在 ATT 汇编中，目标放右边！而且执行完毕后是改写在了 D 上面。
- `subq S, D` : 计算 `D = D - S`。（是用目的地去减来源数据）
- `imulq S, D`: 计算 `D = D * S`。有符号乘法。
- `xorq S, D` : 按位互斥异或 `D = D ^ S`
- `orq S, D`  : 按位或 `D = D | S`
- `andq S, D` : 按位与 `D = D & S`

## 移位操作

移位（Shift）操作是最重要的位级运算：
将一个值往左（向大端）或者右（向小端）推送指定的位数位移。移位数量用 `%cl` 中的前几位截断取模提供。

- **左移指令** (`salq` 或 `shlq`，作用完全相同)：
  `salq k, D`，即将目标 `D` 左移 `k` 位，右边空出来的洞一律补 0！这就是著名的无损乘 2 的 K 次方优化法！

- **右移指令**（因为符号位的问题分为逻辑和算术）：
  1. `shrq`（逻辑右移，Shift Right Logical）：直接干脆，不管三七二十一，在左边因为推开而产生的空位上一律补 0。这适合于**无符号（unsigned）** 数字。
  2. `sarq`（算术右移，Shift Arithmetic Right）：具有高情商手段。它在左侧把原来符号位的数值全部填充蔓延复制下去。这意味着：如果原本是正数，前面填的都是零；如果是负数（首位为 1 因其是补码编码），全都在前面填充 1 保持为负数。这适合于**有符号（signed）**。

## 特殊的算术操作：128位乘法与除法

当我们拿两个 64 位的极大数进行乘法时，产生的数值将无法在任何一个单独的 64位寄存器（如 `%rax`）存放下！按照数学定理它可能蔓延达到 128 位的长度！

这该如何解决？Intel 为这类全长计算提供了一条**特殊且绑定极其特定固定位置的指令：** 单操作数乘法和除法：

- **乘法生成高位**（`mulq S` / `imulq S`）：如果你用这一单操版本乘法指令，编译器默认你有一半乘积就在 `%rax` 当中。计算 `S * %rax` 会把产出的完整 128 位超长字符串**劈成两半**：高 64 位被截住塞给了 `%rdx`（被称为高阶部分），而低 64 位保存在 `%rax`。这叫做联合作答（`%rdx:%rax`）。
- **除法余数联合**（`idivq S` / `divq S`）：这是乘法的逆操作。它默认你用极大体型的双打寄存器 `%rdx:%rax` 拼起来的超强 128 位作为被单方面切割的极大除数。执行除法指令后，它用同样的两个箱子装不同的结论结果：
  **计算得出商的本家**放在 `%rax`。
  **由于除不尽留下的余数**塞在 `%rdx`！

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 对于位运算优化的绝对应用**：
- Java 的 `HashMap` 在计算元素放在哈希桶哪个格子里的时候。并没有使用低效漫长的计算机原生 `div/idiv` （也就是求余数 `%` 算式）操作。它的代码是 `(n - 1) & hash`！这就是基于 `and` 与门位运算。因为 `andq / sub` 等指令只要 1 个 CPU 时钟节拍周期，而即使是超标量计算机器的整数除法，也要生硬吞掉几十个以上的时钟周期。底层运算指令耗时的差距体现出了这些优化的真知灼见。

**`>>>` 无符号右移的 C 语言真面目**：
- 早年间被 Java 题经常拷打的一道：什么是 `>>` 也就是算术右移，以及带三个箭头的 `>>>` 为逻辑右移。
- 回想上面我们写的逻辑分析：这就在 C 和底层的汇编层级中，本质上一个是 C 的 `unsigned` 所被编译器对应的 `shrq`，和 C 本身的 `int` 所默认生成的 `sarq`。由于 Java 没有 `unsigned` 无符号类型，它为了能在 Java 提供 `shrq` 这个指令的能力去在二进制左边填 0，因此只能人工作出一个新式操作符号！

## 🔭 深挖指南

> [!tip] 核心知识点与延伸阅读
>
> **本节最重要的三点**：
> 1. `leaq` 只有加号跟乘号，它**不取内存！只是做基础加乘混算后传寄存器的最妙工具**。
> 2. `add, sub` 中，右边目的地址即作为参与运算方，又被覆写作为存储接受方。
> 3. x86 利用 `%rdx:%rax` 像叠汉堡盒一样完成了超脱单 64 位的 128 级变异结果。
>
> **深挖路径**：
> - 尝试分析：C 语言编译器开启优化选项 `gcc -O2` 和 `gcc -O0` 在一个乘法运算（如乘一个固定的常数16或者31等）时，为什么会直接利用 `sal` 和 `leaq` 替代本原上的 `mul`。
