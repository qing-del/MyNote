## 过程调用的基石：运行时栈 (The Run-Time Stack)

当在高级语言中，**函数调用函数**（如 `P` 调用 `Q`）。为了能够从无数次的嵌套中原封不动地返回最初的状态，且由于一个局部变量只在特定的函数区域内存活，CPU 和 OS 为程序开辟了一块**极其关键的内存工作区：运行时栈（Stack）**。

- 整个栈**朝低地址方向生长**（这是永恒的铁律，指针向下压空间变小）。
- 指向栈顶元素位置的固定角色永远是 `%rsp` 寄存器。
- 把一块属于特定函数使用的栈堆区域称为它的 **栈帧 (Stack Frame)**。

> 类比：你（当前函数 P）正在处理几份文件。突然老板打来电话分配了新任务（调用函数 Q）。为了避免桌子上的文件搞混，你在现有的文件上**叠放一个黑色的托盘（开启新的栈帧）**，在第二层托盘处理老板布置的全套工作（局部变量申请）。老板的任务做完了（函数 Q return 退出），你把整个第二层托盘连同上面的新文件全部扔垃圾桶抽出移除（回收释放栈帧），重新接下去看刚才最初那些摆在第一层的文件（返回 P 并恢复刚才所有状态）。

## 控制转移 (call & ret)

当要进入一个过程（函数）：
- `call Label` 指令：这是一个复合动作指令。第一步：它先把**接下来紧邻着 `call` 下方的那一句原始代码地址（返回地址）**给 `push` 压入当前栈底埋好。第二步：把要访问的函数所在的地址强行暴写入程序跳转计数器 `%rip` 让它起飞去执行！
- `ret` 指令：它恰恰是 `call` 的死对头逆操作。当函数走到结尾时调用，它利用第一步压入的返回地址，直接弹出 (`pop`) 栈顶地址，反写进 `%rip`！实现完美着陆原出发点！

## 数据传送 (参数传递的艺术)

当 `P` 想要调用 `Q(a, b, c, d, e, f, g)` 并传递七个参数怎么办？

x86-64 有一套被写死在系统底层 ABI 中的"参数传递规约"：
- 当只有 **1 个到最多 6 个**的参数时，它不屑于浪费内存压入栈。它强行规定：将参数分别放在这 6 个特殊的寄存器中： 
  **`%rdi, %rsi, %rdx, %rcx, %r8, %r9`**（重要！必须按这个顺序放）。
- 第 7 个以及后续所有溢出超载的极多参数，都会被强行**按相反的顺序入栈存进自己的栈桢空间内**，去供下一任接收者靠偏移寻址去倒腾翻找。
- **返回结果**永远通过且只通过 **`%rax`** 交回调用者。

## 局部存储空间的分配与释放

在 C 中你定义了 `long local_arr[10];` 等一系列繁多对象。
在汇编层级，其实就是通过简简单单个一句：`subq $80, %rsp`。因为数组 10 个 long 就是一共 80 字节。
利用 `%rsp` 指针的向下打洞操作，立刻圈占了 80 字节作为本栈的绝对受保护合法领域！
随后退出函数时，只需要同样执行加法 `addq $80, %rsp`，它就被瞬间回收遗忘干净。

## 寄存器使用惯例 (被调用者保存 vs 调用者保存)

由于通用寄存器一共才 16 个，而所有的函数互相调用时大家都要抢夺使用去计算，怎么确保寄存器不被刚才下去借走它的新函数胡乱抹除覆盖？
规则将寄存器阵营硬分为了两大阵营：

1. **被调用者保存（Callee-saved）** 的寄存器（如 `%rbx, %rbp, %r12...`等）：
   潜规则："你要用随你，但你如果在你的函数里面要动它们，**请在进入函数第一步赶紧去压栈 push 把别人传给你的遗留老值藏好，用完以后自己把它取回放走复原！**" （也就是你在你自己的新托盘上先把别人寄存在你这儿的不变的东西拿胶带封存起来。）
2. **调用者保存（Caller-saved）** 的寄存器（如 `%r10, %r11`）：
   潜规则："这些寄存器是公用擦手白板。任何一个子函数都可以随意覆写掉上面的所有印记而不需要负任何责任。因此如果你（上级函数）觉得自己存的这些东西极其昂贵非常重要，**在发起 call 命令调用别人前去求办事之前，你自己先把它们压栈藏好，别指望别人！**"

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 对于栈的究极放大与管理映射：多线程模型生命线！**
- 我们反复经常说的在 Tomcat 开辟一条 1M 分配栈参数的新 Thread 从而进行 Http 请求响应中。
- Java `StackFrame (栈帧)` 原则完美完全映射并沿袭继承了上面的设计：它的局部变量表 Local Variables（就像上面栈里保存参数及 Caller 临时压入的数据）与方法出口（对应上面由 call 发起的返回地址指向）。
- 当调用大批量递归、亦或是由于 Spring 中的各种极大链条的代理拦截 AOP 等不断套娃嵌套深入！JVM 的栈向底层极速蔓延，最终触发著名的栈越界错误。

**返回值单一与 Java 解法：**
- 汇编中的只通过统一定向的一个小水管 `%rax` 拿返回值，导致了在 C 以及 Java 中：你根本**没法书写返回多个东西的方法（Return a, b, c;）**！如果真想要多个只能把他们装个大箱子包起来（打包为 Object 封装类返回这个箱子的基址引用，亦或是直接向外部传入空数组的引用指针要求函数把产物向箱子里直接灌输！这也就是经常我们在 Spring 的 Dao 层面看到的 `PageBounds` 分页返回以及填充参数封装操作）。

## 🔭 深挖指南

> [!tip] 核心知识点与延伸阅读
>
> **本节最重要的三点**：
> 1. **`%rdi` 开头的 6 员首发猛将用来传参，超出的部分统统送上断头台压入原内存本栈中。**
> 2. **栈是一对下垂生长的倒吊空间，压栈变小 `%rsp` 扣除，出栈长回来！**
> 3. `call` 先压再起飞，`ret` 先谈再落地。这一对连招。
>
> **深挖路径**：
> - 书中提及了一个老旧的被弃用的寄存器叫做 `%rbp` (原本的帧指针)。现代编译器在开了 `-O2` 优化后为何要废弃它而直接计算偏移寻找所有栈内元素？（减少了一个极大的占用。现在全用纯粹推算解决问题）。
