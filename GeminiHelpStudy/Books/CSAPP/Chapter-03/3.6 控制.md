## 控制流的地基：条件码寄存器 (Condition Codes)

CPU 是盲目运行的，直到它学会了在路口观察自己刚才留下的一串特殊的标记。这些标记就是存在处理器内部不可见的几个单比特标识，叫做 **条件码（Condition Codes）**。

它们的作用：在刚才进行的最近的一条算术或比较相关指令之后，由硬件的 ALU （算术逻辑单元）自动去更新这四个比特：
- **`CF` (Carry Flag, 进位标志)**: 刚才最近的操作如果在最高有效位产生了进位或借位。多用于推断**无符号数溢出**的问题。
- **`ZF` (Zero Flag, 零标志)**: 刚才最近操作得出的结果彻彻底底**是不是 0**。是 0 设为 1。
- **`SF` (Sign Flag, 符号标志)**: 最近操作得出的结果**是不是负数**（也就是说结果最高那一位等于 1 时，拉出此警报标志）。
- **`OF` (Overflow Flag, 溢出标志)**: 专门判定**有符号（带一正一负）数**的正溢出或负溢出用的警报。

> [!important] 比较指令与测试指令（cmp 和 test）的本质
> - `cmpq S2, S1`：它的执行和刚才介绍的减法 `subq` 一模一样！都是拿 S1 减去 S2。但是，它产生的那个差值**会被直接扔掉**，完全不存储进入任何地点，只留下它计算差值时带来的四个控制码位 `CF ZF SF OF` 供后人查看流向。
> - `testq S2, S1`：这个指令与 `andq` 一模一样！它相当于计算 `S1 & S2`，也完全扔掉结果值只留那四个标志控制码！如果它们两个完全互斥（为0），则此时就会有 `ZF` 打开！（经常配合检查某人是不是全空或者只拿一个掩码进行特定测验。）

## 控制跳转 (Jump 指令)

最底层的流程更改来自于强大的指令——**跳转 (Jump)**。当执行它时，如果条件符合，它的硬件执行效果是直接把后面的目标参数地扯**蛮力覆盖填写写入了 `%rip` (程序计数器) 中**。迫发改变整个宇宙线向下流转的方向。

- **`jmp` 无条件跳转**：就是我们恶名昭著的 C 的无所不尽的 `goto`！你指哪里它下一行就直接飞向哪里代码。
- **有条件跳转 (Conditional Jumps)**：它们结合刚才由上层 `cmp` 构建后挂在那里的各种奇特标识做出精准判决，比如：
  - `je` （Jump if Equal）：如果相等那就飞到那里去。（相当于上面由于相等得出两个减数结果是0激发了 `ZF=1`）
  - `jne`（不等）、`js`（判定为负）、`jg`（判定源数据巨大大于源参数则进入此处代码）、`jl`（判断低于它）。
  - 这些跳转移出了非常晦涩与对齐的 `C - If` 后侧底层原貌。

## 控制流向汇编：If-Else 指令结构翻译

当你书写了一段很日常的 `if-else` 分支控制块以后，C 编译器会重组你的语义，由于汇编不支持括号等大块代码的模块分割，它使用了**极其直来直去的基于 `goto` 的跳表重装思想**。

```c
long absdiff(long x, long y) {
    long result;
    if (x < y)
        result = y - x;
    else
        result = x - y;
    return result;
}
```

GCC 会转变成这套极其粗鄙但是极其实用并可读的套路格式代码模板去转：

```c
// 伪代码表示真实在它最底层的逻辑跳转控制方式！
    long v;
    if (x >= y) goto false_branch;  // 先对原条件做一个强制取反！（x<y 反向即为大于等于），如果不符合就立即跳转越走躲开该执行。
    v = y - x;                
    goto done;                // 记得这里！处理完了必须有一个再跳回到末尾避免滑入下一步 else 的动作！
false_branch:                 
    v = x - y;                // else 代码在这
done:
    return v;
```

## 向高效转型：条件数据传送 (cmov)

我们上述的这种传统做法其实面临一个在现代处理器上的地雷风险：基于控制的跳传语句，在现在这几十层管线的重度**分支流水预测期**里面常常带来严重错判毁掉的超长计算节拍惩罚损益。

因此编译器对上面这句：要么就你、要么就我的条件传递式进行出击，利用 **`cmov` (条件传送技术)** 去掩盖惩罚：
编译器它不再瞎猜要走上面还是下面。编译器把**两条 if 和 else 的路分别全算一遍得出各自的底牌答案**。
然后把原版必须进行一跳不可的分支结构抹干净，调用类似与 `cmove L`（当刚才的比较认为你小的时候我就把刚才那个算出存在这里底牌移过来传给返回值里面去）。
这一步：**不再做复杂的管线控制改变分支猜测，直接粗放计算再加一个通过单条件开关的寄存器置数。**

> 类比：以前你要送快递，到分岔路口，你要花好几分钟判定要走 A 还是走 B（分支跳转预测）。现在你干脆叫两个快递员：一个沿 A 走到底一个沿 B 走到底进行暴力全铺盖计算。然后在终节点判定：因为最后判定你要选送的区域偏北，所以判定刚才南边的 B 快递员被废弃扔掉，选择 A 快递员的结果被交付（条件控制数据转移）。

## 循环：如何用 if 和 goto 写循环？

所有的 `do-while`、`while` 乃至最最复杂的具有初始化条件的 `for`。他们在被脱出外装和各种编译器分析阶段打成一团血水。它实质就是底层的 `if-else + jmp` 的变形使用。

最为经典的是针对最常用的 `while (test_expr) { body_statement }` 进行优化后重写的经典形态，我们称为 **跳转到中间（Jump-to-middle）** 策略编排方法：

```c
// 原生 while
while(i < 10) {
    p = p * 2;
    i++;
}

// 在 C 往汇编底层编译器中的极丑转换：(被解构为极其高效直接的循环逻辑块)
   goto test;               // 1. 在做任何事之前，你必须先给我跳过去中间去验票做第一次条件！
loop:                       // 2. 这里是 while 本身的 body 花括号内容！
   p = p * 2;
   i++;
test:                       // 3. 这里是整个流程的控制判断处。如果可以那就跳回楼上，不行就直接不管流向最后一步。
   if (i < 10) goto loop;
```

## switch 语句及其底层法宝：跳转表 (Jump Table)

当你在 Java 抑或是 C 内写了一大串根据一个分类代码引发出很多状态机（例如按键分配的 30 个 `case 1 ... case 30`）。
最蠢的方法是一条条使用上面生成的 `if-else` 一路写出一个极长的判定瀑布控制连线，其最糟糕运行所需跳转查验为 O(N)。

这时，编译器施展了 **跳转表（Jump table）魔法（非常重要且精妙）！**：
- 它在内存里强行辟出一块长且整齐的不可动数组内存。这数组里面没有塞数据，塞的是**指令代码每一个 Case 片段那一段的内存起跳偏移地址（Code Poniter）**。
- 它要求你的 switch 的参数只能是不跨度过于离谱的正型数值。在算术开始前，判定该值并截掉前位。
- 直接利用数组索引：以 O(1) 的时间强行加上计算后参数，通过 `jmp *ArrayAddress(, %rax, 8)` 计算索引！去拿到那里面的代码门牌号起跳进去！！

> [!important] Switch 跳转表 O(1) 的神级体现：
> 这个长达几十个 Case 的分类，如果通过跳表技术，你的花费时间：无论是判定查明在 Case 1 ，或者查明你在深不可去、逻辑尽头的 Case 100 处，你执行这一大串 switch 的时间全是一样快的神迹体现！也就是**基于查询与转移的多路映射算法能力。**

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 对于 `switch` 为何要增加 String 等引用的背思**：
- 最古老的 Java 的 `switch` 只被允许能接入 `int/byte/short` 的数（因为那是严格遵守 CSAPP 对编译汇编跳转表的底层的内存连续定型）。
- 但在 Java发展后引入了可以基于非常反汇编底层的： `String` 对照功能。而如果你在这个时候去调用前面提到的对由 Java Class 生成的底图用 `javap` 进行汇编码展示分析的时候。
- 你会赫然发现这个原本只配在机器层面的跳表逻辑技术在 JVM 中被替换转化：它是被编译为通过计算这些字符串独一无二产生的 `hashCode()` 在 `lookupswitch` 及表搜索中进行比对而产生的方法调度！也就是说为了高级语言的功能应用，它进行了基于 JVM 级再做的高维度跳表实现技术。

**警惕分支预测带来的重大性能落差：为什么不能随机化数组后查？**
- 如果你要处理一堆数：全部判定提取其中的正数再把所有的正数相加起来。这个简单的 `if(arr[i] > 0)` 中，如果你提前将这个数组极其干净的进行了从小到大排序？执行会快出好几倍乃至十倍！
- 原因是这块所指的 CPU 底层的具有长流管线的：**分支猜测逻辑**。如果它完全凌乱不堪，CPU 前一刻猜是要走后一刻又被迫做清洗去跳错，时间大损。而如果你把它排的极为有序。CPU 的那一个简单底端的根据历史记录做预估的小脑会判定全对，使代码的管线满载进行，完全规避分支判定的消耗错付。

## 🔭 深挖指南

> [!tip] 核心知识点与延伸阅读
>
> **本节最重要的三点**：
> 1. 原来 CPU 判断的不是 if ，判断的是在背后留下并根据你判定来产生触发或拉起信号报警器——**条件码**（四个基础标识CF等）。
> 2. `jmp` 和一众被绑架的分支选择跳转（如`je`, `jg` 等等）所配合构造形成的各种高级控制程序模板语法（`do-while / for` 等等本质一丘之貉全为 `goto` 化处理）。
> 3. **强用跳转表（Jump table）实现的通过索引一次直接空降执行地点的神级 switch！** 也是通过汇编基础数组定位完成代码运行区映射的重要展示。
>
> **深挖路径**：
> - 为什么《CSAPP》建议当两个运算极其沉重之时且带副作用调用复杂改变内存的时候**不可以随便使用**“不管三七二十一跑出两个再直接选出一个最好成绩”的这个带有 cmov 条件转移命令技巧作为代替？原因会引发你极强的针对 C 和内存防范错变理解认识反推学习。
