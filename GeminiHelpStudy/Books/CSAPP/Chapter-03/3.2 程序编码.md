## 机器级代码与 ISA 抽象

对于写传统 C/C++ 等编译型语言的程序员，操作系统和编译器隐藏了程序的底层工作方式。
要理解机器级代码，必须懂得机器级编程的两种重要**抽象（Abstraction）**：

1. **指令集体系结构（ISA，Instruction Set Architecture）**：
   ISA 规定了处理器的状态、指令的格式以及每条指令对状态的影响。我们将机器程序的行为描述成好像每条指令是按顺序（一句接着一句）执行的，尽管底层硬件通过流水线并发执行。
2. **虚拟内存地址（Virtual Memory Addresses）**：
   机器级程序使用的内存地址是虚拟地址，它看起来像是一个大杀四方的单一字节数组（大号 Byte Array）。它把内存的层级结构（缓存、DRAM、磁盘上的交换区）这堆烂摊子全部挡在幕后。

> 类比：ISA 就像是硬件和软件之间签署的一份**服务协议书（API 接口文档）**。无论 Intel 用什么黑科技（如分支预测、乱序执行）来压榨硅片的性能，只要外面跑的程序能根据这份 ISA 手册原原本本地推演出预期的状态变化，Intel 就算完成了它的工作。

## 可见的处理器状态

在机器级编程中（汇编语言中），我们的视角**不再有变量、不再有对象**，只剩下寄存器、内存和 CPU 中的几个特殊寄存器：

- **程序计数器（PC，x86-64 中称为 `%rip`）**：存放下一条将要执行指令在内存中的地址。
- **整数寄存器（Integer Register File）**：包含 16 个 64 位的具名存储位置（如 `%rax, %rbx` 等）。用来存放局部参数、局部变量或者计算结果。
- **条件码寄存器（Condition Codes）**：保存着最近执行的算术或逻辑指令的状态信息（比如上一个减法结果是不是零？是不是负数？），它是执行 `if` 和 `while` 循环的关键依赖。
- **向量寄存器（Vector Registers）**：用于存放一个或多个 SIMD (单指令多数据流) 的整数或浮点数值。

## 从 C 源代码到机器级代码的编译步骤

假设我们有一个 `mstore.c`
```c
long mult2(long, long);
void multstore(long x, long y, long *dest) {
    long t = mult2(x, y);
    *dest = t;
}
```

通过 GCC 编译器走过这四部曲（回顾第一章内容）：
```mermaid
flowchart LR
    A["C文本\nmstore.c"] -->|C预处理器(cpp)| B["展开的C\nmstore.i"]
    B -->|C编译器(cc1)| C["汇编语言\nmstore.s"]
    C -->|汇编器(as)| D["可重定位目标程序\nmstore.o"]
    D -->|链接器(ld)| E["可执行目标程序\nprog(也就是机器码)"]
```
- **生成汇编代码**：`gcc -Og -S mstore.c` 将产生 `mstore.s`。这里面是纯粹的用字符写的助记符代码。
- **生成目标代码**：`gcc -Og -c mstore.c` 产生 `mstore.o`。一旦变成了 `.o` 它就不再是你能看懂的文本了。它是 01 组成的字节串，比如：`53 48 89 d3 e8 00 00 00`。这就是硬件真正读得懂的机器代码。

## 汇编语言的本质与查看方式

机器代码是**字节序列**。为了方便人类阅读，**汇编语言（Assembly Code）就是这些机器代码的文本表示形式（人类可读的助记符）**。
两者几乎存在一一对应的关系。

> [!important] 逆向工程：如何看懂二进制文件？
> 使用**反汇编器（Disassembler）**。在 Linux 中最强大的工具是 **`objdump`**。

```bash
# 查看目标代码对应的汇编指令
objdump -d mstore.o
```
反汇编器会根据机器级字节串规律重新推导并翻译成了汇编语言：
`53` -> `push %rbx`
`48 89 d3` -> `mov %rdx, %rbx`

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 字节码 vs 机器码 的区别**：
- CSAPP 生成的 `mstore.o` 是真真正正针对你本机 Intel CPU 针脚制定的 **机器码（Machine Code）**，直接通过硬件发号施令。
- `javac` 编译出来的 `.class` 文件叫 **Java 字节码（Bytecode）**。它不是给物理 CPU 用的，它是给**虚拟 CPU（也就是 JVM）**用的。
- 真正负责将这段 Java Bytecode 翻译并在内存里组装成和 `mstore.o` 长的同款 Intel 机器码的，叫 **JVM JIT 即时编译器（如 C2 编译器）**。
- 这也是为什么在 Java 高级调优中，我们会去分析打印 C2 的 JIT 原生汇编结果（`PrintAssembly`）以确认是否有发生寄存器分配和方法内联的原因！架构顶点的眼界，终究会俯冲回 01 代码端。

**反汇编器 `objdump` 的平替**：
- 在 Java 研发中，我们看 class 文件时，使用的 `javap -c` 就是一种专属虚拟机的**反汇编器**。

## 🔭 深挖指南

> [!tip] 核心知识点与延伸阅读
>
> **本节最重要的三点**：
> 1. **ISA 抽象**是软硬件交界的宏伟长城。
> 2. 我们将告别各种优雅的 C/Java 代码结构，进入全盘只有 **内存地址、寄存器、程序计数器（PC）** 等基础状态的世界。
> 3. **反汇编 (`objdump`)** 能够将乱码一样的二进制 01，通过一一映射转为人类能读懂的汇编文本。
>
> **深挖路径**：
> - `ATT` 格式与 `Intel` 格式汇编的区别：GCC 默认使用 ATT 语法（CSAPP 教材基准），而 Windows 上的汇编多见 Intel 格式（如不需要带 % 号）。
