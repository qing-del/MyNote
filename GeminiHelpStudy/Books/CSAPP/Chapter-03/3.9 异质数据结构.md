## 结构体 (Structs)：大杂烩容器

C 语言提供了 `struct` 将可能互不相关的多种类型数据紧紧捆绑压缩在一个大型组合块（Object原型）结构对象中！

在机器级代码视角里，这跟我们对面向对象各种精美设计全无关联：
**它就是向底层申请一块从头到尾足够塞下所有内部材料所需全部字长宽度的连续内存大块**。
- 而编译器在这过程中，将自动计算出你要获取其中第几个子对象材料时所相距这个超大结合体包包最初端点的**字节偏离距离位置（Offset）**！
- 比如这里面如果首先放了一个 4 字节的整数，紧跟着一个 8 字节长数组表（比如名叫 B 吧）。如果我把这段集合包的总起始位置放于 `%rdi`，你要取它那长出来的子结构 `B`，编译器就会写出汇编：`movq 4(%rdi), %rax`（绕过前面的杂碎，精直对 4 处截断拿出来！）它把这些访问工作全部在编译时固定死了转化成了相对基点坐标偏移的取件！不需要任何的执行期的再三验证盘询！！

## 联合体 (Unions)：极端的空间复用

当对于那些互斥不共存且只需要二选一的数据存放：你可以调用极其霸道极简只抢位置用且内部乱套混同合流的结构 `union` ！

一个极其简单原理：
大家都是一起申请的大开槽！
`Struct` 内的所有碎片人手一套专属房，你宽 8 字节我就站你背后也接着顺手圈起自己的新房子空间。而 `Union` 所有包含进它的物件结构体都极其内卷抢房强占！这房子多大完全且仅取决于最大的那个物件体积来造墙定高，然后这堆各式各类五花八门不同型号组件进去全部扎堆站在一开始同门坐标点起点进行使用与覆盖抢位。
这使得它可以被用作奇妙且极其暴力的转换器：塞进去一个极长浮点型的位数数值存储。通过里面并存占道强占空间的相同位数无符号整形去拿！这不相当于实现且完成了一套没有任何操作只有观察提取模式和视图置换的高等位移吗？？这也就是底层进行绕开极长验证规章的极速数据流类型转形！

## 数据对齐 (Data Alignment)：内存的隐形鸿沟

这是整本 CSAPP 在数据这一章能教会后端开发工程师最硬的一份**内存与安全避坑必知指南**。

计算机体系经常不容许其任意拿存这跨度，或者对于极其怪异扭长横跨多个交织取线周期不平整拿走的操作进行重击与速度损毁降配：系统底层设置强制了一整套：**各种基础物件在申请地址必须要遵从的必须的“地址是某 K 大小的整数倍”！的铁线禁令准则（Data Alignment）！**
> 例如一个 8 Byte （双全宽子长度的对象如 Double 以及 Long）。它们落入的分配地址末位必须要是绝对干净清冽的只能是 `...000` (保证是8的整除跨度点)。

如果不能符合！系统以及编译器将主动干预进行**无用留白与大额字节孔隙填充抹泥（Pad）堵满空间顶满推长拉远后面的子集。**

> 类比：你想要一班坐双排四人大车的人和三个人挤小三蹦子的在路上不乱抢道不横穿隔离出事，在规划画他们停车时如果前面卡了个小占一个极小缝位导致大四人跨出横斜在两车车界标杆骑砍在两个段面上！这必定被查！导致极其影响整体提拉交通读取。所以要求不管前面停没停满！大列块双全横的结构全部老老实实在界标线起点整齐开始等待填塞！！中间多出来前面那个落单的人塞不满余下的边角全部强拉起禁止堆放阻隔的空置墙体封杀！！

## 如何利用对齐规则（优化你的内存排布！）

来看看对齐（Padding）对同一组要素产生的巨大不同！！
```c
struct S1 {
    int i;  // 4 字节 
    char c; // 1 字节
    int j;  // 4 字节
} // 按对齐法后，共占据：4 + 1 + (补3空隙) + 4 = 12 字节的大长度

struct S2 {
    int i;  // 4 字节
    int j;  // 4 字节
    char c; // 1 字节
} // 这次占据：4 + 4 + 1 + (不用留太多只最后凑尾封出满块齐宽补3) = 12 ？不如果放在大型连列中！但是可以极致拉减前面无用大段阻断长距离间隔坑位！
```

最核心真理优化口诀是：**把你的所有包含的大数据从最大个从宽排到最小个短边从上依此紧紧靠在一齐！！(Long / Double -> Int -> Short -> Byte/Boolean)** 这绝对带来这结构体包极其压紧致密、对系统缓存无痛的高优且极其顺畅不留气孔的最好状态展现形态去进行全生命存留期节省出大量的应用段开销耗损。

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**JVM 中的黑箱对齐填充：JOL (Java Object Layout) 以及指针压缩**：
- 当我们在 Java 设计高并发下的锁等设计结构（例如 `AtomicLong` 防止由于在同一缓存行伪共享 Cache Line False Sharing 对锁产生的来回穿梭重击缓存颠簸失效）。
- 我们会在其头尾手动作一些毫无意义长宽极度的没用假人变量变量 `long px1, px2 ... px7` 假占空！将其前后人为拉开了极强的数据填充对齐宽阔长廊！这等于是底层在利用硬件这套对其体系规律规避因为离得太近被处理器给打入到一个连坐读取带走的行进而错发高频击穿系统性能缓存机制！（现代 Java 直接引入 `@Contended` 注解完美直接干预由底层向这塞 Padding 空气）。

**Java 究竟为何默认按占用排序排阵：**
- JVM 非常极其的聪明，你在书写一个带一百个字段乱七八糟次序交战的大量大对象类 `Class` 时候，它早就给你做了一套极其恐怖高效重排布优化的手段！！
- 在 `HotSpot` 中 JVM 加载你的这块类之后生成分配堆内大小的蓝图布局模型时（除了保证大且唯一的父子关系结构顺序不能乱）。自动通过 `FieldsAllocationStyle` 按占空（把宽从高到细大下）排了。无需你去专门和在这由于不会优化而在写 C 中挣扎的结构设计者抢占其长位空间调整先后（因为如果自己不进行手编紧凑压缩就会大额暴裂内存孔隙膨胀问题）！但这底层逻辑就是全通。

> **本节最重要的三点**：
> 1. `struct` 在机器级就是一段大块内存连段连续拿的统大盒子加上编译阶段就能写死在代码上的固定跨长位段取向标位相对偏移的获取器模式完成全盘解题。
> 2. `union` 是极度省空间的全体同站一线去占据空间的暴行使用且完全混同共享内层编码流的变形器神器。
> 3. 对齐是对底层极其硬性的规管以及空穴填充所产生的不可视间距膨胀产生的问题机制源基！
