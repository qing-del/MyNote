## C 语言数据类型在 x86-64 上的大小

在进入纷繁复杂的汇编指令前，我们必须明白我们操作的数据块"有多大"。
在 64 位机器（x86-64）上，常见基本 C 数据类型的字节大小与汇编表示如下表：

| C 声明 | x86-64 大小 (字节数) | x86 术语 | 汇编指令后缀 |
|--------|--------------------|---------|------------|
| `char` | 1 | 字节 (Byte) | `b` |
| `short` | 2 | 字 (Word) | `w` |
| `int` | 4 | 双字 (Double word) | `l` |
| `long` | 8 | 四字 (Quad word) | `q` |
| `char*` (指针) | 8 | 四字 (Quad word) | `q` |
| `float` | 4 | 单精度 | `s` |
| `double` | 8 | 双精度 | `l` |

> [!warning] 指针的大小由机器字长决定
> 不论指向的是 1 字节的 `char` 还是极大规模的复杂结构体，所有类型的**指针**在 64 位系统下统统占据 **8 字节 (64比特)** 的空间！这是因为指针存储的是绝对地址，而 64 位 CPU 的寻址总线正是 64 位宽的。

## 汇编代码的操作数大小后缀

GCC 产生的汇编代码（ATT 格式）在指令的末尾往往会带上一个单字母后缀，用于指明该指令处理的数据大小。

```gas
# 例如对于数据移动指令 mov:
movb  ..., ...  # 移动 1 字节 (Byte)
movw  ..., ...  # 移动 2 字节 (Word)
movl  ..., ...  # 移动 4 字节 (Long double word，简称 l)
movq  ..., ...  # 移动 8 字节 (Quad word)
```

**为什么要有这个后缀？**
因为相同的机器码指令在硬件眼中是无状态的，如果你在内存中抓了一把存进寄存器，系统不知道这把该抓 8 个字节（`q`），还是 4 个（`l`）。后缀精确控制了这套读写动作影响的内存槽宽度。

## 为什么 x86 中 "字（Word）" 表示 16位？

在大部分现代标准计算机科学理论中，"字（Word）" 通常指代机器的通用自然位数（比如 64 位机器的一个字长应该是 64 字节）。

但是！由于 x86 这个架构**极致的向后兼容特性（背负了40年历史包袱）**：
- 在 1978 年 16 位架构 8086 刚发布的时候，那时候**16 位**就是机器的主流能力，所以 Intel 把 16 位称为**字（Word）**。
- 后来发展到了 32 位机器（IA32），Intel 为了保证命名体系平滑，只能捏着鼻子管 32 位叫 **"双字 (Double word)"**。
- 到了 64 位时代，也只能继续往上加，管 64 位叫做 **"四字 (Quad word)"** !

> 类比：你买了一杯“中杯”星巴克。本来大中小是很符合逻辑的。结果后来推行了大杯、超大杯。而原来的中杯变成了最小配额单位。在 x86 的宇宙里，`Word = 16 bit` 就是这样的星巴克中杯，千万不能和通常计算机讨论的 64 位词意混淆。

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**JVM 规范与 C 语言平台依赖之争**：
- 在这段 CSAPP 中，指出 `long` 数据在 IA32(32位机) 占 4 字节，而在 x86-64(64位机) 占 8 字节！在 C/C++ 开发中，如果利用 `long` 的长短依赖做了底层移位算法，那么从 32 位机器迁移到 64 位时必定埋雷！
- **Java 的绝对胜利**：正是基于平台痛点，James Gosling 缔造 Java 时，在 JVM 规范里暴力焊死了：无论这台机器是 32 位、64 位 还是烤面包机，只要跑 JVM，所有的 `int` 铁打的是 4 字节，所有的 `long` 铁打的 8 字节！这段兼容性脏乱差完全由 JVM 底层针对不同操作系统的版本通过 `intptr_t` / `jlong` 等宏进行隔离，极大降低了后端程序员的心智负担。

**在底层开发依然有用武之地**：
- 当开发高性能中间件系统如 Netty 操作直接物理内存（`DirectBuffer`）、以及通过 JNI 调用 C 的本地方法库时，掌握这 8 字节的四字特性是避免内存越界（Memory Corruption）的关键。
