## Cache 的通用组织结构

一个 Cache 由 **S 组（Set）** 组成，每组有 **E 行（Line/Way）**，每行存储一个 **B 字节的数据块（Block）**，以及一个**有效位（valid bit）** 和一个 **标记位（tag）**。

$$\text{Cache 容量} = S \times E \times B \text{ 字节}$$

```
Cache 组织结构:

Cache: S=4 组, E=2 路(行/Way), B=8 字节/块

       有效位  标记位    数据块 (8 字节)
         ↓      ↓     ┌──────────────────┐
Set 0: [1] [Tag_A] | b0 b1 b2 b3 b4 b5 b6 b7 |  ← Way 0
       [1] [Tag_B] | b0 b1 b2 b3 b4 b5 b6 b7 |  ← Way 1
         ─────────────────────────────────────
Set 1: [1] [Tag_C] | b0 b1 b2 b3 b4 b5 b6 b7 |
       [0] [  --  ] |          无效           |
         ─────────────────────────────────────
Set 2: [0] [  --  ] |          无效           |
       [0] [  --  ] |          无效           |
         ─────────────────────────────────────
Set 3: [1] [Tag_D] | b0 b1 b2 b3 b4 b5 b6 b7 |
       [1] [Tag_E] | b0 b1 b2 b3 b4 b5 b6 b7 |
```

> 类比：Cache 就像图书馆的**临时书架**，分成若干**书架格（组/Set）**，每格能放几本书（路/Way）。每本书上贴着书号标签（tag）。你来借书，先找到书架格（用地址的 set 位），再逐一对比标签（tag），找到了就是命中，没找到（或格子满了）就要去总书库（主存）借。

## Cache 的三种映射方式

### 直接映射（Direct Mapped）

**E = 1**：每个 Set 只有 1 路（1 行）。

> 类比：每个邮政码区（Set）只有一个**固定的储物柜（1路）**。不同邮政区的包裹会被送到不同的柜子，但**同一邮政区的不同包裹（不同 tag）只能轮流占用同一个柜子**—— 一来一个就踢走前一个。

**优点**：电路实现简单，速度快（只需比对 1 个 tag）。
**缺点**：容易产生**冲突缺失（Conflict Miss）**。

```
示例：两个数组竞争同一组（直接映射下）

int A[8] = {...};   // 假设 A[0] 映射到 Set 2
int B[8] = {...};   // 假设 B[0] 也映射到 Set 2（巧合）

for (int i = 0; i < 8; i++) {
    sum += A[i] + B[i];  // A[i] 和 B[i] 反复互相驱逐 Set 2 中的数据！
}
// 结果：每次访问都是 Miss，命中率 = 0% ← 抖动（Thrashing）
```

---

### 组相联（Set-Associative）

**E > 1**（常见 4-路、8-路组相联）：每组有多路，同一组可同时缓存多个不同 tag 的数据块。

> 类比：每个邮政码区（Set）有 4 个储物柜（4路）。同一邮政区可以同时存放 4 个不同发件人的包裹，冲突概率大幅降低。

**这是现代 CPU Cache 最常用的方式**，L1/L2/L3 通常是 8-路或 16-路组相联。

```
4-路组相联 vs 直接映射（同一场景）:

A[i] 映射 → Set 2, Way 0
B[i] 映射 → Set 2, Way 1  ← 不同的路，互不干扰！

Set 2: [Way 0: A数据] [Way 1: B数据] [Way 2: -] [Way 3: -]
                                   ↑ 同时命中 ✅
```

---

### 全相联（Fully Associative）

**S = 1**（只有一组，E 路等于 Cache 总行数）：任何数据可以放在 Cache 的任意位置。

> 类比：整个图书馆就一个大书架，书可以放在任意位置，管理最灵活。但查找时要逐一比对所有书（所有 tag），找起来最慢（电路代价最大）。

**优点**：完全没有冲突缺失，组利用率最高。
**缺点**：电路复杂，速度慢，面积大。只用在小容量场景（如 TLB）。

---

### 三种方式对比

| 映射方式 | E（路数） | 冲突缺失 | 电路复杂度 | 典型应用 |
|---------|---------|---------|----------|---------|
| 直接映射 | 1 | 高 | 简单 | 早期 CPU Cache |
| 组相联 | 2~16 | 中/低 | 中等 | L1/L2/L3 Cache |
| 全相联 | = 总行数 | 无 | 复杂 | TLB、受害者缓存 |

## Cache 的读过程（地址解析）

一个 m 位的内存地址被分为三段，用于在 Cache 中定位数据：

$$\underbrace{t \text{ 位（标记 tag）}}_{} \underbrace{s \text{ 位（组索引 set index）}}_{log_2 S} \underbrace{b \text{ 位（块偏移 block offset）}}_{log_2 B}$$

```
地址解析示意（m=32, s=4, b=6，S=16组, B=64字节）:

32位地址:
┌──────────────────────────┬────────────┬──────────┐
│      Tag (22位)           │ Set (4位)  │ Block(6位)│
└──────────────────────────┴────────────┴──────────┘
         ↓                      ↓              ↓
   与 Cache 行的          选择哪个组        块内第几字节
   tag 比对               (Set 0~15)       (0~63)
```

**读操作过程**：
1. 用 **s 位** 找到对应的 Set
2. 遍历该 Set 内所有 Way，将各 Way 的 **tag** 与地址的 **t 位** 比对
3. 找到 tag 匹配且 **valid=1** 的行 → **Cache Hit**，用 b 位取出对应字节
4. 没找到 → **Cache Miss**，从下层加载整个 Block

## Cache 的写策略

### 写命中策略

当 CPU 写数据，而该数据已在 Cache 中（写命中）：

| 策略 | 名称 | 行为 | 特点 |
|------|------|------|------|
| **写透（Write-Through）** | Write-Through | 同时写 Cache 和下层内存 | 下层始终一致，但每次写都要访问内存，慢 |
| **写回（Write-Back）** | Write-Back | 只写 Cache，标记为"脏（dirty）"；等被驱逐时再写回 | 写快，但一致性复杂，需要脏位（dirty bit）|

> 类比：
> - **写透**：你在便签和正式文件上**同时记录**，两者永远同步，但每次记录要写两遍。
> - **写回**：你只记在便签上，加个"待正式存档"标记，等临走前才把便签内容录入正式文件——减少了写入次数，但中途可能别人看到的正式文件是旧的。

> [!info] 现代 CPU 几乎全用写回（Write-Back）
> 写透因为每次写操作都要访问主存，带宽利用率低。写回配合脏位是主流选择。

---

### 写缺失策略

当 CPU 写数据，而该数据**不在** Cache 中（写缺失）：

| 策略 | 名称 | 行为 |
|------|------|------|
| **写分配（Write-Allocate）** | 先将数据从下层加载到 Cache，再在 Cache 中写 | 配合写回使用，后续读写都命中 Cache |
| **非写分配（No-Write-Allocate）** | 直接写到下层内存，不把数据加载到 Cache | 配合写透使用，避免 Cache 污染 |

> [!tip] 黄金组合
> - **写回 + 写分配**（Write-Back + Write-Allocate）：现代 CPU 的标准选择
> - **写透 + 非写分配**（Write-Through + No-Write-Allocate）：老式/简单系统

## 真实 CPU 的 Cache 参数

以 Intel Core i7（典型 6 核）为例：

| Cache 层 | 大小 | 路数/组 | Cache Line 大小 | 访问延迟 |
|---------|------|--------|---------------|---------|
| L1 数据 Cache | 32 KB | 8-路 | 64 B | ~4 个时钟周期 |
| L1 指令 Cache | 32 KB | 8-路 | 64 B | ~4 个时钟周期 |
| L2（统一） | 256 KB | 4-路 | 64 B | ~12 个时钟周期 |
| L3（共享） | 8~32 MB | 16-路 | 64 B | ~35-50 个时钟周期 |

```
CPU 芯片内部 Cache 层次示意（双核简化）:

    ┌──────────────────────────────────────────┐
    │              CPU Die                      │
    │  ┌──────────────┐    ┌──────────────┐    │
    │  │   核心 0      │    │   核心 1      │    │
    │  │  ┌─────────┐ │    │  ┌─────────┐ │    │
    │  │  │L1I/L1D  │ │    │  │L1I/L1D  │ │    │
    │  │  └────┬────┘ │    │  └────┬────┘ │    │
    │  │  ┌────┴────┐ │    │  ┌────┴────┐ │    │
    │  │  │   L2    │ │    │  │   L2    │ │    │
    │  │  └────┬────┘ │    │  └────┬────┘ │    │
    │  └───────┼───────┘    └───────┼───────┘    │
    │          └────────────────────┘           │
    │                    ↓                       │
    │           ┌─────────────────┐             │
    │           │  L3 (共享)       │             │
    │           └────────┬─────────┘             │
    └────────────────────┼──────────────────────┘
                         ↓
                   主存 (DRAM)
```

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 中的 Cache Line 伪共享（False Sharing）**：
```java
// ❌ 伪共享问题：两个线程各自更新不同的 long，但它们在同一 Cache Line 上！
class CounterBad {
    volatile long a;  // 8字节
    volatile long b;  // 8字节 → 和 a 在同一个 64字节 Cache Line 里！
}
// Thread1 写 a → 使 Thread2 的 Cache Line 失效 → Thread2 频繁重新加载 b
// 即使 a 和 b 从不被同一个线程读写，也会互相影响 → 性能灾难

// ✅ 解决方案：填充（Padding），强制隔离到不同 Cache Line
class CounterGood {
    volatile long a;
    long p1, p2, p3, p4, p5, p6, p7;  // 7 × 8 = 56 字节填充
    volatile long b;
}
// 或者使用 @Contended 注解（JDK 8+）：
@jdk.internal.vm.annotation.Contended
class CounterBetter {
    volatile long a;
    volatile long b;
}
```

> JUC 中的 `LongAdder` 就是为了解决伪共享，用多个 Cell（每个 Cell 对齐到独立 Cache Line）分散竞争。

**写策略与 JVM 内存模型**：
- `volatile` 写操作 → 相当于强制写透（Write-Through）到"主存"（JMM 意义上的主内存）
- 普通字段写 → 类似写回（Write-Back），CPU 何时刷新到主内存不确定
- 这也是 `volatile` 保证可见性的底层对应关系

**MySQL Redo Log 写策略**：
- `innodb_flush_log_at_trx_commit=1`：每次提交都 fsync → 类比**写透（Write-Through）**，安全但慢
- `innodb_flush_log_at_trx_commit=2`：写 OS Buffer，每秒 fsync → 类比**写回（Write-Back）**，快但有 1s 数据风险
