## TCP/IP 协议族

全球因特网（Internet）是基于 **TCP/IP 协议族** 构建的。

* **IP（Internet Protocol）**：提供基本的命名方案和数据报交付机制。（主机到主机的数据封装，**不可靠**）
* **UDP（User Datagram Protocol）**：对 IP 进行了轻微扩展，能在进程间发送不可靠的数据报。
* **TCP（Transmission Control Protocol）**：在 IP 之上构建的一个复杂的协议，提供了**进程间的可靠的双向数据流**。

> [!info] 程序员视角的因特网抽象
> TCP/IP 原理极其复杂，但作为应用层程序员（编写客户端和服务器），你可以忽略所有底层细节，将其视为具有以下三个特性的黑盒：
> 1. 主机合集被映射为一个 32位 / 128位的 **IP 地址** 集合。
> 2. 这个 IP 地址集合被映射为一组人们可读的 **因特网域名**（如 `www.google.com`）。
> 3. 一台主机上的进程能够通过 **连接（Connection）** 与另一台主机上的进程进行通信。

## IP 地址与字节序问题

在 IPv4 时代，一个 IP 地址是一个 32 位的无符号整数，它存储在结构体 `in_addr` 中。

```c
// IP 地址结构体
struct in_addr {
    uint32_t s_addr; // 大端法网络字节序的 32 位 IP 地址
};
```

> [!failure] 字节序问题（Byte Ordering）
> 如果一台主机的 CPU 用**小端法 (Little-Endian)** 存储数据（如 Intel x86），另一台主机的 CPU 用**大端法 (Big-Endian)**。当小端法主机通过网络发送一个 32 位整数到大端法主机时，大端法主机解析出来的值会完全错误！

为了解决这个问题，因特网规范规定：**网络中传输的所有多字节数据类型必须使用大端法，称为网络字节序（Network Byte Order）！**
主机内存中的字节序称为 **主机字节序（Host Byte Order）**（现代 PC 绝大多数是小端法）。

我们需要使用 Unix 提供的函数进行转换：
```c
#include <arpa/inet.h>
uint32_t htonl(uint32_t hostlong);  // Host to Network Long (32位, 通常用于 IP)
uint16_t htons(uint16_t hostshort); // Host to Network Short (16位, 通常用于端口)
uint32_t ntohl(uint32_t netlong);   // Network to Host Long
uint16_t ntohs(uint16_t netshort);  // Network to Host Short
```

## IP 地址转换函数

我们通常使用点分十进制（Dotted-decimal）字符串来表示 IP，例如 *"127.0.0.1"*。
应用程序需要在**字符串**（如应用程序日志、配置）和**网络字节序的 32 位整数**（传递给底层 socket 接口）之间互相转换。

现代网络编程应使用支持 IPv4 和 IPv6 的通用函数 `inet_pton` 和 `inet_ntop` (p表示 presentation 表达格式，n表示 numeric 数值格式)：

```c
#include <arpa/inet.h>
// 将点分十进制串（src） 转为 网络字节序整数存入 dst 中
int inet_pton(AF_INET, const char *src, void *dst);

// 将网络字节序整数（src） 转为 点分十进制字符串存入 dst 中
const char *inet_ntop(AF_INET, const void *src, char *dst, socklen_t size);
```

## 域名与 DNS

因为 IP 地址很难记忆，因特网定义了一组更加人性化的名称，叫做**域名（Domain Name）**。

**域名系统（DNS, Domain Name System）** 是一个分布式的全球数据库，负责维护**域名与 IP 地址之间的映射表**。

```
因特网域名空间层次结构：
                   (未命名根)
                 /     |       \
              .com   .edu     .gov
               /       |        \
        amazon.com   mit.edu   ...
             /         |
    www.amazon.com  cs.mit.edu
```

*早期 Unix 下常用 `gethostbyname` 查 DNS，在现代代码中已被废弃。目前推荐使用支持通用协议族的 `getaddrinfo` 和 `getnameinfo` 函数。*

## 因特网连接（端口的概念）

一个 TCP 连接之所以叫连接，是因为它是点对点的、全双工的、并且是可靠的字节流。

由于一台机器上可能同时运行着 Web 浏览器、邮件客户端和 SSH 等多个网络应用，当机器网卡收到一个 TCP 数据包时，**它如何知道该交给哪个应用进程去处理？**

这就引出了**端口（Port）**的概念。端口是一个 16 位的无符号整数（范围：0 ~ 65535）。
* **知名端口（Well-known Ports，0~1023）**：保留给标准服务，比如 Web（80）、HTTPS（443）、SSH（22）。服务器必须要有 root 权限才能绑定。
* **临时端口（Ephemeral Ports）**：客户端发起连接时，由内核自动生成的源端口。

## 套接字对 (Socket Pair)

一个连接是由它两端的套接字地址唯一确定的。这对套接字地址称为**套接字对 (Socket Pair)**，可以表示为一个元组：
**(客户端IP : 客户端端口, 服务器IP : 服务器端口)**

```mermaid
flowchart LR
    subgraph 客户端端点
       A[IP: 192.168.1.5<br/>Port: 51213 (临时端口)]
    end
    subgraph 服务端端点
       B[IP: 208.216.181.15<br/>Port: 80 (Web端口)]
    end
    A <-->|TCP 连接| B
```
> [!tip] 核心识别
> **这四点结构（源IP，源Port，目的IP，目的Port）** 是操作系统网络栈用来把每一个独立的网络数据包**分发**到正确的套接字文件描述符上去的唯一凭据！
> 这个概念也解释了为什么一个哪怕只拥有 80 端口和 1 个 IP 的 Web 服务器，也能同时支持成千上万个客户端来连接：因为每个客户端的 (IP, Port) 不同，使得这个"套接字对"元组具有全局唯一性。

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 屏蔽了字节序问题**：
- 在 C 语言中，每次组装网络数据包都需要手动 `htonl/htons`。
- Java 虚拟机在底层自动帮你做好了！因为 Java 的 JVM 规范强制 Java 内存模型**天然是大端法（Big-Endian）** 的，这与网络字节序不谋而合。在 Java 中调用 Netty 写入一个整数，不用去调用任何转换函数。

**DNS 在架构中的扩展（服务发现）**：
- 在传统场景，DNS 负责 `域名 -> 物理 IP` 的转换。
- 在微服务架构（如 Spring Cloud）下，我们用 Nacos/Eureka/Consul 取代了 DNS 的位置进行**服务发现 (Service Discovery)**。这本质是一种更动态、有健康检查和心跳续约的应用层"局域 DNS 注册中心"。

**全双工连接与 WebSocket 原理**：
- 传统的 HTTP 请求-响应模型是单向驱动的，但底层的 TCP 其实是全双工的（双方可以同时互相发消息）。
- Java 中的 WebSocket 协议就是利用了本节的 TCP 属性，在经过一次 HTTP 握手后，退化成裸的 TCP 全双工收发信道建立实时聊天功能。
