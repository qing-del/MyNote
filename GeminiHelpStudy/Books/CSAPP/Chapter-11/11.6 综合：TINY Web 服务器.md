## TINY：麻雀虽小五脏俱全

CSAPP 向我们展示了一段不到 300 行的 C 语言代码 **TINY Web 服务器**。它融合了这几章教授的：网络 Socket 编程、进程与信号控制流、Unix 健壮的大 I/O 以及虚拟内存映射。

**TINY 的能力：**
- 它可以作为一个普通的网页内容提供者。
- 它可以提供静态资源（HTML、TXT、GIF 等图片渲染）。
- 它能像一个现代服务器那样接收并解析 CGI（解析计算两个数字相加并在网页返回执行求和）。虽然不适合用于生产环境（过于简单单薄，存在很多漏洞安全性），但可以作为一个**微缩版的 Tomcat 或 Nginx 核心模型**来进行分析。

## 主函数与接收连接 (main & accept)

```c
int main(int argc, char **argv) {
    int listenfd, connfd;
    char hostname[MAXLINE], port[MAXLINE];
    socklen_t clientlen;
    struct sockaddr_storage clientaddr;

    listenfd = Open_listenfd(argv[1]);  // 初始化用来招揽客人的“大堂经理”描述符
    while (1) {
        clientlen = sizeof(clientaddr);
        // 主进程在这个循环死等到有客户端 tcp 发起联系成功
        connfd = Accept(listenfd, (SA *)&clientaddr, &clientlen); // 返回一对一的连接线
        
        Getnameinfo((SA *) &clientaddr, clientlen, hostname, MAXLINE, port, MAXLINE, 0);
        printf("Accepted connection from (%s, %s)\n", hostname, port);
        
        // 核心转发：执行当前用户的请求命令任务！
        doit(connfd);  
        
        Close(connfd); // 干完一次请求了，关闭它 (这叫做 HTTP 的短连接或事务！)
    }
}
```
**重点**：TINY 是一个**迭代式（Iterative）服务器**。这意味着在执行 `doit()` 时它是串行的，如果有两个用户同时访问，第二个用户将被挂在等待队列里直到前一个干完活 close() 掉，不具备并发能力。（并发能力由第 12 章讲解引入）。

## 事务处理核心函数 (doit)

这部分完成从对网络流的一行一行解析，到提取到底要查阅什么文件。

```c
void doit(int fd) {
    // 1. 从网络流 fd 里面用 RIO 按行读取报文
    rio_readinitb(&rio, fd);
    rio_readlineb(&rio, buf, MAXLINE); 
    
    // 2. 将报文： GET /xxx HTTP/1.1 切分为具体的三个字符串数组提取出来
    sscanf(buf, "%s %s %s", method, uri, version);       

    // 3. 拦截请求判定（如果不是 GET，拒绝响应功能）
    if (strcasecmp(method, "GET")) {                     
        clienterror(fd, method, "501", "Not implemented", "Tiny does not implement this method");
        return;
    }                                                    

    // 忽略剩下的其他繁杂的 Header 头部信息
    read_requesthdrs(&rio);                              

    // 4. 解析 URI！这里判断客户到底是拿静态的 HTML，还是传参想跑动态的 CGI 加法程序
    is_static = parse_uri(uri, filename, cgiargs);       
    
    // 5. 调用系统底层命令：stat 函数检查磁盘文件是不是在
    if (stat(filename, &sbuf) < 0) {                     
        clienterror(fd, filename, "404", "Not found", "Tiny couldn't find this file");
        return;
    }                                                    

    // 6. 确定有东西之后，利用解析判定，分向不同分支处理！
    if (is_static) { 
        // 检查文件是否有读权限？若是普通文件...
        serve_static(fd, filename, sbuf.st_size);        
    } else {
        // 提供动态计算内容程序
        serve_dynamic(fd, filename, cgiargs);            
    }
}
```
它实现了我们在 [[11.5 Web 服务器]] 内描述的一套标准的基于字符串解析的处理模板。任何一门现代 Web 语言也是从这个思路出发的。

## 静态服务实现 (serve_static)

这里的核心并不是教你如何写入 Socket 流，而是一个**极其精彩的 Unix I/O 零拷贝优化**的缩影，利用了你在第 9 章学习的 `mmap`！

```c
void serve_static(int fd, char *filename, int filesize) {
    // ... 发送 HTTP 的响应头部各种 OK 和长短标识等，略

    // 发送主体数据时！
    int srcfd = Open(filename, O_RDONLY, 0); 
    
    // 不用 malloc 和普通 read，直接把文件内容通过虚拟地址建立为内存映射 mmap！
    char *srcp = Mmap(0, filesize, PROT_READ, MAP_PRIVATE, srcfd, 0); 
    Close(srcfd); // mmap映射完了，不需要 fd 就可以关闭，防止文件描述符泄漏泄漏
    
    // 从已经载入虚拟内存的数据缓存上原样倒入 socket 网络流里面发生回传！
    Rio_writen(fd, srcp, filesize);   
    // 用完回收资源
    Munmap(srcp, filesize);           
}
```

> [!tip] 为什么要用 Mmap 而不是普通的由用户创建一个 buf [] 然后 read 放进去再 write 传给 socket？
> 使用我们 [[9.8 内存映射]] 的知识：Mmap 将磁盘文件与虚拟内存绑定，大大避免了内核态和用户态不必要的大量缓冲区数据拷贝往复移动！它的系统开销远小于普通的文件流处理。

## 动态服务实现 (serve_dynamic，即 CGI 的经典落实)

```c
void serve_dynamic(int fd, char *filename, char *cgiargs) {    
    // 发送只有短短四行的头部标识（说明这次返回的是一个代码打印结果）。
    sprintf(buf, "HTTP/1.0 200 OK\r\n"); 
    // 发送头到客户端
    Rio_writen(fd, buf, strlen(buf));
    // 后面不加任何报文尾部响应，因为内容由那个 C 程序单独产生。
    
    if (Fork() == 0) { // 核心！产生一个专门干苦力的子进程！！
        // 子进程将父进程提取到的 cgiargs URL 参数加载到环境变量（这个是进程间共用的特定桥梁）
        setenv("QUERY_STRING", cgiargs, 1); 
        
        // 我们之前学过，1 号描述符就是标准控制台输出 STDOUT，把它强制导引给 fd 这个连接客户端的网络！
        Dup2(fd, STDOUT_FILENO);
        
        // 原地发生一次可执行程序的调包置换。去跑那个叫做 filename 的 C语言生成的文件。
        // 原来里面那个程序不管里面怎么写了个 printf 把两个数加起来输出！输出的字符串会因为上一步顺着掉线发飞网线到达给客户！
        Execve(filename, emptylist, environ); 
    }
    
    // 父进程挂等在这个地方，别让他成为僵尸进程去收尸
    Wait(NULL); 
}
```

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**TINY 是阻塞 I/O 服务器的反面典型（BIO 弊端）**：
- 在这段代码中，如果某个建立好网络连接的恶意客户端，慢吞吞地每隔 100 秒给你发送 HTTP 字符串的一个字母，因为父进程在 `rio_readlineb(&rio, buf)` 由于没有遇到换行一直在 CPU 原地睡眠挂起（Block）。这就导致：这唯一的单线程整个被他一个人占据！！
- 这就是 Java 早期 `java.io.InputStream` 开发的网络应用最容易被攻击的点，称为**"慢速攻击(Slowloris)"**。
- 这也是当今一切工业服务器：如 Go 协程、Java NIO 或 Nginx 底层都强行绑定使用**异步或多路复用（IO Multiplexing）与全事件处理结合**的万恶之源起因。

**Spring MVC DispatcherServlet 与 doit 的魂之契合**：
- 请注意 TINY 中的 `is_static = parse_uri(...)`。这和你在 Java 最常见的拦截控制器（如 `@GetMapping("/api")` 被解析路由，而静态文件在资源文件夹 `Resources` 存放直接返回）是完完全全相同的体系。只不过前者由 Spring 在上方的 JVM 端进行了庞大规模的前置路由表反射建立。
