## Web 基础与 HTTP 简介

Web 客户端和服务器之间的交互用的是一个基于文本的应用层协议，叫做 **超文本传输协议（HTTP，Hypertext Transfer Protocol）**。

* Web 客户端就是网络浏览器（Browser）。
* Web 服务器在后台等待，被前面的套接字抽象中接收到来自浏览器的连接。
* **HTTP 是构建在 TCP 连接基础之上的**：TCP 提供可靠交付，HTTP 提供语意规则（即双方必须交换固定格式的请求行、响应状态等文本文字）。

---

## Web 内容（静态内容与动态内容）

对 Web 服务器而言，所有能给客户端发送的数据被称为**内容（content）**。每个内容片段关联一个 MIME 类型（例如 `text/html`, `image/png`）。

Web 服务器提供内容有两种方式：

1. **服务静态内容（Static Content）**：
   取得一个本来就已经存在的磁盘文件，然后直接发还给客户端。比如用户请求的是一个 `index.html` 或者 `.jpg`。这种请求是在单纯测试后端的 I/O 效率。

2. **服务动态内容（Dynamic Content）**：
   运行一个可执行程序（在 CSAPP 这章特指的是早期的 CGI 程序），然后读取这个程序打印输出的内容，把它发送给客户端。
   用户给出的可能是 `/login?user=abc`，服务器发现这是个程序，传参调用后得到 HTML 再甩给客户端。

> [!tip] 区分标志：URL（通用资源定位符）
> 无论是请求静态的文本，还是运行动态的程序服务器查询。他们都依靠 URL 被寻址。
> 例如：`http://www.google.com:80/index.html` 表明使用 HTTP 在 80 端口要获取根目录下静态文件 `/index.html`。

## HTTP 事务过程详解

因为 HTTP 基于传统的文本行序列，所以我们可以清晰地写出来请求长什么样子。这通常是排查 Web 服务器问题的利器。

### HTTP 请求 (Request)

一个 HTTP 请求由 **一个请求行**、后面跟随 **零个或多个请求报头**（Headers），以及一个用于标记结束的空行（`\r\n`）组成。

```http
GET / HTTP/1.1                    <-- 请求行(Method URI Version)
Host: www.aol.com                 <-- 报头
User-Agent: Mozilla/5.0           <-- 报头
Connection: close                 <-- 报头
                                  <-- 空结束文本行 \r\n (极其重要)
```

- 方法（Method）：最常见的是 GET（请求读取/返回）、POST（提交数据变更）、OPTIONS 等。
- URI 是被请求的具体路径。
- **Connection: close** 是一个古老但常见的报头，告诉服务器在把请求处理响应发回之后，就把这根套接字（connfd）关闭断掉。这种方式非常消耗建立连接的资源；如果带上 `Connection: Keep-Alive` 则长连接被重用。

### HTTP 响应 (Response)

响应的结构与请求呈镜面映射。它由 **一个状态行**，后面跟着 **零个或多个响应报头**，再接一个**空行**，最后才是**响应主体**数据本身组成。

```http
HTTP/1.0 200 OK                   <-- 状态行(Version Status-Code Status-Message)
MIME-Version: 1.0                 <-- 报头
Server: Tiny Web Server           <-- 报头
Content-length: 120               <-- 报头（关键！告诉浏览器需要收到多少字节）
Content-type: text/html           <-- 报头（控制浏览器对主体的渲染方式）
                                  <-- 空行表示报头结束了
<html>...数据本体...</html>          <-- 响应主体 (HTML/图片等二进制或字符串)
```

常见状态码（Status-Code）：
- **200 OK**：成功。
- **301 Moved Permanently**：重定向。
- **403 Forbidden**：不具备资源读权限（比如 UNIX stat 的权限为没读）。
- **404 Not found**：服务器磁盘上的文件不存在找不到。

## 服务端提供动态内容 (CGI 标准)

如何向客户端提供动态内容？C 语言时代的做法是极其纯粹却暴力的模型：**公共网关接口（CGI, Common Gateway Interface）**。

由于这是 C 语言书，早期动态建库的解法是：
1. **客户端如何给服务端参数**：通过 URI 的 `?Key=Value&A=B`。
2. **服务端如何将参数传给子程序**：
   - Web 服务器程序在收到该请求时，`fork()` 派生一个子进程！
   - 父进程设置环境变量（如 `setenv("QUERY_STRING", "Key=Value", 1)`）。
   - 子进程里面执行 `execve` 调用那个针对业务书写的 CGI 可执行文件（比如用 C 或者 Perl 写的）。
3. **子程序的输出如何发回客户端套接字**：
   - Web Server 也就是父进程，在产生那个干活的子进程之前，利用我们在 [[10.9 IO 重定向]] 学习的最强魔法：调用 `dup2(connfd, STDOUT_FILENO)`。
   - 这意味着，**那个 CGI 程序里面任何写往屏幕控制台的 `printf("<html>")` 函数调用，都会由于 stdout 被截胡，全部流向了那个连接客户端的网络套接字！** 

> 类比：你开了一家代办公司（Web 服务器）。之前你只是从资料库搬运现成报告（提供静态内容）。今天客户要一份现算的数据报表（请求动态内容）。于是，你为了接待这个顾客，专门招收了一个员工（fork 出子进程），你把客户的需求单子贴在这个员工的脑门上（环境变量设置参数），并且把员工的工作口径直接连线到了那名客户对面的通讯喇叭里（dup2() 拦截输出）。这名员工算完之后，只需要大声用嘴正常报出来（printf 调用），计算好的报单内容就由于喇叭连线的重定向直接推给客户去结算了！

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Servlet 与 CGI 的架构代差（血泪优化史）**：
- CSAPP 提到的 CGI 虽然实现了逻辑但是由于：**每一个 Web 动态请求，服务器都得在底层 OS 系统去 fork() 进而 execve() 一个重量级的 C 程序进程！** 一旦访问量并发超过 1000 就会让内存和进程陷入崩溃，慢出天际！
- **Java 的 Servlet 机制干碎了 CGI 的垄断**：Tomcat 服务器采用常驻的 JVM 线程池。面对来临请求，不 fork 系统级进程！而是创建一个分配极其廉价的用户态 Java 线程（甚至在今天转变为虚线程），将其解析好的 HttpServletRequest 作为方法传参送入 Controller / Servlet。这不仅保留了 CGI 获取底层字符串数据的全能力，更是性能的百倍跃迁。

**关于头部 Content-length 的致命坑点**：
- 如果 Java Spring 写的响应包的 Content-Length 数值比实际数据长，浏览器拿到响应后发现没读到底，由于是长连接它会一直进行空阻塞傻等在那里，直到出现 Socket Timeout。如果短了则会阶段截断后面的正文字节！框架内部对此帮我们填补了多少工作量可想而知。

**HTTP 的无状态性与 Redis 的崛起**：
- HTTP 是一个彻底无状态协议。每一次建立的 TCP 也是新的事务。服务器完全不认识第二个请求是否是刚才刚点过商品的人。对于需要在前后操作中连续维护客户订单属性的需求，我们在现在的 Java 后端全部引入 Redis，由 Session 标识加分布式缓存存储业务身份完成粘性。
