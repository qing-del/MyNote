## Core i7 的地址翻译硬件

Intel Core i7 处理器的虚拟内存参数：

| 参数 | 值 |
|------|---|
| 虚拟地址位数 | 48 位（实际寻址，64 位规范形式） |
| 物理地址位数 | 52 位（最大支持 4 PB 物理内存） |
| 页面大小 | 4 KB（标准）/ 2 MB / 1 GB（大页） |
| 页表级数 | 4 级（PML4 → PDPT → PD → PT） |
| TLB | L1 ITLB 128 条 / L1 DTLB 64 条 / L2 STLB 1536 条 |
| 页表基地址寄存器 | CR3 |

```
Core i7 的地址翻译全景图:

  CPU ── VA(48位) ──► L1 TLB ──命中──► PA(52位) ──► L1 Cache ──► 数据
                         │                              │
                       未命中                          未命中
                         │                              │
                         ▼                              ▼
                      L2 TLB                        L2 Cache
                         │                              │
                       未命中                          未命中
                         │                              ▼
                         ▼                          L3 Cache
                     四级页表遍历                        │
                    （Page Walk）                     未命中
                         │                              │
                         ▼                              ▼
                     获得 PPN                        主存 DRAM
```

## 四级页表结构

```
48 位虚拟地址的分解:

  ┌──────┬──────┬──────┬──────┬──────────────┐
  │ VPN1 │ VPN2 │ VPN3 │ VPN4 │    VPO       │
  │ 9位  │ 9位  │ 9位  │ 9位  │   12位       │
  └──┬───┴──┬───┴──┬───┴──┬───┴──────────────┘
     │      │      │      │
     ▼      │      │      │
  PML4表    │      │      │  每级 512 条目（2^9）
  (CR3指向) │      │      │  每条目 8 字节（64位）
     │      ▼      │      │  每级页表 = 512×8 = 4KB = 恰好一个物理页！
     │    PDPT表   │      │
     │      │      ▼      │
     │      │    PD 表    │
     │      │      │      ▼
     │      │      │    PT 表
     │      │      │      │
     │      │      │      ▼
     │      │      │  PTE → PPN(40位)
     │      │      │      │
     └──────┴──────┴──────┘
                          + VPO(12位) = PA(52位)
```

```mermaid
flowchart LR
    CR3["CR3 寄存器<br/>PML4 物理基地址"] --> PML4["PML4 表<br/>512 条目"]
    PML4 -->|"VPN1 索引"| PDPT["PDPT 表<br/>512 条目"]
    PDPT -->|"VPN2 索引"| PD["PD 表<br/>512 条目"]
    PD -->|"VPN3 索引"| PT["PT 表<br/>512 条目"]
    PT -->|"VPN4 索引"| PTE["PTE<br/>PPN + 权限位"]
    PTE -->|"PPN + VPO"| PA["物理地址"]
```

> [!info] 页表条目（PTE）的关键位
> 每个 64 位的 PTE 包含：
> | 位域 | 名称 | 说明 |
> |------|------|------|
> | 位 0 | **P（Present）** | 1=页在内存中，0=不在（触发缺页） |
> | 位 1 | **R/W** | 0=只读，1=可读写 |
> | 位 2 | **U/S** | 0=内核态，1=用户态可访问 |
> | 位 5 | **A（Accessed）** | 该页是否被访问过（OS 用于 LRU 判断） |
> | 位 6 | **D（Dirty）** | 该页是否被修改过（写回时需要判断） |
> | 位 63 | **XD（Execute Disable）** | 1=该页不可执行（防注入） |
> | 位 12~51 | **PPN** | 下一级页表或物理页的物理基地址（40位） |

## Linux 进程的虚拟地址空间布局

```
Linux x86-64 进程虚拟地址空间（48 位）:

  0xFFFFFFFFFFFF ┌─────────────────────────┐ ← 高地址
                 │      内核空间            │ 所有进程共享相同映射
                 │   （内核代码、数据、      │ 用户态无法访问
                 │    物理内存直接映射等）    │
  0xFFFF800000000├─────────────────────────┤
                 │    （非规范地址空洞）      │ 不可访问
  0x7FFFFFFFFFFF ├─────────────────────────┤
                 │       用户栈 ↓           │ 向低地址增长
                 │                         │
                 │   共享库 / mmap 区域     │ libc.so, libpthread.so...
                 │                         │
                 │       用户堆 ↑           │ brk / mmap, 向高地址增长
                 ├─────────────────────────┤
                 │    .bss（未初始化数据）   │
                 │    .data（已初始化数据）  │
                 │    .text（代码段）        │
  0x400000       ├─────────────────────────┤ ← 程序入口
  0x000000       └─────────────────────────┘ ← 低地址（NULL 页，不映射）
```

> [!warning] 内核空间映射
> 虽然每个进程的虚拟空间中都包含内核空间（高地址部分），但映射的物理页是**相同的**。用户态进程无法访问内核空间（SUP 位 = 1）。
> 这个设计使得**系统调用时不需要切换页表**——只需从用户态切到内核态，地址翻译机制自动生效。

Linux 内核用 `mm_struct` 和一个 `vm_area_struct` 链表来管理每个进程的虚拟地址空间中的各个区域（Area）：

```
Linux 内核的内存管理数据结构:

  task_struct（进程描述符）
      │
      └──► mm_struct（内存描述符）
              │
              ├── pgd: 一级页表（PML4）物理地址（写入 CR3）
              │
              └── mmap: vm_area_struct 链表
                    │
                    ├── VMA 1: [0x400000, 0x401000) .text  r-x
                    ├── VMA 2: [0x601000, 0x602000) .data  rw-
                    ├── VMA 3: [0x7f...., 0x7f....]  libc  r-x
                    ├── VMA 4: [heap 区域]            rw-
                    └── VMA 5: [stack 区域]           rw-
```

## Linux 的缺页处理

```mermaid
flowchart TD
    A["MMU 触发缺页异常"] --> B["内核调用 do_page_fault()"]
    B --> C{"访问的地址是否在\n某个 VMA 范围内？"}
    C -->|不在| D["发送 SIGSEGV → 段错误 💥"]
    C -->|在| E{"访问权限是否匹配<br/>VMA 的权限？"}
    E -->|不匹配| F["发送 SIGSEGV → 保护故障 💥"]
    E -->|匹配| G["合法缺页 → 正常处理"]
    G --> H["选择牺牲页 → 换入目标页"]
    H --> I["更新页表 → 重新执行指令 ✅"]
```

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**JVM 堆在进程地址空间中的位置**：
- `-Xms/-Xmx` 指定的 JVM 堆位于用户空间的 mmap 区域
- JVM 通过 `mmap` 预留一大块连续虚拟地址 → 但物理页按需分配
- `-XX:+AlwaysPreTouch` 启动时强制触碰所有页面 → 预先触发缺页 → 避免运行时缺页延迟

**`/proc/<pid>/maps` 查看 JVM 内存布局**：
```
典型 JVM 进程的 maps 输出:
00400000-00401000 r-xp ...  java           ← JVM 启动器代码段
7f....-7f....    r-xp ...  libjvm.so       ← JVM 核心库
7f....-7f....    rw-p ...  [heap]          ← JVM 堆（部分，brk 分配）
7f....-7f....    rw-p ...  [anon]          ← JVM 堆（mmap 分配的大块）
7fff...-7fff.... rw-p ...  [stack]         ← 主线程栈
```

**NUMA 感知与内存分配**：
- 多 CPU 系统中，物理内存分布在不同 NUMA 节点
- JVM 的 GC 线程如果频繁访问远端 NUMA 节点的内存 → 性能下降
- `-XX:+UseNUMA` 让 JVM 尽量在本地 NUMA 节点分配内存
