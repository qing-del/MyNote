## 地址空间的定义

**地址空间（Address Space）** 是一个**非负整数的有序集合**：`{0, 1, 2, ..., N-1}`。

如果地址空间中的整数是连续的，我们称之为**线性地址空间（Linear Address Space）**——这是现代系统的标准设计。

> 类比：你有一栋楼，每个房间从 0 号开始编号到 N-1 号。这些编号的集合就是"地址空间"。楼本身不存储东西，但编号让你能唯一标识每个房间。
> CS 术语：地址空间是一种**抽象概念**，它定义了可以寻址的范围。本身不涉及存储内容，只是一个"编号系统"。

## 虚拟地址空间与物理地址空间

现代系统有两个地址空间：

| 地址空间 | 大小 | 由什么决定 | 说明 |
|---------|------|-----------|------|
| **虚拟地址空间** | N = 2^n | CPU 地址位数 n（如 n=64） | CPU 能产生的所有虚拟地址的集合 |
| **物理地址空间** | M = 2^m | 实际物理内存大小 m（如 m=36） | 对应实际 DRAM 中的所有物理位置 |

```
地址空间大小对比:

虚拟地址空间（64位系统，但实际使用 48 位）:
  2^48 = 256 TB 的虚拟空间
  {0, 1, 2, ..., 2^48 - 1}

物理地址空间（假设 16GB 物理内存）:
  2^34 ≈ 16 GB
  {0, 1, 2, ..., 2^34 - 1}

关键洞察：虚拟空间 >> 物理空间
  256 TB >> 16 GB（差了 16384 倍！）
```

> [!important] 虚拟地址空间通常远大于物理地址空间
> 这是虚拟内存的核心设计：让每个进程**以为**自己有巨大的内存空间（虚拟），但实际使用的物理内存远小于此。OS 和硬件协作管理映射关系。

> [!info] x86-64 的 48 位 vs 64 位
> 虽然 x86-64 CPU 的地址线有 64 位，但目前实际只使用低 48 位（256TB）用于虚拟寻址。高 16 位必须是第 47 位的符号扩展（**规范形式地址 Canonical Address**）。
> 这意味着地址空间被分成两半：
> - 低半部分：`0x0000000000000000` ~ `0x00007FFFFFFFFFFF`（用户空间）
> - 高半部分：`0xFFFF800000000000` ~ `0xFFFFFFFFFFFFFFFF`（内核空间）
> - 中间部分是"空洞"，访问会触发故障

## 为什么需要两套地址空间

```mermaid
flowchart TB
    subgraph "没有虚拟内存的世界"
        P1A["进程 A: 地址 0x1000"] --> MEM1["物理内存 0x1000"]
        P2A["进程 B: 地址 0x1000"] --> MEM1
        P1A -.->|💥 冲突!| P2A
    end

    subgraph "有虚拟内存的世界"
        P1B["进程 A: 虚拟地址 0x1000"] -->|翻译| MEM2["物理地址 0x5000"]
        P2B["进程 B: 虚拟地址 0x1000"] -->|翻译| MEM3["物理地址 0x9000"]
        P1B -.->|✅ 隔离| P2B
    end
```

虚拟地址空间解决了三个核心问题：

1. **隔离性**：每个进程有独立的虚拟地址空间，进程 A 的地址 0x1000 和进程 B 的地址 0x1000 映射到不同的物理地址
2. **灵活性**：OS 可以自由安排物理内存的使用，不受进程虚拟地址的约束
3. **简洁性**：每个进程的内存布局可以完全统一（代码段、数据段、堆、栈的位置固定）

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**JVM 进程的虚拟地址空间布局**：
- 每个 JVM 进程独享自己的虚拟地址空间
- 堆（`-Xms/-Xmx`）、元空间（MetaSpace）、线程栈都在虚拟地址空间中各占一段
- 多个 JVM 进程（如微服务）可以同时运行，虚拟地址互不冲突 → 这就是地址空间隔离

**容器与虚拟地址空间**：
- Docker 容器中的进程仍然有完整的虚拟地址空间
- 容器通过 cgroup 限制物理资源使用，但虚拟空间不受限制
- 这就是为什么容器内的 JVM 可能误判可用内存
