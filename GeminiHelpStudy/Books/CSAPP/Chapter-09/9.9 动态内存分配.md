## 为什么需要动态内存分配

程序运行前往往无法确定需要多少内存：
- 数组大小取决于用户输入
- 数据结构的节点数量在运行时变化
- 网络请求的 payload 大小不可预知

> 类比：你搬进新房子，不知道以后会买多少书。你不想一开始就买一个占满整面墙的大书架（静态分配），而是需要时再买一层新架子（动态分配），不需要时拆掉归还（释放）。
> CS 术语：**动态内存分配**允许程序在运行时从堆中申请任意大小的内存块，使用完后释放。

## 堆（Heap）与分配器

**堆**是进程虚拟地址空间中一段可以动态增长的区域，位于未初始化数据段（.bss）之上。

```
堆在进程地址空间中的位置:

  高地址 ────────────────────────
           │      栈 ↓         │
           │                   │
           │  mmap 区域         │
           │                   │
  brk ───► │  ─── 堆顶 ──────  │ ← brk 指针标记堆的当前顶部
           │      堆 ↑         │ ← 向高地址增长
           │                   │
           │  .bss / .data     │
           │  .text            │
  低地址 ────────────────────────
```

**分配器（Allocator）** 维护堆，处理 `malloc` 和 `free` 请求。分配器分两类：

| 类型 | 谁释放 | 例子 |
|------|-------|------|
| **显式分配器** | 程序员手动释放（`free`） | C 的 `malloc/free`，C++ 的 `new/delete` |
| **隐式分配器** | 运行时自动检测并释放（GC） | Java/Go/Python 的垃圾收集器 |

## malloc 和 free

```c
#include <stdlib.h>

// 分配：返回一个至少 size 字节的内存块的指针
// 返回的块按 16 字节对齐（x86-64）
void *malloc(size_t size);

// 释放：释放 ptr 指向的块（ptr 必须是 malloc/realloc 返回的）
void free(void *ptr);

// 使用示例
int *arr = (int *)malloc(10 * sizeof(int));  // 分配 40 字节
if (arr == NULL) {
    // 分配失败（堆空间不足）
    exit(1);
}
// 使用 arr...
free(arr);  // 释放
arr = NULL; // 好习惯：防止悬挂指针
```

> [!warning] malloc 的约束
> 1. 不能控制已分配块的位置和顺序
> 2. 必须**立即响应**请求（不能缓冲或重排请求）
> 3. 只能在空闲内存中分配
> 4. 必须**对齐**返回的块（通常 8 或 16 字节对齐）
> 5. 不能修改或移动已分配的块

## 碎片问题

**碎片（Fragmentation）** 是动态内存分配的核心问题——空闲内存足够，但无法满足分配请求。

```
内部碎片 vs 外部碎片:

内部碎片（Internal）:
  请求 5 字节 → 分配器给了 8 字节（为了对齐）
  ┌──────────────┐
  │ 5B 有效 │3B浪费│  ← 内部的空间浪费
  └──────────────┘

外部碎片（External）:
  堆中有 3 个空闲块，每个 4B = 共 12B 空闲
  ┌────┬──────┬────┬──────┬────┬─────┐
  │used│free 4│used│free 4│used│free 4│
  └────┴──────┴────┴──────┴────┴─────┘
  请求 10B → 失败！虽然总空闲 12B，但没有连续的 10B 空间
  ← 外部碎片：空闲块不连续
```

> 类比：你的停车场（堆）有 3 个分散的空位，每个能停 1 辆小车。来了一辆大巴需要连续 3 个车位 → 停不下（外部碎片）。或者有个车位能停 SUV 但你只停了一辆自行车 → 剩余空间浪费（内部碎片）。
> CS 术语：**内部碎片**由对齐和分配策略导致，可以精确计算。**外部碎片**取决于未来的请求序列，难以预测。

## 分配器的实现策略

### 隐式空闲链表

用每个块的**头部（Header）** 存储块大小和是否已分配：

```
隐式空闲链表的块结构:

  ┌──────────────────────────────────┐
  │ Header: [块大小 | 已分配位 a]     │  4 字节
  ├──────────────────────────────────┤
  │         有效载荷（Payload）        │
  │           或空闲空间              │
  ├──────────────────────────────────┤
  │         填充（Padding，可选）      │
  └──────────────────────────────────┘

  示例: 块大小=32, 已分配=1
  Header = 0x00000021  (32 | 1 = 0x21)
  （利用了对齐保证最低位为 0 的特性来存储分配位）
```

```
隐式空闲链表遍历:

  堆起始                                            堆结尾
  ┌────────┬────────┬────────┬────────┬────────┬───┐
  │ 32/1   │ 16/0   │ 48/1   │ 32/0   │ 64/1   │0/1│
  │(已分配) │(空闲)  │(已分配) │(空闲)  │(已分配) │终止│
  └────────┴────────┴────────┴────────┴────────┴───┘
  ──────────────────────────────────────────────────►
  通过每个块的 Header 中的大小字段，逐块遍历整个堆

  查找空闲块：需要线性遍历整个堆 → O(n) ← 缺点！
```

### 显式空闲链表

在**空闲块**中维护指向前一个和后一个空闲块的指针（双向链表），已分配块不需要这些指针：

```
显式空闲链表:

  只链接空闲块:
  ┌────┐    ┌────┐    ┌────┐
  │ 空闲├───►│空闲├───►│空闲│
  │ 块1 │◄───┤块2 │◄───┤块3 │
  └────┘    └────┘    └────┘
     ↑         ↑         ↑
  ┌──┴──┬──────┴──┬──────┴──┬──────┐
  │free │ alloc  │ free   │ alloc │ free  │  ← 堆中的实际布局
  └─────┴────────┴────────┴──────┘

  优点：查找空闲块只需遍历空闲链表 → 比隐式快得多
  分配时间从 O(总块数) 降低到 O(空闲块数)
```

### 分离空闲链表

**分离适配（Segregated Fits）**：维护多个空闲链表，每个链表只存储特定大小范围的空闲块。

```
分离空闲链表:

  大小类        空闲链表
  ───────────────────────────────
  {1~8}    →   [8] → [8] → [4]
  {9~16}   →   [16] → [12]
  {17~32}  →   [32] → [24] → [20]
  {33~64}  →   [48]
  {65~∞}   →   [128] → [256]

  分配 20 字节:
  1) 查找 {17~32} 链表
  2) 找到第一个 ≥ 20 的块 → [24]
  3) 分配（如果合适可分割）

  优点：近似最佳适配，时间接近 O(1)
```

> [!info] glibc 的 ptmalloc2
> Linux 标准库的 `malloc` 实现（ptmalloc2）使用分离空闲链表：
> - 小块（≤ 512 字节）→ fastbin 或 smallbin 管理
> - 大块（> 512 字节）→ largebin 管理
> - 超大块 → 直接 `mmap` 分配（不走堆）

## 放置策略

当找到多个满足条件的空闲块时，使用哪一个？

| 策略 | 描述 | 优缺点 |
|------|------|--------|
| **首次适配** | 从头遍历，找到**第一个**足够大的空闲块 | 速度快，但前端碎片多 |
| **下一次适配** | 从上次分配位置继续搜索 | 比首次适配更均匀 |
| **最佳适配** | 遍历所有空闲块，选择**最小的**足够大的块 | 碎片少，但搜索慢 |

## 分割与合并

**分割**：如果找到的空闲块远大于请求，将其**分割**为两部分——一部分分配，剩余部分成为新的空闲块。

**合并（Coalescing）**：释放一个块时，检查相邻块是否也是空闲的，如果是就**合并**成一个更大的空闲块，减少外部碎片。

```
合并的四种情况:

  当前释放块: m2

  情况1:  [已分配][  m2  ][已分配]  → 不合并
  情况2:  [已分配][  m2  ][  空闲 ]  → m2 与后块合并
  情况3:  [  空闲 ][  m2  ][已分配]  → 前块与 m2 合并
  情况4:  [  空闲 ][  m2  ][  空闲 ]  → 三块合并为一
```

> [!tip] 边界标记（Boundary Tag）
> 为了高效实现**与前一块的合并**，可以在每个块的**尾部（Footer）** 也存储一份 Header 信息。
> 这样释放当前块时，可以立即查看前一块的尾部来判断它是否空闲 → **常数时间合并**。
> 优化：已分配块不需要 Footer（通过在 Header 中用一个位记录前一块是否已分配）。

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**JVM 的堆内存管理远超 malloc**：
- `malloc` 是通用分配器，JVM 的分配器（如 G1、ZGC）是**面向 GC 优化**的专用分配器
- JVM 使用 **TLAB（Thread-Local Allocation Buffer）** → 每个线程有独立的分配区域 → 无锁分配
- JVM 的对象分配 ≈ **指针碰撞（Bump Pointer）**：在 TLAB 中直接移动指针 → O(1)

**JVM GC 中的碎片问题**：
- CMS GC 使用标记-清除（Mark-Sweep）→ 产生外部碎片 → 可能触发 Full GC 压缩
- G1 GC 使用 Region 化 + 复制算法 → 通过复制消除碎片
- ZGC/Shenandoah 使用着色指针 + 读屏障 → 并发压缩 → 几乎无 STW

**Netty 的 PooledByteBufAllocator**：
- Netty 在用户态实现了类似 malloc 的内存池
- 使用 **Jemalloc 风格**的分离适配 + Slab 分配
- 小对象用 Tiny/Small 缓存，大对象直接分配 → 减少 GC 压力

**MySQL 的内存分配**：
- InnoDB 使用自己的内存管理器（`mem_heap_t`），基于 mmap 和内部链表
- 避免频繁调用系统 malloc/free（系统调用开销大）
