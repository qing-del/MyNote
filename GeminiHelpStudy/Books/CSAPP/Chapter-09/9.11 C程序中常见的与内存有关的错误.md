## 错误详解与示例

### 1. 解引用坏指针

```c
// 经典的 scanf 错误
int val;
scanf("%d", val);    // ❌ 应该传 &val，这里把 val 的值当地址用
// val 未初始化，值可能是 0x12345678 → 写入随机地址 → 段错误或数据损坏
scanf("%d", &val);   // ✅ 正确：传变量的地址
```

### 2. 读未初始化的内存

```c
// 假设 calloc ≠ malloc：calloc 会初始化为 0，malloc 不会
int *arr = (int *)malloc(10 * sizeof(int));
// arr 中的值是垃圾值！不是 0！
for (int i = 0; i < 10; i++) {
    sum += arr[i];   // ❌ 读取垃圾值
}
// 应使用 calloc 或手动 memset(arr, 0, 10 * sizeof(int))
```

### 3. 栈缓冲区溢出

```c
char buf[8];
gets(buf);  // ❌ 如果输入 > 7 个字符 → 溢出写入 buf 之后的栈帧
            // → 覆盖返回地址 → 控制流劫持（安全漏洞！）
```

> [!caution] 缓冲区溢出是历史上最危险的安全漏洞类型
> Morris Worm（1988）、Code Red（2001）、Heartbleed（2014）等重大安全事件的根因。
> 防御手段：栈金丝雀（Stack Canary）、ASLR、NX 位（[[9.5 虚拟内存作为内存保护的工具]]）。

### 4. 假设指针和它们指向的对象大小相同

```c
int **matrix = (int **)malloc(n * sizeof(int));  // ❌ sizeof(int) = 4 字节
// 应该是 sizeof(int *) = 8 字节（64 位系统）！
int **matrix = (int **)malloc(n * sizeof(int *)); // ✅
```

### 5. 造成差一（Off-by-One）错误

```c
int arr[10];
for (int i = 0; i <= 10; i++) {  // ❌ 应该是 i < 10
    arr[i] = 0;                   // arr[10] 越界写入！
}
```

### 6. 引用已释放的内存（悬挂指针）

```c
int *p = malloc(sizeof(int));
*p = 42;
free(p);
// p 现在是悬挂指针（Dangling Pointer）
printf("%d\n", *p);  // ❌ 未定义行为！可能读到旧值、垃圾值、或段错误
```

### 7. 引用不存在的变量

```c
int *foo() {
    int local = 42;
    return &local;    // ❌ 返回栈上局部变量的地址
}                     // 函数返回后栈帧被回收 → 指针指向无效内存

int *p = foo();
printf("%d\n", *p);   // ❌ 未定义行为
```

### 8. 释放块的局部而非整体

```c
int *p = malloc(100);
p += 10;              // 移动了指针
free(p);              // ❌ free 的参数不是 malloc 返回的原始指针
                      // → 未定义行为（通常崩溃）
```

### 9. 内存泄漏

```c
void leak() {
    int *p = malloc(1000);
    // 使用 p ...
    return;           // ❌ 没有 free(p) → 这 1000 字节永远无法回收
}

// 如果 leak() 在循环中调用 →  内存持续增长 → OOM
```

### 10. 重复释放（Double Free）

```c
int *p = malloc(100);
free(p);
free(p);  // ❌ 重复释放 → 破坏堆的数据结构 → 崩溃或安全漏洞
```

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 消除了大部分内存错误，但内存泄漏仍然存在**：
- Java 的"内存泄漏"指对象**不再使用但仍被引用**（无法被 GC 回收）
- 常见场景：
  - 集合类（`HashMap/ArrayList`）持有不再需要的对象引用
  - 监听器/回调未注销
  - `ThreadLocal` 未清理
  - 类加载器泄漏（Web 应用频繁重部署时）

**JNI 层的内存错误**：
- Java 通过 JNI 调用 C/C++ 代码时，C 层的所有内存错误（1~10）都可能发生
- JNI 层的内存泄漏不受 JVM GC 管理 → 需要手动释放
- Netty 的 `DirectByteBuffer` 虽然不走 JNI，但也是堆外内存 → 需要显式释放

**Rust：编译时消除内存错误**：
- Rust 的所有权系统在**编译时**就能检测出悬挂指针、重复释放、内存泄漏等问题
- 无需 GC，也无需手动管理 → 编译期保证内存安全
- 这是 Rust 在系统编程领域（如 Linux 内核模块）受欢迎的原因
