## 为什么需要 RIO

在 [[10.4 读和写文件]] 中我们了解到，`read` 和 `write` 可能返回**不足值（Short Count）**，尤其是网络 socket 和终端 I/O。

直接使用 Unix I/O 编写网络程序，需要程序员**手动编写循环**来处理不足值。这既繁琐又容易出错。

CSAPP 教材提供了一个名为 **RIO（Robust I/O）** 的 I/O 包，自动处理不足值。

```mermaid
flowchart TB
    subgraph "没有 RIO"
        A["应用程序"] -->|"手动循环处理不足值"| B["read / write"]
    end
    subgraph "使用 RIO"
        C["应用程序"] -->|"一次调用"| D["rio_readn / rio_writen"]
        D -->|"内部自动循环"| E["read / write"]
    end
```

RIO 提供两类函数：
1. **无缓冲的输入输出函数**：`rio_readn`、`rio_writen`
2. **带缓冲的输入函数**：`rio_readlineb`、`rio_readnb`

## RIO 的无缓冲输入输出函数

```c
ssize_t rio_readn(int fd, void *usrbuf, size_t n);
ssize_t rio_writen(int fd, void *usrbuf, size_t n);
```

**核心思想**：在循环中反复调用 `read`/`write`，直到传送完所有请求的字节。

```c
// rio_readn 的实现原理（简化版）
ssize_t rio_readn(int fd, void *usrbuf, size_t n) {
    size_t nleft = n;        // 剩余要读的字节数
    ssize_t nread;
    char *bufp = usrbuf;

    while (nleft > 0) {
        if ((nread = read(fd, bufp, nleft)) < 0) {
            if (errno == EINTR)  // 被信号中断 → 重试
                nread = 0;
            else
                return -1;       // 真正的错误
        } else if (nread == 0) {
            break;               // EOF
        }
        nleft -= nread;          // 更新剩余
        bufp += nread;           // 移动缓冲区指针
    }
    return (n - nleft);          // 返回实际读取的总字节数
}
```

```
rio_readn 的执行过程（请求读取 100 字节）:

  第 1 次 read: 返回 40 字节（不足值）  nleft = 60
  第 2 次 read: 返回 35 字节（不足值）  nleft = 25
  第 3 次 read: 返回 25 字节           nleft = 0  ✅ done
  
  总计: 返回 100 字节（拼接完成）
```

> [!important] rio_readn vs rio_writen 的区别
> - `rio_readn`：EOF 时可能返回不足值（因为文件确实没有更多数据了）→ **正常**
> - `rio_writen`：**永远不会返回不足值**（除非出错），因为写入端总可以接受更多数据

## RIO 的带缓冲输入函数

```c
// 初始化 RIO 读缓冲区
void rio_readinitb(rio_t *rp, int fd);

// 从缓冲区读一行（以 '\n' 结尾）
ssize_t rio_readlineb(rio_t *rp, void *usrbuf, size_t maxlen);

// 从缓冲区读 n 个字节
ssize_t rio_readnb(rio_t *rp, void *usrbuf, size_t n);
```

**为什么需要带缓冲版本？**

直接用 `read` 逐字节读取来寻找换行符 `\n` 效率极低——每读一个字节就是一次系统调用。

```
不带缓冲（逐字节读取一行文本）:

  read(fd, &c, 1)  → 系统调用 #1   → 'H'
  read(fd, &c, 1)  → 系统调用 #2   → 'e'
  read(fd, &c, 1)  → 系统调用 #3   → 'l'
  ...
  read(fd, &c, 1)  → 系统调用 #13  → '\n'  ← 终于读完一行！

  → 13 次系统调用，每次需要用户态↔内核态切换 ❌ 效率极低
```

```
带缓冲（一次填充，多次读取）:

  read(fd, internal_buf, 8192)  → 系统调用 #1 → 填充 8192 字节到缓冲区

  然后从缓冲区中逐字符扫描：
  buf[0]='H', buf[1]='e', ..., buf[12]='\n'  ← 找到换行！

  → 只需 1 次系统调用，后续读取都是内存操作 ✅ 高效
```

## RIO 内部缓冲区原理

```c
#define RIO_BUFSIZE 8192

typedef struct {
    int rio_fd;                // 关联的文件描述符
    int rio_cnt;               // 缓冲区中未读字节数
    char *rio_bufptr;          // 下一个未读字节的地址
    char rio_buf[RIO_BUFSIZE]; // 内部缓冲区
} rio_t;
```

```
RIO 缓冲区的工作状态:

  初始状态（缓冲区为空）:
  ┌─────────────────────────────────────┐
  │            rio_buf (8192 bytes)      │
  │  [空空空空空空空空空空空空空空空空]  │
  └─────────────────────────────────────┘
   ↑ rio_bufptr          rio_cnt = 0

  调用 read 填充后:
  ┌─────────────────────────────────────┐
  │  [H][e][l][l][o][\n][W][o] ... [d]  │
  │   ↑ 已读             待读           │
  └─────────────────────────────────────┘
   ↑ rio_bufptr              rio_cnt = 8192

  读取 "Hello\n" 后:
  ┌─────────────────────────────────────┐
  │  [H][e][l][l][o][\n][W][o] ... [d]  │
  │                    ↑ 待读            │
  └─────────────────────────────────────┘
                       ↑ rio_bufptr  rio_cnt = 8186
```

> 类比：RIO 缓冲区就像一个**大碗**。你去火锅店取调料（read 系统调用），每次走去调料台取一小碟很麻烦（逐字节 read），不如拿一个大碗一次性装满（read 8192 字节到缓冲区），然后在座位上慢慢用。大碗用完了再去取一次。
> CS 术语：这就是**用户态缓冲（User-space Buffering）**，通过减少系统调用次数来摊销内核态切换开销。C 标准库的 `stdio`（`fgets`、`fread`）也使用了同样的技术。

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 的 `BufferedInputStream` / `BufferedReader` 就是 RIO 缓冲思想**：
```java
// 没有缓冲：每次 read() 都是系统调用
FileInputStream fis = new FileInputStream("data.txt");
fis.read();  // 系统调用 #1
fis.read();  // 系统调用 #2 ...

// 有缓冲：内部维护 8192 字节缓冲区，和 RIO 完全一样
BufferedInputStream bis = new BufferedInputStream(fis);
bis.read();  // 从缓冲区读，不触发系统调用
bis.read();  // 依然从缓冲区读
```

**Netty 的 `ByteBuf` 就是增强版 RIO 缓冲区**：
- `ByteBuf` 维护 `readerIndex` 和 `writerIndex` → 类似 `rio_bufptr` 和 `rio_cnt`
- 支持动态扩容、引用计数、零拷贝组合（`CompositeByteBuf`）
- 本质思想一致：减少系统调用，在用户态对数据进行分包/拼包

**HTTP 协议解析依赖行读取**：
- HTTP 请求头以 `\r\n` 分隔每行 → 需要"读一行"的能力
- Tomcat/Netty 的 HTTP 解码器内部都实现了类似 `rio_readlineb` 的功能
- `LineBasedFrameDecoder` 是 Netty 中"读一行"的标准实现
