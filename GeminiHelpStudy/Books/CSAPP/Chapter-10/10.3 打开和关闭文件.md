## open 函数

```c
#include <sys/types.h>
#include <sys/stat.h>
#include <fcntl.h>

int open(const char *pathname, int flags, mode_t mode);
// 成功：返回新的文件描述符（保证是当前进程中最小的可用 fd）
// 失败：返回 -1
```

`open` 函数将 `pathname` 指定的文件转换为一个文件描述符，并返回该描述符。

```
open 的执行流程:

  应用程序                         内核
  ────────                       ────────
  fd = open("/tmp/data.txt",     1. 解析路径名
            O_RDONLY);           2. 检查权限
       │                        3. 在打开文件表中创建条目
       │                        4. 在进程 fd 表中分配最小可用 fd
       ▼                        5. 返回 fd 给进程
  fd = 3 ✅
```

> [!important] fd 的分配规则
> 内核总是返回**当前进程中最小的未使用描述符编号**。
> 例如，如果 fd 0, 1, 2 已被占用，新打开的文件获得 fd 3。
> 如果 fd 1 被关闭了，下一次 open 返回的就是 fd 1（不是 fd 3）。
> 这个特性是 [[10.9 IO 重定向]] 的基础。

## flags 参数详解

`flags` 参数指定进程打算如何访问该文件：

| flag | 含义 | 说明 |
|------|------|------|
| `O_RDONLY` | 只读 | 三选一（必选其一）|
| `O_WRONLY` | 只写 | |
| `O_RDWR` | 读写 | |
| `O_CREAT` | 不存在则创建 | 可与上面的 OR 组合 |
| `O_TRUNC` | 已存在则截断为空 | 通常与 `O_WRONLY` 配合 |
| `O_APPEND` | 写入追加到末尾 | 日志文件常用 |

```c
// 典型用法示例:

// 1. 只读打开已有文件
int fd1 = open("data.txt", O_RDONLY);

// 2. 读写打开，不存在则创建，权限 rw-r--r--
int fd2 = open("log.txt", O_RDWR | O_CREAT, 0644);

// 3. 只写打开，截断已有内容（覆盖写）
int fd3 = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);

// 4. 追加写入（日志场景）
int fd4 = open("app.log", O_WRONLY | O_CREAT | O_APPEND, 0644);
```

## 文件权限与 umask

当 flags 中包含 `O_CREAT` 时，需要通过 `mode` 参数指定新文件的**访问权限位**：

```
Linux 文件权限位结构（9 位）:

   Owner     Group     Other
  ┌───────┐ ┌───────┐ ┌───────┐
  │ r w x │ │ r w x │ │ r w x │
  └───────┘ └───────┘ └───────┘
    4 2 1     4 2 1     4 2 1

  例: 0644 = rw-r--r--
      Owner: 读+写 (6=4+2)
      Group: 只读  (4)
      Other: 只读  (4)
```

| 宏定义 | 八进制 | 说明 |
|--------|--------|------|
| `S_IRUSR` | 0400 | Owner 可读 |
| `S_IWUSR` | 0200 | Owner 可写 |
| `S_IXUSR` | 0100 | Owner 可执行 |
| `S_IRGRP` | 0040 | Group 可读 |
| `S_IWGRP` | 0020 | Group 可写 |
| `S_IXGRP` | 0010 | Group 可执行 |
| `S_IROTH` | 0004 | Other 可读 |
| `S_IWOTH` | 0002 | Other 可写 |
| `S_IXOTH` | 0001 | Other 可执行 |

> [!warning] umask 会"屏蔽"部分权限
> 文件的**实际权限 = mode & ~umask**
> 例如 `mode=0666`，`umask=0022`，实际权限 = `0666 & ~0022 = 0644`
> 即 umask 把 Group 和 Other 的写权限"屏蔽"掉了。

```
umask 的工作原理:

  mode  =  0666  =  rw-rw-rw-  （你请求的权限）
  umask =  0022  =  ---w--w--  （要屏蔽的权限）
  ~umask = 7755  =  rwxr-xr-x
  
  实际   =  mode & ~umask
         =  0666 & 7755
         =  0644  =  rw-r--r--  （最终权限）
```

## close 函数

```c
#include <unistd.h>

int close(int fd);
// 成功：返回 0
// 失败：返回 -1
```

`close` 通知内核：进程不再需要这个文件描述符。

> [!failure] 常见错误：忘记 close
> - 未关闭的 fd 会**泄露**，直到进程结束才由内核回收
> - 如果程序持续运行（如服务器），fd 泄露会导致文件描述符耗尽 → `EMFILE` 错误
> - 关闭一个已经关闭的 fd 是**严重错误**（fd 可能已被重新分配给其他文件）

> 类比：关闭文件描述符就像**退房**。你离开酒店时必须退房（close），否则房间一直被占着（fd 泄露）。如果你退了房又去找前台说"我要退 305 房"，而 305 已经住了新客人（fd 被重用），你退的就是别人的房！
> CS 术语：**双重释放（Double Free）** 问题——close 一个已关闭的 fd 是未定义行为，可能导致关闭了另一个正在使用的文件。

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 的 try-with-resources 就是自动 close**：
```java
// Java 7+ 自动调用 close()，防止 fd 泄露
try (FileInputStream fis = new FileInputStream("data.txt")) {
    // 使用 fis
}  // 自动调用 fis.close() → 底层 close(fd)
```

**线上事故：fd 泄露的实际案例**：
- 微服务中如果 HTTP 连接未正确关闭 → 底层 socket fd 泄露
- 数据库连接池中连接未归还 → fd 被耗尽 → 新请求打不开连接
- 排查方式：`lsof -p <pid> | wc -l` 查看进程当前持有的 fd 数量

**umask 与容器安全**：
- Docker 容器中的默认 umask 通常为 0022
- 写入日志文件或配置文件时，权限不当可能导致安全隐患
- 生产环境常通过 `umask 0077` 确保新文件只有 Owner 可读写
