## Unix I/O 的核心思想

Unix 系统最优雅的设计哲学之一：**一切皆文件（Everything is a File）**。所有的 I/O 设备——网络、磁盘、终端——都被建模为**文件**，所有输入输出都被当作对相应文件的**读和写**来执行。

```
Unix I/O 的统一模型:

  磁盘文件 ──┐
  终端      ──┤
  网络套接字  ──┼──► 统一的"文件"抽象 ──► open / read / write / close
  管道      ──┤
  设备      ──┘
```

> 类比：想象一个图书馆（操作系统），无论你要借小说、教材、光碟还是报纸（各种 I/O 设备），都用**同一张借阅卡**（文件描述符）和**同一套流程**（打开、阅读、归还）。你不需要为不同类型的资料学不同的借阅规则。
> CS 术语：这种设计称为 **统一 I/O 接口（Uniform I/O Interface）**，Unix 内核通过 **VFS（虚拟文件系统）** 抽象层将所有设备纳入统一的文件操作语义中。

## 一切皆文件

Linux/Unix 中，所有 I/O 设备都映射到 `/dev` 目录中的文件：

| 设备 | 对应文件 | 说明 |
|------|---------|------|
| 磁盘 | `/dev/sda`, `/dev/nvme0n1` | 块设备文件 |
| 终端 | `/dev/tty`, `/dev/pts/0` | 字符设备文件 |
| 网络 | 套接字（socket）| 通过专用 API 打开 |
| 随机数生成器 | `/dev/random`, `/dev/urandom` | 内核提供的伪设备 |
| 空设备 | `/dev/null` | 写入即丢弃，读取返回 EOF |

> [!info] 文件类型分类
> Unix/Linux 文件系统中的"文件"有三种基础类型：
> - **普通文件（Regular File）**：包含任意数据（文本、二进制）
> - **目录（Directory）**：包含一组**链接（Link）**，每个链接将文件名映射到一个文件
> - **套接字（Socket）**：用于进程间/跨网络通信的特殊文件

## 文件描述符

当进程打开一个文件时，内核返回一个**小的非负整数**，称为**文件描述符（File Descriptor, fd）**。

```
进程打开文件时的数据结构关系:

  进程的文件描述符表:
  fd 0 ──► stdin  (标准输入, 默认键盘)
  fd 1 ──► stdout (标准输出, 默认终端)
  fd 2 ──► stderr (标准错误, 默认终端)
  fd 3 ──► 新打开的文件 A
  fd 4 ──► 网络套接字 B
  ...
```

> [!important] 三个预留的文件描述符
> 每个进程创建时，Shell 会自动为其打开三个文件：
> - **fd 0 → 标准输入（STDIN_FILENO）**
> - **fd 1 → 标准输出（STDOUT_FILENO）**
> - **fd 2 → 标准错误（STDERR_FILENO）**
>
> 定义在 `<unistd.h>` 中

> 类比：文件描述符就像游乐园的**手环号码**。你进入游乐园（进程启动）时，自动获得 3 个手环（0、1、2），分别对应入口（输入）、出口（输出）、应急出口（错误）。每当你去新的游乐项目（打开新文件），就会获得一个新手环（fd 3、4、5...）。你用手环号码就能访问对应设施，不需要记住设施的具体位置。
> CS 术语：文件描述符是进程级的**间接引用（Indirection）**，它抽象了底层的内核数据结构（打开文件表、v-node 表），使用户态程序无需了解内核实现细节。

## 基本 I/O 操作

Unix I/O 只有四个核心操作：

```mermaid
flowchart LR
    A["open()"] --> B["read() / write()"]
    B --> C["close()"]
    A -->|"返回 fd"| B
    B -->|"传入 fd"| C
```

| 操作 | 系统调用 | 说明 |
|------|---------|------|
| 打开 | `open(filename, flags, mode)` | 通知内核准备访问文件，返回 fd |
| 读取 | `read(fd, buf, n)` | 从 fd 读取最多 n 个字节到 buf |
| 写入 | `write(fd, buf, n)` | 从 buf 写入最多 n 个字节到 fd |
| 关闭 | `close(fd)` | 通知内核释放资源，回收 fd |

> [!warning] 每次 read/write 都是系统调用
> 每次调用 `read()` 或 `write()` 都需要**从用户态切换到内核态（上下文切换）**。
> 这就是为什么频繁的小 I/O 操作开销很大——不是数据传输慢，而是**模式切换**的代价。
> 应用层对此的解决方案就是**缓冲 I/O**（如 C 标准库的 `stdio`、Java 的 `BufferedInputStream`）。

## 💡 架构师视角映射

> [!info] 与 Java 后端的联系

**Java 的文件描述符**：
- `java.io.FileDescriptor` 是 JDK 对 Unix fd 的封装
- `System.in`（fd 0）、`System.out`（fd 1）、`System.err`（fd 2）正是三个预留描述符

**Netty 与 fd 的关系**：
- Netty 的 `Channel` 底层就是对 socket fd 的封装
- 基于 epoll 的事件循环本质上是对大量 fd 的**IO 多路复用**监听
- `selector.select()` → 内核 `epoll_wait()` → 返回"就绪的 fd 列表"

**Linux fd 上限与 Java 性能**：
- 每个进程默认 fd 上限为 1024（`ulimit -n`）
- 高并发 Java 服务器（如 Netty）需要调大该值（如 65535）
- 否则当连接数超过 fd 上限 → `Too many open files` 报错
