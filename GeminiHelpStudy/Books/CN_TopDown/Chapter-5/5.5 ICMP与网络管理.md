## ICMP：互联网控制报文协议

在网络通信中，不可避免会遇到“目的地不可达”、“网络拥塞”或是“跳数耗尽”。当 IP 数据报因为这些特殊原因遭遇底层死亡或丢失时，**没有任何高层面的服务或者管理员知道究竟出了什么事**。

这就非常需要一种能在**路由器和主机之间**互传“差错日志”的协议，它就是：
**ICMP（Internet Control Message Protocol，互联网控制报文协议）**

> [!important] ICMP 到底是哪一层的？
> 这是一个经典谜题。虽然名字叫“XX控制协议”，但在体系结构分层上，**ICMP 通常被认为是 IP 的一部分（位于网络层）**，而不是传输层。
> 因为 ICMP 报文也必须封装到 IP 数据报的有效载荷中（类似 OSPF，IP 把协议字段标为 `1` 作为它的识别符），并不基于 TCP 和 UDP。

### ICMP 报文的核心结构
它主要只提供：
1. **类型字段 (Type)**（指示出了什么错误大类）
2. **代码字段 (Code)**（在大类的分支下的具体细致死因）
3. 和引起该次错误的那个**死亡 IP 首部的拷贝加上前 8 个字节（为了让你辨认原本导致它死去的是哪条包）**！

| 类型 (Type) | 代码 (Code) | 描述含义 | 经典常发场景 |
|-------------|-------------|----------|--------------|
| **0** | **0** | 叫声回应（Echo Reply，Ping 的响应报文） | 调试连通性 |
| **3** | **1** | 主机不可达（Host Unreachable） | 本地路由不知道目的地网络，无路可走 |
| **3** | **3** | 端口不可达（Port Unreachable） | 目的地主机的网卡收下了 IP，但向上丢给指定 UDP/TCP 端口时，发觉对应微服务没被启动监听（常遇到连接已拒或拒绝访问出错）。 |
| **8** | **0** | 请求回应（Echo Request，Ping 的起始包） | |
| **11** | **0** | 寿命 TTL 超时（TTL Expired in Transit，引发报废） | 防路由死环、也是 Traceroute 核心机制！ |

### Ping 和 Traceroute 的原理

这两款网管工具最经典的应用，就是基于那极其简单的几个 ICMP 标识构成的：

**1. Ping**
通过你向目的地发送一个 `Type 8 (Echo Request)`。当它到达另一头主机的 IP 层时，由于是标准内置程序，目标 IP 系统底层不需要上层应用插手，直接强制自己发送一个对应的 `Type 0 (Echo Reply)` 的拷贝还给你——这就是所谓的网络可回声连通行测试。

**2. Traceroute / Tracert (网络路由路线探测器)**
我要怎么知道我发往谷歌的一包数据中途经过哪些中继的省份路由设备机房节点？它是精妙地利用了 ICMP 中的 **`Type 11 - TTL 超时`**！

> [!tip] 巧妙的 Traceroute (路由追踪) 阴谋
> - 我发出第一个到 Google 的 UDP 数据包，但我故意将外壳 IP 部分原本可设 64 的跳跃数（**TTL 值设为 `1`**）。
> - 它刚出门来到家里大门的第一台路由器——路由器说要减一（TTL 剩 0，包死刑丢去垃圾堆）。路由器于是发我一份 **“ICMP Type 11，TTL 死刑罚没通知”**！这封警告信上有那台路由器的名字，**我们就这样逼他说出了大名！获得了第 1 跳真实所在的设备和地理位置**！
> - 然后我发出第二包同名数据报，将其伪装寿命提升到（**TTL 设为 `2`**）。这一包平安跨过第一站，但是到达光纤上的第二个节点时正好死亡并再次触发报错发送警告，**我们就这样记录了第 2 站的名字！**
> - ……以此类推不断逼停下一跳节点发送报错，直至有最后一次能安稳抵达目的地并在里面终结为 `Type 3 端口不可达` 返回目标宣告测试已圆满完赛，沿途经过的所有省份路由器的大名与延迟全部尽收眼底了。

## 网络管理与 SNMP

### 为什么需要统一网络管理方案？

如果我们手里有上百万台服务器、交换机，想要监控这些网关哪几台正在濒临极限断裂？哪些机柜掉线或拥堵需要报修？这时候仅仅只会有自动分发数据的 OSPF 已远远无法解忧，我们需要一套能在全球范围被各硬件大厂承认的高空视角体系方法和统一的系统接口结构语言——这就是**SNMP 协议**及其架构思想管理网络体系学。

### 系统工作组成

> [!note] Agent 与 MIB（管理信息库）
> 像阿里云与公司里的 IDC 运维机柜的后台核心服务器管理系统架构如下：
> 1. **管理服务器（Managing Server，总指挥）**：位于机房外的大屏幕集中运维机（网络运维中心，NOC）。
> 2. **被管设备探针代理（Agent）**：藏于每台被采购的设备（不管出厂厂家是华三、锐捷还是思科）系统内里的代码微核（网络探针后台代理程序）。
> 3. **网络管理协议，也就是 SNMP（Simple Network Management Protocol）自身**。
> 4. **MIB（管理信息库，Management Information Base）**: 用一套最死板但是世界通用的树形标签结构记录设备的参数（像字典定义，比如：“系统内存余量” 必须存放在特定标签标号段落下，各个代理去拉这些库房字段名进行翻译和呈现给上面的大指挥系统中心观看）。

**SNMP 请求-响应通信（Pull，拉取）**
管理中心（总机）主动定时采用 **UDP** 数据报包方式向各地探针端查询 MIB 特定条目的当前值并要求更改某些运行权限的参数，做到了远程操盘万站群集。

**SNMP 陷阱通信（Trap，主动推送报警）**
但如果代理在半夜发现电源被破坏或者风扇暴死转速狂掉线呢，总局如果还在睡梦里几秒发周期获取时已经晚了，所以代理 Agent 需要马上利用自己的反客为主权利进行告密触发式紧急发送一封急报——称之为发送系统陷阱探案（Trap）给总机系统。

> [!info] 💡 架构师视角映射
> - **微服务监控：Prometheus + Grafana 的现代翻版**：传统硬件时代的 SNMP + MIB + 探针。由于到了今天的云计算时期已经极为古老和落伍！今天取代它这一思想做同样概念监制的其实是 Prometheus 的 Exporter 体系！你在各个业务节点侧通过启动 Prometheus Agent （充当 SNMP 被管探针代理）。你的拉取采集 Pull 每三秒抓取服务器 Metric `/metrics` 状态上报数据（SNMP里的定期去取 MIB值）。或者触发 AlertManager 进行（Trap 报警回调和阻绝处理）。
> - **Ping 禁用策略对公有云的影响**：有些云主机被强制封锁防火墙对 ICMP 类型 Echo Request 的接听许可能力，这也就是即便某主机没有挂机但是仍旧表现出在客户端永远显示 “Request timeout (Ping不通)” 且让你以为服务器已关机的假面效果的原因（因为禁止了发送 Ping的回传反馈来拒绝探测请求防攻击和爆破探测机制）。
