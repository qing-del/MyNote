## 通俗类比：公司组织架构合并提拔

当部门员工的入职和解雇导致“全公司红黑性质”被打破时，我们需要通过“高层职位互换”来修复属性，同时**绝对不能搞乱上下级（二叉搜索树大小）的总体次序**。

想象一条部门汇报链：员工 $\alpha <$ 主管 1（X）$<$ 员工 $\beta <$ 部门经理（Y）$<$ 员工 $\gamma$。
为了削减 Y 的职权（左旋）：
- 让 X 原地顶替 Y 当上部门经理。
- X 把原本手里管着的员工 $\beta$ 踢还给 Y。
- Y 委屈巴巴成了 X 的下属（接管员工 $\beta$）。
整个过程，$\alpha < X < \beta < Y < \gamma$ 这种左小右大的“二叉搜索树天平”完全没变！这套升降级的人事变动，我们叫做 **旋转**。

## 核心精讲：保持 BST 拓扑的旋转 (Rotations)

改变节点的颜色和修复指针结构的过程，是维护红黑树特性的关键。而更改内部连接的动作称为**旋转 (Rotations)**。旋转是一种 **局部操作 (Local Operation)**，它能改变结构的深度而不改变**中序遍历 (Inorder Traversal)** 的性质：保持原本二叉搜索树的从小到大属性。

旋转主要分为两种，互为逆向操作：
1. **LEFT-ROTATE(T, x)**: 左旋
2. **RIGHT-ROTATE(T, y)**: 右旋

```mermaid
flowchart TD
    subgraph 左旋之后 (LEFT-ROTATE on X) / 右旋之前 (RIGHT-ROTATE on Y)
        Y(Y) --> X(X)
        Y --> C[γ]
        X --> A[α]
        X --> B[β]
    end

    subgraph 右旋之后 (RIGHT-ROTATE on Y) / 左旋之前 (LEFT-ROTATE on X)
        X1(X) --> A1[α]
        X1 --> Y1(Y)
        Y1 --> B1[β]
        Y1 --> C1[γ]
    end

    X1 -- "LEFT-ROTATE(X)" --> Y
    Y -- "RIGHT-ROTATE(Y)" --> X1
```

在调用 `LEFT-ROTATE(T, x)` 时，我们假设它的右孩子 $y$ 绝对不是 `NIL`。
- 左旋的主旨：以 $x$ 到 $y$ 的链为支轴进行，使 $y$ 成为该子树的新根。
- 原本 $y$ 的左子树 $\beta$，变成了 $x$ 的右子树（因为 $\beta$ 卡在 $x$ 和 $y$ 之间，$x < \beta < y$）。

### 旋转的伪代码核心逻辑

由于红黑树存在双向指针（`parent`, `left`, `right`），每一次完美的旋转必须要精确切断重连 3 组边（6 个方向的指针赋值）：

1. **处理从根开始下放的一条外层边 (β 的挂载)**：把 $y$ 的左子树（也就是 $\beta$），接给 $x$ 当右子树。并修改 $\beta$ 的父亲。
2. **处理最高指挥官的外挂边 (Y 顶替 X 的上级)**：让 $y$ 的父亲指针直接指向原本 $x$ 的父亲。如果 $x$ 是整棵树的根，就让 $y$ 当新根。
3. **处理两人之间的互换连接边 (X 下放归属)**：让 $x$ 挂在 $y$ 之下成为 $y$ 的左孩子（因为 $X < Y$）。

**时间和复杂度分析**：
指针的三加三断纯粹是赋值代码操作，其时间复杂度为严格的 **$O(1)$ 常数时间**。旋转仅修改了树内部一小段指针关联，所以不会触碰除被操作节点及其周围几根骨架外的额外节点！这为红黑树极限性能奠定了拓扑基础。

## 💡 架构师视角映射：`TreeMap` 的指针游戏

如果你翻开 JDK `java.util.TreeMap` 的源码，在底层会找到下面这段惊人重合的代码：
```java
private void rotateLeft(Entry<K,V> p) {
    if (p != null) {
        Entry<K,V> r = p.right;
        p.right = r.left;           // 步骤 1：β 过继给 x
        if (r.left != null)
            r.left.parent = p;
        r.parent = p.parent;        // 步骤 2：上级交接
        if (p.parent == null)
            root = r;
        else if (p.parent.left == p)
            p.parent.left = r;
        else
            p.parent.right = r;
        r.left = p;                 // 步骤 3：X 降级成为 Y 的左节点
        p.parent = r;
    }
}
```

这是一场纯粹的指针杂耍游戏。我们在做多线程或者底层工程时必须学会“纸上得来终觉浅，绝知此事要手敲”。`TreeMap` 在多线程非安全环境下，如果并发产生两个 `rotateLeft` 会直接毁坏树骨架，造成成环内存泄漏，这也是为什么在 JUC 并发包下绝不可能用 `TreeMap` 跑锁，而是换成并行的 `ConcurrentSkipListMap` (跳表机制)。
