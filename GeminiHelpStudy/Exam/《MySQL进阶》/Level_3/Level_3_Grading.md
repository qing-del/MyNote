#### 扣分点（-1 分）

**T7 的快照读结果实际上应该是 `amount = 150.00` 但原因比你描述的更复杂一层。**

在 T6，事务 A 执行了 `UPDATE orders SET order_status = 9 WHERE user_id = 20`，这条 UPDATE 修改了 `id=3` 和 `id=5` 两行的 `order_status` 字段。对于 `id=5` 这行，事务 A 做了**当前读**（UPDATE 是当前读），它会读到最新的物理版本（事务 B 写入的 `amount=999.00`），然后基于这个最新版本进行修改（只改 `order_status`，不改 `amount`）。

所以在 T6 之后，聚簇索引中 `id=5` 的最新版本变成了：
```
(id=5, user_id=20, order_status=9, amount=999.00, trx_id=事务A的trx_id)
```

然后在 T7，事务 A 再做快照读 `SELECT amount FROM orders WHERE id=5`：
- 最新版本：`trx_id=事务A的trx_id` → `trx_id == creator_trx_id` → **可见！**
- 因此事务 A 在 T7 读到的 `amount` 实际上是 **999.00**，而不是 150.00！

**这是本题最容易踩的坑。** UPDATE 是当前读，它直接基于最新物理版本进行修改，修改后的新版本记录的 `trx_id` 是事务 A 自己的。事务 A 随后的快照读会通过 `creator_trx_id` 匹配命中这个版本，因此读到的是被事务 B 修改过的 `amount=999.00`，加上事务 A 自己修改的 `order_status=9`。

> ⚠️ **这就是 MySQL MVCC 中著名的"UPDATE 打断快照"现象**：当你在 RR 隔离级别下先做快照读（看到旧值），然后做了一次 UPDATE（基于当前最新值修改），再做快照读时，你会看到一个"混合"了其他事务修改的版本——因为 UPDATE 的当前读强制将你拉到了最新版本线上。

我只扣了 1 分是因为你的版本链推演逻辑本身是正确的，只是遗漏了 "UPDATE 是当前读会改变版本链顶端" 这个关键交叉效应。这是一个高级坑，能踩到说明题目出到了它该有的深度。

### Part B - Q3a/b：二级索引加锁（7/15）

#### 你的回答核心观点

> "InnoDB 会在二级索引得到的区间上加 X 锁，对应 (3,5) 区间，是 GAP 锁"

#### ❌ 关键错误

**1. 锁的区间表述完全错误（-4 分）**

你写的加锁区间 `(3,5)` 是用聚簇索引的 id 来描述的，但我们在讨论的是 **`idx_user_id` 二级索引树**上的加锁行为。在二级索引树上，叶子节点的排列顺序是 `(user_id, id)` —— **先按 user_id 排序，user_id 相同时按 id 排序**。

`idx_user_id` 的叶子节点顺序是：
```
(10, 1) → (20, 3) → (20, 5) → (30, 8) → (30, 10) → (40, 15) → (50, 20)
```

当 `WHERE user_id = 20` 命中 `(20, 3)` 和 `(20, 5)` 两条记录时，InnoDB 加的锁应该描述为基于**这棵二级索引树的节点位置**。

**2. 加锁类型描述不精确（-4 分）**

在 RR 隔离级别下，InnoDB 对非唯一二级索引使用 **Next-Key Lock**（临键锁）而不是纯 GAP 锁。你的实践结果已经告诉你了真实答案：

```
idx_user_id | RECORD | X       | 20, 3     ← Next-Key Lock on (20,3): 锁住 (10,1) 到 (20,3) 的区间 + 记录本身
idx_user_id | RECORD | X       | 20, 5     ← Next-Key Lock on (20,5): 锁住 (20,3) 到 (20,5) 的区间 + 记录本身  
idx_user_id | RECORD | X,GAP   | 30, 8     ← Gap Lock on (30,8): 只锁住 (20,5) 到 (30,8) 之间的间隙
PRIMARY     | RECORD | X,REC_NOT_GAP | 3    ← 聚簇索引上 id=3 的纯记录锁
PRIMARY     | RECORD | X,REC_NOT_GAP | 5    ← 聚簇索引上 id=5 的纯记录锁
```

让我拆解这个加锁逻辑：

**在 `idx_user_id` 上：**
- **(20, 3)** 命中：加 **Next-Key Lock**，锁定区间 `((10,1), (20,3)]`（左开右闭）
- **(20, 5)** 命中：加 **Next-Key Lock**，锁定区间 `((20,3), (20,5)]`（左开右闭）  
- 扫描继续到下一条记录 **(30, 8)** 发现 `user_id=30 ≠ 20`，不再匹配：加 **Gap Lock**（纯间隙锁），锁定区间 `((20,5), (30,8))`（左开右开，不含记录本身）

这三把锁合在一起，覆盖的完整区间是 `((10,1), (30,8))`，其中 `(20,3)` 和 `(20,5)` 两个记录点被 X 锁住。

**为什么要锁 `(20,5)` 之后到 `(30,8)` 之前的间隙？** 因为 RR 下必须防止幻读——如果有人在这个间隙中插入一条 `(user_id=20, id=6)` 的新记录，事务 A 再次执行 `WHERE user_id=20` 就会多看到一行，即幻读。

**3. 你在实践后的困惑（+1 奖励分）**

> "GAP 锁咋是锁 ((20,5), (30,8)) 这个二级索引区间，防止插入吗？那干鸡毛不防止插入中间，而是防止插入尾部？"

虽然表述粗犷，但这个追问本身说明你在**认真对照理论和实践的差距**。答案是：中间的 `(20,3)` 到 `(20,5)` 之间已经被两个 Next-Key Lock 联合覆盖了，自然也是被保护的。`(20,5)` 到 `(30,8)` 的 Gap Lock 保护的是 `user_id=20` 范围的"右边界之外"——防止有人在这里插入一条新的 `user_id=20` 的记录。三把锁合在一起才是完整的防线。

#### Q5：死锁判断（6/9）

**扣分（-3）：** 你没有用**等待图（Wait-for Graph）**来形式化地证明"没有死锁"。题目明确要求"画出事务等待图"。正确的等待图应该是：


#### 你的附言

> "如果将 INSERT 的 id 改成 6，我就不会做了。是形成死锁，但是我不是很清楚 InnoDB 是怎么处理死锁的"

**你的直觉完全正确！** 如果改成 `id=6`：
- 事务 X 的 Gap Lock：(5, 8)
- 事务 Y 的 Gap Lock：(10, 15)
- 事务 X 插入 `id=11` → 落在 (10,15) → 被 Y 阻塞
- 事务 Y 插入 `id=6` → 落在 (5,8) → 被 X 阻塞
- **X → Y → X** 形成环 → **死锁！**

InnoDB 的处理方式：死锁检测器（**Wait-for Graph 算法**）会在发现锁等待时遍历等待图，检测是否存在环路。检测到死锁后，InnoDB 会选择**代价较小的事务**（通常是 undo log 更少、修改量更小的那个）作为 **victim（牺牲者）**，强制回滚它，释放其持有的所有锁，让另一个事务继续执行。被回滚的事务会收到错误 `ERROR 1213 (40001): Deadlock found when trying to get lock; try restarting transaction`。

**你能识别出这个变体会死锁，说明你的间隙锁直觉已经建立起来了。只差临门一脚——把死锁检测的处理机制补上。**

### 你最需要深化的 3 个点

1. **二级索引上的 Next-Key Lock 和 Gap Lock 区别。** 当 WHERE 条件命中二级索引的记录时，加的是 Next-Key Lock（区间 + 记录本身）；扫描到第一条不匹配的记录时，退化为 Gap Lock（只锁间隙，不锁记录本身）。这是 RR 级别下防止幻读的完整防线。

2. **UPDATE 是当前读（Current Read），会"打断"快照。** 这是 MVCC 中最容易被忽略的交叉效应——先快照读看到旧值，中间做了一次 UPDATE（基于最新版本修改），再快照读看到的是自己修改后的版本（包含了其他事务的最新数据）。

3. **死锁检测器的 Wait-for Graph 算法和 victim 选择。** 当 InnoDB 检测到等待图中存在环路时，会选择 undo log 最少的事务回滚。这个机制你需要实际构造一个死锁场景去观察 `SHOW ENGINE INNODB STATUS` 中的 `LATEST DETECTED DEADLOCK` 部分。
