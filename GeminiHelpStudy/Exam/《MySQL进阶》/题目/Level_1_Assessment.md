## 【Part 1：InnoDB 物理与内存架构】

### 题 1：Buffer Pool 的“灵异”换页
**场景记录：** 
我们有一张存放历史日志的冷数据表 `log_history`，总共有 5000 万行数据，平时极少被业务直接访问。某天，一个后台脚本意外启动，开始每隔 5 分钟执行一次 `SELECT COUNT(*) FROM log_history;`。监控面板显示，每次执行这句全表扫描 SQL 时，核心业务的读写响应时间经常会出现偶发性的断崖式下跌，Buffer Pool 的 `free buffers` 骤降，同时 `evicted without access`（未被访问即被淘汰的页数）指标异常上升。
**请回答：**
这句全表扫描 SQL，到底在 Buffer Pool 的 LRU 链表中引发了什么内部运作？请结合 InnoDB 对 LRU 链表的冷热分离机制（Old / New Sublist）及 `innodb_old_blocks_time` 参数，给出底层的推移图景，并解释为什么会产生大量 `evicted without access` 现象以及影响核心业务的原因。

### 题 2：DYNAMIC 行格式与 16KB 页的尴尬
**场景记录：** 
一张表设定了 `ROW_FORMAT=DYNAMIC`。假设 InnoDB 的一个数据页大小严格为 16KB（16384 Bytes）。你往这张表插入了一行数据，其中包含一个 10,000 Bytes 大小的 `VARCHAR` 字段。16KB 的数据页减去固定格式的 Header 等元信息后理论上也完全装得下这段 10,000 Bytes 的实际数据。但在用工具物理窥探数据页结构时，却发现这 10,000 Bytes 的数据并没有被完整保存在当前数据页中，而是产生了一个 20 Bytes 的溢出页指针（Overflow Pointer），数据被放到了溢出页。
**请回答：**
为什么明明 16KB 的数据页还有大把的物理剩余空间，InnoDB 却依然选择将这 10,000 Bytes 的字段变为“行溢出”处理？这背后是基于 InnoDB 引擎当初设计时，对于一页内“最少容纳行数”的什么硬性假设？如果允许一条巨大会塞满数据页的数据存在当前页，会直接导致 B+ 树的物理结构退化成什么样？

## 【Part 2：B+ 树索引本质与执行成本】

### 题 3：子查询加速深度分页的 B+ 树漫游指北
**场景记录：**
表 `users` (id PK, name VARCHAR, index `idx_name(name)`)。
**慢查询 SQL (耗时 5 秒):** 
```sql
SELECT * FROM users ORDER BY name LIMIT 100000, 10;
```
**优化后的 SQL (耗时 0.1 秒):**
```sql
SELECT * FROM users u 
INNER JOIN (SELECT id FROM users ORDER BY name LIMIT 100000, 10) tmp 
ON u.id = tmp.id;
```
**请回答：**
这两句 SQL 返回的结果是完完全全相同的。请严格从引擎遍历 B+ 树的扫描路径（先走哪棵树、扫描多少个节点、发生了多少次回表导致的随机 I/O）详细推演这两句 SQL 的底层执行流程。为什么加了一层看似复杂的内部临时表和 JOIN 操作之后，反而彻底抹平了性能瓶颈？关键的成本区别发生在哪里？

### 题 4：范围查询导致排序索引失效的结构迷局
**场景记录：**
表 `orders` 建立了一个复合二级索引 `idx_status_time(status, create_time)`。
**查询 A：** 
```sql
SELECT id FROM orders WHERE status = 1 ORDER BY create_time DESC LIMIT 10;
```
**查询 B：** 
```sql
SELECT id FROM orders WHERE status > 1 ORDER BY create_time DESC LIMIT 10;
```
通过 EXPLAIN 可以看到，查询 A 的 Extra 仅仅是 `Using index`，而查询 B 出现了 `Using filesort`。
**请回答：**
请不要用“因为有范围查询所以最左前缀匹配断了”这种浅尝辄止的面经口诀来敷衍我。请真正在你的脑海中画出 `idx_status_time` 这棵 B+ 树的叶子节点物理分布状态，从节点内数据的绝对有序性和相对有序性角度，严格推导出为什么查询 A 可以直接顺着叶子节点链表拿数据完成排序，而查询 B 拿到的数据必须是在 MySQL 内存里重新进行耗时的排序操作。

### 题 5：优化器的“算账”逻辑与南辕北辙
**场景记录：**
表 `users` 共有 1000 万行数据。存在一个二级索引 `idx_age(age)`。执行 SQL：
```sql
SELECT * FROM users WHERE age = 22;
```
查询优化器（Optimizer）最终在执行计划中决定抛弃 `idx_age` 索引，直接选择了对聚簇索引全表扫描（Full Table Scan）。而你通过统计信息分析发现，`age = 22` 的用户大约占全表的 20%（即 200 万行）。
**请回答：**
即便二级索引 `idx_age` 只包含 `age` 字段和主键 `id`，其叶子节点占用的总物理数据页和体积远远小于聚簇索引（包含所有列）。按理说在这个小树里扫描 200 万行数据应当是非常轻量的操作。请从 MySQL 磁盘 IO 种类（“顺序 I/O” 与 “随机 I/O”）以及 “MRR/回表真实成本” 的角度说明，优化器底层的 Cost-Based 计算模型为什么要抛弃这条虽然体积小，却具有陷阱的访问路径。

### 题 7：千面数据——Undo Log 版本链寻址推演
**场景记录：**
事务隔离级别为最经典的 `REPEATABLE READ (RR)`。
表 `account`，某一行初始数据记录：`(id=1, balance=100)`。
时间线如下：
- **T1**：事务 A 开启，执行 `SELECT balance FROM account WHERE id=1;`，查到的结果必然是 100。**事务 A 未提交。**
- **T2**：事务 B 开启，执行 `UPDATE account SET balance=200 WHERE id=1;`，**事务 B 顺利提交。**
- **T3**：事务 C 开启，执行 `UPDATE account SET balance=300 WHERE id=1;`，**事务 C 顺利提交。**
- **T4**：事务 A 第二次执行 `SELECT balance FROM account WHERE id=1;`。
**请回答：**
显然查到的结果依然是 100。请结合聚簇索引叶子节点中存在的隐藏字段（`trx_id` 和 `roll_pointer`）以及在物理 Undo 表空间中真实存在的 Update Undo Log 的链表结构，一步一步推演出：事务 A 的执行器在拿到了 RR 级别下特定机制生成的 Read View 后，是如何跟当下聚簇索引上的最新物理记录版本 `balance=300` 发生碰撞的？又是如何顺藤摸瓜在一节一节的链表指针中回溯历史，并在什么判定条件下，恰好停留并读出了那个正确的结果。

### 题 8：一个温水煮青蛙的长事务与清道夫的怠工
**场景记录：**
一个高频更新和写入的电商核心主库（TPS 达到 6000 以上）。某天深夜，某个周边服务的定时任务代码发生了严重的死循环，但其在进入死循环前开启了一个事务：
```sql

SELECT id FROM some_table LIMIT 1;
-- 随后代码卡住，事务既未 COMMIT 也未 ROLLBACK
```
此后几个小时内，整个 MySQL 数据库的物理磁盘空间用量暴走，专门对应的独立表空间或默认系统表空间体积如同吹气球一样膨胀一百多 GB；同时很多完全无关的表，甚至仅仅做少量更新的数据页也开始表现出极度低效的检索性能。
**请回答：**
这个长时间未提交的跨夜事务仅仅只是做了一个及其无关紧要的、甚至连锁都没加的一行记录查询。它何德何能导致整个系统级别的存储空间大爆炸？
这究竟是怎么阻塞且影响了 MySQL 内部非常重要的 Purge 操作（清除不再被任何活跃版本引用的 Undo Log 的后台线程）？具体讲述为什么这个极早的旧 Read View，导致所有产生改变的表，它们的老版本 Undo Log 犹如垃圾山般堆积而无法被回收。
