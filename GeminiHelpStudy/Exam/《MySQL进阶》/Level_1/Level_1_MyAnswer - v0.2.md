### 题1
答：
- 全表扫描SQL会令Buffer Pool中的free链表中的“空闲页”去读取大量数据 - 这是free链表骤降的原因
- 将读取到数据的“数据页”插入到URL链表的`midpoint`位置（old的头部）
- 刚进入到old区的数据页，只要处于`innodb_old_blocks_time`参数时间内被访问是不会进入到热区的
- 但是由于读取的数据很多，且buffer pool中的内存是有限的（默认old区只能占用3/8的空间），这个时候会不断的在URL的old区尾部淘汰数据页 -- `evicted without access`飙升的原因
- 极端情况：如果**持续出现单位时间内**，读取数据页到old区的数据量 > URL中old区淘汰数据页的速度，那么就会导致old区“缓存击穿”导致出现**缓存污染**，使得热区尾部要开始淘汰核心业务数据页来给old区腾空间

- 如果真的一页存储这条数据行，会导致一页只能存一个数据（
    我们可以计算：令16KB ≈ 16000 bytes;（粗略计算下顺带掐去页头页尾和页目录的影响）

- 会使 B+树 退化成一颗多叉搜索树，丢失 B+树 高扇出低树高的特点，同时会导致查询一条数据就是进行一次随机IO，因为由数据结构与算法中学习到的链表的内存分布，其内存分布并不是连续的，所以在磁盘中，存储方式是 B+树 非叶子节点核心存放是地址和一些关键标识数据（例如聚簇索引树会放置叶子节点“页”中的第一条数据行的`id`值），然后 B+树 的叶子节点是一条“页”链表，这条链表在扇出值=1时，就是一条普通的链表，每查询一条数据，进行偏移的时候就会进行一次随机IO来获取下一页的数据，会导致存储层做大量的随机IO影响性能


### 题3
答：
- 第一条SQL语句分析
    - 先通过二级索引有序获取所有数据（二级索引B+树）
    - 会一共扫描100010条数据
    - 每一条数据都是先去二级索引上进行搜索得到聚簇索引（聚簇索引B+树）
        - 因为是`order by name`，所以可以直接在通过二级索引B+树
        - 找到第一个叶子节点，然后在链表上进行遍历获得数据
        - 然后一条一条通过IO传输到Service
    - 然后再拿着聚簇索引进行回表查询拿到数据行
        - 因为Service层一看是`select *`
        - 所以会拿到一条数据就进行一次回表的随机IO查询
        - 这里的回表IO是传给聚簇索引B+树`id`值进行索引搜索来查询
    - 然后通过IO将这条数据通过IO传给Service层
    - 就这样重复执行100010次
    > 这里一共进行了100010次回表的随机IO
    - 最后Service层检查条件之后，丢掉100000条数据，然后相当于做了100000次无效的IO传输；严重影响性能

- 第二条SQL语句分析
    - 会先通过二级索引B+树获取100010条数据，但是不会发生回表（因为**索引覆盖**）（查到一条就发送一条给Service层）
        - Service层拿到数据的时候检查SQL语句的子查询发现`select id`
        - Service层这个时候不会发起回表查询
        - 直到100010条二元数据组全部传输完毕之后
    - Service进行筛选得到最后10条数据，然后对10条聚簇索引`id`进行回表查询
    > 如果这里是联合索引还会更快，因为会带有MRR加速（前提是数据量大的时候效果才会比较明显），变成顺序IO
    - 然后最后将这10条有效数据行传给Service层
    > 这里一共进行了10次回表的随机IO

- 因为加入join之后，在子查询中使用了**索引覆盖**，不会让每次在二级索引树中查询到的聚簇索引都发生一次回表查询，减少了随机IO的性能损耗（因为磁头移动去读取数据是很慢的）；
- 关键成本发生在**减少了大量的无效回表随机IO查询**

- 在 B+树 中，联合索引构建的这颗索引树，如示例中的其实是遵顼这个排序规则的

```cpp
bool cmp(const node& a, const node* b) {
    if(*a.status == *b.status) return *a.create_time < *b.create_time;
    return *a.status < *b.status;
}

signed main() {
    node leave[n]; // 代表所有叶子节点
    sort(leave, leave+n+1, cmp);
    return 0;
}
```
- 其实这个就是自底向上来看 B+树 （自顶向下我也会，因为这个代码好敲也简短）；
- 接下来就可以是以每页的数据行记录数来分割每部分，然后往上构建整颗B+树了
- 上面的节点就是存放在这棵树中存放的是：(status, create_time, id)（因为不唯一，所以要保证每个节点都不一样的话，后面还会带个`id`，当然还会存放偏倚的地址值）

- 而对于`查询B`来说，除了是`status>1`这个条件其实没那么容易做精确匹配还有一个问题，就是我每个不同的`status`下查询出来不同组的`create_time`是有序的（分组看的话），如果是整体一起，那么就是无序的，只能说是局部有序，那就只能是拉到排序缓存区中进行一次排序了；

### 题5
答：
- 按照我的理解，成本的排序可以是这样的`磁头移动（随机IO） >> 顺序IO`
- 结合题目提示的话，只能说`SQL优化器`发现通过`where age=22`这个条件搜索得出来的聚簇索引
- 顺序是特别乱的（因为二级索引不唯一的情况，所以很可能查询出来的id是这样的\[1, 5, 2, 8, 3, 9, 4, 7, 6, 10\]，因为在二级索引树中，并不是类似于联合索引那样会根据两个索引排序，真实情况是这样的->{
    ```md
    - age=1
        - (1, 7)
        - (1, 2)
        - (1, 6)
    - age=2
        - (2, 5)
        - (2, 3)
        - (2, 9)
    ```
}），那么在`SQL优化器`的理解里，我回表200w次的随机IO，不如做多几次顺序IO，这样更快
- 而且同时还能减少存储层和Service层之间使用IO传输的次数，因为IO传输也是很慢的，想要性能高，IO传输次数也要少
- 因为多次顺序IO操作中，磁头只需要按下去，一路顺着读就行了，不需要过多的磁头移动（物理过程，超级慢），读完一次移动一下就可以了
- 而在多次随机IO操作中，磁头必须要不停的进行移动，基本是读取一次就需要进行一次移动，直到读取完毕所有数据

- 还有一种原因，就是做`MRR`排序之后200w次回表查询的代价依旧大于做全表扫描的代价；

答：
- 首先客户端发起这条SQL请求的时候完成TCP三次握手，建立连接并完成缓存区的创建
- 然后MySQL会对这条SQL语句解析并丢进执行器开始运作，此时Service层和存储层建立连接和完成缓存区的创建
- 在存储层查询数据的时候会通过滑动窗口阻塞机制来设置最大缓存量
- 伪代码如下：
```txt
func isFull(nowCacheSize) { 通过Service层传过来的 ACK 判断与IO通道的最大容量来判断最终大小 }

main:
    while(next_line() != null) {
        row_data = read(); //[Disk -> Cache]
        if (row_data == null) break;
        nowCacheSize += row_data.size();
        if (isFull(nowCacheSize)) {
            flush();    // InnoDB -> Service
        }
    }
```
- 在Service层中同上代码逻辑对客户端发送数据
- 但是我们这里要加入执行器拉去数据的逻辑，继续使用伪代码
```txt
func screen(date_set) { 过滤掉不符合条件的数据行 }

main:
    do {
        data_set = 调用 InnoDB 获取数据集;
        screen(data_set);

        // ... 一些列将数据集写入到发送缓冲区并测满之后发送给客户端的逻辑代码
        data_set.clear();
    }while(InnoDB.next_line_isNotNull())
```

- 现在再说一下如果是客户端主动断掉了连接会发生什么
- 在切断连接之后，客户端将不再会返回ACK给MySQL
- MySQL的Service层根据上次接收到的ACK解析并通过比较函数`min(ACK.max_size(), IO_Tunnel.max_size())`最大装载量之后，发送数据，直到`all_data_size == ACK.max_size()`，（这里的ACK指代客户端断连前最后一次发送的ACK响应）
- 此时这个进程就会卡住了，也就是因为客户端的缓冲区满了而被阻塞 -> send() -- from MySQL.Service to Client 阻塞 -> MySQL.Service的消息缓冲区由于在接收InnoDB读取的数据行，然后满了 -> InnoDB.send() 被阻塞
- 后续就是MySQL尝试若干次重连无法连接上客户端之后，就终结这次SQL查询并释放掉所有资源



### 题7
答：
- 这个问题其实涉及到了`快照读`的机制
- 在`RR`隔离级别下，事务只会为第一条`select`语句创建一个`ReadView`，后面的所有查询都只是复用这张“快照”
- 假设事务A的`trx_id=1`，那么在它的`ReadView`版本
    - `m_ids`是空的(因为我这里假设它是第一个事务，没有其他事务在活动；如果有的话那肯定是不为空的，而是包含所有当前在活跃中的事务)
    - `max_trx_id=2` -- 下一个即将被分配的事务id
    - `min_tex_id=-m_ids.isEmpty() ? max_trx_id : m_ids.getMin()` 
    - `creator_trx_id=0` -- 快照读默认的（除非发生修改）
- 尽管是后续的事务B(trx_id=2)和事务C(trx_id=3)，那么他们修改完毕之后
- 事务A再次进行读取的时候，去读最新版本的时候，发现`c.trx_id >= a.max_trx_id`这个条件是为`true`，说明是这个事务是事务A之后才创建的，不能读，然后通过`roll_pointer`去获取下一个版本的地址并继续四步判定法；读取事务B的修改依旧是同理；直到读取到`trx_id=1`的，发现`a.max_trx_id > a.trx_id > a.min_trx_id`并且`m_ids.contain(a.trx_id) = false`，即为可以获取这个版本，那么就会读取到这条数据，即 `balance = 100` 的这条数据；


### 题8
答：
- 分析题目，这个是一个高并发的电商数据库，那么我们可以分析
- 首先在事务开启之后，会获得一个事务id，这里我们假设它是`trx_id=100`
- 我们假设这个数据库是默认的RR隔离级别
- 在这个高并发系统中，我推测TPS应该是数据库每秒钟收到的业务请求数量
- 那么在这么大量的请求下，单位时间内会创建大量的`ReadView`
- 然后在这些快照版本中`m_ids`都会包含那个卡住并处于活跃状态的事务id，只要它一直不释放，那么这个Purge线程就一直很难正常工作
- 因为Purge线程是检查目前所有快照版本中最小的`min_trx_id`（大于这个版本的很可能还处于活跃状态），导致被卡在了很古老的时间点无法运作
- 然后会将大量的“没用”的快照版本写入到`Undo Log`中，然后无法回收，导致文件暴涨几GB
- 然后由于版本链叠加，并且活跃的事务由于之前卡住的事务导致的逐渐增加，会形成一个很长很长的，导致不管你是哪里的查询，只要是要通过MVCC去读取准确数据的话，都需要去遍历一条极长的版本链，导致性能下降的问题
