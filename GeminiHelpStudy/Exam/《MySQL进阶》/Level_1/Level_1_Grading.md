### 题 1：Buffer Pool 的"灵异"换页（5/12）

#### 你的逻辑漏洞

你的回答暴露了一个核心问题：**你并没有真正理解 InnoDB LRU 链表的冷热分离（Old/New Sublist）机制，而是在用朴素的"标准 LRU"思维去硬套。** 你自己也坦诚了这一点，态度值得肯定。但考试不给态度分。

具体问题：

1. **Free 链表的定位严重错误。** 你写"free: 业务数据页→业务数据页→..."。Free 链表中存放的是**空闲的、尚未被使用的数据页**，不是业务数据页。业务数据页既然已经被加载使用了，它一定在 LRU 链表或 Flush 链表上，绝对不在 Free 链表上。Free 链表是一个"空白页仓库"。
2. **`innodb_old_blocks_time` 的作用理解反了。** 你说"超过这个界限就做清除操作"，这完全不对。`innodb_old_blocks_time` 是一个**时间窗口（默认 1000ms）**，它的含义是：一个数据页首次被加载到 LRU 的 Old 区（冷区）后，必须在 Old 区存活超过这个时间窗口，且在此期间再次被访问，才有资格被提升到 New 区（热区）头部。如果在这个时间窗口内没有被再次访问，它就一直留在 Old 区，等着被淘汰。
3. **没有解释"为什么核心业务受影响"的真正原因。** 你只说了"别的业务在排队等 IO"，这太模糊了。

#### 满分标准答案

InnoDB 的 LRU 链表并非教科书上的标准 LRU，而是一个**冷热分离的两段式 LRU**：

```
LRU 链表结构:
[New Sublist (热区, ~5/8)]  ←→  [midpoint]  ←→  [Old Sublist (冷区, ~3/8)]
     高频访问的热数据页                              新加载 / 低频访问的数据页
```

**全表扫描发生时的精确推移过程：**

1. 执行 `SELECT COUNT(*) FROM log_history` 时，InnoDB 需要将这张 5000 万行表的**所有数据页**逐个从磁盘加载到 Buffer Pool。
2. 每个新加载的数据页**首先被插入到 LRU 链表的 midpoint 位置**（即 Old 区的头部），而**不是整个 LRU 的头部**。这就是冷热分离的精髓——新来的页默认是"冷"的。
3. `innodb_old_blocks_time`（默认 1000ms）此时发挥关键作用：一个页被放入 Old 区后，如果在 1000ms 内再次被访问，它仍然**不会**被提升到 New 区。全表扫描就是典型的"扫一遍就走"的访问模式——每个页被读取一次后就再也不会被访问了，时间间隔远远不到 1000ms 内的二次访问。
4. 因此，这些全表扫描加载的冷数据页始终被困在 Old 区，不断涌入又不断被后来的新页从 Old 区尾部挤出去。**这就是 `evicted without access` 指标飙升的精确原因**——页被加载到 Old 区，从未被提升到 New 区（没有"第二次访问"），就直接从 Old 区尾部被驱逐了。

**为什么核心业务受影响：**

虽然冷热分离机制保护了 New 区的热数据不会被直接替换掉，但 Old 区本身也占据了 Buffer Pool 约 3/8 的空间。当全表扫描疯狂地向 Old 区灌入冷数据页时：
- Old 区原本缓存的一些**业务上偶尔需要但不算高频的温数据页**会被挤掉。
- 如果 Buffer Pool 总容量不够大，Free 链表迅速耗尽后，LRU 的淘汰压力传导到整个链表的尾部，核心业务下一次需要访问那些被挤掉的温数据时，就必须重新发起磁盘 I/O，引发响应时间的断崖式下跌。
- 极端情况下，如果 `innodb_old_blocks_time` 被不当地设置为 0，全表扫描的页甚至可能侵入 New 区，那就是灾难级的全面污染。

### 题 2：DYNAMIC 行格式与 16KB 页的尴尬（8/12）

#### 你的逻辑漏洞

你正确回答了核心结论：**InnoDB 要求一页至少容纳两条记录**，并且指出了如果一页只存一条记录，B+ 树会退化。但存在以下不足：

1. **退化目标说得不够精确。** 你说"退化成一棵多叉搜索树"——这个描述太模糊了。本质上，如果一页只有一条记录，叶子节点之间的双向链表就失去了"范围扫描"的意义（每读一条记录就要跳到下一页），B+ 树的叶子层退化成了**链表**，范围查询退化为逐页跳转的随机 I/O，B+ 树的核心优势——叶子节点的顺序扫描能力——被彻底摧毁。更准确地说，每个叶子节点只有一条记录，这棵树在叶子层的行为接近一棵普通 **B 树**甚至链表，完全丧失了 B+ 树"一页内批量顺序读取多条记录"的 I/O 红利。
2. **没有解释具体的溢出判定计算逻辑。** 为什么是"至少两条"？具体来说，InnoDB 的硬性约束是：**一个数据页至少要能容纳 2 行用户记录**。16384 Bytes 减去页头、页尾、Page Directory 等元信息后，剩余可用空间大约在 16000 Bytes 以内。要塞下 2 行记录，每行的"页内存储部分"就不能超过大约 8000 Bytes。你的那行数据如果行内部分超过了这个阈值（10000 Bytes 显然超了），即使页还有空间，InnoDB 也必须将大字段推到溢出页中，只在当前页保留一个 20 Bytes 的指针，确保当前行在页内的实际占用足够小，从而为第二行留出空间。

#### 满分标准答案要点补充

- InnoDB 的设计哲学：**每页至少 2 行记录**，这保证了 B+ 树叶子节点至少有最基本的"批量"意义。
- `DYNAMIC` 行格式下，当行的总长度使得"一页放不下两行"时，会将变长字段（VARCHAR / TEXT / BLOB）完整推到溢出页（Off-page），仅在原数据页保留一个 20 Bytes 的溢出指针。
- 如果破坏了这个约束：B+ 树叶子节点变成每页仅存一条记录 → 范围扫描每读一条记录就发生一次页切换（随机 I/O）→ 叶子层的顺序连续读的红利完全消失 → 性能灾难。

3. **没有提到一个关键数字：回表阈值。** 经验法则是：当查询结果集超过全表约 **15%-30%** 的行数时，优化器几乎一定会放弃二级索引而选全表扫描。20% 恰好在这个危险区间内。

### 题 6：拔掉网线后的执行器（3/12）

#### 你的逻辑漏洞

你非常坦诚地承认了这部分知识的欠缺，所以我不会在"态度"上为难你。但答案本身的推理有诸多不准确之处。

1. **你猜测 MySQL "分批传" 或 "报错被拦截" ——方向对了但机制全错。** MySQL 确实不会一口气把 1 亿行全读到内存再发。但原因不是"分批 IO"或"最大数据包限制"，而是执行器本身就是**逐行拉取、逐行发送**的流式处理模式。
2. **你对发送缓冲区、TCP 滑动窗口的追问**说明你对网络层协同的知识确实是空白的。

#### 满分标准答案

这个问题涉及 MySQL Server 层执行器 + 网络层 + TCP 协议的三层协同：

**第一层：执行器的流式拉取模型**
MySQL 的执行器**不会**一次性把 1 亿行数据全部从引擎层读出来放到内存里。它的工作模式是一个循环：
```
while (true) {
    row = engine->read_next_row();   // 从引擎层拉一行
    if (row == NULL) break;
    net_buffer_write(row);           // 写入网络发送缓冲区
    if (net_buffer_full) {
        net_flush();                 // 缓冲区满了，通过 TCP 发出去
    }
}
```
`net_buffer_length`（默认 16KB）是 Server 层的网络发送缓冲区。执行器每拉一行就往里塞，塞满了就 flush 一次给 TCP socket。

**第二层：TCP 的滑动窗口与反压**
`net_flush()` 调用底层的 `send()` 系统调用，把数据写入操作系统的 TCP 发送缓冲区（`SO_SNDBUF`）。TCP 协议有**滑动窗口**机制：
- 正常情况下，客户端持续消费数据，不断发回 ACK 并维持一个较大的接收窗口（rwnd），发送端可以持续发送。
- **客户端断开连接后**：没有人消费 TCP 接收缓冲区的数据 → 客户端 TCP 栈的接收窗口（rwnd）迅速缩小为 0 → 发送端（MySQL 所在服务器）的 TCP 栈发现窗口为 0，**停止发送** → 操作系统的 TCP 发送缓冲区满 → `send()` 系统调用阻塞 → `net_flush()` 阻塞 → 执行器的写入调用阻塞 → **整个执行循环停住了**。

**第三层：超时检测与连接清理**
执行器停住后，MySQL 并不会永远等下去。TCP 层在持续重传和探测（keepalive probe）后，最终会检测到连接已断开（RST 或超时），返回一个写入错误。Server 层的连接管理器捕获这个错误后，会触发：
- 清理该连接的查询状态
- 回滚未提交的事务
- 关闭连接、释放资源

**结论：MySQL 不会 OOM。** 执行器被 TCP 反压机制从网络层一路"卡"回来，停在了"往 net_buffer 写入"这一步上。然后通过 TCP 超时或 `wait_timeout` 被清理掉。整个过程中，MySQL 内存中最多只持有一个 `net_buffer`（默认 16KB）和引擎层当前扫描涉及的少量 Buffer Pool 页面，远远不会 OOM。

### 题 7：MVCC 版本链寻址推演（8/13）

| 事务 | trx_id | 操作 |
|:---:|:---:|:---|

- `max_trx_id = 101`（下一个将要分配的事务 id）
- `creator_trx_id = 0`（纯读事务）

**T3 之后，聚簇索引上 `id=1` 这行的物理版本链：**

```
聚簇索引叶子节点（最新版本）:
┌────────────────────────────────────────┐
│ id=1, balance=300, trx_id=300          │
│ roll_pointer ──→ Undo Log 版本 2       │
└────────────────────────────────────────┘
                    │
                    ▼
Undo Log 版本 2（由事务 C 的 UPDATE 产生）:
┌────────────────────────────────────────┐
│ id=1, balance=200, trx_id=200          │
│ roll_pointer ──→ Undo Log 版本 1       │
└────────────────────────────────────────┘
                    │
                    ▼
Undo Log 版本 1（由事务 B 的 UPDATE 产生）:
┌────────────────────────────────────────┐
│ id=1, balance=100, trx_id=90 (初始写入)│

└────────────────────────────────────────┘
```

**T4 时，事务 A 执行第二次 SELECT，寻址过程：**

1. **读到最新物理记录：** `balance=300, trx_id=300`。
   - 判断：`trx_id=300 >= max_trx_id=101` → 说明这个版本是在 Read View **创建之后**才产生的事务，**不可见**。
   - 动作：沿 `roll_pointer` 跳到 Undo Log 版本 2。

2. **读到 Undo Log 版本 2：** `balance=200, trx_id=200`。
   - 判断：`trx_id=200 >= max_trx_id=101` → 同样是 Read View 创建之后的事务，**不可见**。
   - 动作：沿 `roll_pointer` 跳到 Undo Log 版本 1。

3. **读到 Undo Log 版本 1：** `balance=100, trx_id=90`。
   - 判断：`trx_id=90 < min_trx_id=101` → 说明这个版本的事务在 Read View 创建之前**已经提交**，**可见**！
   - 动作：返回 `balance=100`。✅

**这就是 MVCC 版本链寻址的完整物理过程。**

### 题 8：长事务阻塞 Purge（8/12）

#### 评价

你的推理方向完全正确——这可能是你所有答案中"直觉最准"的一道。你准确地识别了：
- 罪魁祸首不是锁，而是**未提交的长事务持有了一个极早的 Read View**。
- 该 Read View 阻止了 Purge 线程回收 Undo Log。
- 结果是 Undo Log 垃圾堆积、磁盘空间膨胀。

#### 扣分点（-4 分）

1. **"内存中充斥着快照版本"这个说法不准确。** Undo Log 并不存在内存中（除了 Buffer Pool 缓存的部分），它们存储在**Undo 表空间**的磁盘文件中。所以膨胀的是磁盘空间，不是"内存被快照版本挤满"。你在后半句修正说"丢进磁盘中"，但前后矛盾。

2. **没有解释"检索性能下降"的具体机制。** 当 Undo Log 版本链极长时，任何需要通过 MVCC 读取历史版本的查询（包括其他正常事务的快照读），都需要沿着 `roll_pointer` 逐节遍历这条极长的版本链，才能找到对自己可见的版本。版本链越长，每次读操作的延迟越高。这才是"无关表检索也变慢"的精确原因。

3. **没有解释 Purge 的判断机制。** Purge 线程的工作原理是：它会找到当前系统中所有活跃 Read View 中最小的 `min_trx_id`（也称为"低水位线"/"low water mark"）。任何 `trx_id` 大于等于这个低水位线的 Undo Log 版本，都**有可能**被某个活跃的 Read View 引用，因此**不能被回收**。你那个 `trx_id=100` 的长事务的 Read View，它的 `min_trx_id` 极小，直接把低水位线钉在了一个很古老的位置 → 之后所有事务（6000 TPS × 数小时 = 几千万个事务）产生的 Undo Log 全都在低水位线之上 → **无一能被 Purge 清除** → 垃圾山就这样堆起来了。
