### 题1
答：
- 全表扫描SQL会令Buffer Pool中的free链表去尽可能的利用IO去读磁盘中的页数据；
- 当会话访问这些数据的时候，LRU链表就会将它们（被访问的页数据）从free链表中放入到链表的头部
- 由于5000w行数据是一个很大的量，内存也有可能是一下子装不下的，所以读的时候就会出现频繁的IO，同时导致缓冲池中全都是充斥着这些仅仅会被访问一次的数据
- 导致别的业务语句在排队等IO，而无法直接从缓冲池中读取需要的数据
- 直到执行`select count(*) from log_history`的结果查询完毕之后
- 别的业务数据需要重新从磁盘利用IO重新读取需要的数据
- 就在此时LRU链表中的与`log_history`相关的数据页会因为没被访问，逐渐被业务访问的页数据刷到后面
- 直到超过`innodb_old_blocks_time`界限的时候，就会对超过这个界限LRU链表后面的数据页做清除操作

- 我认为的流程图
```txt
# 查询`log_history`前
free: 业务数据页->业务数据页->...
LRU: 业务数据页->业务数据页->...(innodb_old_blocks_time的界限)-(后面的是需要被淘汰的)->业务数据页->业务数据页->...

# 查询`log_history`时
free: log数据页->log数据页->...
LRU: log数据页->log数据页->...(innodb_old_blocks_time的界限)-(后面的是需要被淘汰的)->业务数据页->业务数据页->...

# 查询`log_history`后
free: 业务数据页->业务数据页->...
LRU: 业务数据页->业务数据页->...(innodb_old_blocks_time的界限)-(后面的是需要被淘汰的)->log数据页->log数据页->...
```

> 我在buffer pool的时候没有学的这么细，我只知道free链表是读数据，flush是刷数据，LRU链表是做数据页淘汰的
> 我也不知道冷热分离机制是什么，我感觉就是应该是被访问多的和被访问少的会被分别处理的机制；
> 这里是靠我以前学习算法的时候学过URL淘汰规则和结合一部分我已学的Buffer Pool知识猜和推测的答案，

### 题3
答：
- 第一条SQL语句分析
    - 先通过二级索引对所有数据进行分组操作（二级索引B+树）
    - 会一共扫描100010条数据
    - 每一条数据都是先去二级索引上进行搜索得到聚簇索引（聚簇索引B+树）
    - 然后再拿着聚簇索引进行回表查询拿到数据行
    - 然后通过IO将这100010条数据全部通过IO传给Service层
    > 这里一共进行了100010次回表的随机IO

- 第二条SQL语句分析
    - 会先通过二级索引B+树获取100010条数据，但是不会发生回表（因为**索引覆盖**）
    - 此时通过IO传输100010条数据给Service层
    - Service进行筛选得到最后10条数据，然后对10条聚簇索引`id`进行回表查询
    > 如果这里是联合索引还会更快，因为会带有MRR加速，变成顺序IO
    - 然后最后将这10条有效数据行传给Service层
    > 这里一共进行了10次回表的随机IO

- 因为加入join之后，在子查询中使用了**索引覆盖**，不会让每次在二级索引树中查询到的聚簇索引都发生一次回表查询，减少了随机IO的性能损耗（因为磁头移动去读取数据是很慢的）；
- 关键成本发生在**减少了大量的无效回表随机IO查询**


### 题4
答：
- 先说我对于联合索引的理解
- 在 B+树 中，联合索引构建的这颗索引树，如示例中的其实是遵顼这个排序规则的
- 我用cpp代码来展示数组排序之后的节点（其实就是链表的样子）
```cpp
bool operator< cmp(node* a, node* b) {
    if(a.status == b.status) return a.create_time < b.create_time;
    return a.status < b.status;
}

signed main() {
    node leave[n]; // 代表所有叶子节点
    sort(leave, leave+n+1, cmp);
    return 0;
}
```
- 其实这个就是自底向上来看 B+树 （自顶向下我也会，因为这个代码好敲也简短）；
- 接下来就可以是以每页的数据行记录数来分割每部分，然后往上构建整颗B+树了
- 上面的节点就是存放在这棵树中存放的是：(status, create_time, id)（因为不唯一，所以要保证每个节点都不一样的话，后面还会带个`id`，当然还会存放偏倚的地址值）

- 至于为什么会失效，首先在`查询A`中的`where status = 1`其实是可以通过前缀索引进行精确匹配的，这个时候只需要匹配到最后一个满足`status=1`的节点，然后通过`Backward Scan`的方式就可以得到需要的查询结果了；（其实老师，如果联合索引没有触发`Backward Scan`，那么`查询A`依旧是会进行`using filesort`，我看似乎是有一些版本的MySQL是不会反向扫描的，我也不太确定，可能是我记不清了）
- 而对于`查询B`来说，除了是`status>1`这个条件其实没那么容易做精确匹配还有一个问题，就是我每个不同的`status`下查询出来不同组的`create_time`是有序的（分组看的话），如果是整体一起，那么就是无序的，只能说是局部有序，那就只能是拉到排序缓存区中进行一次排序了；


### 题5
答：
- 按照我的理解，成本的排序可以是这样的`磁头移动（随机IO） >> 顺序IO`
- 结合题目提示的话，只能说`SQL优化器`发现通过`where age=22`这个条件搜索得出来的聚簇索引
- 顺序是特别乱的，那么在`SQL优化器`的理解里，我回表200w次的随机IO，不如做多几次顺序IO，这样更快
- 而且同时还能减少存储层和Service层之间使用IO传输的次数，因为IO传输也是很慢的，想要性能高，IO传输次数也要少
- 因为多次顺序IO操作中，磁头只需要按下去，一路顺着读就行了，不需要过多的磁头移动（物理过程，超级慢），读完一次移动一下就可以了
- 而在多次随机IO操作中，磁头必须要不停的进行移动，基本是读取一次就需要进行一次移动，直到读取完毕所有数据

- 还有一种原因，就是做`MRR`排序的代价大于做全表扫描的代价；

> 我并没有学习`SQL优化器`的成本计算原理，我只知道它会根据计算代价来选择最优的
> 所以我这里知识结合了我现有的知识做了一部分的推测

### 题7
答：
- 这个问题其实涉及到了`快照读`的机制
- 在`RR`隔离级别下，事务只会为第一条`select`语句创建一个`ReadView`，后面的所有查询都只是复用这张“快照”
- 假设事务A的`trx_id=1`，那么在它的`ReadView`版本
    - `m_ids`是空的
    - `max_trx_id=1`
    - `min_tex_id=-∞` -- 不知道具体，反正就是比 1 小
    - `creator_trx_id=0` -- 快照读默认的（除非发生修改）
- 尽管是后续的事务B(trx_id=2)和事务C(trx_id=3)，那么他们修改完毕之后
- 事务A再次进行读取的时候，去读最新版本的时候，发现`c.trx_id > a.max_trx_id`说明是这个事务是事务A之后才创建的，不能读；读取事务B的修改依旧是同理；直到读取到`trx_id=1`的，发现`a.max_trx_id > a.trx_id > a.min_trx_id`并且`m_ids.contain(a.trx_id) = false`，即为可以获取这个版本，那么就会读取到这条数据；


### 题8
答：
> 其实这题一开始我差点从“锁”的方向进行思考了，但是我又感觉矛盾，因为普通读不会加锁；
> 后面看了提示半天之后，突然反应过来是后台线程中的`Purge`线程无法正常工作了，因为`Undo log`中生成的版本快照一直存在活跃事务版本，所以无法回收和清除导致的

- 分析题目，这个是一个高并发的电商数据库，那么我们可以分析
- 首先在事务开启之后，会获得一个事务id，这里我们假设它是`trx_id=100`
- 我们假设这个数据库是默认的RR隔离级别
- 在这个高并发系统中，我推测TPS应该是数据库每秒钟收到的业务请求数量
- 那么在这么大量的请求下，单位时间内会创建大量的`ReadView`
- 然后在这些快照版本中`m_ids`都会包含那个卡住并处于活跃状态的事务id，只要它一直不释放，那么这个Purge线程就一直很难正常工作
- 然后内存中大量充斥着各种“没用”的快照版本，然后无法回收
- 挤满内存之后InnoDB又会将这些东西丢进磁盘中，然后开始疯狂的占用磁盘IO的这条线
- 导致后续更新需要做IO的时候就要跟它抢资源，导致出现极少量更新也出现卡顿的情况；
