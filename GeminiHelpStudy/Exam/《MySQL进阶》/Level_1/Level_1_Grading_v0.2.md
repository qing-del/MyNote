### 题 5：优化器的"算账"逻辑（9/13，v0.1: 9 分）

#### 评价

很遗憾，这道题的分数**没有变化**。

你的核心论述和 v0.1 完全一致：随机 I/O >> 顺序 I/O，200 万次回表太贵，全表扫描更合算。方向依然是对的，但**新增的内容都属于复述和强调，没有实质性的新深度。**

#### 依旧缺失的关键点（-4 分不变）

1. **"二级索引查出来的 id 是乱序的"——这个说法依然有误。**（-1.5）

   你在 v0.2 中画了一个图：
   ```
   age=1: (1,7), (1,2), (1,6)
   age=2: (2,5), (2,3), (2,9)
   ```
   **这张图是错的。** 在二级索引 `idx_age(age)` 的叶子节点中，同一个 `age` 值下的主键 `id` 是**严格升序排列**的。正确的物理分布是：
   ```
   age=1: (1, id=2), (1, id=6), (1, id=7)   ← id 升序
   age=2: (2, id=3), (2, id=5), (2, id=9)   ← id 升序
   ```
   二级索引先按 `age` 排序，`age` 相同时按**主键 id 升序**排列——这是 InnoDB 的硬性设计（因为二级索引叶子节点需要存储主键来回表，且需要保证唯一性，所以 id 是有序的）。

   但这里 id 有序不有序**不是问题的核心**。核心是：这些 id 对应的**聚簇索引记录分散在聚簇索引 B+ 树的各个不同数据页上**。id=2 的完整行在某一页，id=6 的完整行可能在距离很远的另一页。每次回表都是根据 id 去聚簇索引做一次主键点查，**命中的数据页是随机分布的**——这才是 200 万次随机 I/O 的根源。

2. **MRR 仍然没有展开说明。**（-1.5） 你提了一句"MRR 排序后 200w 次回表的代价依旧大于全表扫描"，和 v0.1 几乎一样。MRR 的核心原理是：先从二级索引收集一批 id → 在内存中按主键排序 → 再去聚簇索引按排好序的 id 顺序回表，将随机 I/O 转化为近似顺序 I/O。你需要指出的是：即便 MRR 生效，200 万行的排序成本 + 最终访问的聚簇索引页数依然是一个天文数字，优化器计算后总 Cost 仍不如全表顺序扫描。

3. **回表阈值未提及。**（-1） 经验法则：结果集超过全表 15%-30% 时，优化器几乎一定放弃二级索引。

> 我注意到你说"暂时先不打算学这么深入的 SQL 优化器内容"。作为教授我尊重你的学习节奏规划，但分数不会因此调整——这正好是你未来需要补的债。

- ✅ 正确指出最终结局：MySQL 尝试重连/重传若干次后超时，终结查询并释放资源。
- ✅ 整体链路的**方向和结论**是完全正确的。

#### 扣分点（-4 分）

1. **TCP 滑动窗口的细节理解有偏差。**（-1.5）

   你写"存储层查询数据的时候会通过滑动窗口阻塞机制来设置最大缓存量"——滑动窗口是 **TCP 协议层**的机制，不是 InnoDB 存储层的概念。滑动窗口的作用对象是 **MySQL Server ↔ 客户端** 之间的 TCP 连接，而不是 InnoDB 与 Server 层之间的内部通信。InnoDB 和 Server 层之间是进程内函数调用，根本不走 TCP。

   正确的反压路径是：
   ```
   TCP 接收窗口(rwnd)归零 → OS 的 TCP 发送缓冲区(SO_SNDBUF)满
   → send() 系统调用阻塞 → net_flush() 阻塞 → net_buffer 写满
   → 执行器的 write_row 循环被卡住 → 执行器停止向引擎层调用 read_next_row()
   ```

3. **缺少 `net_buffer_length` 的精确角色描述。**（-1） `net_buffer_length`（默认 16KB）是 Server 层的网络发送缓冲区。执行器每取一行就往 net_buffer 里写，写满 16KB 就调用 `net_flush()` 通过 TCP socket 发出去。这是请求方向上离引擎最近的第一级缓冲，你的伪代码里隐含了，但没有明确指出这个参数的名字和大小。

### 题 8：长事务阻塞 Purge（10/12，v0.1: 8 分）

#### 进步

- ✅ **关键突破：你补上了 Purge 线程的判断机制。** 你写道："Purge 线程是检查目前所有快照版本中最小的 `min_trx_id`（大于这个版本的很可能还处于活跃状态），导致被卡在了很古老的时间点无法运作"——这正是 v0.1 中被扣分的核心遗漏，也是本题最关键的得分点之一。你理解了**低水位线（low water mark）**的概念。
- ✅ 你也补上了"版本链极长导致查询性能下降"的机制。

#### 扣分点（-2 分）

1. **"将大量的'没用'的快照版本写入到 Undo Log 中"——因果关系依然有些颠倒。**（-1）

   准确的因果链是：
   - 大量事务在执行 UPDATE/DELETE 时，**InnoDB 引擎自动为每次修改生成 Undo Log 记录**（这是事务修改的正常副产品，不是"写入快照版本"）。
   - 这些 Undo Log 记录存储在**Undo 表空间**的磁盘文件中。
   - 正常情况下，Purge 线程会持续清理那些"不再被任何活跃 Read View 引用"的旧 Undo Log。
   - 但因为那个古老的长事务的 Read View 钉住了低水位线 → Purge 无法判定这些 Undo Log 是安全可回收的 → Undo Log 文件只增不减 → 磁盘暴涨。

   你的描述给人一种"系统主动把快照版本塞进 Undo Log"的错觉，但实际上 Undo Log 是事务修改过程中**被动、自动**产生的，而不是由谁主动"写入"的。问题不在于"写入"，而在于"无法清除"。

2. **"活跃的事务由于之前卡住的事务导致的逐渐增加"——这句话有歧义。**（-1） 那个卡住的长事务并不会导致其他活跃事务的数量增加（其他事务正常提交完全不受影响）。它影响的是 **Purge 线程的清理范围**——不管其他事务提交了多少，只要那个老 Read View 还活着，它们产生的 Undo Log 就不能被 Purge 回收。清晰地区分"活跃事务数量增加"和"不可回收的 Undo Log 数量增加"非常重要。
