## 一、DDL 设计（28/35）

### ✅ 值得肯定

1. **表关系梳理先行。** 你在写 DDL 之前先画了 Mermaid ER 图，梳理了 10 张表的关系。这种"先画图再建表"的习惯在工业界非常重要——太多工程师上来就 `CREATE TABLE`，结果建到一半发现表关系不对再回头改。（+2 加分）

2. **数据类型选型有思考过程。** 你对每个字段都给出了选型理由——`TINYINT` 因为不超过 128、`VARCHAR(30)` 因为影院名字的最大长度、`DECIMAL(9,2)` 因为电影票不超过百万。这种"带着业务约束选类型"的方法论完全正确。

3. **冗余字段有意识。** `schedules` 表中冗余了 `theater_id` 和 `duration`（本可以通过 JOIN 影厅表和电影表获得），你在注释中解释了原因（避免 JOIN）。这是正确的反范式决策。

4. **`theater_movie_relationship` 中冗余了 `movie_name`。** 虽然违反第三范式，但你给出了合理理由：用户搜电影名时需要直接命中这张关系表，避免再 JOIN 电影表。

### ❌ 扣分项

**1. 主键语法错误（-1 分）**

多处 `PRIMARY KEY id` 应该是 `PRIMARY KEY (id)`。这虽然是语法笔误，但在正式 DDL 中会直接报错。

**2. `schedule_seats.position` 用 TINYINT 过于紧凑（-2 分）**

你写 `position tinyint not null comment '场次座位编号'`。TINYINT 的无符号范围是 0-255，有符号是 -128~127。题目中说每个影厅有 120 个座位，有符号 TINYINT 只到 127。如果未来有大型影厅（如 IMAX 巨幕 300 座），你就溢出了。更安全的选择是 `SMALLINT UNSIGNED`（0~65535）或 `INT`。

另外，`position` 这个字段名不太合适——它暗示一个一维编号，但影院座位通常是二维的（行号+列号，如"5排8座"）。更好的设计是 `row_no TINYINT, col_no TINYINT` 两个字段，或者 `seat_label VARCHAR(8)`（如 "E08"）。

**3. 缺少关键字段（-2 分）**

- `schedule_seats` 表缺少 `locked_at DATETIME` 字段——用于记录座位被锁定的时间，是 15 分钟超时释放的基础数据。你在后面的超时处理方案中提到了"利用时间和状态筛选超时订单"，但座位表本身没有锁定时间戳，你只能通过 JOIN 订单表获取——这增加了不必要的 JOIN 复杂度。
- `schedule_seats` 表缺少 `locked_by BIGINT`（锁定者的订单 ID 或用户 ID），用于释放时验证身份。

**4. 电影拼写（-1 分）**

全文 `moive` 应为 `movie`。这不只是拼写问题——在实际项目中，表名/字段名的拼写错误一旦上线就成了永久的技术债，所有 API、Mapper、DTO 全部跟着错。

**5. `ticket_orders.order_no` 的生成方式有安全隐患（-1 分）**

你的订单编号生成方式：
```java
String.format("%10d", userId) + "#" + System.currentTimeMillis()/1000
```
这个方式有两个问题：
- 同一秒内同一用户可能发起两个订单 → 编号冲突
- `userId` 被直接暴露在订单编号中 → 信息泄露

更好的做法是使用 UUID、雪花算法（Snowflake）、或业务前缀 + 日期 + 自增序列的组合。

## 二、索引设计（20/25）

### ✅ 值得肯定

1. **每个索引都有业务场景说明。** 你不仅列出了索引，还说明了它服务于哪个查询场景、为什么选择这个字段顺序。这正是题目要求的。

2. **`idx_theater_movie_begin_hall`** 的设计非常棒。你参考了猫眼的真实业务流程（选影院 → 选电影 → 按时间排序 → 展示影厅），索引字段顺序完美匹配查询路径，且 `begin_time` 在第三位可以天然完成排序（避免 filesort）。`hall_id` 放在最后做覆盖索引。（+3 加分）

3. **`idx_schedule_position_status_price`** 的覆盖索引设计思路正确——选座页面需要的字段全在索引中，零回表。

4. **`idx_create_status`** 为定时任务扫描超时订单服务——这说明你在设计索引时不仅考虑了用户查询路径，还考虑了后台的维护任务。

### ❌ 扣分项

**1. `idx_create_status` 字段写错了（-2 分）**

你写的是 `(create_id, status)`，但表中没有 `create_id` 字段，应该是 `(create_at, status)` 或 `(status, create_at)`。而且对于超时订单扫描任务，查询条件是 `WHERE status = 1 AND create_at < NOW() - INTERVAL 15 MINUTE`——按最左前缀原则，应该是 **`(status, create_at)`**，先用 `status=1` 等值匹配，再用 `create_at` 范围过滤。你的字段顺序反了。

**2. `idx_user_status_create_schedule_amount` 过于宽泛（-2 分）**

这是一个 5 列的覆盖索引。虽然避免了回表，但：
- 每个用户的订单量通常不多（你自己也说了），所以 `(user_id)` 单列索引的过滤效果已经足够好，后面的列主要是做覆盖。
- 5 列覆盖索引的维护成本（每次 INSERT/UPDATE 都要更新这棵索引树）与收益不成正比。在订单表这种写入频繁的表上，过宽的索引会拖慢写性能。

**3. `schedule_seats` 缺少并发核心索引（-1 分）**

你的选座 SQL 是 `SELECT ... FROM schedule_seats WHERE id IN (...) FOR UPDATE`。这走的是主键，没问题。但如果查询"某场次的所有可售座位"，你需要 `WHERE schedule_id = ? AND status = 1`。你的联合索引 `(schedule_id, position, status, price)` 中 `position` 在 `status` 前面，如果查询不指定 `position`（展示所有位置），`status` 的过滤要在 Server 层做。更好的设计是 `(schedule_id, status, position, price)`。

## 三、并发选座方案（22/40）

### ✅ 正确的核心决策

1. **选择悲观锁（`SELECT ... FOR UPDATE`）✅。** 你给出了清晰的选型理由：电影票绝对不能超卖，乐观锁在高并发下会导致大量 CAS 重试，悲观锁直接阻断并发。在这个场景下是正确的选择。

2. **使用 `FOR UPDATE NOWAIT` ✅。** 非常关键的一个细节——你用了 `NOWAIT` 而不是纯 `FOR UPDATE`。`NOWAIT` 的含义是"如果拿不到锁，立即返回错误而不是等待"。在 500 人抢票场景中，如果不加 `NOWAIT`，499 个请求会阻塞在数据库上等待锁释放，占满连接池导致整个系统瘫痪。`NOWAIT` 让拿不到锁的请求快速失败，配合前端的"系统繁忙"提示，这是正确的工业级做法。（+3 加分）

3. **全局异常捕获器处理死锁。** 你提前考虑了死锁场景并设计了兜底方案。

4. **批量插入检查 `max_allowed_packet`。** 这说明你知道 MySQL 连接层的参数对批量操作的限制。

### ❌ 关键安全隐患

**1. 多座位选座的原子性未完全保证（-6 分）⚠️**

你的 SQL 是：
```sql
SELECT status FROM schedule_seats WHERE id IN (seat1, seat2) FOR UPDATE NOWAIT;
```

这里有一个**严重的并发安全问题**：`IN` 列表中的多行加锁顺序。

当 A 选座位 `(1, 2)`，B 选座位 `(2, 3)` 时：
- 如果 InnoDB 按 id 升序加锁：A 先锁 id=1，B 先锁 id=2 → A 尝试锁 id=2 被 B 阻塞（但你用了 NOWAIT 所以 A 直接失败）→ 从结果来看是安全的
- 但如果你**没有用 NOWAIT**（或在某些场景去掉了 NOWAIT），就可能发生：A 锁了 id=1 等待 id=2，B 锁了 id=2 等待 id=3 中已被 A 的其他事务锁住的行 → 潜在死锁

你用了 `NOWAIT` 在一定程度上规避了死锁问题（拿不到立即失败），但代价是：**在高并发下，只要有一个座位被别人正在处理，你的整个选座请求就失败了**，即使另一个用户选的座位集和你没有任何交集。这会导致高峰期用户体验极差——大量"系统繁忙"。

更好的方案是：**保证加锁顺序！** 在 Java 层将 `seatIds` 按升序排序后再传给 SQL，这样所有事务都按相同的 id 顺序加锁，根除死锁可能性，同时可以去掉 `NOWAIT` 让非交集请求正常排队。

```java
Collections.sort(submitOrderDTO.getSeatIds()); // 关键一行！
```

**2. 缺少"座位状态双重检查"（-4 分）**

你的流程是：
1. `SELECT ... FOR UPDATE NOWAIT` 拿到座位状态
2. 在 Java 层检查 `allMatch(status == 1)`
3. `UPDATE ... SET status = 2`

这个流程的逻辑是对的，但你缺少了一个关键的**防御性 SQL 写法**：应该在 UPDATE 中也带上条件检查：

```sql
UPDATE schedule_seats SET status = 2, locked_at = NOW() 
WHERE id IN (...) AND status = 1;
```

然后检查 `affected_rows` 是否等于预期数量。如果不等，说明有座位在你 SELECT 和 UPDATE 之间被修改了——虽然在 `FOR UPDATE` 加锁下这理论上不会发生，但**防御性编程**是资金/票务系统的铁律：永远不要依赖单一机制的正确性。

**3. 超时释放方案存在时间窗口漏洞（-4 分）**

你的方案是用 `Spring Task` 每分钟扫描超时订单。这意味着：
- 一个座位最多可能被锁定 **15 分钟 + 1 分钟（扫描间隔）= 16 分钟**
- 在扫描任务执行和座位释放之间，如果有新用户查询这个座位，会看到它是"锁定"状态（但实际上已经超时应该释放了）

更严谨的做法是：**将超时判断下沉到查询层**。查询可售座位时，不仅检查 `status = 1`，还检查 `status = 2 AND locked_at < NOW() - INTERVAL 15 MINUTE`：

```sql
SELECT * FROM schedule_seats 
WHERE schedule_id = ? 
  AND (status = 1 OR (status = 2 AND locked_at < NOW() - INTERVAL 15 MINUTE));
```

这样即使定时任务还没跑到，用户也能看到已超时的座位并尝试选购。定时任务作为兜底清理即可。

**4. 事务边界过大（-4 分）**

你的 `@Transaction` 注解挂在整个 `submitOrder` 方法上，意味着从"查座位状态"到"插入订单"到"插入订单详情"全部在同一个数据库事务中。

在 `FOR UPDATE` 场景下，事务持续时间 = 锁持有时间。你的事务中间做了大量 Java 逻辑（创建对象、计算金额、生成订单号），这些 CPU 密集操作在锁内执行会**延长锁持有时间**，降低并发吞吐量。

更好的做法是：
1. **事务外**做所有计算（生成订单号、构建对象、校验参数）
2. **事务内**只做最核心的三步：`SELECT...FOR UPDATE` → `UPDATE 座位状态` → `INSERT 订单` → `INSERT 订单详情` → `COMMIT`

将事务做到尽可能**短小精悍**。

# 场景二：钱包余额扣减与转账（思路点评，不计分）

你在 Part 2 中列出了 5 个核心点的思路。我逐一点评，告诉你哪些方向是对的、哪些需要调整，供你未来有空时展开。

### 思路 1：余额不能小于 0

> 在数据库表中加入 `balance >= 0.00` 约束

**✅ 方向正确，但实现方式需要调整。** MySQL 的 `CHECK` 约束在 8.0.16+ 才真正生效（之前的版本解析但不执行）。更可靠的做法是在 UPDATE 语句中直接带条件：

```sql
UPDATE wallets SET balance = balance - 150.00 WHERE user_id = ? AND balance >= 150.00;
```

如果 `affected_rows = 0`，说明余额不足，扣款失败。这比 CHECK 约束更高效（减少一次 SELECT），而且是原子操作。

### 思路 2：幂等性防重

> 将订单生成环节和支付环节分开，生成唯一 API 支付密钥

**✅ 方向完全正确。** 工业界的标准做法就是：业务层生成一个全局唯一的**幂等键（Idempotency Key）**，在交易流水表中设置 UNIQUE 约束。第二次请求携带相同的幂等键尝试插入时，UNIQUE 约束违反 → 直接拒绝 → 保证只扣一次。

### 思路 3：转账原子性

> 如果 Java 代码层可以处理的话... 否则用存储过程

**⚠️ 需要纠正。** 你提到了 `@Transactional` 自调用失效的问题（`this.xxx()` 绕过代理），说明你对 Spring AOP 的底层机制有了解。但这不是选择存储过程的理由。

正确的工业方案是：**在 Service 层用 `@Transactional` 注解包裹转账操作，内部按固定顺序（如先扣 id 小的、再加 id 大的）加锁**，避免死锁。自调用的问题可以通过注入自身代理 `@Lazy` 或提取到另一个 Service 来解决。存储过程在现代微服务架构中已经很少使用，原因是调试困难、版本管理复杂、且无法利用 Java 的异常处理和日志体系。

> ⚠️ **转账场景中最关键的一行代码——你没有提到：**
> 
> **对两个账户按 user_id 升序排序再加锁！** 如果 A→B 和 B→A 同时发生，且两个事务分别先锁自己的账户再锁对方的 → 经典死锁。解决方案和你在场景一中缺失的一样：**保证加锁顺序**。

### 思路 4：流水记录

> 利用 Spring AOP 切面类自动记录

**⚠️ 危险！** AOP 切面做日志记录（比如操作审计日志）没有问题，但做**资金流水**是不安全的。原因是：

AOP 切面和业务方法可能不在同一个事务中（取决于切面的 `@Order` 和事务传播属性）。如果余额扣了但 AOP 切面的流水插入失败（或反过来），就会出现"余额扣了但流水没记"的灾难。

**正确做法：流水插入和余额变更必须在同一个 `@Transactional` 事务中完成，而不是通过 AOP 异步处理。**

```java
@Transactional
public void deduct(Long userId, BigDecimal amount, String idempotencyKey) {
    // 1. 插入流水（带幂等键唯一约束）
    // 2. 扣减余额（带 balance >= amount 条件）
    // 同一事务：要么全成功，要么全回滚
}
```
