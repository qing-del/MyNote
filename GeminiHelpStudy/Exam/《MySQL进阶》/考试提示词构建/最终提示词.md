## 二、考核框架总览

|阶段|考核主题|核心知识点|
|---|---|---|
|**Level 1**|底层架构与执行原理|InnoDB 物理结构（页/行/Buffer Pool）、B+树本质、SQL 执行流程、Undo Log 在 MVCC 中的角色|
|**Level 2**|查询优化与调参实战|EXPLAIN 深度解析、索引失效场景（含最左前缀）、Join 算法、深度分页优化原理|

|**Level 4**|业务建模与并发防范|表结构 DDL 设计、索引策略、悲观锁/乐观锁方案选型、事务边界控制|

### 📌 Level 3：并发控制与锁推演（硬核实战）

```
[开始 Level 3]

现在进入高并发与锁的场景考核。我将在本地 MySQL 终端中进行心智推演或实操验证。

请给我设计 2 个具体的事务并发场景：给出 2~3 个并发执行的 Transaction（包含明确的时间线 T1, T2...），指定具体的表结构（含索引类型）和隔离级别（默认使用 Repeatable Read）。

要求我完成：
1. 结合 MVCC 的 Read View 机制，推断各事务在快照读时能读到哪个版本的数据。
2. 结合具体索引结构，推断当前加的是 Record Lock、Gap Lock 还是 Next-Key Lock，加锁区间是什么。
3. 判断哪个事务会被阻塞，或是否会触发死锁（Deadlock），并从底层依据解释。

出题建议：可以涉及在辅助索引上加锁时回溯聚簇索引的场景，或构造一个需要判断 Gap Lock 边界的场景，以考察真实的加锁细节。

强烈建议：对于不确定的锁行为，在本地开两个 Session 手动执行 BEGIN + 具体 SQL，用 SELECT * FROM performance_schema.data_locks 观察真实加锁结果，验证你的推演是否正确。

请给出题目和时间线，等待我的回答。
```

   - 写出关键事务的伪代码或 SQL 流程，体现事务边界控制（何时 BEGIN / COMMIT / ROLLBACK）。

请给出项目背景和需求，等待我提交设计方案。

## 四、使用建议

**考核时的心态：** 把每次回答当成真实的技术面试，先用脑子里沉淀的逻辑框架推导，哪怕细节记不清，也要把排查思路和推演过程完整写出来——教授会根据你的思维过程评分，而不只是看结论对不对。

**Level 3 的实操价值：** 锁的行为在不同索引结构下差异很大，纯心智推演容易出错。遇到不确定的场景，一定要在本地开两个终端实际验证，这是建立锁直觉最快的方式。

**关于 Redo Log：** 如果教授在讲解崩溃恢复时提到了 Redo Log，你可以结合自己笔记里的内容作答，但不必主动延伸到 Binlog 相关的主从同步方向。
